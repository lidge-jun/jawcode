# 00.01 — 다음 단계: TTFT 우선 수리 순서

> 상태: ⬜ 착수 대기
> 상위 문서: [00_moc_toolcall_loop_reformation.md](./00_moc_toolcall_loop_reformation.md)
> 작성일: 2026-06-13
> 입력: 사용자 "첫 토큰까지의 출력시간이 개오래걸려, 이렇게 로딩하니까 좀 빠르고" —
> 검증 2차에서 TTFT 병목 위치가 확정됨에 따라 실행 순서를 TTFT 우선으로 재배열.

검증 2차로 우선순위가 바뀌었다. 당초 P0이던 "WS 수명주기 수리"는 여전히 필요하지만,
실측 결과 WS+델타는 대형 컨텍스트에서도 일단 붙으면 정상(라운드 3~5초)이었다. 체감
고통의 본체는 **콜드 스타트 TTFT**(resume 첫 라운드 2분 15초)와 **idle 워치독 오발**
(`Provider stream stalled` → 라운드 재시작)이다. 따라서 실행 순서는 TTFT 직결 항목부터.

## T1 — 콘텐츠 프리웜 (TTFT 본체 수리)

네이티브는 프리웜 때 실제 페이로드를 `generate=false`로 보내 **서버 프리필을 미리
완료**한다 (`codex-rs/core/src/client.rs:15` "WebSocket prewarm is a v2-only
`response.create` with `generate=false`; it waits for completion", `:1401`). jawcode
프리웜은 핸드셰이크만 연다 (`packages/ai/src/providers/openai-codex-responses.ts:1609-1650`).

작업:

1. `prewarmOpenAICodexResponses`에 `generate: false` + 현재 히스토리 페이로드 전송 단계
   추가. 완료 응답의 `response_id`를 `state.lastResponseId`/`lastRequest`에 시드해서
   **사용자의 첫 메시지가 델타로 나가게** 만든다.
2. 호출 시점 확장 — 현재는 세션 생성 1회뿐(`packages/coding-agent/src/sdk.ts:2003-2032`):
   - `--resume` 직후 (이번 실측에서 2분 15초를 만든 지점)
   - 턴 종료 후 idle 재진입 시 (WS idle 5분 타임아웃 `:90` 대비, 사용자가 타이핑하는 동안
     재연결+재프리필)
3. 프리웜 결과를 stats에 기록 (`prewarmed` 플래그는 있음 — 프리필 완료 여부 추가).

수용 기준: resume 직후 첫 응답 라운드가 "프리웜 완료 후 메시지 전송" 시나리오에서
2분대 → 추론 시간 수준(수십 초 이내)으로 단축.

## T2 — idle 워치독 보정 (stall 오발 제거)

`Provider stream stalled while waiting for the next event` =
`register-builtins.ts:166,211-219`의 클라이언트 idle 워치독. high effort + 대형 컨텍스트의
무이벤트 구간(추론 침묵)과 충돌해 정상 라운드를 끊고 재시작시킨다.

작업:

1. `LazyStreamLimits` 패턴(`register-builtins.ts:182-191`, gemini용 선례 있음)으로
   `openai-codex`에 effort 연동 floor 추가 — high/xhigh일 때 first-event/idle 타임아웃 상향.
2. stall 발생 시 재시작 비용이 로그에 보이게: 끊긴 라운드의 경과 시간과 재시도 횟수 기록.

수용 기준: high effort 대형 컨텍스트 세션에서 stall 오발 0건 (서버 실제 장애 제외).

## T3 — 가시성 (P0-2 + 로그 보강)

1. 푸터에 transport 표시 (`websocket` / `sse(fallback)`) — 데이터는
   `getOpenAICodexTransportDetails`(`provider-details.ts:71-83`)에 이미 존재.
2. `[codex] codex websocket request` 디버그 로그에 **delta/full 여부와 input 아이템 수**
   추가 — `recordCodexWebSocketRequestStats`(:1829-1845)가 이미 계산하는 값을 로그 라인
   (:705-712)에 한 줄 싣기만 하면 됨. 검증 2차에서 델타 여부를 추정으로 판정해야 했던
   한계의 직접 해소.
3. WS→SSE 폴백 시 1회성 notice (anthropic fast-mode auto-fallback 패턴,
   `agent-session.ts:6056-6063`).

## T4 — WS 수명주기 수리 (구 P0-1)

오전 prewarm 실패(같은 밀리초 3건)의 재발 방지. 검증 1·2차에서는 재현 안 됐지만 구조는
그대로다.

1. `getOrCreateCodexWebSocketConnection`(:2163-2172) — CONNECTING + auth 일치 시 기존
   `#connectPromise` 합류 (현재는 무조건 `close("reconnect")`).
2. 실패 시 `disableWebsocket` 영구 강등 대신 `PI_CODEX_WEBSOCKET_RETRY_BUDGET`(:231-236)
   라운드 단위 소진.
3. 같은 세션 키에 대한 동시 connect 직렬화.

## T5 — 후순위 (P1 잔여)

- `x-codex-turn-state` 송수신 수리 (검증 1·2차 전 요청 `sentTurnStateHeader: false`).
- `prompt_cache_retention` codex 백엔드 실효 검증 (`request-transformer.ts:48` 타입만 존재).
- SSE 폴백 본문 압축(zstd), `"# Juice: 0"` 조건부 주입 정리(`openai-responses-shared.ts:765-771`).
- 멀티계정 전환 경계 보호 (전환은 턴 경계로만 + notice).
- 푸터 tok/s를 턴 평균/순간 속도로 분리, TTFT 별도 표기 (P2).

## 검증 시나리오 (T1~T2 완료 후)

```bash
# 1) resume 콜드 스타트: 프리웜 동작 확인 후 첫 메시지 TTFT 측정
PI_CODEX_DEBUG=1 PI_CODEX_WEBSOCKET=1 jwc --resume <대형 세션 id>
# 2) idle 5분 초과 후 다음 메시지 TTFT (콘텐츠 프리웜의 재연결 경로)
# 3) high effort 장시간 추론 라운드에서 stall 오발 없음
```

비교 기준(검증 2차 베이스라인): resume 첫 라운드 2분 15초 / 워밍 후 라운드 3.3~5.4초 /
stall 1회. T1·T2 후 첫 라운드가 워밍 라운드 수준에 수렴해야 성공.
