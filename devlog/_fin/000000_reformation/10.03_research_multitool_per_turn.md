# 10.03 — 연구: 멀티툴/턴 ("reasoning → tool tool tool") + 렌더링 병목 측정

> 상태: 조사 ✅ (260613, Sonnet 3병렬 + 직접 측정) / 이식 ⬜
> 상위: [10_moc](./10_moc_general_improvements.md)
> 입력: 사용자 "codex cli는 추론→tool→추론→tool만 아니라 추론→tool tool tool도 도는 것 같다,
> 후속 리서치 / codex·claude code·opencode 전부 / tui ansi 렌더링 병목 가능성".

## A. 렌더링 병목 측정 — ANSI 렌더는 병목이 아니다

10회 하드 Write(각 120줄) 워크로드, gpt-5.5 high effort:

| 조건 | wall | 비고 |
|---|---|---|
| headless (`jwc -p`) 단독 | 274.8s | 10/10 |
| TUI 단독 | 273.4s | 10/10 |
| headless ×3 동시(6-way) | 276.5 / 277.7 / 297.1s | wall 유효 (파일은 스크립트 cd 누락으로 launch cwd에 생성됨) |
| TUI ×3 동시(6-way) | 286.1 / 295.2 / 298.2s | 10/10 |

판정:
- **TUI ≈ headless (0.5% 차이)** — ANSI 렌더링은 병목이 **아니다**. G5 수정(O(n²) 재렌더 제거)
  이후 렌더가 WS 소비를 막지 않는다.
- 6-way 자기경합에서도 274s → 276~298s로 **소폭 증가에 그침** — priority 풀이 동시성 부하를
  잘 흡수. 여기서도 TUI/headless wall이 완전히 겹침 → 렌더 중립 재확인.
- 병목은 전부 **모델측**(high effort 추론 + 토큰 생성). 측정 워크로드 기준 라운드당 수십 초는
  추론·생성 시간이지 클라이언트가 아니다.

## B. "tool tool tool" 메커니즘 5종 + 업스트림 비교

### B0. 한눈 요약 (배칭을 만드는 두 축: 프롬프트 유도 / mid-stream)

| 에이전트 | 배칭 프롬프트 유도 | mid-stream 디스패치 | 기본 병렬 | 모델제어 직렬 파라미터 |
|---|---|---|---|---|
| codex-rs | ❌ (구조만, "실무선 1개/턴" 자인) | ✅ tokio::spawn | ✅ (모델 capability 게이트) | per-tool write-lock |
| claude-code | ✅ 시스템프롬프트 "maximize parallel" | ✅ StreamingToolExecutor | ✅ | per-tool safety flag |
| opencode | ✅ "MUST 한 메시지에 묶어라" | ✅ (AI SDK execute) | ✅ | 없음(권한만) |
| gemini-cli | ✅ 프롬프트 + **`wait_for_previous` 파라미터** | ❌ (스트림 완료 후) | ✅ | **모델이 `wait_for_previous:true`** |
| **jawcode** | ❌ | ❌ (post-stream) | ✅ (기계만, 실측 미사용) | exclusive 선언(툴 고정) |

**4/5가 배칭 프롬프트 유도를 가졌고 jawcode만 없다.** mid-stream은 5종 중 3종(codex/CC/opencode).
gemini-cli는 jawcode처럼 post-stream인데도 **프롬프트 유도만으로** 배칭을 잘 만든다 → 갭1(프롬프트)
이 갭2(mid-stream)보다 선결·고ROI임을 외부 사례가 재확인.

### B1. 업스트림 pi vs jawcode 포크 — 갭은 전래(inherited)

| 차원 | 업스트림 pi (`_upstream_gjc`) | jawcode 포크 | 판정 |
|---|---|---|---|
| shared/exclusive 스케줄러 | `agent-loop.ts:1211-1229` | 동일 위치·동일 코드 | 전래 |
| 툴 디스패치 시점 | post-stream | 동일 | 전래 |
| `parallel_tool_calls` 기본 | unset(API 기본 true) + custom 툴 게이트 | 동일 (주석까지 거의 동일) | 전래 |
| 배칭 프롬프트 유도 | **없음** (subagent 레벨만) | **없음** (동일) | 전래 |
| exclusive 툴 집합 | bash/write/edit/ssh/eval/vim/debug/todo-write | 동일 + `recipe` 추가 | 전래(+1) |

**판정: "배칭 유도 없음 + post-stream"은 업스트림 pi 설계에서 그대로 물려받음.** 포크 회귀 아님.
read/glob/grep/ls/web-search는 양쪽 다 `shared`(병렬 가능) — 스케줄러는 준비돼 있고 모델이
배칭만 안 한다. 수리는 `system-prompt.md` 프롬프트 변경(코드 변경 아님).

## C0. "tool tool tool" 메커니즘 상세 (4종)

| 항목 | codex-rs | claude-code | opencode | **jawcode (현재)** |
|---|---|---|---|---|
| 멀티툴 허용 | `parallel_tool_calls` (모델 capability 게이트, 기본 false→모델 지원 시 true) | Anthropic API: 한 메시지 N개 tool_use | AI SDK 기본 | `parallel_tool_calls=true` (custom 툴 있으면 턴 전체 false) |
| 실행 동시성 | FuturesOrdered + `RwLock` (parallel-safe=read lock 병렬, 그 외 write lock 직렬) | StreamingToolExecutor + `isConcurrencySafe` (read-only 병렬, Bash는 read-only일 때만, write 직렬) | AI SDK가 모든 execute 동시 호출 (안전성 게이트 없음, 권한만) | shared/exclusive 스케줄러 (**codex RwLock과 동일 패턴**) |
| 디스패치 시점 | **mid-stream** (`OutputItemDone`에 tokio::spawn) | **mid-stream** (스트림 루프서 `addTool`→`processQueue`) | mid-stream (SDK가 tool-call에 execute) | **post-stream** (응답 완료 후 records 실행) ← 갭 |
| 결과 배칭 | N개 → 단일 후속 요청 | 병합돼 단일 user 메시지 | AI SDK step | N개 → 단일 후속 요청 |
| 프롬프트 유도 | **없음** (구조적만; 코드 주석: "실무선 대개 1개/요청") | **명시적** ("call multiple tools in parallel, maximize parallel") | **명시적** ("MUST send single message with multiple tool calls") | **없음** ← 갭 |
| 툴 출력 절단 | head+tail 중간절단, user msg 20K 토큰 캡 | per-tool 30K/20K/∞, msg 200K 합계, 100K 토큰 캡, 디스크 스필 | 2000줄/50KB 디스크 스필 + 위임 힌트 | bash 아티팩트 스필 (부분) |
| 폭주 방지 | — | sibling abort (Bash 에러 시만) | doom-loop 감지 (동일 툴+입력 3연속→ask) | — |

핵심 통찰:
- **codex조차 프롬프트 유도가 없어 "실무선 대개 한 턴 한 툴"** (codex 주석 명시). 사용자가 본
  codex의 "tool tool tool"은 주로 read-only/shell 배치가 가끔 나오는 경우 + mid-stream
  디스패치로 체감이 빠른 것.
- **배칭을 가장 확실히 만드는 건 프롬프트 유도** (CC·opencode). jawcode는 실행 기계
  (shared/exclusive = codex RwLock 동급)는 이미 있는데 **유도 문구가 없다.**

## C. jawcode의 두 갭과 이식안

### 실증 검증 (260613 11:57, 라이브) — 갭1 확정

독립 6개 파일 read 작업("doc1~doc6 읽고 총 줄 수")을 jwc(gpt-5.5 high)로 실행, 디버그 로그
WS 요청 수 집계:

```
total WS requests: 7   (mode=delta items=1 × 7)
  → read 6회가 각각 별도 턴 + 최종 답변 1턴 = 7턴
  → 정답 "Total lines: 18"
```

read는 `shared`(병렬 가능) 툴인데도 **모델이 한 턴에 하나씩** 처리했다. 실행 스케줄러는
배칭을 지원하지만 **모델이 멀티툴을 안 내보내서** 안 쓰인다. codex/CC였다면 6 read를 한 턴에
묶어(병렬) ~2턴이면 끝날 작업이 7턴. high effort에선 턴당 추론 오버헤드가 붙으므로 이
3~4배 턴 수 차이가 곧 체감 지연이다. → **갭1(프롬프트 유도)이 read-heavy 작업 라운드 수의
직접 원인임이 코드+실측으로 확정.**

### 갭1 — 프롬프트 유도 부재 (난이도 하, ROI 최상)

jawcode 시스템 프롬프트에 독립 툴 병렬 호출 유도가 없다. CC 문구가 모범:
> "You can call multiple tools in a single response. If independent, make all
> independent tool calls in parallel. Maximize parallel where possible. If
> dependent, call sequentially."

- 이식: `packages/coding-agent/src/prompts/system/system-prompt.md`에 위 취지 1문단 추가.
- 효과: read/grep/glob 같은 `shared` 툴이 한 턴에 배치→병렬 실행(기존 스케줄러가 이미 지원).
  write류는 `exclusive`라 직렬 유지(정확). 즉 읽기 중심 턴에서 라운드 수 자체가 감소.
- 주의: 10-write 같은 write-heavy는 exclusive라 배칭해도 직렬 — 이건 정상(파일 경합 방지).

### 갭2 — mid-stream 디스패치 부재 (G9, 난이도 상)

codex·CC 둘 다 `OutputItemDone`/tool_use 파싱 즉시 실행 시작. jawcode는 응답 완료 후.
연속 툴에서 라운드당 수 초 절감 여지. 단 abort/부분결과/결과순서 보장 재설계 필요 →
단독 사이클(이미 G9로 deferred). 갭1을 먼저 하면 비용 대비 효과가 더 크다.

### 부수 이식 후보

- **gemini `wait_for_previous` 파라미터** — 갭1 프롬프트와 함께 가면 강력. 모든 툴 스키마에
  `wait_for_previous: boolean` 추가 + 프롬프트에 "의존 시 true". jawcode는 이미 exclusive로
  구조적 직렬을 하므로, 모델제어 직렬은 read/grep 등 shared 툴을 "이건 순서가 중요" 케이스에
  모델이 직접 직렬화하게 해줌. 난이도 중.
- doom-loop 감지(opencode) — 동일 툴+입력 N연속 시 중단. 난이도 하.
- 툴 출력 head+tail 스필 통일(G8) — 멀티툴 턴의 컨텍스트 팽창 억제.

## C2. 턴당 추론 비용 레버 (effort/추론 재사용) — jawcode는 이미 최적

추론 비용을 줄이는 두 축: ① 라운드당 비용 ② 라운드 수. ①을 codex/CC와 비교:

| 기법 | codex | claude-code | **jawcode** |
|---|---|---|---|
| `previous_response_id` + delta input | ✅ (`client.rs:1062-1092`) | ❌ (개념 없음) | ✅ (`buildAppendInput`, 실측 mode=delta) |
| `include: reasoning.encrypted_content` | ✅ | n/a | ✅ (`request-transformer.ts:163-166`) |
| 라운드 간 reasoning params 고정 | ✅ (변경 시 체인 끊김) | — | ✅ |
| 프리웜으로 첫 턴 즉시 delta | ✅ | — | ✅ (T1, landed) |
| 기계적 턴 effort 자동 다운시프트 | ❌ (없음) | ❌ (백그라운드 호출만 disabled) | ❌ |

판정:
- **라운드당 추론 비용의 #1 레버(`previous_response_id`+delta+encrypted_content)는 jawcode가
  이미 보유** — 실측으로 확인(read 6턴 전부 mode=delta items=1). 서버가 암호화 추론 상태를
  이어받아 매 라운드 추론을 재도출하지 않음.
- **effort 자동 다운시프트는 어느 에이전트도 메인 루프에서 안 한다** — jawcode 갭 아님. 게다가
  effort를 라운드 중간에 바꾸면 `previous_response_id` 체인이 끊겨 풀 재전송이 되므로
  (codex `get_incremental_items` 가드) 긴 체인에선 오히려 손해.
- 결론: **턴당 비용은 이미 최적. 남은 유일한 큰 레버는 "라운드 수"** = 갭1(배칭). 이게 최종
  권고가 갭1 단일 1순위인 결정적 근거.

## D. 권고 순서

1. **갭1 프롬프트 유도** — 1문단, 즉효, 위험 최소. read-heavy 턴 라운드 수 감소.
2. G8 툴출력 스필(기존 10_moc) — 멀티툴 결과 배칭 비용 절감.
3. doom-loop 감지 — 낭비 라운드 차단.
4. G9 mid-stream 디스패치 — 단독 사이클, 최대 효과·최대 위험.

측정 근거: A절에서 **병목은 모델 연산**으로 확정됐으므로, 클라이언트 개선의 실익은
"라운드 수 감소"(갭1·G8)가 "라운드당 단축"(G9)보다 크다. 갭1이 1순위인 이유.

## E. 최종 결론 (5종 + 업스트림 + 라이브 실측 종합)

1. **렌더링은 병목 아님** (TUI≈headless, A절). 6-way 동시성에서도 유지.
2. **fast(priority)는 실효** — 약 1.5배 빠름(별도 측정, [00.03](./00.03_impl_closeout.md)). echo는 무의미.
3. **턴당 추론 비용은 이미 최적** — jawcode가 `previous_response_id`+delta+encrypted_content를
   이미 사용(C2, 실측 확인). codex와 동급, CC보다 우위.
4. **유일하게 큰 미개척 레버 = 라운드 수 = 갭1(배칭 프롬프트 유도).** 코드+실측+5종 비교+업스트림
   판정 4중으로 확정. 4/5 에이전트가 프롬프트 유도를 갖고 jawcode만 없음. gemini-cli가 post-stream
   인데도 프롬프트만으로 배칭 성공 → 갭1 단독으로 충분, 코드 변경 불필요(`system-prompt.md` 1문단).
5. **mid-stream 디스패치(G9)는 2순위** — 효과 있으나 라운드 수 감소보다 작고 위험 큼.

**실행 1순위 후보였던 갭1 — 실측 결과 기각(보류).**

`system-prompt.md` `<tools><policy>`에 배칭 유도 1문단을 추가하고 동일 6-read 작업을 재실측:

| 조건 | 턴 수 |
|---|---|
| 프롬프트 유도 전 | 7 (실측) |
| 프롬프트 유도 후 | **7 (3/3 동일)** ← 효과 0 |

→ **gpt-5.5는 배칭 프롬프트 유도에 반응하지 않는다.** 코드 리서치의 codex 주석("parallel
enabled여도 실무선 1개 아이템/요청")과 정합 — 멀티툴 배칭은 **모델 속성**이지 프롬프트로 못
바꾼다. 효과 0인데 프롬프트 토큰만 늘리므로 **변경 되돌림(revert).** 갭1은 **보류**로 이동.

판정 보정: "라운드 수" 레버는 적어도 gpt-5.5에서는 **프롬프트로 도달 불가**. 남는 가능성은
구조적 강제(예: 모델이 안 묶어주면 클라가 read 배치를 강제하는 별도 메커니즘)인데, 이는
모델 의도를 거스르는 침습적 변경이라 ROI/위험이 나쁘다 → 보류.

### 잔존 결론 (클라이언트 레버 소진)

- 렌더링: 병목 아님 (수정 완료분으로 충분)
- fast: 실효 (이미 동작)
- 턴당 추론비용: 이미 최적 (previous_response_id+delta, jawcode 보유)
- 라운드 수: gpt-5.5에선 프롬프트로 못 줄임 (실측 기각)
- → **남은 체감 지연은 모델 연산(high effort)** 이며, 클라이언트가 더 줄일 여지는 작다.
  사용자 레버는 effort 하향(`config.yml :high → :medium`) — 코드가 아니라 설정.

선택적 보강(보류): gemini식 `wait_for_previous`(모델제어 직렬)는 배칭이 안 되는 이상 무의미.
