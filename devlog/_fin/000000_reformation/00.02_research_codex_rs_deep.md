## A. 외부 요인 — 서버측 성능 저하 (먼저 읽을 것)

**우리 증상 ①(1.7 tok/s)과 동일한 현상이 네이티브 Codex에서도 광범위하게 보고되고 있다.**
즉 급성 느림의 상당 부분은 클라이언트 개혁으로 못 고치는 업스트림 QoS다. 개혁의 목표는
"업스트림이 정상일 때 네이티브와 동급"으로 잡아야 하고, 비교 측정은 반드시 같은 시간대에
네이티브 codex와 나란히 돌려 격리해야 한다.

- GPT-5.4/5.5 **Fast가 ~1 tok/s, TTFT 10~30초+** — 2026-05-24~26경 시작, ChatGPT Pro 20x
  구독, low effort·새 세션·최소 컨텍스트에서도 재현, 미해결(open).

> 출처: [GPT-5.4 Fast and GPT-5.5 Fast are extremely slow (~1 token/s) · openai/codex #24585](https://github.com/openai/codex/issues/24585)

- gpt-5.5 xhigh 턴이 **첫 출력 전 30분 정지** 후 정상 재개 — 추론 침묵 구간이 서버측에서도
  극단적으로 길어질 수 있음 (우리 워치독 오발 문제와 직결).

> 출처: [Codex Desktop: gpt-5.5 xhigh turn stalled 30m before first output · openai/codex #24260](https://github.com/openai/codex/issues/24260)

- 네이티브도 **WS 실패 시 5회×15초=75초를 소모한 뒤에야 HTTP 폴백** — 워크어라운드로
  `supports_websockets = false` 프로바이더 설정. 우리 P0(빠른 폴백 + 가시성)와 같은 문제
  계열이며, 네이티브조차 폴백 지연을 겪는다는 증거.

> 출처: [TUI "Timeout waiting for child process" — 75s startup delay before HTTP fallback · openai/codex #22634](https://github.com/openai/codex/issues/22634)

- 원격 컴팩션 엔드포인트(`/responses/compact`)가 **30~60분 행 또는 504** — WS 본 응답은
  정상인데 컴팩션만 별도 HTTP 경로라 막힘. jawcode가 원격 컴팩션을 이식할 경우 주의.

> 출처: [Codex Desktop: /responses/compact can hang · openai/codex #24618](https://github.com/openai/codex/issues/24618)

- 2026년 5월 이후 Codex 응답 지연이 공개 신호 기준으로 악화 추세, OpenAI 상태 페이지도
  5/8·5/11 GPT-5.5 지연 인시던트 기록.

> 출처: [Codex Web Response Latency Has Worsened in Measurable Public Signals Since May 2026 — SmartScope](https://smartscope.blog/en/blog/codex-web-response-latency-may-2026/)

**판정 보정**: 검증 1차(41.7 tok/s)·2차(워밍 후 3~5초 라운드)는 정상 시간대였고, 증상
①·③(1.7 tok/s, 스트림 중간 붕괴)은 위 인시던트 패턴과 부합한다. → 증상 ③의 유력 후보를
"(a) 서버 QoS"로 상향. 10_moc G5 판별 테스트에 "같은 시각 네이티브 codex 대조군" 추가.

## B. codex-rs 직접 분석 — 네이티브 파라미터 기준표

jawcode 워치독·재시도 값을 맞출 때의 기준 (직접 확인, file:line):

| 파라미터 | 네이티브 값 | 위치 | jawcode 현재 |
|---|---|---|---|
| stream idle timeout | **300초(5분)** 기본, 서버가 원격 설정으로 덮어쓰기 가능 | `model-provider-info/src/lib.rs:26,309`, `config/src/thread_config/remote.rs:183` | 공통 워치독이 더 타이트 → stall 오발 (G1) |
| stream 재시도 예산 | **5회** (사용자 설정 상한 별도) | `model-provider-info/src/lib.rs:27,30` | `PI_CODex_WEBSOCKET_RETRY_BUDGET` 기본 5 — 동급 |
| request 재시도 예산 | **4회** | `model-provider-info/src/lib.rs:28` | — |
| WS connect timeout | 15초 (5회 = #22634의 75초) | `core/src/client.rs:155` | 10초 (`openai-codex-responses.ts:89`) |
| rate-limit 헤더 파싱 | Codex 전용 헤더 패밀리 → 윈도별 스냅샷 구조화 | `codex-api/src/rate_limits.rs:22-47` | usage fetch로 별도 조회만 |

추가 직접 확인 사항:

- **원격 컴팩션**: `core/src/compact_remote.rs`, `compact_remote_v2.rs` — 컴팩션을 서버
  엔드포인트에 위임 (로컬 요약 비용 0). 단 A절 #24618처럼 해당 엔드포인트 자체가 불안정한
  시기가 있으므로 이식 시 로컬 폴백 필수.
- **서버 푸시 설정**: `thread_config/remote.rs` — idle timeout 등 전송 파라미터를 서버가
  세션별로 내려줄 수 있는 구조. 서버가 "지금 느리니 기다려라"를 클라이언트에 전달하는
  통로가 있다는 뜻 — jawcode는 이 채널이 없어 고정값으로 오판.
- **rate-limit 스냅샷**: 응답 헤더에서 사용량 윈도(주간/시간별)를 매 요청 파싱해 사용자에게
  노출 — 스로틀 임박을 사전에 보여줄 수 있는 재료. jawcode 이식 후보 (G3 가시성과 연계).

## C. 탈출구 — Codex SDK 위임 옵션

너무 느리거나 전송 계층 유지비가 과하면, openai-codex 프로바이더에 한해 **턴 전체를
네이티브 바이너리에 위임**하는 선택지가 있다:

- `@openai/codex-sdk`(TypeScript)는 codex CLI를 스폰해 **JSONL stdio로 이벤트를 교환**하는
  래퍼. `startThread()` → `run()`/`runStreamed()`(async generator)로 턴 단위 실행,
  스레드 이어가기 지원.

> 출처: [@openai/codex-sdk — npm](https://www.npmjs.com/package/@openai/codex-sdk), [SDK – Codex | OpenAI Developers](https://developers.openai.com/codex/sdk)

장점: WS·델타·프리웜·서버 푸시 설정·향후 서버측 수정까지 **전부 자동 상속**. 전송 계층
유지비 0.
단점: 툴 실행·샌드박스·루프 제어가 codex 것으로 대체 — jawcode 고유 툴/스킬/권한 계층과의
통합이 어그러짐. 사실상 "jawcode가 codex의 TUI 셸"이 되는 구조라, **전 제품이 아니라
특정 모드(예: `--codex-native` 플래그)로만** 검토할 것.

판단 기준: T1~T4(00.01) 수리 후에도 같은 시간대 네이티브 대비 체감 격차가 유의미하게
남으면 그때 모드로 도입.

## D. 심층 스윕 결과 (서브에이전트, 검증 표기 ✅ = 직접 확인)

체감 지연 영향 순. A~B절과 중복 없는 신규 항목만.

### D1. WS permessage-deflate 압축 ✅

모든 WS 프레임에 `permessage_deflate` 확장 적용 (`codex-api/src/endpoint/responses_websocket.rs:542-549`
직접 확인). 토큰 델타 JSON의 반복 구조에 효과 큼. **Bun 네이티브 WebSocket은 클라이언트
permessage-deflate 미지원이므로 이식 시 `ws` 패키지(perMessageDeflate: true) 검토 필요** —
jawcode가 현재 비압축 프레임이라면 동일 스트림에서 바이트 수 배수 차이.

### D2. 턴 경계 WS 세션 캐싱 (Drop → 반납)

턴 종료 시 `ModelClientSession`이 Drop에서 `cached_websocket_session`으로 반납, 다음 턴이
`take_cached_websocket_session()`으로 회수 (`core/src/client.rs:378-384,408-423,949-955`).
jawcode도 세션 키별 `state.connection` 보존으로 **동등 구조 있음** — 차이는 idle 닫힘 후
재연결 시점뿐 (T1 콘텐츠 프리웜으로 해결).

### D3. 툴 mid-stream 디스패치 — codex도 한다 ✅

`OutputItemDone` 도착 즉시 툴 future를 `in_flight`(FuturesOrdered)에 push, `response.completed`
후 drain (`core/src/session/turn.rs:1875-1876` 직접 확인). **CC(10.01 §1)와 codex 둘 다
mid-stream 시작** — jawcode만 응답 완료 후 일괄 실행. 10 밴드 G9의 근거 보강.

### D4. 재시도 정책 정밀값

- 요청 레벨: 4회, base **200ms** ×2^n ±10% 지터, `retry_429: false` ✅
  (`model-provider-info/src/lib.rs:252-258` 직접 확인 — **429는 자동 재시도 금지**, 사용량
  한도로 표면화). 5xx·전송 오류만 재시도.
- 스트림 레벨: 5회 소진 시 **세션 영구 HTTP 폴백** (`core/src/responses_retry.rs:22-79`) —
  jawcode의 영구 강등과 동일 정책이되 예산 소진 후라는 차이.
- 인스트림 `rate_limit_exceeded`는 에러 메시지의 `"try again in Xs"`를 정규식 파싱해 그
  시간만큼 대기 (`codex-api/src/sse/responses.rs:487-511`).
- WS 60분 연결 한도(`websocket_connection_limit_reached`)는 무지연 재연결.
- 미드스트림 resume은 **없음** — 끊기면 전체 재요청 (단 같은 턴 내 재연결은
  `previous_response_id` 델타).

### D5. 서버 혼잡 분류 + 인스트림 rate-limit 텔레메트리 — 증상 ③ 판별 도구

- `server_is_overloaded` / `slow_down` 코드는 Retryable이 아닌 **별도 ServerOverloaded
  타입**으로 분류해 "서버 혼잡" UX로 표시 (`codex-api/src/api_bridge.rs:46-57`).
- 백엔드가 **스트림 중간에 `codex.rate_limits` 이벤트**로 primary(15분)/secondary(일간)
  윈도의 `used_percent`·`reset_at`을 푸시 (`codex-api/src/rate_limits.rs:132-163`,
  `sse/responses.rs:720-726`). **jawcode가 이 이벤트를 받아 노출하면 증상 ③(중간 레이트
  붕괴)이 서버 스로틀인지 즉석 판정 가능** — 10_moc G5의 최우선 판별 수단으로 승격.

### D6. 토큰 효율

- MCP 툴 스키마가 정규화 JSON **4,000바이트** 초과 시 다단 압축: 설명 제거 → `$defs` 제거
  → 깊이 2 초과 붕괴 ✅ (`tools/src/json_schema.rs:192-213` 직접 확인).
- 툴 출력은 head+tail 중간 절단 + `"Total output lines: N"` 프리픽스
  (`utils/output-truncation/src/lib.rs:12-19`). 컴팩션 시 사용자 메시지 캡 20K 토큰
  (`core/src/compact.rs:49`).

### D7. 기타 운영 디테일

- 프리웜 하드 타임아웃 = WS connect timeout(15초), 전용 모듈 ✅
  (`core/src/session_startup_prewarm.rs` 존재 확인) — T1 설계 시 참조.
- 원격 컴팩션은 unary POST `/responses/compact`, 타임아웃 = idle×4(기본 20분)
  (`codex-api/src/endpoint/compact.rs:33-68`).
- WS 업그레이드 426 응답 → 세션 영구 HTTP 폴백 (`core/src/client.rs:1419-1422`).
- `x-responsesapi-include-timing-metrics` 옵트인 헤더 + 요청 body에
  `x-codex-ws-stream-request-start-ms`(클라 송신 시각) — 서버측 지연 귀속용
  (`core/src/client.rs:134-145,1667-1682`). G2(지표) 이식 후보.
- 전송 튜닝 부재 확인: API 경로에 H2/TCP_NODELAY 강제 없음, WS는 Nagle 유지(tungstenite
  권장) — 소켓 레벨 마법은 없다. 격차는 전부 프로토콜 레벨(델타·압축·프리웜)에서 난다.
