## 1. 스트리밍 툴 실행 (mid-stream dispatch) — 이식 가치 최상

`StreamingToolExecutor` — 스트림이 끝나기 전에, `tool_use` 블록이 파싱되는 즉시 해당 툴
실행 시작. 모델 출력과 툴 실행이 실시간 인터리브. ✅ (`services/tools/StreamingToolExecutor.ts`,
`isConcurrencySafe` 필드 :26,84,105 확인)

- 동시성 규칙: 읽기 전용 툴(Read/Grep/Glob/WebFetch/WebSearch/LSP 등)은 항상 병렬,
  Bash는 읽기 전용 판정 통과 시에만 병렬, Write/Edit/Agent/MCP는 배타.
- Bash 실패 시 형제 툴 일괄 취소(sibling abort).
- jawcode 현재: 응답 완료 후 일괄 실행 (`agent-loop.ts:1215-1229`의 shared/exclusive
  스케줄링은 있으나 mid-stream 시작은 없음). **연속 툴 라운드에서 라운드당 수 초 절감 여지.**

## 2. 워치독·재시도 기준값 — G1 캘리브레이션 재료

- 스트림 idle 워치독: **90초** (45초에 경고), 바이트 0 기준. ✅ (`services/api/claude.ts:1869-1877`)
- 청크 간 30초 스톨은 **로그만** (abort 안 함).
- 재시도: 최대 10회, base 500ms 지수 + 25% 지터, 상한 32초. 529는 3연속 후 모델 폴백.
- 백그라운드성 호출(querySource 비전경)은 529 시 **즉시 포기** — 재시도 증폭 방지.
- `Retry-After` 헤더 존중.
- 시사점: CC조차 90초인데 codex-rs는 300초(00.02 B절) — gpt-5.5 high의 추론 침묵은
  Anthropic 모델보다 길다. jawcode G1은 **프로바이더×effort별 floor**가 정답.

## 3. 컴팩션 2단 구조 — 마이크로컴팩션 먼저

- **마이크로컴팩션**: 풀 컴팩션 전에 오래된 툴 결과만 `'[Old tool result content cleared]'`로
  치환 (Read/Bash/Grep/Glob/WebSearch/WebFetch/FileEdit/FileWrite 대상). 비용 거의 0.
- 풀 컴팩션 트리거: 매 루프 진입 시 `contextWindow - max(maxOutput, 20_000) - 13_000`
  초과 검사. ✅ (`services/compact/autoCompact.ts:62,70,76` — `AUTOCOMPACT_BUFFER_TOKENS=13_000`,
  연속 실패 3회 서킷브레이커)
- 토큰 기준은 로컬 추정이 아니라 **직전 응답의 input_tokens 필드**.
- jawcode 시사점: 대형 컨텍스트가 TTFT·재전송 비용의 곱셈 항이므로, 마이크로컴팩션
  도입만으로 00 밴드 전 항목의 상수를 줄인다.

## 4. 사이드콜 은닉 패턴 — 메인 스트림 뒤에 숨기기

- 툴 사용 요약(Haiku): 툴 배치 직후 발사하되 **await 안 함** — 다음 루프 반복 시작점에서
  소비. ~1초 지연이 5~30초 메인 턴 안에 숨음.
- 메모리 프리페치·스킬 디스커버리도 동일 패턴 (쿼리 진입 시 발사, 툴 완료 후 소비).
- jawcode 현재: title-generator가 첫 메시지와 **동시 발사** (검증 1차 로그 09:23:06.903/933,
  같은 계정 한도 공유). 발사는 유지하되 미해결 Promise로 두고 라운드 경계에서 소비 +
  429/529 시 즉시 포기 태그.

## 5. 툴 결과 크기 규율

- 툴당 50K chars 기본 — 초과분은 디스크로, 모델엔 `<persisted-output>` 경로 참조.
  ✅ (`constants/toolLimits.ts:13` — `DEFAULT_MAX_RESULT_SIZE_CHARS = 50_000`)
- 메시지당 합계 200K chars / 400K bytes 상한.
- jawcode 시사점: append/캐시 안정성(G4)과 시너지 — 히스토리에 거대 툴 출력이 들어가지
  않으면 델타 직렬화·재전송 비용 자체가 작아진다.

## 6. 캐시 규율 (Anthropic 캐시지만 원리는 이식 가능)

- 캐시 브레이크포인트: 시스템 프롬프트 세그먼트별 + 툴 스키마 마지막 + **히스토리 마지막
  메시지 1곳**. fire-and-forget 포크는 `skipCacheWrite`로 메인 캐시 오염 방지.
- 기능 플래그성 헤더는 **세션 시작 시 래치** — 도중 토글로 prefix가 흔들리는 것 자체를
  봉쇄 (TTL 플립당 ~20K 토큰 낭비 방지).
- jawcode 시사점: G4(직렬화 안정성)의 설계 선례. "Juice: 0" 조건부 주입 같은 mid-session
  구조 변경은 CC 기준으로는 금지 패턴.

## 7. 투기 실행 (speculation) — 장기 아이디어

다음 턴 예측을 읽기 전용 포크 에이전트로 미리 실행 (`services/PromptSuggestion/speculation.ts`
✅ 존재 확인, SAFE_READ_ONLY_TOOLS 한정, skipCacheWrite). 메인 모델 대기 시간에 다음 턴
재료를 선확보. 이식 비용 크므로 백로그.
