# 10.04 — 증상 진단: 버스트 시 출력 둔화 / idle 회복 (5.5)

> 상태: 진단 ✅ (260613, 측정 기반) / 판별 테스트 ⬜ (사용자 승인 대기)
> 상위: [10_moc](./10_moc_general_improvements.md)
> 입력: 사용자 "출력이 끊겨서 호출됨, 대화 이어나가면 특히 5.5에서 출력속도 현저히 느려짐.
> 조금 기다리면 빨라짐. 짧은 단위로 끊어치면 느려지는 특성?"

## 증상

- 연속 짧은 턴(끊어치기) → 출력속도 둔화 (라이브 푸터 16.6 tok/s).
- 조금 기다림(idle) → 다시 빨라짐.
- 대화가 길어질수록 5.5에서 심화. 출력이 중간에 끊긴 듯 보임("ansi gap" 직감).

## 가설별 측정·판정

| 가설 | 측정/증거 | 판정 |
|---|---|---|
| 서버 quota 스로틀 | 활성 세션 로그 `codex.rate_limits` **primary=6%/secondary=3% 내내 평탄** | 기각 (할당량 안 참) |
| 클라 라운드당 O(컨텍스트) clone+stringify (P2) | 마이크로벤치 `/tmp/clonebench.ts`: 445KB body에서 clone×2+stringify+baseline = **0.6ms/round** | 기각 (sub-ms, 무시가능) |
| 긴-텍스트 마크다운 O(n²) 렌더 | `assistant-message.ts:193` 델타마다 전체 텍스트 `new Markdown` 재파싱. **실재 버그**지만 **출력 길이**에 묶임 | 부분 — 버스트/idle 시그니처와 불일치. 별개 항목으로 상존(긴 단일 출력엔 영향). |
| **서버 단기 처리량 스로틀** (token-bucket류) | "버스트 고갈→idle 충전"의 교과서적 패턴. quota 윈도우(6%)와 별개 차원. codex #24585(GPT-5.5 Fast 간헐 ~1tok/s)와 정합 | **유력** |

> 출처: [GPT-5.4/5.5 Fast extremely slow · openai/codex #24585](https://github.com/openai/codex/issues/24585)

## 판정

측정으로 클라이언트 후보 2개(quota, clone)를 기각. 남은 유력 원인은 **ChatGPT sub/priority
백엔드의 단기 처리량 스로틀** — 버스트로 순간 처리량이 깎이고 idle에 회복. 이는 우리 지표 밖
(quota used_percent와 다른 차원)이라 직접 못 봤지만, "끊어치면 느림/기다리면 빠름" + quota
평탄 + 스트리밍 tok/s 저하가 모두 이 가설로만 동시에 설명됨. **클라이언트가 못 고치는 외부
요인.** (긴-텍스트 마크다운 O(n²)는 별개로 상존 — 긴 단일 응답 렌더엔 영향, 본 증상의 주인은 아님.)

## 판별 테스트 결과 (260613, Sonnet 3 병렬 버스트) — 서버 스로틀 기각

calm 베이스라인: 짧은 작업(read+1문장) **6.1s/호출, 2턴**.

3-way 동시 × 4 연속(=12 호출, headless `jwc -p`, 렌더 없음):

| 워커 | median | max |
|---|---|---|
| A | 5.35s | 7.19s |
| B | 5.10s | 6.40s |
| C | 5.00s | 8.11s |

**버스트에서도 베이스라인과 동일(5~6s), 둔화 0.** 추가로 이전 6-way 동시 10-write도
274→276-297s로 거의 안 늘었음. → **headless는 어떤 버스트로도 사용자 증상 미재현.**

**판정 갱신: 서버 단기 처리량 스로틀 기각.** 사용자 증상은 **TUI(인터랙티브)에서만** 발생 =
렌더 경로. headless(`-p`)는 렌더를 안 하므로 구조적으로 재현 불가. 범인은 **마크다운 O(n²)
렌더**(아래) — 코드로 확정된 유일한 TUI 전용 둔화 경로. "대화 길수록/긴 출력일수록 심화"와
정합("끊어치면"은 누적 렌더 백로그, "기다리면 빠름"은 백로그 드레인으로 설명).

⚠️ **수리 주의**: 렌더 경로는 **보호된 TUI**(메모리 `tui-visual-design-protected`: welcome/
scroll/folding user-curated). 코얼레싱/캐시 패치는 스트리밍 시각 동작을 깨지 않는 범위로
한정하고 before/after 시각 검증 필수. 보호 영역 변경은 사용자 확인 후 진행.

## (구) 판별 테스트 (정의적) — 위에서 실행 완료

서버 vs 클라 최종 분리:
- **동일 버스트를 headless(`jwc -p`, TUI 렌더 없음)로** 연속 실행 → 그래도 둔화하면 **서버**
  (렌더·클라 무관), TUI에서만 둔화하면 **렌더**. clone은 양쪽 동일하므로 이미 기각됨.
- 또는 같은 시각 네이티브 codex-cli로 버스트 → 동일 둔화면 백엔드 공통.

## 완화책 (서버측이면)

- effort 하향(`config.yml :high→:medium`) — 턴당 토큰·부하 감소로 스로틀 체감 완화.
- 연속 끊어치기 대신 한 턴에 요청 묶기(단 5.5는 멀티툴 배칭 안 함 — 10.03).
- 혼잡 시간대 회피. (sub 백엔드 특성)

## ★ 근본 원인 유력 — fast(priority) 모드의 서버측 회귀 (사용자 가설 확인)

웹 검증: gpt-5.5/5.4 **fast 모드가 최근 Standard만큼 느려진 문서화된 서버측 회귀** 다발 보고.

| 이슈 | 내용 |
|---|---|
| openai/codex #19241 | "GPT-5.5 Fast가 Standard 속도처럼, 사용량은 증가分 청구" |
| openai/codex #24422 | "GPT-5.5 Fast가 갑자기 Standard만큼 느림 + 긴 thinking/context/search stall" |
| openai/codex #18692 | "GPT-5.4 Fast가 Standard와 차이 없는데 크레딧 2x" |

> 출처: [#19241](https://github.com/openai/codex/issues/19241), [#24422](https://github.com/openai/codex/issues/24422), [Speed – Codex](https://developers.openai.com/codex/speed)

- fast는 gpt-5.5에서 **2.5x 크레딧** 소모(공식). 현재 회귀로 **속도 이득 소멸 + 비용 2.5x** = 순손해.
- #24422의 "긴 stall + 끊김"이 사용자 증상("끊겨서 호출/둔화")과 정확히 일치.
- 이게 앞선 모든 측정과도 정합: headless 테스트는 fast 영향이 적은/혼합된 순간이라 재현 안 됨;
  내 초기 A/B(priority 84 vs default 56)는 fast가 정상 작동한 좋은 윈도우. 회귀는 **간헐적**.
- **권고: `/fast off`** (또는 `serviceTier: none`). 사용자가 이미 `config.yml: serviceTier: none`으로
  전환함 — 정확한 조치. jawcode에서 none이면 `shouldSendServiceTier`=false → priority 미전송
  (`ai/src/types.ts:218-225`). `/fast` 토글은 `slash-commands/builtin-registry.ts:719`.
- 클라이언트 수리 불가(서버측). jawcode 측 개선 여지: fast 회귀 감지 시 자동 폴백 — anthropic은
  이미 있음(`clearAnthropicFastModeFallback`), openai-codex엔 없음 → 후보(난이도 중).

## ★★ 최종 실증 확인 (사용자 머신, 260613)

`serviceTier: none`(fast off)으로 전환 후 사용자 직접 확인:
> "고점은 낮지만 둔화는 안 걸림"

= **fast ON: 고점↑ but 간헐적 stall/둔화 / fast OFF: 고점↓ but 안정(둔화 0).** 문서화된
gpt-5.5 fast 회귀와 정확히 일치. **근본 원인 = fast 모드 서버측 회귀 확정** (클라 무관).

트레이드오프(사용자 선택):
- **fast on**: 정상 윈도우엔 ~1.5x 빠름(내 A/B 84 vs 56) but 회귀 윈도우엔 stall + 2.5x 비용.
- **fast off (현 설정)**: 천장 낮지만 안정적, 비용 1x. 회귀가 해소될 때까지 권장.

## 코드 레벨 확정 — fast는 순수 passthrough (codex-rs 심층)

codex-rs 전수 조사 결과:
- `service_tier:priority`는 요청 JSON에 주입만(`core/src/client.rs:769`). **tier 전용 처리 0**:
  idle 타임아웃·재시도·abort·복구 전부 tier 무관(`codex-api/src/sse/responses.rs`).
- 응답의 `service_tier` echo를 **읽지 않음** → 요청 priority vs 응답 default 불일치 감지 없음.
- `codex.rate_limits`에 tier 필드 없음 — priority 전용 윈도우 없음.
- 코드/주석/CHANGELOG 어디에도 fast 불안정 언급 없음.
- 기본값: `default_service_tier: null`(gpt-5.5/5.4) — fast는 **명시 설정/토글 시에만** 활성.

→ **fast 둔화/끊김은 100% 서버측.** 클라(codex·jawcode)는 스로틀된 priority 스트림을 일반
idle 타임아웃과 구분조차 못 함. 유일한 완화 = tier를 default로(fast off). 사용자 가설 + 실증 +
코드 3중 확정.

## 웹 리서치 — 속도 워크어라운드/대안 (fast 대신)

**fast가 회귀 중인 지금, 속도를 되찾는 실효 레버 = reasoning effort 하향.** (fast off로 고점이
낮아진 사용자에게 직접 처방.)

- **effort 곡선(커뮤니티 벤치)**: minimal ~0.1x · low ~0.3x · **medium 1x(기준)** · high 3~5x ·
  xhigh 8~15x 토큰/시간. → **현재 jawcode 기본 `:high`를 `:medium`(또는 단순작업 `low`)로 내리면
  3~5배 체감 단축.** gpt-5.5 공식 기본도 medium.
- gpt-5.5는 동일 작업에 **출력 토큰 ~40% 적음** → per-token 속도 같아도 작업당 벽시계 짧음.
- **gpt-5.4-mini**: Codex에서 2x+ 빠름, 입력 토큰 3.3x 저렴 — 단순작업 최적.
- **Codex Spark(gpt-5.3-Codex-Spark)**: Cerebras, **1000+ tok/s**(~15x), Pro 한정·지능은 5.3급.
  빠른 프로토타입/타겟 편집용.
- **#14204 확인**: `service_tier=fast`가 ChatGPT websocket 백엔드에서 **조용히 드롭**("Unsupported
  service_tier: fast" 반환, 2026-03 보고, 미해결 종료) → 우리가 본 **echo=default가 정상(거부)**
  임을 뒷받침.

속도 옵션 MLB 20-80 (체감 인터랙티브 지연 기준, 80=최速):

| 옵션 | 속도 | 품질 | 비용효율 |
|---|---|---|---|
| Codex Spark | 80 | 55 | 60 |
| gpt-5.4-mini, effort=low | 72 | 55 | 75 |
| gpt-5.4 + /fast(구독) | 68 | 65 | 45 |
| gpt-5.5, effort=low/minimal | 65 | 72 | 60 |
| gpt-5.4, effort=medium | 58 | 65 | 65 |
| gpt-5.5 + fast(구독) | 55 | 75 | 35 (회귀로 신뢰도↓) |
| gpt-5.5, effort=high (현 설정) | 35 | 80 | 30 |

**처방(사용자)**: fast off 유지 + `config.yml: modelRoles.default`의 effort를 `:high → :medium`로.
단순작업 많으면 `gpt-5.4-mini` 프로파일 병행. 정 빠름 필요 + Pro면 Spark.

## 웹 리서치 — service_tier echo 메커니즘 (이전 결론 정밀 보정)

공식 문서: `response.service_tier`는 **실제 처리된(granted) tier**를 보고하며 요청값과 다를 수 있음.

- **API 키(종량) 경로**: `priority` 요청 → 응답 `default`면 **진짜 다운그레이드 신호**
  (ramp rate limit 초과 시 — ≥1M TPM에서 15분 내 +50% 급증 → standard 과금 + echo=default).
- **ChatGPT OAuth/Codex 경로(jawcode 사용)**: 내부 tier명이 `"fast"`(→ upstream `priority` 매핑).
  OAuth/Responses 라우트가 tier 메타를 동일하게 안 돌려줄 수 있어 **echo=default/auto가 거부인지
  메타 누락인지 미문서화**(CLIProxyAPI #1933이 정확히 이 현상). 네이티브 codex는 echo 미독.
- priority는 **API 키 기능**, ChatGPT Pro "fast"는 별개 구독 라우팅. 구독엔 API 크레딧 미포함.
  priority 공개 latency SLA 없음(대형 계약 한정).

**이전 결론 보정**: "echo=default는 무의미"는 **OAuth 경로 한정 참**(모호 + codex 무시) — `⚡?`
revert는 OAuth에서 여전히 타당. 단 **API-키 경로에선 echo=default가 실제 다운그레이드 신호**라
"무의미"는 틀림. jawcode는 OAuth라 실무상 영향 없음.

> 출처: [Create chat completion — service_tier echo | OpenAI API Reference](https://developers.openai.com/api/reference/resources/chat/subresources/completions/methods/create), [Priority processing | OpenAI API](https://developers.openai.com/api/docs/guides/priority-processing), [Configuration Reference — Codex (fast→priority 매핑)](https://developers.openai.com/codex/config-reference), [priority returns default on Codex OAuth · CLIProxyAPI #1933](https://github.com/router-for-me/CLIProxyAPI/issues/1933)

## 별개 상존 항목 — 긴-텍스트 마크다운 O(n²)

`assistant-message.ts:171-239` `updateContent`가 텍스트 델타마다 `#contentContainer.clear()`
후 누적 전체 텍스트로 `new Markdown(fullText)` 생성. Markdown 캐시는 전체 텍스트 키라 토큰마다
미스 → 전체 lexer+highlight 재실행 = 출력 길이에 O(n²). **본 증상(버스트/idle)의 주인은
아니나**, 긴 단일 응답에서 렌더 둔화를 유발하는 실재 버그. 수리안: 스트리밍 중 렌더 코얼레싱
(델타마다가 아니라 ~16-32ms 프레임마다) 또는 활성 블록 tail-only 파싱. 별도 검증 후 처리 권고.
