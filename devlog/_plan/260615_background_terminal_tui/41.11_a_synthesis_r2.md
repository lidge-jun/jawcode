## Finding

ARCH-A7: plan pseudocode used `manager.getJob(row.id, ownerFilter)`, but `AsyncJobManager.getJob` accepts only `(id: string)`. Owner scoping must be manual before reading metadata or adding ids.

## Evidence

- `packages/coding-agent/src/async/job-manager.ts` defines `getJob(id: string): AsyncJob | undefined`.
- Existing owner scoping in tools checks `job.ownerId` manually or uses APIs that accept an owner filter (`cancel`, `readOutputSince`).

## Plan change

Update `resolveBackgroundIds()` algorithm steps 2 and 3:

- call `manager.getJob(row.id)` with one argument;
- only use the job if `!ownerFilter?.ownerId || job.ownerId === ownerFilter.ownerId`;
- when adding `currentJobId` / `historicalJobIds`, resolve each with `manager.getJob(id)` and apply the same owner check;
- keep `readOutputSince(jobId, offset, ownerFilter)` and `cancel(..., ownerFilter)` as the owner-enforcing operation APIs.

## Residual risk

None beyond implementation discipline; the revised algorithm now matches the real `AsyncJobManager` signature and owner model.
