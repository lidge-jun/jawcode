## Finding responses

- **F1 accepted.** Successful completion visibility is now explicitly handled as structured TUI/overlay visibility through terminal rows retained until async job retention or acknowledgement. No assistant prose injection.
- **F2 accepted.** Failed and cancelled jobs are modeled as attention terminal states and retained until acknowledgement.
- **F3 accepted.** Snapshot count semantics are defined: active = running/paused plus crons; terminal-unacknowledged = completed/failed/cancelled retained rows; footer visible count = active + terminal-unacknowledged. Failure-only/cancelled-only cases render `needs attention`; completion-only renders `completed`.
- **F4 accepted.** `ctrl+x` no longer delegates to current `handleBackgroundCommand()` because that backgrounds the whole interactive app. The plan now adds a safe `handleForegroundBackgroundCommand()` wrapper that shows `No foreground job to background` until a real foreground-detach API exists.
- **F5 accepted.** `status-line.ts` is added as an explicit modify target. The existing status-line jobs segment remains monitor/cron-oriented in the first slice; composer footer is the canonical all-background summary to prevent divergent counts.
- **F6 accepted.** Footer copy is aligned with status: `running`, `completed`, or `needs attention`.
- **F7 accepted.** Subagent inclusion now uses `type === "task" || metadata.subagent` before monitor/bash classification, so metadata-tagged subagent jobs are not lost.
- **F8 accepted.** Overlay membership rules are defined: active rows plus unacknowledged terminal rows; acknowledged completed/failed/cancelled rows can disappear after acknowledgement/retention.
