## WONDER — what's still missing?

- Real terminal viewport position is still not directly observable. The chosen fix avoids duplicate/pushed rows by using the safe viewportRepaint-only path for `ProcessTerminal`, but it cannot preserve append-growth for genuinely bottom-pinned real terminals until a reliable hook exists.
- The tests prove xterm-backed semantics and byte-path discrimination (`\x1b[20;1H` vs viewport repaint). They do not fully emulate Ghostty/macOS scrollback behavior under every live streaming cadence.
- Follow-up classification after live use: a minor display glitch that self-heals on the next chat input remains deferred in `../260614_performance/73_scroll_repaint_timing_followup.md`. It is a repaint timing edge, not evidence to undo the unknown-viewport fix or restore broad tick-based rendering.
- If future performance work further narrows repaint paths, `#previousLines`, `#viewportTopRow`, and physical cursor bookkeeping remain sensitive areas.

## REFLECT — how to improve the spec

- The earlier plan should have explicitly treated `undefined` viewport state as a separate third state rather than folding it into bottom-like behavior.
- Acceptance criteria now need to include branch-discriminator byte assertions when a bug depends on physical terminal behavior, not just visible tail-row presence.
- The ontology should distinguish logical transcript correctness from physical terminal write safety: a logically correct frame can still duplicate or push rows if the renderer chooses the wrong physical append path.
