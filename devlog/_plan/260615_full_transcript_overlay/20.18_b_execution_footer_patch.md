# 20.18 B — bash/eval full transcript footer patch

## Root cause

`BashExecutionComponent` and `EvalExecutionComponent` already render all available output lines in expanded/full transcript mode, but `#updateDisplay()` still computed `hiddenLineCount` from the collapsed preview window. The status footer then emitted a misleading `… N more lines (ctrl+o to expand)` hint even though Ctrl+T had already rendered the full historical output.

## Code changes

- `packages/coding-agent/src/modes/components/bash-execution.ts`
  - When building the status footer, pass `hiddenLineCount: this.#expanded || hasSixelOutput ? 0 : hiddenLineCount`.
  - This preserves existing sixel suppression and prevents false hidden-line counts for expanded/full transcript bash output.
- `packages/coding-agent/src/modes/components/eval-execution.ts`
  - When building the status footer, pass `hiddenLineCount: this.#expanded ? 0 : hiddenLineCount`.

## Test changes

- `packages/coding-agent/test/session-transcript-replay.test.ts`
  - Existing `replays completed bash and python execution output without streaming caps` still asserts first/last markers for 150-line bash and python histories.
  - Added assertions that full transcript output does not contain `ctrl+o to expand` and does not match `/… \d+ more lines/`.
