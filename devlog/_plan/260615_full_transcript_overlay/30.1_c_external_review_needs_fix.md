## Findings and decisions

### Finding 1 — Read replay out of order

Decision: accept as code issue.

Evidence:

- Test fixture calls tools in order: generic → read → bash → eval.
- `G` tail assertion currently checks `OLD_READ_LINE_19`, meaning replay placed read after later tool calls.
- Root cause: `session-transcript-replay.ts` defers creating `ReadToolGroupComponent` until the `toolResult`, so the read group is appended at result time rather than tool-call time.

Fix:

- Create/update the `ReadToolGroupComponent` when a historical read `toolCall` is encountered.
- Add it to `items` immediately to preserve assistant tool-call order.
- Map `pendingTools.set(content.id, readGroup)` at tool-call time so the later result populates the existing in-order component.
- Add order assertions: generic < read < bash < eval < live tool; `G` should show `OLD_EVAL_LINE_19` and `LIVE_TOOL_MARKER`, not read.

### Finding 2 — Historical bashExecution/pythonExecution replay uses streaming append path

Decision: accept as code issue.

Evidence:

- `session-transcript-replay.ts` replays `bashExecution`/`pythonExecution` by `appendOutput(message.output)` then `setComplete(...)`.
- Those components support `setComplete(..., { output, truncation })`; replay should restore completed historical output directly, not re-stream it through bounded append paths.

Fix:

- Update `replayStandardMessage()` for `bashExecution` and `pythonExecution` to pass output/truncation into `setComplete`.
- Add regression coverage with historical bash/python execution output when practical.

### Finding 3 — FullTranscriptOverlay cache can show stale live/current output

Decision: accept as code issue.

Evidence:

- `FullTranscriptOverlayComponent.#transcriptLines()` returns cached lines by width only.
- Overlay manually renders source components; child invalidation does not necessarily invalidate overlay cache.

Fix:

- Remove the early width-only cache return or constrain caching to immutable sources.
- Add a test mutating a source component between renders at the same width; second render must show updated output.

### Finding 4 — Generic JSON full transcript keeps expanded depth/scalar caps

Decision: accept as code issue.

Evidence:

- `ToolExecutionComponent` removes JSON line count cap in full transcript mode but still uses expanded-mode depth/scalar limits.

Fix:

- In full transcript mode, use unbounded or transcript-specific JSON depth/scalar limits.
- Add regression for nested JSON / long scalar full transcript output.
