# 70.7 — P2.0 A-stage audit synthesis round 1

Planner verdict: PASS (`70.5_a_p2_0_planner_audit_r1.md`)
Architect verdict: FAIL (`70.6_a_p2_0_architect_audit_r1.md`)

## Root cause

The P2.0 plan had become precise enough for plan coherence but still mapped several counters to abstract phase names rather than existing code-owned hook points. The Architect failures are valid integration findings: some planned keys (`sdk.resume.startupFlags.scan`, navigation transcript render rows, `frame.emittedLines`, text placeholders) would either duplicate existing metrics owners or force ad hoc source changes during B-stage.

## Finding decisions

- PLANNER-A1 accepted: add an explicit Scope boundary list.
- PLANNER-A2 accepted: add registry API mapping.
- PLANNER-A3 accepted: done checklist will name wrap/visible placeholders.
- PLANNER-A4 accepted with compromise: transcript materialization rows are cache-miss/initial-or-unknown only; navigation gets a separate `navigation.input` counter because scroll does not rematerialize lines on cache hits.
- PLANNER-A5 accepted: all placeholder rows explicitly carry `placeholder: "true"`.
- PLANNER-A6 accepted: add `perf.write.error` registry row and assertion.
- PLANNER-A7 accepted: flush owner is `TUI.stop()` only for runtime lifecycle plus explicit helper in tests.
- PLANNER-A8 accepted: `frame.emittedLines` becomes a placeholder until the diff/write owner is selected.
- PLANNER-A9 accepted: disabled-mode tests use isolated temp paths and assert zero appended rows/bytes.
- PLANNER-A10 accepted: SDK counter names are changed to existing spans; no mid-implementation helper-only fallback.

- ARCH-A1 accepted: P2.0 keeps JSONL text counters as TUI-frame placeholders and leaves `utils.ts` `recordHelper("text.visibleWidth")` as the real existing owner. No `utils.ts` migration in this slice.
- ARCH-A2 accepted: replace `sdk.resume.startupFlags.scan` with exact existing spans: `sdk.session.context.build`, `sdk.session.branch.scan`, `sdk.session.workspaceTree.start`, and `sdk.session.ttsr.discover` as needed. No invented startup-flags module.
- ARCH-A3 accepted: transcript line materialization counters attach to `#transcriptLines` cache-miss path; navigation uses a separate `transcript.overlay.navigation.input` counter in `handleInput`.
- ARCH-A4 accepted: split session open counters into `headerProbe.file.load` and `hydrate.file.load` / entry counts, documenting current double load instead of hiding it.
- ARCH-A5 accepted: rename resume-specific context counter to `sdk.session.context.build`, or gate it on existing-session only if implementation chooses a resume-only hook. Plan now prefers the non-resume-specific key.
- ARCH-A6 accepted: `session.context.entries.materialized` is instrumented exactly at `SessionManager.buildSessionContext()` around its current `getEntries()` call.
- ARCH-A7 accepted: `TUI.stop()` is the runtime flush owner and tests set `JWC_PERF_LOG` to a temp path.
- ARCH-A8 accepted: tests must enable/reset `renderMetrics` directly or set env before import; no assumption that post-import env changes affect singleton construction.
- ARCH-A9 accepted: `frame.emittedLines` is placeholder/deferred, not aliased to `frame.lines`.
- ARCH-A10 accepted: add `bun --cwd=packages/coding-agent run check` to gates.
- ARCH-A11 accepted: coding-agent hooks use process-wide `renderMetrics` singleton.
- ARCH-A12 accepted: prepared-line placeholders come only from the TUI frame hook; no prepared-line module.

## Residual risk

The main residual risk is implementation ergonomics for synchronous JSONL flush from `TUI.stop()`. B-stage must pick an implementation that preserves the synchronous stop contract and emits no interactive noise. If that proves infeasible, B-stage must revise the plan before claiming DONE.
