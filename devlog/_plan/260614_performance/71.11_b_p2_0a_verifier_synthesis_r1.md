# 71.11 — P2.0a B-stage verifier synthesis round 1

Verifier verdict: NEEDS_FIX

Source: `agent://59-P20aBVerifier`

## Issues

1. **Event-row cardinality not fully bounded.** `recordEvent()` bounded snapshot counter keys but emitted original `source`, original counter keys, and original labels. The plan requires source/counter/label cardinality to follow bounded overflow policy.
2. **Test coverage incomplete.** Missing coverage for default `resolvePerfLogPath()` using `getLogsDir()`, event-row source/counter/label overflow, `PI_TUI_METRICS=1` in-memory-only stop behavior, `JWC_PERF=1` flush intent, repeated stop/flush idempotence/no duplicates, and explicit `process.memoryUsage()` non-use by event recording.

## Root cause

The implementation treated the bounded counter map as the canonical cardinality guard, but the JSONL row itself is also an externally consumed artifact and must be bounded independently. The tests validated common JSONL rows but not overflow or every flush-gating branch.

## Decisions

- Fix implementation: add bounded event source/counter/label maps and retain event row fields through `retainedLabel()` before buffering.
- Fix reset: clear the new event-boundary maps.
- Fix tests:
  - Assert default path equals `path.join(getLogsDir(), "perf.jsonl")`.
  - Assert event rows overflow source/counter/label names to `other`.
  - Assert `PI_TUI_METRICS=1` / programmatic enablement without `JWC_PERF` or `JWC_PERF_LOG` does not auto-flush from `TUI.stop()`.
  - Assert `isPerfJsonlFlushEnabled({ JWC_PERF: "1" })` is true; use actual stop-flush test with `JWC_PERF_LOG` temp path for file safety.
  - Assert repeated stop/flush does not duplicate rows.
  - Spy on `process.memoryUsage()` around `recordCounter()` / `recordEvent()`.
