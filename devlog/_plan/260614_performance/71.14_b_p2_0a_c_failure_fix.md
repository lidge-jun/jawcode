# 71.14 — P2.0a B retry: C-stage adversarial fixes

Date: 2026-06-15

## Trigger

C-stage adversarial review `agent://62-P20aCAdversarial` returned FAIL with four B-route findings:

1. Programmatic `renderMetrics.enable()` + `TUI.stop()` without `JWC_PERF` / `JWC_PERF_LOG` lacked an isolated disk no-append assertion.
2. Empty `JWC_PERF_LOG=""` counted as flush intent while `resolvePerfLogPath()` fell back to the default log path.
3. The integration test matched `frame.lines` only when the value was `> 0`, over-constraining a counter that may validly be zero.
4. `metrics.ts` module header still described only the older `PI_TUI_METRICS` surface.

## Root cause

Implementation/test boundary issue. The P2.0a runtime sink was sound, but two edge contracts were under-specified in tests and one flush-intent predicate accepted an empty override string.

## Fix

- `packages/tui/src/metrics.ts`
  - `isPerfJsonlFlushEnabled()` now treats `JWC_PERF_LOG` as intent only when it is a non-empty string.
  - Module header documents `JWC_PERF=1`, `PI_TUI_METRICS=1` in-memory compatibility, bounded `jwc.perf-events/1` JSONL helpers, and non-empty `JWC_PERF_LOG` stop-flush intent.
- `packages/tui/test/metrics.test.ts`
  - Added `isPerfJsonlFlushEnabled({ JWC_PERF_LOG: "" }) === false`.
- `packages/tui/test/perf-events-p2_0.test.ts`
  - Programmatic-enable and `PI_TUI_METRICS` compatibility tests now seed an isolated perf-log file and assert it remains unchanged after `TUI.stop()`.
  - `frame.lines` row matching now checks for numeric key presence rather than `> 0`.
