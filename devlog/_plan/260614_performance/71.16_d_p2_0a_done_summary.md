## Acceptance criteria met

- `JWC_PERF=1` enables process-wide metrics collection.
- `PI_TUI_METRICS=1` remains an in-memory compatibility enablement path and does not auto-flush JSONL from `TUI.stop()` by itself.
- `JWC_PERF_LOG` overrides the target log path only when non-empty.
- Event rows use compact `jwc.perf-events/1` JSONL with UTC timestamps, source, counters, and optional labels.
- Event/counter retention is bounded and overflows to `other` / `eventsDropped`.
- Successful flush appends one JSON object per line, drains buffered events, and is idempotent across repeated `TUI.stop()`.
- Flush failures do not throw in interactive use, increment `perf.write.error`, and retain buffered events.
- The real TUI render path emits `tui.frame`, `tui.preparedLine`, and `tui.text` rows from the post-truncation anchor.
- Existing P1.5.1 input render priority tests remain green.
- No coding-agent/session/sdk hooks, P2.2 prepared-line cache behavior, P2.1 assistant child reuse behavior, render scheduling rewrite, scroll/fill/gap rewrite, or welcome visual rewrite was introduced in P2.0a.

## REFLECT — spec improvements

- The original broad P2.0 plan mixed too many surfaces for one A-stage; future measurement plans should split sink/lifecycle from individual subsystem counters up front.
- Acceptance criteria should distinguish flush intent from path resolution explicitly, including empty-string environment values.
- Disk no-append assertions should always seed isolated files when the contract is “no automatic flush.”
- Placeholder counter rows should specify whether value constraints require key presence, numeric type, zero, or positive count.
