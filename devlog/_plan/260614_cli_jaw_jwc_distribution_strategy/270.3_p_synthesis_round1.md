# 270.3 — P critic synthesis round 1

> Critic receipt: `270.2_p_critic_round1.md`
> Verdict: ITERATE
> Plan under revision: `270.1_p_jwc_selector_exposure_plan.md`

## Original plan intent

Plan a narrow 270 implementation that exposes JWC in cli-jaw Manager's primary runtime selector and verifies `/api/cli-registry` JWC visibility without touching 280 default runtime, 300 Code UI, or 305 Jaw mode native attach.

## Root-cause category

Verification contract was too soft. The plan identified the correct code paths and mutations, but left the live API and visible UI checks optional/ambiguous. That would let B-stage claim completion after metadata-only unit tests, even though the parent 270 contract requires registry API and dropdown evidence.

## Evidence checked locally

- `270_jwc_ui_cli_selector.md:52-58` explicitly requires `curl ... /api/cli-registry | jq '.jwc'` and a browser dropdown check.
- `320_post270_current_state_triage.md:24-37` says 270 must check cli-jaw selector/registry paths together with current jawcode package evidence.
- `../cli-jaw/src/routes/settings.ts:310-311` wraps `buildLiveCliRegistry()` in `ok(res, ...)`; response shape must be pinned by inspecting `../cli-jaw/src/http/response.ts` during implementation.
- `../cli-jaw/public/manager/src/settings/pages/components/agent/RuntimeHeader.tsx:50-56` uses `PRIMARY_CLIS` both for visible ordering and `collapsedAfter`.
