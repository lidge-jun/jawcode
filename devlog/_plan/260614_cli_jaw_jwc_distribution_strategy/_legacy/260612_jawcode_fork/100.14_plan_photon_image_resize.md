100.13 감사에서 `Bun.Image` 부재가 high로 잡혔고, 당시엔 의존성 결정이 사용자 몫이라 헤더
metadata만 셰임하고 실제 리사이즈는 미구현으로 뒀다. 레퍼런스 4종(gemini-cli·codex-cli·
opencode + sharp)을 비교한 결과 **네이티브 libvips(sharp)는 아무도 안 쓰고**, 같은 Bun
런타임인 **opencode가 photon(WASM)으로 실제 리사이즈**를 한다는 게 확인됐다. WASM은
플랫폼별 네이티브 바이너리가 없어 esbuild Node 번들과 잘 맞으므로, opencode와 동일한
`@silvia-odwyer/photon-node`로 `Bun.Image` 셰임의 transform을 채운다.

## 왜 이 방식인가 (레퍼런스 근거)

| 리포 | 런타임 | 리사이즈 | 방식 |
|------|--------|---------|------|
| gemini-cli | Node | ❌ | 패스스루 (인프로세스 리사이즈 안 함) |
| codex-cli | Rust | ✅ | 순수 Rust `image` 크레이트 (네이티브 X) |
| **opencode** | **Bun** | ✅ | **photon WASM** (`@silvia-odwyer/photon-node@0.3.4`) |
| (불채택) sharp | Node | — | 네이티브 libvips ~19MB·플랫폼별 빌드 — 레퍼런스 0 |

photon WASM = 단일 `.wasm`(1.9MB) + JS glue, 플랫폼 분기 없음. sharp의 멀티플랫폼 네이티브
빌드 복잡도를 피하면서 실제 decode/resize/encode 가능.

## 우리 케이스가 opencode보다 단순한 이유

- **Bun 런타임**(jwc CLI/TUI): 네이티브 `Bun.Image` 사용 → 셰임도 photon도 **안 돌아감**.
- **Node dist-node 번들**에서만 셰임이 활성(`shims/index.ts`: `!globalThis.Bun`일 때만 설치).
- 따라서 opencode가 Bun **컴파일 바이너리** 때문에 했던 `globalThis.__OPENCODE_PHOTON_WASM_PATH`
  글로벌 경로 패치가 **불필요**하다. photon을 esbuild **external**로 두면, 런타임에 Node가
  node_modules에서 `photon_rs.js`를 로드하고 그 CJS가 `__dirname` 기준으로
  `photon_rs_bg.wasm`을 `readFileSync` → 자연 해소.

## photon 0.3.4 API (디스크 실측)

순수 CJS(`main: photon_rs.js`, `module` 필드 없음). require 시점에 동기 WASM init
(`new WebAssembly.Module/Instance` + `__wbindgen_start()`), 별도 `init()`/`initSync()` 없음.

| API | 시그니처 | 용도 |
|-----|---------|------|
| `PhotonImage.new_from_byteslice(Uint8Array)` | static | 인코딩된 이미지 bytes 디코드(포맷 자동) |
| `img.get_width()` / `get_height()` | | 치수 |
| `resize(img, w, h, SamplingFilter.Lanczos3)` | free fn | 새 PhotonImage 반환 |
| `img.get_bytes()` | | PNG 인코드 |
| `img.get_bytes_jpeg(quality)` | | JPEG 인코드 (0–100) |
| `img.get_bytes_webp()` | | WebP 인코드 (품질 인자 없음) |
| `img.free()` | | **WASM 힙 수동 해제 (필수)** |

검증(plain Node, 260613): 16×16 PNG → decode → resize 8×8 → PNG/JPEG/WebP 전부 바이트 생성 OK.
1×1 등 degenerate 입력은 `new_from_byteslice`가 WASM panic → 셰임 try/catch로 원본 폴백.

## 호출부 — 무수정 (`image-resize.ts`)

업스트림 `coding-agent/src/utils/image-resize.ts`는 `Bun.Image` 체인을 사용:
```
new Bun.Image(buf).metadata() → {width,height,format}
new Bun.Image(buf).resize(w,h).png()|.jpeg({quality})|.webp({quality}).bytes()
```
정책: maxWidth/Height 1568, maxBytes 예산, fast-path(이미 작으면 통과) → PNG/JPEG/WebP 최소
선택 → 품질 래더 → 치수 래더. `GJC_NO_WEBP`로 WebP 제외. **이 파일은 안 건드린다** —
`Bun.Image` 셰임에 photon 백엔드만 채워 체인을 실제 동작시킨다 (리베이스 친화·최소 diff).

## 구현 (`packages/jwc/src/shims/bun-image.ts` 재작성)

`NodeBunImage`를 lazy-photon 백엔드로:
- `metadata()`: **헤더 파싱 유지** (WASM 로드 없이 빠른 fast-path 판정; 100.13 자산 재사용).
- `resize(w,h)`: 목표 치수 기록 (체인).
- `png()` / `jpeg({quality})` / `webp({quality})`: 출력 포맷+품질 기록 (체인).
- `bytes()` (terminal): `await import("@silvia-odwyer/photon-node")` lazy 로드(캐시) →
  `new_from_byteslice` → (resize 지정 시) `resize(...,Lanczos3)` → 포맷별 `get_bytes*`
  → `free()` 2개 → Uint8Array 반환. decode/encode 실패는 throw → image-resize가 catch.
- `arrayBuffer()`: bytes() 위임.
- photon 미지정 transform(체인만 하고 bytes 호출)이면 metadata는 헤더로, bytes는 photon.

빌드(`build-node.ts`): `@silvia-odwyer/photon-node`를 **external** 목록에 추가
(better-sqlite3와 동일 — node_modules 런타임 해석, `.wasm`을 번들에 인라인하지 않음).
`packages/jwc/package.json` dependencies에 `@silvia-odwyer/photon-node@0.3.4`.

## 함정 (Sonnet 보고 반영)

1. WASM 경로: external + node_modules `__dirname`로 해소 (opencode 글로벌 패치 불필요 — Bun
   컴파일 바이너리 한정 이슈였음).
2. `with { type: "file" }` 문법은 Node esbuild에서 금지 → import 안 씀, 동적 import로 로드.

4. `free()` 미호출 시 WASM 힙 누수 → resize 결과 + 원본 둘 다 free.
5. CJS interop: `await import(...)`의 default/namespace 양쪽 대응(`mod.default ?? mod`).
