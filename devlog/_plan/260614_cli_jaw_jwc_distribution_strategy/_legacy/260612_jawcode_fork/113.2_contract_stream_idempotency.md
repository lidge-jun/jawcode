## 조항 1 — SSE 리플레이 멱등성

**규칙**: 모든 `/api/jwc/*` SSE 이벤트 envelope는 다음 필드를 **처음부터** 가진다.

```jsonc
{
  "topic": "jwc",
  "event": "jwc-invalidate",        // 또는 스트림 이벤트명
  "id": "evt-<run>-<seq>",          // 이벤트 고유 id (재전달 식별)
  "sseReplay": false,               // 재전달 시 true — 클라이언트는 중복 적용 금지
  "cursor": { "textLen": 0, "traceSeq": 0 },  // run 스코프 누적 커서
  "rev": 8
}
```

- 클라이언트는 `id` 중복 수신 시 no-op, `sseReplay: true`면 누적 커서 비교 후 부족분만 적용한다.
- 근거: cli-jaw 260612 docs 06-09 + 260613 P1-b — 끊긴 연결의 지연 재전달이 중복 버블/도구 셀을 만든 실사고. 재전달분은 `[worker-replay …]` 마커 패턴과 동형으로 식별 가능해야 한다.

## 조항 2 — keep-alive는 데이터 이벤트로

**규칙**: SSE 코멘트(`: ping`)는 **금지** — JS `EventSource`/fetch-reader에 보이지 않아 클라이언트 워치독이 불가능하다.

```jsonc
{ "topic": "system", "event": "ping", "ts": 1765584000000 }
```

- 서버는 N초 주기 ping 데이터 이벤트를 발행하고, 클라이언트는 **2.5 × 주기** 무수신 시 강제 재연결한다 (staleness 워치독).
- 근거: cli-jaw 260613 Phase 4 — 코멘트 기반 keep-alive로는 죽은 연결을 클라이언트가 감지 못 한 실증.

## 조항 3 — 스냅샷 하이드레이션 멱등성

**규칙**: 모든 스냅샷 GET 응답(`/api/jwc/models`, `/sessions/:id/state`, …)의 **표시용 배열에 안정 식별자 없는 항목 금지**.

- 항목은 `id`(또는 `stepRef`/`seq`)를 필수로 가지며, 클라이언트 머지는 식별자 기반 — ordinal(배열 인덱스) 머지 금지 (ordinal 발산 가드).
- 동일 시그니처 스냅샷 재수신은 no-op 스킵 (시그니처 = `rev` 또는 content hash).
- 근거: cli-jaw 260613 Phase 1-c/3.

## 조항 4 — 장시간 호출은 202 + 폴링

**규칙**: 예상 소요가 수십 초를 넘을 수 있는 작업 API(워커 위임, 대량 export 등)는 블로킹 응답 금지 — `202 {jobId}` + result 폴 + **완료 전 저장된 verdict 동봉** 패턴 고정.

- 근거: cli-jaw 260613 Phase 6 — undici `headersTimeout` 300s에서 블로킹 HTTP가 죽으며 클라이언트 타임아웃 캐스케이드 발생. ACP host의 장시간 호출도 동일 함정 ([112.3 §S1 주의](./112.3_plan_code_mode_impl.md)).
- dispatch는 always-poll — 폴링 주기는 계약이 아니라 클라이언트 구현 자유.

## 조항 5 — audience 게이트 설계

**규칙**: 표면에 보여야 할 진행 신호는 처음부터 **public + sanitized 레인**으로 설계한다. internal 전용 스트림에 UI-필요 신호를 실으면 "클릭해야 렌더" 버그의 뿌리가 된다.

- 이벤트 정의 시 `audience: 'public' | 'internal'`을 명시 필드로 계약에 포함 — cli-jaw `bus.broadcast(type, data, audience)` 시그니처와 동형.
- 직원(employee) 스트림이라도 진행률·상태 요약은 public 레인에 sanitized 사본을 발행한다.
- 근거: cli-jaw 260613 docs 00-03 (직원 스트림 비가시성 사고).
