# 100.01 — Node 포팅 착수점 고정: dist-node 빌드 스켈레톤

> 상태: ✅ **완료 (260613)** — 빌드 green(23.7MB ESM), Node import는 계약대로 `bun`에서
> 차단(100.02 신호). 실행 기록은 §실행 결과.
> 선행: [100.00_prep_gate_cleanup.md](./100.00_prep_gate_cleanup.md)
> 상위 문서: [100_moc_node_porting.md](./100_moc_node_porting.md)

## 결론

100.00 다음 작업은 **바로 모든 Bun API를 치환하는 것**이 아니라, Node 포팅의 산출물 위치와 빌드 진입점을 먼저 고정하는 것이다.

100.01의 목표는 아래 한 줄이다.

> `packages/jwc/dist-node/sdk.js`를 100 밴드의 Node 산출물 계약으로 확정하고, esbuild 기반 Node 22 빌드 스켈레톤이 `@gajae-code/coding-agent/sdk`를 번들 대상으로 잡는지 확인한다.

이 단계는 아직 완전한 Node 실행 성공을 요구하지 않는다. 실패해도 된다. 대신 실패가 **어떤 Bun API/문법에서 시작되는지**를 재현 가능한 로그로 고정해야 한다.

---

## 왜 100.01이 필요한가

100 MOC에는 아직 열린 질문이 남아 있다.

| 질문 | 100.01 결정 |
|------|-------------|
| Q1 `packages/jwc/dist-node/` vs `packages/coding-agent/dist-node/` | **`packages/jwc/dist-node/`로 고정** |
| Q2 vitest vs `node:test` | 100.01에서는 결정하지 않음. Node 테스트 단계인 100.04 이후에서 `vitest` 기본값 유지 검토 |
| Q3 `Bun.serve` 범위 | 100.01에서는 구현하지 않음. 빌드 실패 표면에 걸리는지만 기록 |
| Q4 듀얼 런타임 | **듀얼 유지**. Bun TUI 경로는 건드리지 않음 |
| Q5 ACP 선행 여부 | 100.01에는 영향 없음. Node SDK import 가능성이 선행 기반 |

즉 100.01은 “대규모 포팅 시작”이 아니라 **Node 빌드 계약 고정 + 첫 실패 표면 확보**다.

## 범위

### 포함

- `packages/jwc/dist-node/`를 Node 산출물 경로로 확정
- `packages/jwc/scripts/build-node.ts` 신설
- `packages/jwc/package.json`에 `build:node` 스크립트 추가
- esbuild 설정 최소 골격 작성
  - `platform: "node"`
  - `target: "node22"`
  - `format: "esm"`
  - entrypoint: `packages/jwc/src/sdk.ts`
  - outfile: `packages/jwc/dist-node/sdk.js`
  - loader: `{ ".md": "text", ".html": "text" }`
  - define: `{ "Bun.env": "process.env" }`
- 첫 빌드 실행 결과를 이 문서에 기록

### 제외

- `packages/jwc/src/shims/` 전체 구현
- `Bun.file`, `Bun.write`, `Bun.spawn`, `Bun.sleep` 등 API 치환
- `bun:sqlite` → `better-sqlite3` 치환
- `Bun.JSONL.parseChunk` 자체 파서 구현
- vitest/node:test 전환
- 실 프로바이더 스트리밍 검증

## 구현 순서

### 1. 현재 상태 확인

```bash
git status --short
bun run check:ts
bun run check:jwc-ui
```

통과 기준:

- 100.00 변경이 포함된 상태에서 `check:ts`, `check:jwc-ui`가 green
- 예상 외 변경이 있으면 먼저 분리하거나 문서에 기록

### 2. 빌드 의존성 확인

`esbuild`가 현재 workspace catalog/devDependencies에 있는지 확인한다.

- 있으면 기존 버전 사용
- 없으면 100.01에서 추가할지, Bun 내장 `Bun.build`로 임시 스켈레톤을 만들지 결정

기본값은 **esbuild 명시 의존성 추가**다. 100 MOC가 esbuild를 계약으로 잡았기 때문이다.

### 3. `packages/jwc/scripts/build-node.ts` 작성

초기 스켈레톤은 “성공 보장”보다 “실패 표면 고정”을 우선한다.

필수 속성:

```ts
await build({
  entryPoints: ["src/sdk.ts"],
  outfile: "dist-node/sdk.js",
  bundle: true,
  platform: "node",
  target: "node22",
  format: "esm",
  loader: {
    ".md": "text",
    ".html": "text",
  },
  define: {
    "Bun.env": "process.env",
  },
  external: [
    "better-sqlite3",
    "node-tar",
    "strip-ansi",
    "json5",
    "xxhash-wasm",
    "@gajae-code/natives",
    "markit-ai",
  ],
});
```

주의:

- `ENGINE_NAME` define 금지
- `packages/tui/`를 직접 entry로 넣지 않음
- CLI entry(`src/cli-entry.ts`)가 아니라 SDK entry(`src/sdk.ts`)를 빌드
- 첫 단계에서는 shim inject를 넣지 않아도 된다. inject는 100.02에서 추가해도 된다.

### 4. `packages/jwc/package.json` 스크립트 추가

```json
{
  "scripts": {
    "build:node": "bun scripts/build-node.ts"
  }
}
```

기존 `bundle` 스크립트는 Bun CLI 번들용이므로 유지한다.

### 5. 첫 빌드 실행

```bash
bun --cwd=packages/jwc run build:node
```

가능한 결과:

| 결과 | 처리 |
|------|------|
| 성공 | `node -e "import('./packages/jwc/dist-node/sdk.js').then(m => console.log(Object.keys(m)))"` 실행 |
| 실패: text import | loader 설정 누락/문법 문제이므로 100.01에서 수정 |
| 실패: `Bun.*` runtime reference | 정상적인 다음 단계 신호. 100.02 shim inject 대상으로 기록 |
| 실패: `bun:sqlite` resolve | 100.02/100.03 sqlite shim 대상으로 기록 |
| 실패: package external 문제 | 100.01에서 external 목록 보정 |

## 산출물

100.01 완료 시 변경 파일은 원칙적으로 아래 2개다.

| 파일 | 내용 |
|------|------|
| `packages/jwc/scripts/build-node.ts` | Node 22 SDK 번들 스켈레톤 |
| `packages/jwc/package.json` | `build:node` 스크립트 |

빌드가 성공하면 아래 산출물이 생성될 수 있다.

| 경로 | 처리 |
|------|------|
| `packages/jwc/dist-node/sdk.js` | gitignore 정책 확인 후 커밋 여부 결정 |

현재 `packages/jwc/files`에는 `dist`만 포함되어 있으므로, publish 대상까지 고려하려면 추후 `dist-node` 포함 여부를 별도 결정한다. 100.01에서는 publish 정책을 바꾸지 않는다.

## 완료 기준

100.01은 아래 조건을 만족하면 완료다.

- `packages/jwc/package.json`에 `build:node`가 존재
- `packages/jwc/scripts/build-node.ts`가 존재
- `bun --cwd=packages/jwc run build:node`를 실행했고 결과가 문서에 기록됨
- 빌드 실패 시, 첫 실패 원인이 100.02 이후 작업 항목으로 분류됨
- `bun run check:jwc-ui` green 유지
- 가능하면 `bun run check:ts` green 유지

100.01의 성공은 **Node SDK가 완전히 import되는 것**이 아니다. 그것은 100.03~100.04의 완료 기준이다.

## 실행 결과 (260613)

- esbuild `^0.25.0` devDependency 추가 (0.21은 import attribute 자체를 거부 — 171 에러).
- 1차 빌드 실패 표면: `with { type: "text" }` 미지원 + `bun`(26)·`bun:sqlite`(11)·`bun:ffi`(2)
  미해석. 실측 임베드 확장자: .md 137 · .txt 15 · .py 3 · .lark 2 · .html 1.
- 플러그인 `text-import-attributes`로 해소: 텍스트 5종 확장자 onResolve(`args.with.type==="text"`
  가드)+onLoad(text). **부수 발견** — `with { type: "json" }`은 esbuild가 JSON-modules 스펙
  (default-only)으로 처리해 `utils/src/dirs.ts:17`의 named import(`engines`,`version`)가 깨짐
  → json attribute도 플러그인이 클레임해 named-export json 로더로 우회.
- `bun`/`bun:sqlite`/`bun:ffi`는 **의도적 external** — 빌드는 통과시키고 Node 런타임
  실패점으로 남김 (100.02 셰임 별칭 대상).
- 최종: `dist-node/sdk.js` 23.7MB ESM 생성. `node v24 import` → `Cannot find package 'bun'`
  (계약된 다음 신호). `.gitignore`에 `packages/jwc/dist-node/` 추가, rebrand 스캐너
  ignoredDirs에 `dist-node` 추가 (산출물 내 upstream 토큰 오탐 방지).
- 게이트: `check:jwc-ui` exit 0 · `check:ts` exit 0.

## 100.01에서 하지 말아야 할 것

- `packages/tui/` 시각/스크롤 코드를 건드리지 않는다.
- `ENGINE_NAME = "gjc"`를 바꾸지 않는다.
- `@gajae-code/*` 패키지 스코프를 바꾸지 않는다.
- 모든 Bun API를 한 번에 치환하려고 하지 않는다.
- `dist-node` publish 정책을 100.01에서 확정하지 않는다.
- 실패 로그 없이 “나중에 셰임하면 됨”이라고 넘기지 않는다.
