## 왜 이 문서인가

050 밴드에서 IPABCD 런타임(CLI·슬래시·상태머신·스테이지 프롬프트)이 완전히 구현됐지만 **모델이 그 존재를 알 방법이 없다**. 현재 시스템 프롬프트는 "exactly four default workflow skills"라고 선언하고 있고, 그 이후 어디에도 `orchestrate` 표면이 언급되지 않는다. 모델에 "너가 할 수 있는 것들"을 물으면 `ralplan`/`ultragoal`/`team` 어휘로만 답한다(260612 스크린샷). 동시에, 사용자가 "pabcd 진행해"라고 말해도 모델은 `/orchestrate` 슬래시가 아니라 스스로 shell 명령을 내릴 수 있다는 사실을 모른다.

이 플랜은 세 모듈로 그 빈 자리를 채운다:

- **M1** — 시스템 프롬프트 `<jwc-runtime>` 블록 전면 개정: discovery 레이어. 모델이 세션 시작 시점에 orchestrate 표면을 인식하게 한다.
- **M2** — 매 턴 스테이지 헤더 주입: 지속성 레이어. 컴팩션 후에도 단계 컨텍스트가 살아남는다.
- **M3** — 스테이지 프롬프트 말미 자가 전이 문장: 자가 전이 레이어. 모델이 `jwc orchestrate <stage>`를 스스로 실행한다.

세 모듈의 관계는 cli-jaw의 "상주 안내(M1) → 매 턴 헤더(M2) → 자가 전이(M3)" 4층 구조를 동형 이식한 것이다(`prompt_flow.md` §cli-jaw 주입 구조 대조 참조).

### 구현 사양

#### Step 1: `#buildPabcdStageMessage()` 메서드 추가

위치: `agent-session.ts` — `#buildGoalModeMessage()` (라인 4511) 바로 다음에 신규 private 메서드 삽입.

```typescript
async #buildPabcdStageMessage(): Promise<CustomMessage | null> {
    const cwd = this.sessionManager.getCwd();
    const sessionId = this.sessionManager.getSessionId();
    const result = await readPabcdState(cwd, sessionId).catch(() => null); // 방어적 이중화 — readPabcdState는 설계상 throw하지 않음
    if (!result) return null;
    if (!result.ok) return null; // corrupt state — skip silently
    const envelope = result.value;
    if (!envelope.active) return null;

    const stage = (envelope.current_phase ?? "").toUpperCase();
    if (!stage || stage === "COMPLETE") return null;

    const STAGE_LABELS: Record<string, string> = {
        I: "INTERVIEW",
        P: "PLANNING",
        A: "PLAN AUDIT",
        B: "BUILD",
        C: "CHECK",
        D: "DONE",
    };
    const label = STAGE_LABELS[stage] ?? stage;

    // Gate chip: audit_status / verification_status from ctx
    const ctx = envelope.ctx ?? {};
    const gateChips: string[] = [];
    if (stage === "A") {
        const audit = ctx.audit_status ?? "pending";
        if (audit !== "pass") gateChips.push(`audit=${audit}`);
    }
    if (stage === "B") {
        const verif = ctx.verification_status ?? "pending";
        if (verif !== "done") gateChips.push(`verification=${verif}`);
    }
    const gateStr = gateChips.length > 0 ? ` · ${gateChips.join(" · ")}` : "";

    // Next-action hint
    const NEXT_HINTS: Record<string, string> = {
        I: "Complete requirements gathering, then the user runs `orchestrate p`.",
        P: "Present plan draft and wait for approval. When approved, run `jwc orchestrate a`.",
        A: "Run audit subagents. On PASS + user approval, run `jwc orchestrate b`.",
        B: "Implement the plan. On DONE, run `jwc orchestrate c`.",
        C: "Run gates. All green → run `jwc orchestrate d` automatically.",
        D: "Summarize cycle, then run `jwc orchestrate d --complete`.",
    };
    const hint = NEXT_HINTS[stage] ?? "";

    const content = `[PABCD — ${stage}: ${label}${gateStr}]\n${hint}`.trimEnd();

    return {
        role: "custom",
        customType: "pabcd-stage-context",
        content,
        display: false,
        attribution: "agent",
        timestamp: Date.now(),
    };
}
```

**설계 결정 기록:**

- `display: false` — cli-jaw `getPrefix()` 동형. 사용자 UI에 노출하지 않고 모델 컨텍스트에만 주입.
- `customType: "pabcd-stage-context"` — plan-mode-context / goal-mode-context 패턴 일관.
- **corrupt state → skip silently**: `readPabcdState`가 `{ ok: false }` 반환 시 null 반환. 사용자 세션을 깨지 않는다.
- **read-every-turn**: pabcd-state.json은 1~2 KB의 소형 파일. 매 턴 읽기 오버헤드는 무시 가능하고, mtime 캐시 도입은 stale 헤더 발생 위험이 있다. 단순성 우선(cli-jaw `getPrefix()` 동형 접근). 향후 프로파일링 결과에 따라 mtime 캐시를 추가할 수 있다.
- `attribution: "agent"` — plan/goal 모드 메시지와 동일.

**plan 모드 + pabcd 동시 활성 시 중복 컨텍스트 동작**: plan 모드와 pabcd가 동시 활성인 경우 두 context message(plan-mode-context + pabcd-stage-context)가 모두 주입된다 — 충돌이 아니라 중복 컨텍스트이며 의도된 동작. 순서는 기존 push 블럭(agent-session.ts:4759-4766) 순서를 따른다.

**적용 범위 — boss 세션 한정**: 적용 범위는 boss 세션이다. 서브에이전트 세션은 sessionId가 달라 `readPabcdState`가 null을 반환하므로 자연히 주입 0 (cli-jaw가 boss 세션만 prefix하는 것과 동형). 브랜드 게이트는 별도로 두지 않음 — 상태 파일 경로가 `.jwc/`라 jaw 오케스트레이션에서만 존재.

#### Step 2: `prompt()` 내부 메시지 조립 경로에 M2 호출 추가

위치: 라인 4763–4766 직후 (`goalModeMessage` push 직후, `options?.prependMessages` push 직전):


#### Step 3: import 추가

`agent-session.ts` 상단 import 블록에:
```typescript

```

현재 `orchestrate-state.ts`는 `agent-session.ts`에서 import되지 않는다. 이 import가 신규 의존성이므로 순환 참조 여부를 확인해야 한다. `orchestrate-state.ts`는 `path`, `zod`, 내부 `skill-state/`, `gjc-runtime/state-schema`, `gjc-runtime/state-writer`만 import하므로 순환 없음이 예상된다.

#### sendPabcdStageContext() 공개 메서드 (선택)

goal 모드의 `sendGoalModeContext()` 패턴과 동형으로 `sendPabcdStageContext()` 공개 메서드도 추가한다. 이는 향후 pabcd 단계 전이 시 즉시 헤더를 주입하는 경로로 사용될 수 있다.

```typescript
async sendPabcdStageContext(options?: { deliverAs?: "steer" | "followUp" | "nextTurn" }): Promise<void> {
    const message = await this.#buildPabcdStageMessage();
    if (!message) return;
    await this.sendCustomMessage(
        {
            customType: message.customType,
            content: message.content,
            display: message.display,
            details: message.details,
            attribution: message.attribution,
        },
        options ? { deliverAs: options.deliverAs } : undefined,
    );
}
```

### M2 테스트 계획

신규 테스트 파일: `packages/coding-agent/test/pabcd-stage-header.test.ts`

테스트 목록:
- `"returns null when no state file exists"`: tmpDir에 pabcd-state.json 없을 때 `#buildPabcdStageMessage()` → null.
- `"returns null when envelope.active is false"`: `active: false` 상태 파일 → null.
- `"returns stage header for active stage B"`: `active: true, current_phase: "b"` → content가 `[PABCD — B: BUILD]` 포함.
- `"includes gate chip when audit_status is pending in stage A"`: A 단계 + `ctx.audit_status: "pending"` → content에 `audit=pending` 포함.
- `"includes gate chip when verification_status is needs_fix in stage B"`: B 단계 + `ctx.verification_status: "needs_fix"` → `verification=needs_fix` 포함.
- `"omits gate chip when a→b gate is satisfied"`: A 단계 + `ctx.audit_status: "pass"` → gate chip 없음.
- `"skips silently on corrupt JSON"`: 잘못된 JSON → null (예외 throw 없음).
- `"skips silently when active is complete"`: `current_phase: "complete"` → null.

**구현 방법**: M2 로직을 순수 함수 `buildPabcdStageContent(envelope)` 로 분리해 별도 모듈(`session/pabcd-stage-header.ts`)에 둔다. `agent-session.ts`의 `#buildPabcdStageMessage()`는 `readPabcdState()`로 파일을 읽은 뒤 `buildPabcdStageContent(result.value)`를 호출한다. 테스트는 `buildPabcdStageContent`를 직접 단위 테스트한다(8-case 목록 아래 참조).

## M3 — 스테이지 프롬프트 자가 전이 문장

### 배경

현재 `orchestrate-{i,p,a,b,c,d}.md` 6개 파일은 사용자가 `/orchestrate <stage>`를 직접 실행하는 것을 전제로 한다. 다음 두 파일은 이미 자가 전이 힌트가 있다:

- `orchestrate-p.md` 마지막 줄: `⛔ When approved, run: \`orchestrate a\``
- `orchestrate-a.md` 마지막 줄: `⛔ When approved, run: \`orchestrate b\``
- `orchestrate-b.md` 마지막 줄: `⛔ When approved, run: \`orchestrate c\``
- `orchestrate-c.md` 마지막 줄: `When done → run: \`orchestrate d\``
- `orchestrate-d.md` 마지막 줄: `When done → run: \`orchestrate d --complete\` (or start a new cycle with \`orchestrate i\`)`

**문제**: 이 힌트들은 모두 수동 안내("run: `orchestrate`")이지 자가 전이 명령("run it yourself via the shell tool")이 아니다. cli-jaw `orchestration.md:47`의 "YOU advance phases by running the exact `cli-jaw orchestrate` shell command. No other method." 패턴이 없다.

추가로 `orchestrate-i.md`에는 자가 전이 힌트 자체가 없다 — "Ready for planning. Run `orchestrate p` to proceed."라고 제안만 한다.

### 각 파일별 추가 문장

#### `orchestrate-i.md` — 현재 마지막 줄 (라인 16):
```
- Auto-transition I→P is forbidden; the user (or main session, in goal mode with evidence) must invoke `orchestrate p` explicitly.
```

이 줄 직전, line 15 직후 (`- Returning to I from any later stage...` 행 다음)에 다음 문장을 삽입한다:

```
- If the user says "pabcd 진행해" or otherwise requests to advance, run `jwc orchestrate p` via the shell tool yourself after confirming requirements are sufficient.
```

(I→P 자가 전이는 cli-jaw와 달리 D050-2에 의해 조건부 허용: 요건이 충분하면 모델이 전이 가능)

#### `orchestrate-p.md` — 현재 마지막 두 줄:
```
⛔ STOP. Present the plan and WAIT for user approval. No project-source mutation in P — only the devlog plan file and the sanctioned ralplan writer.
⛔ When approved, run: `orchestrate a`
```

두 번째 줄을 다음으로 교체:
```
⛔ When the user approves the plan, run `jwc orchestrate a` yourself via the shell tool.
```

#### `orchestrate-a.md` — 현재 마지막 두 줄:
```
⛔ STOP after reporting. WAIT for user approval.
⛔ When approved, run: `orchestrate b`
```

두 번째 줄을 다음으로 교체:
```
⛔ When the user approves (audit_status=pass is required), run `jwc orchestrate b` yourself via the shell tool.
```

Audit-prompt 연계: 라인 12의 `orchestrate audit-prompt planner` / `orchestrate audit-prompt architect` 안내는 이미 올바른 형태로 존재한다. 변경 불필요.

#### `orchestrate-b.md` — 현재 마지막 두 줄:
```
⛔ STOP after reporting. WAIT for user approval.
⛔ When approved, run: `orchestrate c`
```

두 번째 줄을 다음으로 교체:
```
⛔ When the user approves (verification_status=done is required), run `jwc orchestrate c` yourself via the shell tool.
```

#### `orchestrate-c.md` — 현재 마지막 줄:
```
When done → run: `orchestrate d`
```

교체:
```
When all gates are green → run `jwc orchestrate d` yourself via the shell tool immediately (no user approval required for C→D).
```

(C는 자동 전이 스테이지: 모든 게이트 통과 시 모델이 자동으로 D로 전이. 사용자 승인 대기 없음.)

#### `orchestrate-d.md` — 현재 마지막 줄:
```
When done → run: `orchestrate d --complete` (or start a new cycle with `orchestrate i`).
```

교체:
```
When done → run `jwc orchestrate d --complete` yourself via the shell tool to close the orchestration. To start a new cycle: run `jwc orchestrate i`.
```
