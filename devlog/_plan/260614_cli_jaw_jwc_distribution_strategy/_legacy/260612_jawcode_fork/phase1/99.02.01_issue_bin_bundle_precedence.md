# 99.02.01 — issue: dev 심링크가 stale 번들을 실행 (bin 번들 우선 버그) + update 배너 오발

> 상태: ✅ 수정 완료 (260612 밤). 소속: [99.02 정리](./99.02.00_plan_parallel_landing.md) 패키지.
> 입력: 사용자 "dist symlink여서 바로 적용되어야 하는 거 아니야?" + "symlink인데 update 배너 계속 뜨는 것도 이상해"
> (260612 22시, 083.7 컴포저 핀이 실행 중 세션에 반영 안 되던 조사에서 파생).

`~/.bun/bin/jwc` 심링크는 라이브 소스를 가리키는데도 src 변경이 실행에 반영되지 않았다. 원인은
심링크가 아니라 **bin의 로드 순서**: P12(063.1, 퍼블리시 독립화)가 만든 `dist/jwc.bundle.js`가
존재하면 워크스페이스에서도 무조건 번들을 실행했다 — 주석 의도("workspace checkouts fall back to
live sources")와 구현(try bundle → catch src)이 정반대였던 버그. 번들은 마지막 `bun run bundle`
시점의 스냅샷이라, 그 이후의 모든 src 변경(083.7 컴포저 핀 포함)이 조용히 무시됐다.

## 수정 ①: `packages/jwc/bin/jwc.js` — 워크스페이스 우선 프로브

```js
const workspaceCli = new URL("../../coding-agent/src/cli.ts", import.meta.url);
if (existsSync(workspaceCli)) await import("@gajae-code/coding-agent/cli");  // 워크스페이스 = 라이브
else await import("../dist/jwc.bundle.js");                                  // 퍼블리시 설치 = 번들
```

- 퍼블리시 tarball은 `files: ["bin","dist"]`라 `../../coding-agent` 프로브가 실패 → 번들 경로 유지
  (퍼블리시 거동 무변경). 워크스페이스에서는 dist가 있어도 항상 라이브 소스.
- 검증: `bun bin/jwc.js --version` → `jwc/0.4.4` (src 로드), `--help` 정상.

## 수정 ②: update 배너 오발 — dev 체크아웃에서 체크 스킵

`main.ts checkForNewVersion`은 `registry.npmjs.org/@gajae-code/coding-agent/latest`(업스트림
0.4.5)를 로컬 VERSION(0.4.4)과 비교해 배너를 띄웠다. dev 심링크 설치에서는 ① 비교 자체가 무의미하고
② 안내대로 `jwc update`를 실행하면 dev 링크를 덮어칠 위험. 가드 추가:

```ts
// test/는 어느 퍼블리시 tarball에도 없음 (업스트림: src/scripts/examples, jwc: bin/dist)
if (existsSync(new URL("../test", import.meta.url))) return;
```

- 업스트림이 **src를 그대로 배포**하므로 "/src/ 경로" 감지는 오발 — tarball에 없는 `test/` 존재가
  워크스페이스의 안전한 마커.

## 잔여 (이관)

- **브랜드 인식 업데이트 타깃**: jwc 퍼블리시 설치(번들)는 여전히 업스트림 패키지를 비교 — `jwc`
  npm 패키지(0.1.0) 기준 비교 + 번들에 jwc 버전 임베드가 정답. 릴리즈 게이트 성격 → **99.05 행으로 이관**.
- stale 번들 자체는 무해해짐(워크스페이스에서 미사용) — 퍼블리시 시 `bun run bundle`이 재생성.
- ⚠️ 실행 중 프로세스는 어떤 경우에도 핫리로드되지 않음 — src 변경 반영은 **재시작**이 단위
  (083.6 §4의 "재시작 후 재현" 절차와 동일).
