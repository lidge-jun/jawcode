## 3. [확정 제안] — 전략 B (+ 루트명 유지)

**내부 스코프 `@gajae-code/*` 보존, 대외 퍼블리시 표면은 `jwc` 단일화, 루트 name 현행 유지.**

근거: ① 체리픽 비용 최소 — 업스트림이 `@gajae-code/*` import 파일을 건드려도 충돌 없음 ② `packages/jwc`가 이미 유일한 사용자 진입점 (`npx jwc`) ③ npm org 등록 불필요 ④ g002/rebrand 가드의 업스트림 일치 유지.

**포기**: `@gajae-code/*` npm namespace에 포크 publish 불가 (업스트림 org 소유) → **M1이 필수 선결** (jwc 독립 퍼블리시 가능화).

## 4. 구현 모듈

### M1 — `packages/jwc` 의존 분리 (선결, 우선순위 최고)

현재 `jwc`가 `@gajae-code/coding-agent: catalog:` 의존 → npm에 해당 패키지 부재 시 `npm install jwc` 실패.
- 선택지 1 (번들): prebuild에서 coding-agent를 bun build로 번들, dist ship — tarball 수십 MB
- 선택지 2 (resolved deps): publish 시 버전 치환 — npm에 @gajae-code/* 존재해야 동작 (불가)
- → **[열린 질문 Q1]** 선결. 대상: `packages/jwc/package.json`, `scripts/ci-release-publish.ts`

### M2 — `packages/jwc` 독립 bin 강화

`packages/jwc/src`의 `@gajae-code/coding-agent` import 2곳을 번들 경로로 전환. 대상: `packages/jwc/src/index.ts`, `bin/jwc.js`.

### M3 — 가드 `WORKSPACE_SCOPE` 상수 추출 (미래 전환 대비)

`verify-g002-gates.ts:17,23,228` + `rebrand-inventory.ts:34` + `release.ts:243` + `sync-versions.ts:24` 하드코딩 → `const WORKSPACE_SCOPE = "@gajae-code"` 단일 상수화. 값 동일하므로 업스트림 충돌 범위는 해당 스크립트 1파일로 국한. `:193 codingName.includes("gajae")` 어설션도 상수 기반으로 (Q4).

### M4 — `bun.lock` 재생성 + 로컬 검증

`bun install` → `check:tools` → `test:release` → `verify-g002-gates`.

### M5 — 퍼블리시 드라이런

`bun scripts/ci-release-publish.ts --dry-run` + `cd packages/jwc && npm pack --dry-run` — tarball에 bin/dist 포함, `@gajae-code/*` 의존 미노출(M1 후), 버전 정책 확인.
