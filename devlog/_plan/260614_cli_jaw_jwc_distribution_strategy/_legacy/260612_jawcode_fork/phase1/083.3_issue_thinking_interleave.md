# 083.3 — issue: 추론(thinking)이 도구 뒤에 와도 맨 위 블록에 계속 누적됨

> 상태: ✅ 수정 완료 (A안 세그먼트 분할, 2026-06-12). 소속: 083 ([083_moc_tui_output.md](./083_moc_tui_output.md)).

## 적용된 수정 (A안)

- **라이브** (`event-controller.ts`): `#segmentStartIndex` 추가. `#handleMessageUpdate`에서 마지막
  toolCall 이후 가시 thinking/text가 등장하면 새 AssistantMessageComponent 세그먼트를 chatContainer
  끝에 추가하고 streamingComponent 포인터 교체. `updateContent`는 `#segmentSlice()`(세그먼트 시작
  이후 블록만)로 호출 — `#handleMessageEnd` 최종 렌더도 동일 슬라이스. 세그먼트 컷 시 위쪽 도구를
  즉시 minimize(083.1과 결합 — 다음 메시지를 기다리지 않고 post-tool 추론 시작 시점에 접힘).
- **리플레이** (`ui-helpers.ts`): assistant content를 toolCall 경계에서 `flushSegment()`로 분할 —
  [세그먼트, 도구들, 세그먼트, …] 순서로 추가. 에러/abort 푸터는 마지막 세그먼트만 유지(중간
  세그먼트는 stopReason "stop"으로 덮음), `setUsageInfo`는 마지막 세그먼트에 부착.
> 입력: 사용자 "추론이 tool 다음에 왔으면 그 다음에 위치해야 되는데 맨 처음 추론 block에서 계속 이어지고
> 있다" + 스크린샷 (서사형 추론 전부가 상단에 뭉치고, 도구 행들은 그 아래 일렬).

한 assistant 메시지 안에서 모델이 `[추론1, 도구A, 추론2, 도구B, …]`로 인터리브해도, TUI는
**추론1+추론2를 전부 상단의 단일 어시스턴트 블록에 합쳐 렌더**하고 도구 행들만 그 아래로 쌓는다.
시간 순서(추론→도구→추론)가 화면에서 사라져 "왜 이 도구를 불렀는지"의 문맥이 끊긴다.

## 원인 메커니즘 (코드 사실)

1. `message_start`(assistant)에서 **streamingComponent 1개**를 chatContainer에 추가 — 위치 고정.
   (`event-controller.ts:264-268` `#handleMessageStart`)
2. `message_update`마다 `streamingComponent.updateContent(전체 메시지)` — 컴포넌트는 자기
   `#contentContainer`를 **clear 후 메시지의 모든 text/thinking 블록을 통째로 재렌더**.
   (`event-controller.ts:317`, `assistant-message.ts:138-146` — clear → 전 블록 재추가)
3. toolCall 블록은 별도 `ToolExecutionComponent`로 **chatContainer 끝에 append**.
   (`event-controller.ts:327-` 스트리밍 / `:486-` 실행 시작)
4. → 같은 메시지에서 도구 호출 *이후* 도착한 thinking 델타도 (2)에 의해 **이미 위에 있는**
   어시스턴트 블록 안으로 들어감. 도구 행 아래에 새 추론 블록이 생기지 않음.

리플레이(`ui-helpers.ts` assistant 분기)도 동일 구조: `addMessageToChat(message)` 1회(텍스트·추론
전부) → toolCall 컴포넌트들 순차 추가 → 인터리빙 소실.

대조: Codex/Claude Code는 **콘텐츠 블록(셀) 단위 렌더** — thinking/text/tool이 도착 순서대로 각자
셀이 되어 시간 순서가 보존된다.

## 영향

- 추론↔도구 인과 문맥 소실 (특히 추론 많은 모델: cursor composer, thinking 계열에서 체감 큼).
- 083.1 자동 접힘과 결합하면 "위에 거대 추론 벽 + 아래 한 줄 도구 목록"으로 더 도드라짐 (스크린샷).

### A안 구현 메모 (착수 시)

- 분기점: `#handleMessageUpdate`에서 content를 스캔해 "마지막 toolCall 인덱스" 이후 블록 집합이
  비어있지 않고 현재 세그먼트가 toolCall 이전 범위면 → 새 세그먼트 생성.
- `updateContent(message, fromBlockIndex?)` 시그니처 확장 또는 세그먼트별 가상 메시지 슬라이스 전달.
- `setUsageInfo`/`#lastAssistantComponent`는 **마지막 세그먼트**에 부착 (`#handleMessageEnd`).
- 리플레이(`ui-helpers`): assistant content를 toolCall 경계로 슬라이스해 [어시스턴트 세그먼트, 도구들,
  어시스턴트 세그먼트, …] 순서로 추가.
- 점검: thinking 토글 rebuild(`hideThinkingBlock`), optimistic user message, 스트리밍 중 abort 경로,
  `#lastThinkingCount` 리셋 로직(`:319-325` — read 그룹 분리용)과의 상호작용.

## 검증 (수정 시)

- 단위: `[thinking, toolCall, thinking]` 픽스처 → chatContainer 자식 순서가 [세그먼트1, 도구, 세그먼트2].
- 리플레이 동일 순서. thinking 토글 on/off 왕복 무회귀.
- e2e: 추론 많은 모델로 도구 여러 개 턴 → 추론이 각 도구 사이에 위치.
