# 086.2 — issue: 스크롤백에 TUI 조각 잔존 + `/redraw` 수동 복구 커맨드

> 상태: ✅ 완화 적용 (2026-06-12, fork — `/redraw` 커맨드). 렌더러 단일 원인은 086.1과 동일하게 미특정.
> 소속: 086 비주얼 밴드. 선행: [086.1](./086.1_issue_banner_scroll.md) — 동일 계열(스크롤백 아티팩트),
> xterm 헤드리스 재현 7종 무결 → 렌더러 기본 경로는 자가치유 구조임을 이미 확인.
> 입력: 사용자 스크린샷 — 라이브 TUI 프레임 위 스크롤백에 welcome 박스·Flow keys·옛 테이블 행이
> 찢긴 채 누적. "이거 자꾸 위에 TUI 조각이 남는데 해결할 방법 없나?"

VS Code 터미널에서 긴 세션을 돌리면 라이브 프레임 위 스크롤백에 이전 렌더의 조각(배너,
표 일부)이 박제된다. 086.1 조사에서 렌더러의 차등/fullRender/리사이즈 경로는 헤드리스 재현
전부 무결했으므로, 남는 용의자는 실환경 변수들이다: VS Code xterm.js의 리사이즈 reflow(줄
재래핑이 렌더러의 줄 위치 가정과 어긋남), IME 하드웨어 커서 시퀀스의 상대 이동 드리프트(한글
입력 상시), 스크롤백 리밋, suspend/resume 재인쇄 타이밍. 단일 원인 특정에는 라이브 로그
(`PI_DEBUG_REDRAW`)가 필요하다 — 대신 원인과 무관하게 즉시 복구할 수단을 추가했다.

---

## 1. 복구 수단이 이미 코드에 있었다

`tui.ts` `requestRender(force=true)`는 `#previousLines/width/height`를 리셋해 다음 렌더를
**전체 클리어(`2J H 3J` — 화면+스크롤백) + 트랜스크립트 전량 재인쇄**로 강제한다. Ctrl-Z
복귀·외부 에디터 종료가 이미 이 경로로 화면을 수복한다(`interactive-mode.ts:563,1496`).
즉 조각이 생겨도 자가수복 트리거만 있으면 되는데, **사용자가 누를 수 있는 트리거가 없었다**
(ctrl+l은 jwc에서 모델 선택에 배정됨).

## 2. 패치 — `/redraw` 내장 커맨드 (fork)

`slash-commands/builtin-registry.ts`에 TUI 전용 커맨드 추가 (083.4 `/effort` 전례):

```ts
{ name: "redraw", description: "Force a full screen repaint (clears stale TUI fragments)",
  handleTui: (_c, runtime) => { runtime.ctx.editor.setText(""); runtime.ctx.ui.requestRender(true); } }
```

조각이 보이면 `/redraw` 한 번으로 스크롤백까지 싹 지우고 현재 트랜스크립트를 다시 그린다.
멀티플렉서(tmux 등)에서는 기존 fullRender 정책대로 3J를 생략하므로 안전.

검증: `cli-command-surface.test.ts` 10 pass(커맨드 표면 회귀), biome clean.
주의: `3J`는 스크롤백을 지우므로 TUI 밖에서 쌓인 다른 출력도 사라진다 — 의도된 트레이드오프.

## 3. 잔여

- 라이브 재현 시 `PI_DEBUG_REDRAW=1`로 fullRender 사유 로그를 떠서 원인 특정 (086.1 §잔여와 공동).
- 자동화 후보: 리사이즈 폭주(VS Code 패널 드래그) 감지 시 디바운스 후 자동 force render — 보류.
