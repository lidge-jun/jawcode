# 113.1 — REST + SSE 계약: `/api/jwc/*` 어댑터

> 상태: 🟡 v1 — 독립 감사 대기
> 소속: [113 plan_web_native_surfaces](./113_plan_web_native_surfaces.md) 계약화. 착수 전제: 110/111 in-process 부착 완료.
> 범위: cli-jaw 서버에 추가되는 `GET|POST /api/jwc/*` 라우트 + 기존 `GET /api/events` SSE 채널 확장.

---

113이 확정한 원칙 — **복잡한 의미론은 jwc 엔진에, 웹은 뷰만 재사용** — 을 구체적인 와이어 계약으로 문서화한다.
모든 엔드포인트는 읽기(스냅샷)와 쓰기(변경)를 분리하고, SSE는 무효화 신호만 담는다.

## 1. 계약 버전 헤더

```
X-Jwc-Contract: 1
```

- 모든 `/api/jwc/*` 요청/응답에 포함. 서버는 미지원 버전(현재 1 이외)에 `400 {"error":"unsupported_contract"}` 반환.
- 클라이언트는 요청 시 헤더 전송 필수; 미전송 → 서버가 v1로 간주(관대한 처리).

## 2. 스냅샷 리소스 (GET — 읽기 전용)

### 2.1 모델 카탈로그

```
GET /api/jwc/models
```

**응답 스키마 (JSON)**

```jsonc
{
  "rev": 12,                      // 글로벌 단조 카운터 (모델 목록 변경 시 증가)
  "groups": [
    {
      "id": "claude",             // 프로바이더 그룹 식별자 (claude | codex | local)
      "label": "Claude",          // UI 표시명
      "models": [
        {
          "selector": "anthropic/claude-sonnet-4-5",   // formatModelString() 결과: provider/id
          "label": "Claude Sonnet 4.5",                 // model.name
          "provider": "anthropic",                      // model.provider
          "thinkingLevels": ["off","low","medium","high","max"], // ThinkingConfig.levels 또는 minLevel~maxLevel 파생
          "capabilities": {
            "input": ["text", "image"],   // model.input
            "reasoning": true             // model.reasoning
          }
        }
      ]
    },
    {
      "id": "codex",
      "label": "Codex",
      "models": [ /* ... */ ]
    },
    {
      "id": "local",
      "label": "Local",
      "models": [ /* ... */ ]
    }
  ]
}
```

**코드 실사 보정사항**

| 필드 | 원래 스케치 | 실사 결과 | 확정 |
|------|------------|----------|------|
| `selector` | `selector` | `formatModelString(model)` = `"${model.provider}/${model.id}"` | `"provider/id"` 형식 |
| `thinkingLevels` | `thinkingLevels[]` | `ThinkingConfig.levels?: readonly Effort[]` 또는 `minLevel~maxLevel` 범위 파생; 모델에 `thinking` 없으면 `["off"]` | 배열, 항상 포함 |
| `capabilities` | 스케치에 없음 | `model.input: ("text"\|"image")[]`, `model.reasoning: boolean` | 추가 |
| 그룹 매핑 | claude/codex/local | `PROVIDER_DESCRIPTORS`에 그룹 개념 없음 → 어댑터가 provider 문자열 접두사로 분류 (`anthropic*` → claude, `openai-codex*` → codex, `ollama/llama.cpp/lm-studio` → local, 나머지 → 별도 그룹) | 어댑터 책임 |
| `provider` 값 | `"claude"\|"codex"\|"local"` | `model.provider` 실제 값 (`"anthropic"`, `"openai-codex"`, `"ollama"` 등) | `model.provider` 그대로 |

**그룹 분류 규칙 (어댑터)**

```
anthropic, kiro              → id: "claude"
openai-codex                 → id: "codex"
ollama, llama.cpp, lm-studio → id: "local"
그 외                        → id: provider 문자열 그대로 (동적 그룹)
```

### 2.2 세션 상태 스냅샷

```
GET /api/jwc/sessions/:id/state
```

**응답 스키마**

```jsonc
{
  "rev": 7,                           // 세션별 단조 카운터
  "model": {
    "selector": "anthropic/claude-sonnet-4-5",   // 현재 session.model -> formatModelString()
    "thinkingLevel": "high"                       // session.thinkingLevel (ThinkingLevel enum)
  },
  "serviceTier": "priority",          // session.serviceTier (ServiceTier | null)
  "pendingModelSwitch": {             // 턴 중 변경 시 존재, 없으면 키 누락
    "selector": "anthropic/claude-opus-4",
    "thinkingLevel": "max"
  },
  "pabcd": {                          // 활성 pabcd 없으면 null
    "stage": "b",                     // NativePabcdEnvelopeParsed.current_phase
    "active": true,                   // envelope.active
    "gates": []                       // 향후 확장 예약 — 현재 빈 배열
  },
  "usage": {
    "tokens": 42180,                  // ContextUsage.tokens (null이면 키 값 null)
    "contextWindow": 200000,          // ContextUsage.contextWindow
    "contextPct": 21.1                // ContextUsage.percent (null이면 null)
  }
}
```

**코드 실사 보정사항**

| 필드 | 원래 스케치 | 실사 결과 | 확정 |
|------|------------|----------|------|
| `effort` | 별도 필드 스케치 | `thinkingLevel`이 effort와 동의어; `session.thinkingLevel` (ThinkingLevel enum) | `model.thinkingLevel`에 통합 |
| `serviceTier` 타입 | 스케치 언급 | `ServiceTier = "auto"\|"default"\|"flex"\|"scale"\|"priority"\|"openai-only"\|"claude-only"` | 열거형 문자열 그대로 직렬화 |
| `pendingModelSwitch` | 스케치 언급 | `#pendingModelSwitch: {model, thinkingLevel?}` (interactive-mode.ts L337) | 객체 또는 키 누락 |
| `pabcd.gates` | 스케치 언급 | 현재 `NativePabcdEnvelopeParsed`에 gates 필드 없음 — `ctx` 내부에만 존재 | 빈 배열로 예약 |
| `usage.contextPct` | `contextPct` | `ContextUsage.percent` | 필드명 `contextPct`로 어댑터가 매핑 |

### 2.3 설정 스키마

```
GET /api/jwc/settings/schema
```

**응답 스키마**: `settings-schema.ts`의 `SETTINGS_SCHEMA` 객체를 경로 → 정의 맵으로 직렬화.

```jsonc
{
  "rev": 3,
  "settings": {
    "theme.dark": {
      "type": "enum",
      "values": ["default", "minimal", "compact", "full", "nerd", "ascii", "custom"],
      "default": "default",      // default: undefined 인 경우 → 문자열 "default" (열거형 첫 값 또는 명시적 표기)
      "ui": {
        "tab": "appearance",
        "label": "Dark theme",
        "description": "Color scheme for dark mode.",
        "options": [             // UiEnum.options (존재할 경우)
          { "value": "default", "label": "Default" }
        ]
      }
    },
    "defaultThinkingLevel": {
      "type": "enum",
      "values": ["off", "minimal", "low", "medium", "high", "xhigh", "max"],
      "default": "default",      // ← default: undefined → "default" 표기 규칙 (아래 참조)
      "ui": { "tab": "model", "label": "Default reasoning effort", "description": "..." }
    },
    "shellPath": {
      "type": "string",
      "default": "default",      // ← default: undefined → "default"
      "ui": { "tab": "editing", "label": "Shell path", "description": "..." }
    }
  }
}
```

**`default: undefined` → `"default"` 표기 규칙** (99.20.04 핫픽스 확립)

> settings-schema.ts에서 `default: undefined`인 항목은 "값이 설정되지 않은 상태 = 시스템 기본값 사용"을 의미한다.
> REST 직렬화 시 `undefined`를 그대로 노출하면 JSON에서 키가 사라지므로, 어댑터는 이를 문자열 `"default"`로 치환한다.
> 클라이언트는 `"default"` 수신 시 해당 항목을 미설정 상태로 처리한다.

## 3. 변경 엔드포인트 (POST — 쓰기)

### 3.1 엔드포인트 표

| 엔드포인트 | 바디 | 성공 | SSE 도메인 |
|-----------|------|------|-----------|
| `POST /api/jwc/sessions/:id/model` | `{selector, thinkingLevel?}` | `200` 또는 `202` | `model` |
| `POST /api/jwc/sessions/:id/tier` | `{serviceTier}` | `200` | `model` |
| `POST /api/jwc/settings` | `{path, value}` | `200` | `settings` |

### 3.2 세션 모델 변경

```
POST /api/jwc/sessions/:id/model
Content-Type: application/json

{
  "selector": "anthropic/claude-opus-4",   // provider/id 또는 canonical alias
  "thinkingLevel": "high"                  // optional, ThinkingLevel enum
}
```

**응답**

| 상황 | 상태코드 | 바디 |
|------|---------|------|
| 즉시 적용 (턴 없음) | `200` | `{"applied": true, "pending": false, "rev": 8}` |
| 턴 중 — 다음 턴 경계 적용 | `202` | `{"applied": false, "pending": true, "rev": 8}` |
| 모델 미발견 | `404` | `{"error": "model_not_found"}` |
| API 키 없음 | `422` | `{"error": "no_api_key", "provider": "anthropic"}` |

**202 의미론**: 어댑터가 `session.setModel()`을 호출하면 `#pendingModelSwitch`가 설정되고 `flushPendingModelSwitch()`는 다음 턴 시작 시 호출됨 (interactive-mode.ts L1149). 웹은 "다음 턴부터 적용" 배지 1개로 노출.

### 3.3 서비스 티어 변경

```
POST /api/jwc/sessions/:id/tier
Content-Type: application/json

{ "serviceTier": "priority" }
```

**응답**: `200 {"applied": true, "rev": 9}` — `ServiceTier` 열거형 외 값은 `422 {"error": "invalid_tier"}`.

### 3.4 설정 변경

```
POST /api/jwc/settings
Content-Type: application/json

{ "path": "defaultThinkingLevel", "value": "high" }
```

**유효성 검사**: 스키마(`settings-schema.ts`) 기준.
- `type: "enum"` → `values[]` 내 값만 허용.
- `type: "boolean"` → `true | false` 만 허용.
- `type: "string"`, `"number"`, `"array"`, `"record"` → 타입 일치.
- `"default"` 값은 특수값 — 해당 경로를 `undefined`(미설정)으로 리셋.

**응답**

| 상황 | 상태코드 | 바디 |
|------|---------|------|
| 성공 | `200` | `{"applied": true, "rev": 4}` |
| 경로 없음 | `404` | `{"error": "unknown_path"}` |
| 값 타입 오류 | `422` | `{"error": "invalid_value", "path": "...", "allowed": [...]}` |

## 5. SSE 무효화 계약

### 5.1 이벤트 형식

기존 `GET /api/events` 채널을 재사용한다. cli-jaw의 `BusEvent` 형식에 따라 **`event:` 필드 없이** 페이로드만 JSON으로 전송된다.

```
id: 42
data: {"topic":"jwc","event":"jwc-invalidate","domain":"model","session":"sess-abc","rev":8}

id: 43
data: {"topic":"jwc","event":"jwc-invalidate","domain":"settings","rev":4}
```

**필드 정의**

| 필드 | 타입 | 설명 |
|------|------|------|
| `topic` | `"jwc"` | cli-jaw EventTopic 확장 (신규 추가) |
| `event` | `"jwc-invalidate"` | 고정값 |
| `domain` | `"model"\|"settings"\|"quota"\|"pabcd"\|"usage"` | 무효화 대상 도메인 |
| `session` | `string?` | 세션 도메인이면 세션 ID, 글로벌이면 키 누락 |
| `rev` | `number` | 무효화 이후 예상 스냅샷 rev |

**도메인 → 재조회 매핑**

| `domain` | 재조회 엔드포인트 |
|---------|----------------|
| `model` | `GET /api/jwc/sessions/:id/state` |
| `settings` | `GET /api/jwc/settings/values` |
| `quota` | `GET /api/jwc/quota` |
| `pabcd` | `GET /api/jwc/sessions/:id/state` |
| `usage` | `GET /api/jwc/sessions/:id/state` |

### 5.2 rev 발급 단위

- **세션별 도메인** (`model`, `pabcd`, `usage`): 세션별 단조 정수. 세션 A와 B의 rev는 독립.
- **글로벌 도메인** (`settings`, `quota`, `models`): 서버 전역 단조 정수.
- rev는 서버 재시작 시 0 초기화 가능. 클라이언트는 항상 스냅샷 rev와 비교하므로 단조성만 보장되면 된다.

### 5.3 재연결 핸드셰이크

> **프런트 머지 0 원칙**: 프런트는 델타를 받지 않는다. 무효화 신호를 받으면 해당 도메인의 스냅샷 전체를 다시 GET한다.

재연결 절차:

1. `EventSource`가 재연결되면 `Last-Event-ID`는 cli-jaw의 `replaySince()` 메커니즘이 처리한다.
2. 재연결 후 클라이언트는 **보유 중인 rev와 스냅샷 rev를 비교**한다 — 스냅샷 rev가 크면 즉시 렌더 교체.
3. `jwc-invalidate` 재전송은 멱등이다 — 같은 rev의 무효화 신호를 복수 수신해도 스냅샷 재조회는 동일 결과를 반환하므로 무해.
4. 서버 재시작으로 rev가 초기화된 경우: 클라이언트 보유 rev > 스냅샷 rev이면 전체 갱신.

**Last-Event-ID 사용 방침**: `/api/events`가 이미 `Last-Event-ID` 기반 재전송을 지원한다. `jwc-invalidate` 이벤트가 재전송되더라도 프런트가 멱등 재조회를 하기 때문에 무해. `replay_gap` 신호 수신 시 프런트는 모든 jwc 도메인 전체 재조회.

### 5.4 미지원 `domain` 무시 규칙

클라이언트는 알 수 없는 `domain` 값을 수신하면 무시한다 (forward compatibility). 서버는 새 도메인을 추가할 때 클라이언트 버전 확인 없이 발행 가능.

## 6. 모달 상호배제 규약

**서버는 409를 반환하지 않는다.**

동시 편집 충돌은 UI 레이어가 구조적으로 방지한다:

- 웹 팝업이 열린 상태에서는 TUI 입력이 불가 (모달 오버레이).
- TUI 오버레이가 활성 상태에서는 반대 표면 팝업이 불가.
- 서버의 역할: **last-write 백스톱** — 두 클라이언트가 동시에 쓰더라도 마지막 쓰기가 적용되고 SSE로 양쪽에 무효화 신호 전송.

설계 근거: 113 확정 결정 4 "충돌은 UI 모달 상호배제가 구조적으로 차단" — 서버 레벨 잠금은 비용 대비 효용 없음.

## 8. 열린 질문 (113 승계)

1. **Workers 탭 통합 순서** — 기존 cli-jaw `/model` 팝업에 Boss 탭을 "추가"하는 것이 최소 경로인지, 아니면 `/api/jwc/models` 응답에 Workers 모델도 포함해야 하는지.
2. **rev 발급 세분화** — 현재 설계는 세션별 단조이지만, 세션 내 도메인별(model vs pabcd vs usage)로 rev를 분리할지 단일 세션 rev로 통합할지 미결.
3. **설정 `requiresRestart` 메타** — `settings-schema.ts`에 `requiresRestart?: boolean` 추가 여부. 웹에서 바꾼 설정 중 TUI 재시작이 필요한 항목 구분 표시 필요 (현재 스키마에 없음).
