## 0b. 실소스 검증 결과 — 시퀀스 교정 [중요]

1. **스크롤백 캡처 조건**: primary screen + **스크롤 리전 top=row 1** (`CSI 1;N r`)일 때만 리전 상단 이탈
   라인이 스크롤백 진입. top>1 부분 리전은 전 터미널에서 **조용히 폐기**. 따라서 히스토리 리전은 항상 1행부터.
2. **커밋 메커니즘**: 커서를 히스토리 리전 **바닥**(뷰포트 바로 윗행)에 두고 라인마다 `\r\n` →
   리전이 위로 스크롤되며 top 라인이 스크롤백으로 이탈 → 바닥 빈 행에 새 라인 write. **RI(`\x1bM`)가 아님**.
3. **RI의 실제 용도**: Phase 1 — 뷰포트가 화면 바닥에 붙지 않았을 때 `CSI {vpTop+1};{rows} r` 리전에서
   RI×N으로 **뷰포트 자체를 아래로 밀기**. 뷰포트가 바닥이면 Phase 1 생략.
4. **DECSTBM은 무조건 커서를 1;1로 홈** — 커서 스냅샷은 첫 DECSTBM 이전에 떠야 하고, 복원은 ResetScrollRegion
   (`\x1b[r`) 이후 plain MoveTo로 (Codex는 DECSC/DECRC 대신 last_known_cursor_pos 스냅샷+MoveTo).
5. **능력 감지**: Codex는 프로브 없이 env 기반 분기 (TERM_PROGRAM 등) + **Zellij만 raw-append 특수 모드**
   (ZELLIJ env — 부분 리전 soft-wrap 미준수). tmux는 표준 경로(tmux 자체 히스토리에 적재됨 — 의도 동작).
   iTerm2는 "Save lines to scrollback when an app status bar is present" 설정(기본 ON) 의존.
6. **pre-wrap 보강**: URL 단독 라인은 wrap 생략(클릭 가능 보존), 연속행 hanging indent 수동 구현 필요,
   `hard: true` + 후행 공백 제거.

## 3. 구현 스케치

1. **`tui.ts`에 커밋 프리미티브**: `insertHistoryLines(lines: string[])` —
   ① 커서 저장 ② `CSI {top};{bottom} r`로 스크롤 리전을 뷰포트 위로 제한 ③ 리전 상단에서 `\x1bM`×N으로
   공간 확보 후 라인 write ④ 리전 해제 + 커서 복원. 동기화 출력(`?2026`)으로 감싼다.
2. **Container 경계**: chatContainer를 `committedChat`(가상 버퍼에서 제외)과 `liveChat`으로 분리하지 않고,
   **컴포넌트에 `committed` 플래그** — frame 조립 시 committed는 스킵, 커밋 순간에만 insertHistoryLines로 1회 방출.
   순서 보장: 커밋은 항상 프레임 최상단부터 연속 prefix만 가능(중간 커밋 금지) — Codex와 동일한 단조성.
3. **랩 정책**: 커밋 라인은 write 시점 폭으로 pre-wrap(`Bun.wrapAnsi`) — 리사이즈 시 리플로우 포기(Codex PreWrap과 동일).
   이미지(Kitty/iTerm 프로토콜) 셀은 커밋 불가 → 라이브 존 체류 또는 플레이스홀더 커밋 후 오버레이 열람.
4. **B2-lite 상호작용**: 커밋 레인에서 프레임 = 라이브 존(상시 ≤ viewport 근방) → 오버플로 케이스 소멸,
   sticky gap(§11/§12) 로직은 virtual 레인 전용으로 격리. ViewportFill은 라이브 존 바닥 핀 용도로 유지.
5. **인터랙티비티 재배선** (083.8 §6c): ctrl+o/t·포커스 모드(ctrl+shift+up)는 비커밋(라이브 존) 컴포넌트만 순회.
   과거 셀은 alt+t 트랜스크립트 오버레이. `/resume` replay는 전량 커밋 형태로 방출(폴딩 최종형) — 현행과 동일 체감.

## 3b. P2 구현 설계 [확정 260613 — diff 렌더러와의 통합 핵심]

**문제**: diff 렌더러는 프레임을 row 0부터 그린다. B2-lite는 fill 블랭크를 프레임 *안에* 넣어 바닥 핀을
구현하므로 프레임 길이 = 화면 높이 → liveZoneTop이 항상 0이 되어 커밋 불가. 또한 커밋된 픽셀이 화면
위쪽에 남아 있는 동안 fill이 수축하면 diff가 그 행을 덮어쓴다(콘텐츠 손실).

**해법**: **fill 영역 = 히스토리 리전**으로 재해석.

1. `liveZoneTop` = 프레임 내 fill 행 수 (= 화면높이 − 라이브 콘텐츠 행 수). 커밋은 `CSI 1;liveZoneTop r`
   리전에서 수행 — 라이브 존(fill 아래 전부)은 불가침.
2. **`#committedScreenRows` 불변식**: 라이브 존 바로 위에 쌓인 "아직 스크롤백 미진입" 커밋 픽셀 행 수.
   diff 입장에서 그 행들은 fill 블랭크("")라서 prev==new → 절대 재페인트 안 됨 (보호가 공짜).
3. **성장 시 스크롤-아웃 규칙**: 라이브 콘텐츠가 자라 fill이 줄면(liveZoneTop′ < committedScreenRows),
   페인트 전에 히스토리 리전을 `\r\n`×(committedScreenRows − liveZoneTop′)로 먼저 밀어 커밋 픽셀을
   위로 보내고(최상단 행은 스크롤백 진입) 빈 행을 확보한다.
4. **커밋 시점 = 셀 확정 시점** (스크롤아웃 대기 아님): 유저 메시지 add 시, 도구 폴딩 확정 시, 세그먼트
   컷 시 즉시 `tui.commitLines(렌더 결과)` — 프레임(chatContainer)에는 아예 안 들어감. 프레임은 상시
   [fill][활성 세그먼트+라이브 도구][컴포저]로 소형 유지.
5. **폴백**: liveZoneTop=0(활성 콘텐츠가 화면 초과 — 긴 thinking 테일)이면 commitLines가 false 반환 →
   호출자는 virtual 레인 방식(chatContainer add)으로 폴백. P3 점진 커밋이 이 케이스를 해소.
6. 커밋 라인은 프레임 파이프라인과 동일한 전처리(pre-wrap/truncate/LINE_TERMINATOR) 후 방출 — 스크롤백
   픽셀이 diff 경로 출력과 동일 보장.
