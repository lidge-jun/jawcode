# 081.11 — issue: composer bash 대량 작업이 타임아웃 루프에 빠짐 (har_struct 치환 사례)

> 상태: ✅ 옵션 A+B 구현 (2026-06-12, fork — §4). C는 A의 discipline 조항에 포함.
> 소속: 081 밴드 (composer 도구 규율). 관련: [081.8](./081.8_issue_composer_anchor_fabrication.md)
> (discipline 프롬프트), [081.9](./081.9_issue_composer_autocontinue_no_stop.md) (과잉 재검증 습성).
> 입력: 사용자 — "직접 작업하라고, grep만 주구장창 하지 말고. composer 모델에서 bash 오류 또 계속 떠."

composer로 `har_struct → struct_har` 일괄 리네임을 시키자 bash가 3연속으로
타임아웃(120s/180s/120s 취소)을 맞으며 같은 검색을 변형 반복했다. 관찰된 패턴은 단순
"bash 버그"가 아니라 **모델의 셸 작성 습성 × bash 도구 타임아웃의 합작 루프**다.

---

## 1. 증거 (사용자 세션 페이스트, 시간순)

| # | 명령 | 결과 | 문제 |
|---|---|---|---|
| 1 | `rg -l 'har_struct' … \| while read f; do perl -pi …; done; rg -c … \| head -5 \|\| echo "no har_struct left"` | "no har_struct left" + ⟦Timeout: 120s⟧ | **`\|\| echo` 폴백이 타임아웃 kill을 "치환 완료"로 둔갑** — rg가 매치 없음(exit 1)이든 도중 사살이든 같은 메시지. 모델은 완료로 믿음 |
| 2 | `find . -type f \( -name '*.md' … \) ! -path './node_modules/*' -print0 \| xargs -0 grep -l` | 100+ 파일에 har_struct 잔존 발견, exit 1 | #1이 사실 미완료였음이 드러남. `! -path`는 **출력 필터일 뿐 디렉토리 하강을 안 막음**(-prune 아님) → node_modules 전체 순회 |
| 3 | `find … -exec grep -l 'har_struct' {} \;` 후 while-perl | ⟦Timeout: 180s⟧ | `-exec \;` = **파일당 grep 프로세스 1개** — 모노레포에서 수만 프로세스 |
| 4 | `for d in …; do find \| while read; do grep -q && perl; done; done` | ⟦Timeout: 120s⟧ 취소 | 같은 일을 더 느린 3중 루프로 재시도 — 파일당 grep+perl 2프로세스 |

병행 습성: 본작업(치환) 전후로 read/grep 검증만 반복("grep만 주구장창") — 081.9에서 본
과잉 재검증과 동일 계열.

## 2. 메커니즘 정리

1. **타임아웃 식별 실패**: bash 도구가 120s(기본)에 명령을 자르면 모델은 부분 출력만 본다.
   `\|\| echo` 같은 폴백과 결합하면 실패가 성공 메시지로 변조된다 — 이후 추론 전체가 오염.
2. **비효율 셸 패턴 3종**: ① `! -path`로 prune 흉내(하강 차단 안 됨) ② `-exec cmd \;`
   파일당 포크 ③ 파이프 while-loop 한 줄 처리. 모노레포(node_modules 수십만 파일)에서는
   전부 타임아웃 직행.
3. **루프 강화**: 실패하면 같은 접근의 *더 느린* 변형으로 재시도 — 빠른 도구(rg)로 돌아가지
   않고 find 계열로 하강.

참고: #519(cursor ms→s 단위)와는 무관 — 이 세션은 xai 직결 + jwc 네이티브 bash이며 타임아웃
동작 자체는 정상(120s 기본). 문제는 모델이 그 한도를 **모르고 설계**한다는 것.

## 3. 대응 옵션 (미적용)

| # | 옵션 | 내용 |
|---|---|---|
| A | **discipline 프롬프트 보강** (081.8 `COMPOSER_EDIT_DISCIPLINE_PROMPT` 또는 별도 블록) | "셸 명령은 기본 120s 안에 끝나야 한다. 대량 검색은 rg(자동 ignore), find는 `-path … -prune -o`로 하강 차단, `-exec \;` 대신 `+`/xargs. 타임아웃·취소된 명령의 출력은 결과로 믿지 말 것. 같은 명령의 느린 변형 재시도 금지 — 범위를 쪼개라." |

## 4. 구현 (2026-06-12, fork)

- **B (호스트)**: `coding-agent/src/tools/bash.ts` — 타임아웃 시 모델용 에러가 부분
  출력만 담던 두 경로(`#buildResultText`, interactive 경로)를 `formatTimedOutResult`로
  통일: 항상 "Command KILLED after Ns timeout — did NOT complete / output PARTIAL /
  narrower scope or higher timeout" 공지를 선두에 박는다. **이게 본 사건의 직접 원인
  픽스** — 기존 코드는 부분 출력이 있으면 타임아웃 표시 없이 그대로 전달했다.
- **A (composer 한정, C 포함)**: `composer-discipline.ts` 프롬프트에 2조항 추가 —
  ① 타임아웃 kill 출력은 부분이며 폴백 메시지를 완료로 믿지 말 것, 재시도는 느린 변형이
  아니라 좁힌 범위로 ② rg 우선, find는 `-prune`, `-exec {} +`/xargs, bulk 치환 정석
  `rg -l | xargs perl -pi -e` 한 줄.
