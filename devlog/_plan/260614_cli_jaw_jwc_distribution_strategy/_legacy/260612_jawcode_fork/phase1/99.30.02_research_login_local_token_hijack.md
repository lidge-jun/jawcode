# 99.30.02 — `/login` 로컬 토큰 가로채기 제거 — 저장 자격증명 유무 기반 if 분기

> 상태: 조사 ✅ / 설계 ✅ (260613 사용자 2차 확정: `local` 서브커맨드 없이 if 분기) /
> **구현 ✅ (260613)** — ai 스위트 1265 pass·기존 실패 oauth-xai 1건 부수 해소·tsc 0.
> 소속: [99.30.00 MOC](./99.30.00_moc_feature_improvement.md). 입력: 사용자 260613 "온보딩
> plain 상태에서는 로컬 토큰 자동 인식이 맞지만, `/login xai`를 명시했을 땐 원래 OAuth
> 로직을 따라야 한다." → 2차: "저장된 게 있다 → OAuth(로컬 ❌) / 없다 → 로컬 시도 /
> 로컬도 없다 → OAuth."

`/login <provider>`는 사용자가 **명시적으로 새 OAuth 인증을 요청**하는 행위인데, 현재
anthropic·openai-codex·xai 세 provider의 login 함수가 모두 **로컬 CLI 토큰 감지를 무조건
선행**해서, 로컬 토큰이 존재하는 한 진짜 OAuth 플로우에 도달할 수 없다. 반대로 이 자동
인식이 정작 어울리는 자리인 **온보딩(plain 상태)에는 아무 배선이 없다** — 기능이 자리를
잘못 잡은 것. 해법은 명시 로그인에서 가로채기를 제거하고, 임포트는 `/login <provider>
local` 명시 커맨드(+추후 온보딩 자동 인식)로 옮기는 것이다.

## 1. 현황 — 가로채기 구조 (3 provider 공통 패턴)

```ts
// utils/oauth/xai.ts:191-206 (anthropic.ts:170-185 · openai-codex.ts:160-179 동일 골격)
export async function loginXai(ctrl) {
    const local = detectGrokCliToken();
    if (local) { …import/refresh 후 return; }   // ← 진짜 OAuth 도달 불가
    return new XaiOAuthFlow(ctrl).login();
}
```

| provider | login fn | 로컬 소스 (`local-token-detect.ts`) | 만료 시 |
|----------|----------|--------------------------------------|---------|
| anthropic | `anthropic.ts:170` | macOS 키체인 `Claude Code-credentials` (`:12,48-68`) / linux secret-tool | refresh 시도, 실패 시 fallthrough |
| openai-codex | `openai-codex.ts:160` | `~/.codex/auth.json` (`:19-46`) | 동일 |
| xai | `xai.ts:191` | `~/.grok/auth.json`의 `https://auth.x.ai::` 키 (`:89-113`) | 동일 |

진입 체인: `/login xai` (`builtin-registry.ts:922-942`) → `showOAuthSelector("login", id)` →
`#handleOAuthLogin` (`selector-controller.ts:1146`) → `authStorage.login(provider, ctrl)`
(`auth-storage.ts:1341`) → 위 login fn. **어느 단계에도 임포트를 끌 옵션이 없다.**
coding-agent 측(온보딩 포함)에서 `detect*` 직접 호출은 **0건** — 자동 인식은 오직 명시
로그인 내부에만 산다.

## 2. UX 문제 정의

1. **명시 의도 위반**: 사용자가 재로그인을 원하는 전형적 이유(토큰 깨짐·계정 전환·스코프
   갱신)에서 전부 실패한다. `/logout xai` 후 `/login xai` 해도 **같은 로컬 토큰이 다시
   수입되는 루프**.
2. **침묵 가로채기**: `onProgress("Found Grok CLI token, importing automatically")` 한 줄이
   유일한 고지 — 099.20.07 감사에서 본 채팅 오염 라인이라 눈에 안 띈다. 사용자는 브라우저가
   열리길 기대하는데 아무 일 없이 끝난다.
3. **외부 CLI와 계정 결합**: grok-cli에서 계정을 바꾸면 jwc도 조용히 따라간다. 본 세션
   사례: jwc의 xai 자격증명(account `73861c41…`)이 `~/.grok` 토큰과 동일 계정 — 임포트
   유래로 추정.
4. **온보딩 공백**: 정작 plain 상태(자격증명 0)에서는 로컬 토큰이 있어도 아무 제안이 없다.

## 3. 해법 (260613 사용자 2차 확정 — `local` 서브커맨드 폐기, 상태 기반 if 분기)

```
/login <provider>:
  저장된 자격증명 있음   → 진짜 OAuth (로컬 무시)    ← 재로그인 = 계정전환/토큰갱신 의도
  저장된 것 없음        → 로컬 토큰 시도             ← 첫 로그인 = 온보딩 흐름 자체
  로컬 없음/refresh 실패 → 진짜 OAuth                ← silent 폴백 (첫 로그인이므로 자연스러움)
```

1차안(`/login <provider> local` 서브커맨드 단독)과의 차이: 기본 동작은 **커맨드 표면 추가
없이** 상태로 의도를 판별하고 — 자격증명이 이미 있는 사용자가 `/login`을 다시 치는 것
자체가 "새로 인증하겠다"는 신호, 빈 상태는 온보딩이므로 자동 인식이 기대 동작 — §2-4의
온보딩 공백도 첫 `/login`이 해소한다. 여기에 3차 확정(260613)으로 **`/login <provider>
local` 명시 플래그를 병행 추가**: 자격증명이 있어도 로컬 토큰을 강제 재임포트하고 싶을 때
(예: grok-cli에서 계정을 바꾼 직후 동기화) 쓰는 escape hatch다.

내부 모드는 3단 (`utils/oauth/types.ts` `LocalTokenImportMode`):

| 모드 | 트리거 | 동작 |
|------|--------|------|
| `off` | 저장된 자격증명 있음 (기본) | 진짜 OAuth 직행, 로컬 무시 |
| `fallback` | 저장된 것 없음 (첫 로그인) | 로컬 임포트 시도 → 없거나 refresh 실패 시 OAuth |
| `only` | `/login <provider> local` 명시 | 로컬만 — 없거나 refresh 실패 시 **명시 에러** (silent OAuth 전환 금지) |

판정은 `authStorage.login()` 최상단 단일 지점 — 각 case의 `remove(provider)`가 증거를
지우기 **전에**: `opts?.importLocal ? "only" : this.has(provider) ? "off" : "fallback"`.
