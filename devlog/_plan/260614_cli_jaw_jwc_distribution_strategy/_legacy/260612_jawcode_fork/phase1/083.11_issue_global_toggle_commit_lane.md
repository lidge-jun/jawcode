# 083.11 — issue: Ctrl+O / Ctrl+T 전역 접기 토글이 커밋 레인 과거 스크롤백을 닫지 못함

> 상태: 🔴 사용자 실기기 버그 기록 (260614)
> 보고: jwc TUI에서 `ctrl+o`로 전체가 열린 뒤 `ctrl+o`로 다시 닫으면, 직전 사용자 입력 이후 현재 턴까지만 닫히고 위쪽 과거 출력은 열린 채 유지된다. `ctrl+t` thinking 토글도 동일 증상.
> 영향: commit render lane 기본 ON 세션의 장기 대화. 사용자는 전역 토글이라고 인식하지만 실제로는 “라이브 프레임 + 미커밋 현재 턴”에만 가역적으로 적용된다.

## 관찰된 증상

1. 긴 세션에서 `ctrl+o`를 눌러 도구/thinking 출력이 전체 펼쳐진 상태가 된다.
2. 다시 `ctrl+o`를 누르면 최근 구간(대략 “이전 내 대화 보냄” 이후 현재 대화/턴)만 접힌다.
3. 그보다 위쪽의 과거 출력은 펼쳐진 픽셀 상태 그대로 남는다.
4. `ctrl+t`도 thinking 블록에 대해 같은 비대칭을 보인다.

## 코드 조사

핫키 배선:

- `packages/coding-agent/src/config/keybindings.ts`
  - `app.tools.expand` = `ctrl+o`
  - `app.thinking.toggle` = `ctrl+t`
- `packages/coding-agent/src/modes/controllers/input-controller.ts`
  - `toggleToolOutputExpansion()`:
    - `const expanded = !this.ctx.toolOutputExpanded`
    - `this.ctx.thinkingExpanded = expanded`
    - `setToolsExpanded(expanded)` 호출
  - `setToolsExpanded()`:
    - `chatContainer.children`와 `liveToolContainer.children`의 `setExpanded(expanded)`만 호출
  - `toggleThinkingBlockVisibility()`:
    - `chatContainer.children` 중 `AssistantMessageComponent`만 `setThinkingExpanded(expanded)` 호출

커밋 레인 동작:

- `packages/coding-agent/src/modes/utils/ui-helpers.ts:67` `commitFinalizedBacklog()`
  - 다음 prompt submit 직전에 완료된 `chatContainer.children` prefix를 `ctx.ui.commitLines(lines)`로 터미널 scrollback에 직접 기록한다.
  - 성공하면 `child.committed = true`로 표시한다.
- `packages/tui/src/tui.ts:229` `Container.render()`
  - `child.committed`면 render tree에서 skip한다.
  - 주석 그대로 “committed pixels live in the scrollback — the frame must not carry a second copy.”
- `packages/tui/src/tui.ts:1178` `commitLines()`
  - 커밋된 라인은 diff-rendered frame이 아니라 terminal scrollback의 immutable pixels가 된다.

따라서 `ctrl+o` / `ctrl+t`는 객체 상태(`setExpanded`, `setThinkingExpanded`)는 바꾸지만, 이미 `commitLines()`로 scrollback에 찍힌 과거 픽셀은 다시 그릴 수 없다. 그 결과 현재 라이브 프레임에 남은 미커밋 구간만 닫히고, 과거 커밋 구간은 열린 채 유지된다.

## 원인 요약

083.9/083.10 commit render lane의 핵심 불변식은 “과거 transcript는 scrollback에 단조 append하고 다시 지우지 않는다”이다. 그런데 083.5의 `ctrl+o`/`ctrl+t` UX는 “전역 출력 상태가 가역적”이라는 모델을 유지한다. 두 모델이 충돌한다.

특히 사용자가 펼친 상태에서 턴 경계를 넘으면, 펼쳐진 렌더 결과가 커밋 픽셀로 굳는다. 이후 접기 토글은 미커밋 객체에는 적용되지만 커밋 픽셀에는 적용 불가라서 전역 토글이 부분 토글처럼 보인다.

## 재현 시나리오

1. `JWC_COMMIT_LANE` 기본 ON 환경에서 jwc TUI 실행.
2. 도구 출력 또는 thinking이 있는 턴을 만든다.
3. `ctrl+o` 또는 `ctrl+t`로 펼친다.
4. 새 사용자 메시지를 제출해 `commitFinalizedBacklog()`가 직전 턴을 scrollback에 커밋하게 한다.
5. 다시 `ctrl+o` 또는 `ctrl+t`로 접는다.
6. 기대: 전체 transcript가 접힌 상태로 일관됨.
7. 실제: 새 prompt 이후 미커밋/라이브 구간만 접히고, 커밋된 위쪽 과거 scrollback은 펼친 픽셀 그대로 남음.

## 수정 방향 후보

### A. 커밋 전 항상 canonical collapsed 상태로 강제

턴 경계에서 `commitFinalizedBacklog()`가 픽셀을 찍기 전에 커밋 대상 prefix를 canonical collapsed 상태로 렌더한다.

- 장점: commit lane의 scrollback 불변식 유지.
- 장점: 과거 scrollback은 항상 압축된 transcript라 장기 세션 밀도가 안정적.
- 단점: 사용자가 펼쳐 놓은 상태를 그대로 scrollback에 보존하지 않는다.
- 구현 포인트:
  - `commitFinalizedBacklog()`가 `child.render(width)` 전에 tool/thinking/todo 계열을 접힌 상태로 렌더하도록 일시 상태 전환 또는 별도 `renderCommitted(width)` 프로토콜 도입.
  - `ToolExecutionComponent` minimized/expanded 상호작용과 `AssistantMessageComponent` thinking collapsed summary를 별도 검증.

### B. 커밋 레인 사용 중 전역 토글 문구/동작을 “라이브 구간 토글”로 축소

커밋된 scrollback은 바꿀 수 없음을 UX로 명확히 하고, `ctrl+o`/`ctrl+t`가 미커밋 구간에만 적용되도록 상태/힌트/문서를 맞춘다.

- 장점: 가장 작고 안전한 변경.
- 단점: 사용자가 기대한 “전체 닫기” 문제는 해결하지 못한다.
- 구현 포인트:
  - status: `Tool output: collapsed (current turn; committed scrollback unchanged)`류 힌트.
  - hotkeys 문서와 devlog에 commit lane 제한 명시.

### C. 전역 토글 시 full rebuild / scrollback clear로 과거 픽셀 재작성

`ctrl+o`/`ctrl+t` 시 세션 전체를 새 접힘 상태로 렌더하고 scrollback을 지운 뒤 다시 쓴다.

- 장점: 사용자 기대와 가장 일치.
- 단점: 083.9 commit lane의 핵심 목표(3J 회피, scrollback 보존, 장기 세션 무깜빡임)를 깨뜨린다.
- 단점: tmux/zellij/native scrollback에서 사용자가 올려 본 히스토리를 날릴 수 있다.
- 판정: 기본안으로 부적합. explicit “reflow transcript” 같은 별도 명령이면 가능.

## 비교 조사: Codex RS / Claude Code

### Codex RS (`~/Developer/codex/openai-codex/codex-rs`)

Codex RS는 jwc와 비슷하게 finalized history를 터미널 scrollback에 직접 쓴다. 하지만 전역 “접기/펼치기”를 같은 표면에 얹지 않는다.

확인한 코드:

- `codex-rs/tui/src/app/event_dispatch.rs`
  - `AppEvent::InsertHistoryCell`에서 `transcript_cells.push(cell.clone())` 후 `insert_history_cell_lines(...)`로 scrollback에 쓴다.
- `codex-rs/tui/src/app/resize_reflow.rs`
  - 모듈 주석: app은 `HistoryCell`을 source로 저장하지만 normal chat view에는 finalized history를 terminal scrollback에 쓴다.
  - resize 시 `transcript_cells` source로 `clear_scrollback_and_visible_screen_ansi()` 후 재방출한다.
  - `reflow_transcript_now()`도 같은 source-backed rebuild 경로다.
- `codex-rs/tui/src/app/input.rs`
  - `ctrl+t`는 `Overlay::new_transcript(self.transcript_cells.clone(), ...)`로 alternate-screen transcript overlay를 연다.
  - `alt+r` `toggle_raw_output`은 `apply_raw_output_mode()` → `reflow_transcript_now()`로 전체 scrollback을 source에서 재작성한다.
- `codex-rs/tui/src/keymap.rs`
  - default `open_transcript = ctrl-t`
  - default `toggle_raw_output = alt-r`
  - snapshot상 `ctrl-o`는 “Copy the last agent response to the clipboard”다.

판정:

- Codex RS의 normal chat scrollback은 커밋 픽셀이라 동일하게 사후 부분 접기에는 부적합하다.
- 대신 `ctrl+t`는 “접기 토글”이 아니라 전체 transcript overlay다.
- 전체 표시 모드 변경이 필요한 경우(`alt+r` raw/rich)는 source-backed full reflow로 scrollback을 clear + rewrite한다. 이건 resize/raw-output 같은 명시적 재방출 작업이지 매번 접기 토글 UX가 아니다.
- jwc의 `ctrl+o`를 “전체 transcript 접기/펼치기”로 유지하려면 Codex식 source-backed reflow가 필요하지만, 그건 083.9 commit lane의 무깜빡임/scrollback 보존 목표와 충돌한다. Codex도 이 비용을 raw-output/resize 같은 특수 이벤트로 제한한다.

### Claude Code (`~/Developer/codex/claude_code/orideop_src`)

Claude Code는 `ctrl+o`를 normal scrollback 픽셀 토글로 쓰지 않는다. `ctrl+o`는 prompt screen ↔ transcript screen 전환이고, transcript screen 내부는 React/Ink virtual list가 source messages를 다시 렌더한다.

확인한 코드:

- `src/keybindings/defaultBindings.ts`
  - `ctrl+t`: `app:toggleTodos`
  - `ctrl+o`: `app:toggleTranscript`
- `src/hooks/useGlobalKeybindings.tsx`
  - `app:toggleTranscript` 핸들러가 `setScreen(s => s === 'transcript' ? 'prompt' : 'transcript')`를 수행한다.
  - transcript 진입/종료 시 `setShowAllInTranscript(false)`로 transcript 전용 상태를 리셋한다.
  - transcript 전용 `ctrl+e`는 `showAllInTranscript`만 토글한다.
- `src/components/Messages.tsx`
  - `isTranscriptMode = screen === 'transcript'`
  - transcript mode는 “truly unfiltered” 주석이 붙어 있고 prompt mode의 brief/drop 필터를 우회한다.
  - fullscreen에서는 `expandedKeys`로 message별 verbose toggle을 관리한다. `onItemClick`이 key를 Set에 넣고 빼며, `MessageRow`에는 `verbose || isItemExpanded(...)`가 전달된다.
  - virtual scroll active 시 `VirtualMessageList`가 전체 transcript를 source messages 기반으로 렌더한다.
- `src/components/VirtualMessageList.tsx`
  - props에 `onItemClick`, `isItemExpanded`가 있고, 주석도 “click a message to toggle verbose rendering for it”라고 설명한다.
  - scroll/search/height cache가 source messages 기반으로 동작한다.
- `src/components/messages/AssistantThinkingMessage.tsx`
  - `const shouldShowFullThinking = isTranscriptMode || verbose`
  - prompt mode에서는 thinking 요약, transcript mode 또는 per-message verbose에서는 전문 표시.
- `src/components/CtrlOToExpand.tsx`
  - `CtrlOToExpand` 힌트는 `app:toggleTranscript`/`ctrl+o`를 “to expand”로 보여주지만, virtual list 안에서는 숨긴다. 즉 prompt screen의 “전체 transcript로 가라”는 안내지 scrollback 픽셀을 접고 펴는 조작이 아니다.

판정:

- Claude Code의 `ctrl+o`는 jwc식 인라인 전역 토글이 아니라 별도 transcript screen 진입/종료다.
- transcript screen에서는 source messages를 다시 렌더하므로 닫으면 prompt screen으로 돌아갈 뿐, 과거 scrollback 픽셀을 펼친 상태로 굳히는 문제가 없다.
- message별 펼침은 virtual list 내부 상태(`expandedKeys`)라 해당 screen 안에서만 의미가 있고, normal prompt scrollback을 재작성하지 않는다.

## 비교 결론

Codex RS와 Claude Code 둘 다 “이미 터미널 scrollback에 박힌 과거 픽셀을 일반 접기 토글로 계속 가역 조작한다”는 UX를 기본으로 두지 않는다.

- Codex RS: scrollback은 finalized history append. 전문은 `ctrl+t` overlay. 전체 재방출은 raw-output/resize 같은 특수 reflow에서만 수행.
- Claude Code: `ctrl+o`는 transcript screen 전환. source messages 기반 virtual list에서 다시 렌더하고, prompt screen scrollback 자체를 접고 펴지 않는다.

따라서 jwc는 둘 중 하나로 정렬해야 한다.

1. **Codex식**: normal scrollback은 canonical collapsed로 커밋하고, full transcript는 `alt+t` overlay/별도 screen에서 본다.
2. **Claude식**: `ctrl+o`를 inline 전역 토글이 아니라 transcript screen/overlay 진입으로 바꾸고, prompt screen은 collapsed canonical 유지한다.

현재 jwc 구조에서는 1번이 더 작은 변경이다. `ctrl+o`/`ctrl+t`는 “현재 미커밋/live 구간 토글”로 축소하고, commit 시 과거는 항상 collapsed canonical로 고정하는 쪽이 두 비교 대상과 가장 잘 맞는다.
사용자 후속 결정(260614): Codex식 1번으로 진행한다. `ctrl+t` thinking 매핑은 제거하고, `ctrl+o`가 현재 미커밋/live tool+thinking만 토글한다. full transcript overlay/screen은 별도 후속 phase로 남긴다. canonical 상세 실행 계획은 [260614_tui_codex_live_toggle](../../../../260614_tui_codex_live_toggle/00_moc.md)에 둔다.

## 권장안

A를 기본 수정안으로 본다. commit lane은 과거 픽셀을 되돌릴 수 없으므로, 커밋 시점의 픽셀을 항상 collapsed canonical 형태로 고정해야 `ctrl+o`/`ctrl+t`가 장기 세션에서 사용자를 속이지 않는다. 펼친 전문은 이미 `alt+t` transcript overlay와 개별 tool focus가 담당한다는 083.1/083.5 설계와도 맞다.

## 검증 포인트

- 단위/헤드리스:
  - 커밋 대상 tool이 `toolOutputExpanded=true`인 상태에서도 `commitFinalizedBacklog()`가 collapsed height로 `commitLines()`에 전달되는지 확인.
  - 커밋 대상 assistant thinking이 `thinkingExpanded=true`인 상태에서도 collapsed summary로 커밋되는지 확인.
  - 미커밋 현재 턴은 기존처럼 `ctrl+o`/`ctrl+t`로 즉시 펼침/접힘 가능해야 한다.
- 실기기:
  - 긴 세션에서 `ctrl+o` 펼침 → 새 prompt 제출 → `ctrl+o` 접힘 후 위쪽 scrollback이 애초에 collapsed 형태로 남는지 확인.
  - `alt+t` transcript overlay는 여전히 full transcript를 보여야 한다.
