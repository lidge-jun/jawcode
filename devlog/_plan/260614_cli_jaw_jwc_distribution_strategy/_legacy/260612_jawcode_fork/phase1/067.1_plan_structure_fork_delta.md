## 2. fork-delta 인덱스 설계

### 2.1 신규 `structure/fork-delta.md` — 스키마

| 열 | 설명 |
|---|---|
| 경로 | 저장소 상대경로 |
| 종류 | `NEW` / `HARD-EDIT` / `REMOVED` / `INVERTED-GUARD` / `SOFT-EDIT` |
| 밴드 | devlog MOC 밴드 (085.5, 060, …) |
| devlog 문서 | 변경 근거 위치 |
| upstream merge 지침 | `CONFLICT-EXPECTED` / `AUTO` / `MANUAL-REVIEW` / `N/A` |
| 보존 경계 | upstream에서 건드리면 안 되는 패턴 (HARD-EDIT 시) |

종류 정의: NEW(업스트림에 없음 — 자동 보존) / HARD-EDIT(산문·식별자 직접 수정 — 충돌 고확률) / REMOVED(재추가 시 re-delete) / INVERTED-GUARD(가드 논리 반전 — 테스트 동반) / SOFT-EDIT(`${APP_NAME}` 동적화 등 — 저위험).

## 3. 규약 [확정 제안]

### 3.1 커밋 트레일러

```
Fork-Delta: <종류> <저장소상대경로>
```

- HARD-EDIT·INVERTED-GUARD·REMOVED 포함 커밋: **필수**
- NEW·SOFT-EDIT: 선택 (fork-delta.md 갱신으로 대체 가능)
- 파싱: `git log --format=%(trailers:key=Fork-Delta)` — 자동 인덱스 갱신 발판
- C1~C13 소급 불필요 (초기 엔트리로 대체)

### 3.2 소스 인라인 마커 — **채택 유보**

`// [jawcode-fork]` 인라인 마커는 채택하지 않음 (충돌 면적 증가, 프롬프트 md 런타임 노출 위험). 예외: 파일 상단 1행 한정 보존 경계 주석(`// [jawcode-fork] preserved: ...`)만 허용.

### 3.3 INDEX.md 동기화 규칙 — fork-delta.md를 Tier 1 등재 + 동기화 표에 "HARD-EDIT/NEW/REMOVED/INVERTED-GUARD 파일 변경 → fork-delta.md" 행 추가.

## 5. 유지보수 규칙

| 조건 | fork-delta.md 갱신 |
|---|---|
| HARD-EDIT·INVERTED-GUARD·REMOVED 커밋 | 필수 (동행) |
| NEW 파일 추가 | 필수 (지침 N/A) |
| SOFT-EDIT만 | 선택 (밴드 일괄) |
| 업스트림 리베이스 전 | CONFLICT-EXPECTED 사전 점검 필수 |

리베이스 전: `grep CONFLICT-EXPECTED structure/fork-delta.md` ↔ `git diff upstream/main --name-only` 대조. 체리픽: 대상 커밋이 HARD-EDIT 경로를 건드리는지 사전 확인 → 충돌 시 보존 경계 열 참조.

upstream 기여 후보 구분: NEW 중 버그픽스 성격(local-token-detect 등)은 PR 후보 / 브랜딩 전용(agent-identity, cli-jaw-vocab, stage-skill-map, goal-runtime)은 비대상.
