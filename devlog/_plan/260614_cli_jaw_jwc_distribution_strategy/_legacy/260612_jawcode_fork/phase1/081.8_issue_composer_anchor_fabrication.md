# 081.8 — issue: composer 모델의 hashline 앵커 날조로 edit 연쇄 거부

> 상태: ✅ 수정 완료 (2026-06-12). 소속: 081 밴드 (cursor/composer 도구 규율).
> 입력: 사용자 "edit이 자꾸 실패하는 오류 — cursor grok 프롬프트 패치 계열을 더 보강해야돼."
> 📤 **업스트림 PR: [Yeachan-Heo/gajae-code#528](https://github.com/Yeachan-Heo/gajae-code/pull/528)** (dev 대상,
> 브랜치 `feat/composer-edit-discipline`, 커밋 1개). 참고: 선행 #518(모델 엔트리)·#519(timeout 단위) 모두 **머지됨**.
> 업스트림판은 cursor.ts에 HOST_OVERRIDE가 없어 `buildCursorSystemPromptJsons`에 discipline 블록만 선두 삽입
> (fork판은 HOST_OVERRIDE 다음). 나머지(모듈·openai 주입·테스트)는 동일 포팅.

xai 직결로 붙인 `grok-composer-2.5-fast`가 hook-selector/ask 작업 세션에서 edit을
네 번 연속 거부당했다 ("Edit rejected: N anchors do not match"). 원인은 모델이 jwc의
hashline 앵커 규약을 무시하는 것 — `sed -n`으로 파일을 읽고(앵커 없는 출력) 앵커
해시를 **지어내서** edit을 날린다. 기존 패치 계열(cursor.ts HOST_OVERRIDE, progrok
TOOL_DISCIPLINE)은 xai 직결 경로에 하나도 적용되지 않는 공백이 있었고, 이번에
`composer-discipline.ts`를 신설해 openai-completions와 cursor RPC 양쪽에서 composer
모델 요청 맨 앞에 앵커/edit 규율 시스템 블록을 주입하도록 했다.

## 1. 증거 (세션 2026-06-12T05-38-51 / 019eba57)

`~/.gjc/agent/sessions/-Developer-new/2026-06-12T05-38-51-338Z_*.jsonl`, edit 17회 중
거부 4회 (rec 233/262/267/274). 전부 `xai/grok-composer-2.5-fast`.

| rec | 행동 | 문제 |
|---|---|---|
| 226→228 | `sed -n '598,625p'`로 읽고 edit | sed 출력엔 앵커 없음 — 이전 read 기억으로 작성 |
| 229 | edit 성공했지만 범위 밖 줄 중복 재방출 | hashline.md의 anti-pattern 그대로 |
| 232→233 | `≔617hp..619fc` 전송 | 실제 617은 `617ca` — **해시 날조** (`hp`는 옛 613줄 해시를 줄번호만 +4 산술 이동) |
| 202, 238 등 | python heredoc(pathlib write_text)으로 파일 직접 수정 | 대역외 쓰기 — 기존 앵커 전부 무효화, read-cache 복구(`recovery.ts`)도 스냅샷 불일치로 거부 |
| 239 | heredoc 안에 추론 산문이 섞여 SyntaxError | reasoning 누수 (081.2 계열 동일 습성) |
| 261~274 | 거부 메시지가 새 앵커를 보여줘도 기억으로 재작성 | 거부 출력 미활용 → 재실패 |

핵심: `computeLineHash`(hash.ts:140)는 **줄 내용만으로** 해시를 만들므로 모델이
머리로 계산할 수 없고, 산술 이동(줄번호만 보정)도 불가능하다. 복구 경로
(`tryRecoverHashlineWithCache`)는 모델 앵커가 캐시 스냅샷과 정확히 일치해야만
작동하는데, 날조 해시·대역외 수정은 둘 다 이 전제를 깬다.

## 2. 패치 계열 공백

| 경로 | 기존 보호 | composer-2.5-fast 적용 여부 |
|---|---|---|
| cursor RPC (`providers/cursor.ts`) | CURSOR_HOST_OVERRIDE_PROMPT (081.6) | ❌ xai 직결이라 미경유 |
| progrok 프록시 (002_proxy/05_progrok) | TOOL_DISCIPLINE + effort 스트립 | ❌ baseUrl이 api.x.ai 직결 |
| xai 직결 (openai-completions) | **없음** | ← 이번 세션 경로 |

## 3. 수정 (jwc fork)

신설 `packages/ai/src/providers/composer-discipline.ts`:
- `isComposerHarnessModel(modelId)` — id에 "composer" 포함 여부 (progrok 게이트와 동일).
- `COMPOSER_EDIT_DISCIPLINE_PROMPT` — 관찰된 실패 모드 6개를 정조준한 영어 지침:
  read/search 도구로만 읽기(셸 출력엔 앵커 없음), edit/write 도구로만 수정(대역외
  쓰기 금지), 앵커는 최신 출력에서 verbatim 복사(해시 계산 불가), 수정 후 기존 앵커
  실효, 거부 메시지의 새 앵커 사용, 커맨드 문자열에 추론 섞지 않기.

주입 지점 (둘 다 호스트 시스템 프롬프트가 있을 때만, 맨 앞 고정):
- `openai-completions.ts` `convertMessages()` — composer면 `systemPrompts.unshift(...)`.
  멀티 시스템 미지원 호환 모드에서도 join 순서상 선두 유지.
- `cursor.ts` `buildCursorSystemPromptJsons(systemPrompt, modelId?)` — HOST_OVERRIDE
  다음, 호스트 프롬프트 앞에 별도 블록. 옵셔널 인자라 기존 호출/테스트 무변경,
  블롭 캐시도 엔트리 단위라 prefix 캐시 유지.

## 5. 남은 일 / 한계

- 프롬프트 규율은 완화책이다. 모델이 그래도 어기면 도구 측 강제(예: composer 모델
  한정 bash에서 파일 출력/수정 명령 차단)가 다음 단계 — 별도 결정 필요.
- progrok TOOL_DISCIPLINE에도 동일 앵커 조항을 백포트하면 opencode 등 다른 클라이언트도
  보호됨 (002_proxy 쪽, 미적용).
- 거부 메시지를 받았을 때 모델이 새 앵커를 쓰도록 거부 텍스트 자체에 한 줄 더
  강조하는 것도 고려 (hashline/anchors.ts `rejectionHeader`).
