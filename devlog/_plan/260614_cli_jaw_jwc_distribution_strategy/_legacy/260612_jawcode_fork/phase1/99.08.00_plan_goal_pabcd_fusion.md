# 99.08.00 — plan: goal↔pabcd 합성 (표시 + 기록)

> 상태: ✅ **구현 완료** (260613 — 99.04.05 C5 헤더 goal 병기 + C6 전이 체크포인트로 랜딩; 99.06 스윕에서 표기 현행화). 방향 확정(표시+기록 둘 다)은 260612 인터뷰.
> 소속: [99.00.00 MOC](./99.00.00_moc_stabilization.md) 패키지 99.08. 선행: 99.03 (M2 헤더 레일이 토대).
> 입력: 실사용 마찰 1위 — "아니 /goal이 계속되어야하는데 지금 ipabcd에서는 작동안하잖아"
> ([99.00.02](./99.00.02_analysis_cli_jaw_usage.md) §마찰). 동형성 검증(260612): goal 모드와 pabcd는
> 완전히 독립된 상태 슬롯, 합성 로직 0줄·소유자 부재였음 → 본 밴드가 소유자.

goal(오래 달리는 목표)과 pabcd(단계 파이프라인)는 실사용에서 **동시에** 돈다 — goal이 "무엇을"이고
pabcd가 "어떻게"다. 그런데 jwc에서 둘은 서로를 모른다: goal 모드 컨텍스트는 pabcd 단계를 모르고,
orchestrate 전이는 goal 체크포인트에 흔적을 남기지 않는다. cli-jaw에서도 같은 단절이 마찰 1위였다.
본 플랜은 두 상태를 **합치지 않고**(별개 슬롯 유지) 두 지점에서 **엮는다**.

## 설계 (확정 방향)

### A. 표시 — M2 헤더에 goal 상태 병기

99.03 M2의 `pabcd-stage-context` custom message를 확장: goal 모드 활성 + pabcd 활성이 동시면 한 줄로 병기.

```
[GOAL: <objective 요약 ≤40자> · PABCD — B: BUILD · verify=pending]
```

- goal만 활성 / pabcd만 활성이면 각자 기존 형식 유지 (goal 모드는 기존 goal-mode-context가 이미 담당 —
  중복 objective 전문 주입은 피하고 헤더에는 요약만).
- 구현 위치: `session/pabcd-stage-header.ts`(99.03.01이 확정한 순수 함수)에 goal 파라미터 추가 —
  `buildPabcdStageContent(state, goal?)`. 99.03 구현 시 시그니처만 미리 열어두면 본 밴드는 1-인자 추가로 끝.

### B. 기록 — orchestrate 전이 시 goal 체크포인트 자동 기록

`runNativeOrchestrateCommand`의 stage 전이 성공 지점(persist 직후, `orchestrate-runtime.ts:344` 부근)에서
활성 goal이 있으면 체크포인트 1건 자동 append:

```
summary: "pabcd <from>→<to> (audit=<status>)"   evidence: [pabcd-state path, plan_ref?]
```

- 경로: goal-runtime의 update/checkpoint 동사 재사용 (`jwc goal update` 내부 함수 직호출 — CLI 셸아웃 금지).
- 역방향 없음: goal 전이가 pabcd를 건드리지는 않는다 (D130-1 단방향 결과 동일 — 파이프라인이 보고하고
  ledger가 받는다).
- goal 비활성 시 no-op (게이트 1줄).

## 비범위

- 두 상태의 단일 스키마 통합 ❌ (별개 슬롯 유지 — 인스턴스/세션·boss/로컬 원칙과 정합)
- cli-jaw 측 goal↔orchestrate 수리 ❌ (M2 130 밴드에서 D130-1 주입으로 자연 해소)

## 완료 기준

- goal+pabcd 동시 활성 세션에서: ① 매 턴 헤더에 둘 다 보임, ② `jwc orchestrate b` 전이 후
  `jwc goal history`에 체크포인트 자동 기록 확인 (e2e).
- goal만/pabcd만 활성일 때 무간섭 회귀 0.
- 실사용 재현 시나리오("goal 진행 중 pabcd 단계 진입 후 계속") 마찰 해소 확인.
