# jwc "커밋 시점 접기" 설계 초안

> **상태**: 🟡 설계 초안 — 소스 실측 기반 (Codex inline 모드 청사진)

---

## 왜 이게 필요한가

현행 jwc는 도구 셀이 완료될 때 **사후 수축(retroactive collapse)**을 쓴다. 활성 셀을 먼저 expanded 상태로 그렸다가, 완료 신호가 오면 setMinimized(true)로 접고 viewport를 다시 그린다. 이 수축 물리가 §9 floor 계산(`083.1_plan_tool_autocollapse.md`)과 §10 compaction 로직(`083.6_issue_autocollapse_editor_jump.md`, `083.7_plan_composer_bottom_pin.md`)의 복잡도를 낳는다.

근본 문제: **히스토리가 단조 성장(monotonic growth)하지 않는다**. 이미 그려진 행이 나중에 줄어들 수 있고, 이것이 composer 앵커 이탈, 스크롤 점프, 재렌더 아티팩트의 원인이 된다.

## Codex inline 모드 청사진 — 소스 실측 결과

Codex `codex-rs/tui`의 commit-time folding 메커니즘을 소스에서 확인했다 (상세: `01_co_tui.md` §보강).

### 핵심 불변식

1. **스크롤백 = 최종형 전용 공간**: `insert_history_hyperlink_lines_with_mode_and_wrap_policy()` (`tui/src/insert_history.rs`)는 ANSI escape sequence (`SetScrollRegion` + `\x1bM` Reverse Index)로 viewport 위쪽 스크롤백에 직접 write한다. **한 번 write된 행은 절대 수정되지 않는다.**

2. **뉴라인 게이트**: `StreamCore::push_delta()`는 `\n`이 포함된 청크가 왔을 때만 stable 큐에 enqueue한다. partial 줄은 tail(활성 라이브 존)에 머문다.

3. **Compact-from-birth**: ExecCell은 `should_flush()`(모든 call 완료)가 true가 되는 순간 compact form(최대 5줄 head + 5줄 tail + `…+N lines` 엘립시스)을 한 번만 스크롤백에 write한다. Running 중에는 viewport 활성 셀에만 존재하고 스크롤백에는 없다.

4. **활성 라이브 존**: viewport의 `current_tail_lines()` 영역. StreamController의 `enqueued_stable_len`이 render의 경계를 나눈다. tail = `rendered_lines[enqueued_stable_len..]`.

### 레이아웃 구조 (실측)

```
╔══════════════════════════════════╗
║  스크롤백 (실 터미널 scrollback)  ║  ← insert_history_lines()로만 write
║  committed cells (최종형)         ║    write 후 불변
╠══════════════════════════════════╣
║  inline viewport (라이브 존)      ║  ← ratatui draw()가 소유
║  ├ active stream tail             ║
║  ├ active exec cell               ║
║  └ bottom_pane (composer + 상태) ║
╚══════════════════════════════════╝
```

스크롤백은 단조 성장한다. 뒤로 줄어드는 일이 없다.

## jwc 대응 설계 — "라이브 존" 패턴 도입

### 기본 아이디어

도구 셀의 preview를 **composer 클러스터 위 "라이브 존"**에서 렌더한다. 완료 시 compact 최종형 1블록만 `chatContainer`(히스토리 영역)에 append한다.

```
[jwc 현행]           [목표 상태]
chatContainer        chatContainer
  ├ cell A             ├ cell A (committed, compact)
  ├ cell B (active,    ├ cell B (committed, compact)
  │  expanded)         │
  └ cell C             └ cell C (committed, compact)
                     liveZone  ← 신규
                       └ cell D (active preview)
                     composer
```

"라이브 존"은 Codex의 inline viewport tail 개념과 동일하다. 차이는 jwc가 현재 React 기반이라는 점 — statusContainer 인접, ViewportFill 아래에 고정 영역으로 두면 된다.

### 구현 스케치

**1. event-controller `#handleToolExecutionStart` 변경**

현행: `chatContainer.addChild(activeToolCell)` → 즉시 히스토리에 삽입

변경: `liveZone.setActiveCell(activeToolCell)` → 라이브 존에만 렌더

**2. 완료 시 마이그레이션**

```
on ToolExecutionComplete:
  compactCell = activeToolCell.toCompactForm()  // head 5 + tail 5 + +N lines
  liveZone.clearActiveCell()
  chatContainer.addChild(compactCell)           // 히스토리에 최종형 append
```

이 시퀀스가 Codex의 `ExecCell::should_flush()` → `flush_active_cell()` → `add_boxed_history()` 패턴에 해당한다.

**3. Ctrl+O / 포커스 링 정합**

- 라이브 존의 activeToolCell은 포커스 링에서 제외 (스크롤백 대상 아님)
- Ctrl+O(transcript overlay)는 committed cells + 현재 activeCell live tail을 보여줌 (Codex의 `TranscriptOverlay` 패턴 참고)
- 완료된 compactCell은 포커스 링에서 expand-on-focus가 가능 (enter = expand, esc = collapse)

## §9·§10과의 관계 변화

| 메커니즘 | 현행 역할 | 목표 상태 |
| --- | --- | --- |
| §9 floor 계산 | 수축 후 composer 위치 보정 필수 | 히스토리 단조 성장 → 불필요해짐 |
| §10 compaction | 과도 성장 방지 최후 수단 | 안전망으로 강등 (edge case 전용) |

"커밋 시점 접기"가 정착되면 §9는 삭제 가능성이 높고 §10은 극히 긴 대화에서만 필요한 optional 기능이 된다.

## 열린 질문

1. **스트리밍 어시스턴트 텍스트도 라이브 존으로?** Codex는 assistant text streaming도 tail 개념으로 viewport에만 그리고, newline 단위로 stable 큐에 커밋한다. jwc도 동일하게 가면 streaming text도 retroactive 변경이 없어진다. 단, 현재 streaming text는 이미 줄 단위로 추가하므로 큰 차이는 없을 수 있음.

2. **thinking(내부 추론) 블록은?** `083.5_plan_thinking_collapse.md` 참고. thinking은 완료 즉시 접힌 형태("Thought for Xs")로 커밋하면 동일 패턴 적용 가능.

3. **라이브 존 높이 제약**: 도구 출력이 매우 길면 라이브 존이 화면을 넘칠 수 있다. Codex는 viewport height를 동적 조정 (`scroll_region_up`)으로 해결. jwc는 라이브 존 max height를 고정하고 내부 스크롤로 처리 가능.

4. **여러 도구 동시 실행**: concurrent tool calls 시 라이브 존에 여러 activeCell이 쌓일 수 있음. 순서 보장 방법 설계 필요.
