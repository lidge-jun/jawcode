## 0. 핵심 결론 — 우려했던 것보다 훨씬 깨끗하다

멀티 cwd N세션의 최대 우려였던 **전역 상태 오염은 ACP 팩토리가 이미 방어**하고 있다:

```ts
// main.ts:269 createAcpSessionFactory — 세션마다:
const nextSettings = await args.settings.cloneForCwd(cwd);      // ① 설정: cwd 스코프 클론 (settings.ts:368)
const nextSessionManager = SessionManager.create(cwd, ...);     // ② 세션 저장: cwd별
await args.createSession({ cwd, sessionManager, settings, ...}) // ③ cwd 명시 — getProjectDir() 폴백 미사용
```

- `sdk.ts:461/470/797`의 `cwd ?? getProjectDir()` 폴백은 **cwd를 안 넘긴 호출자**용 — ACP 경로는 항상 명시.
- 공유되는 것은 `authStorage`·`modelRegistry`뿐 — D112-2(인스턴스 소유) 의도와 일치.
- 세션 영속/복구도 기성품: `loadSession`(+`#replaySessionHistory`) · `resumeSession` · `listSessions`(페이지네이션) ·
  `unstable_forkSession` · `closeSession` — `acp-agent.ts:452-530`. **상주 child 크래시 복구 = loadSession 재호출**.

## 3. 비블로커 (확인 완료)

- **동시 prompt 비간섭**: 세션별 `ManagedSessionRecord` 격리(`acp-agent.ts:377`), 알림은 sessionId 태깅 —
  한 세션의 스트림이 다른 세션 입력을 막는 구조 없음.
- **설정 격리**: `Settings.cloneForCwd`(settings.ts:368)가 세션 cwd의 프로젝트 설정/SYSTEM.md를 반영.
- **메모리 백엔드**: 기본 off + spawn 시 env 명시(112.1) — D112-2 일회용 원칙 충족.
- **권한 플로우**: ACP permission 요청이 sessionId로 태깅되어 멀티 세션 라우팅 가능 — cli-jaw 서버가
  웹 클라이언트별 중계만 구현하면 됨 (112.1 P1 범위).
