# 100.1 — Bun 셰임 전수 인벤토리 (M2 Node 포팅 게이트)

> 상태: 🟡 v1 — 전수 인벤토리 실측 기반. 독립 감사 대기.
> 상위 문서: [100_moc_node_porting.md](./100_moc_node_porting.md)
> 측정 기준일: 2026-06-12 (업스트림 jawcode `agent` 브랜치)
> 포팅 스코프: `packages/ai` + `packages/agent` + `packages/coding-agent` (비TUI 경로) + `packages/utils`. TUI 경로(`/tui/`)는 제외.

M2의 실질적 관문은 **`createAgentSession()`이 Node 22에서 스트리밍 왕복 1회를 완주하는 것**이다. 그 전제는 엔진 패키지(`ai`, `agent`, `coding-agent` 비TUI, `utils`)가 Bun 전용 API를 하나도 런타임에 노출하지 않는 상태다. 이 문서는 그 현황을 측정하고 셰임 전략을 제안한다.

업스트림은 빌드 산출물 없는 raw `.ts` 배포 구조다. `check:node20-baseline` 스크립트는 "Node 지원 허위 주장 방지" 가드일 뿐이며 실제 호환을 보장하지 않는다(구 01 리서치 확인). 카운트는 `grep -rn` 실측값이고, 테스트 파일은 별도 구분했다.

## 2. 셰임 매핑 표

| # | Bun API | Node 22 대응 | 방법 | 비고 |
|---|---|---|---|---|
| A | `Bun.file(p).text()` | `fs.readFile(p, 'utf8')` (`node:fs/promises`) | 셰임 가능 (런타임 분기) | `.json()` → `JSON.parse(await readFile)`, `.exists()` → `stat` try/catch, `.stat()` → `stat()`, `.bytes()` → `readFile` Buffer |
| B | `Bun.write(p, data)` | `fs.writeFile(p, data)` | 셰임 가능 (런타임 분기) | `Bun.file(p).write()` 체인도 동일 |
| C | `Bun.spawn(cmd, opts)` | `child_process.spawn` 또는 `execa` | 셰임 가능 (런타임 분기) | `stdin/stdout` 옵션 매핑 필요; `stt/recorder.ts` pty 경로는 별도 검토 |
| D | `bun:sqlite` | `better-sqlite3` (cli-jaw 기보유, D8 확정) | 셰임 가능 (런타임 분기) | API 근접 — `Database(path)`, `.prepare().get/all/run`. sync API 일치. `Statement` 타입만 교체 |
| E | `Bun.sleep(ms)` | `timers/promises` `setTimeout(ms)` 또는 `new Promise(r => setTimeout(r, ms))` | 셰임 가능 (런타임 분기) | `sdk.ts:1064` 이미 플래그됨; `Promise.race([op, sleep(ms)])` 패턴 그대로 동작 |
| F | `Bun.hash(s)` (bigint) | `xxhash-wasm` 또는 `farmhash` (npm) | 셰임 가능 (런타임 분기) | bigint 반환 인터페이스 유지 필요; `.xxHash32(s, seed)` → xxhash-wasm 동등 API |
| F | `Bun.CryptoHasher("sha256")` | `node:crypto` `createHash("sha256")` | 셰임 가능 (런타임 분기) | `.update(s).digest("hex")` API 동일 |
| F | `new Bun.SHA256()` | `node:crypto` `createHash("sha256")` | 셰임 가능 (런타임 분기) | `blob-store.ts` 3곳 |
| G | `import x from "…" with { type: "text" }` | **esbuild `loader: { '.md': 'text', '.html': 'text' }` 빌드 치환** | 빌드 치환 (esbuild) | 런타임 분기 불가 — 빌드타임 import assertion. 158 곳 일괄 처리. 단일 esbuild 설정으로 해결. |
| H | `Bun.JSONL.parseChunk` | 자체 구현 (`utils/src/stream.ts` 내부화) | 재작성 필요 | `parseChunk(input, start, stop)` 시그니처 보존 자체 파서; `stream.test.ts` 통과가 검증 기준 |
| I | `Bun.JSON5.parse` | `json5` npm (`JSON5.parse`) | 셰임 가능 (런타임 분기) | `json5` 패키지 추가 필요 |
| J | `Bun.serve({fetch, port})` | `node:http` `createServer` 또는 `fastify` | 재작성 필요 | **M2 엔진 포팅 스코프 밖** — auth-broker/gateway 서버는 cli-jaw 측 Node 코드. coding-agent의 `bridge-mode.ts:616`, `eval/py/tool-bridge.ts:35`만 스코프 안. `node:http` 얇은 래퍼로 대체 가능 |
| K | `Bun.stdin.text()` / `.stream()` | `process.stdin` (ReadableStream Node 22) | 셰임 가능 (런타임 분기) | `rpc-mode.ts`는 cli-jaw 서버에서 직접 호출되지 않으므로 우선순위 낮음 |
| L | `Bun.stripANSI(s)` | `strip-ansi` npm | 셰임 가능 (런타임 분기) | 2곳, 간단 |
| M | `Bun.semver.order` | **제거** (Node 빌드 진입점에서 이 체크는 무의미) | 빌드 치환 (조건 분기 제거) | cli.ts는 Bun 진입점 전용. Node dist-node/ 진입점은 다른 파일 |
| N | `Bun.env.X` | `process.env.X` | 빌드 치환 (esbuild define) 또는 셰임 | esbuild `define: { 'Bun.env': 'process.env' }` 로 일괄 치환 권장 |

| P | `Bun.gc(true)` | `// no-op in Node` | 셰임 가능 (런타임 분기) | `debug/profiler.ts` 1곳; 셰임에서 빈 함수 제공 |
| Q | `process.isBun` / `globalThis.Bun` | N/A (0곳) | 해당 없음 | 기존 분기 없음 — 셰임 주입 후 추가 예정 |

## 3. 포팅 전략

### 3-1. 단일 코드베이스 + 런타임 분기 셰임 (권장)

로드맵 "Bun.* 셰임 + tsc/esbuild 트랜스파일" 노선을 따른다. TUI가 Bun에 의존하므로 Node 단일화가 아닌 **듀얼 런타임**이 기본값이다.

```
packages/jwc/src/shims/
  bun-file.ts       ← Bun.file → fs/promises 래퍼
  bun-write.ts      ← Bun.write → fs.writeFile 래퍼
  bun-spawn.ts      ← Bun.spawn → child_process.spawn 래퍼
  bun-sleep.ts      ← Bun.sleep → setTimeout Promise 래퍼
  bun-hash.ts       ← Bun.hash → xxhash-wasm 래퍼 (bigint 인터페이스 유지)
  bun-crypto.ts     ← Bun.CryptoHasher / Bun.SHA256 → crypto 래퍼
  bun-jsonl.ts      ← Bun.JSONL.parseChunk 자체 구현
  bun-json5.ts      ← Bun.JSON5 → json5 npm 래퍼
  bun-ansi.ts       ← Bun.stripANSI → strip-ansi 래퍼
  bun-archive.ts    ← Bun.Archive → node-tar / fflate 래퍼
  bun-stdin.ts      ← Bun.stdin → process.stdin 래퍼
  bun-serve.ts      ← Bun.serve → node:http createServer 래퍼 (2곳)
  bun-gc.ts         ← Bun.gc → no-op
  index.ts          ← 글로벌 주입: if (!globalThis.Bun) { globalThis.Bun = shimObject; }
```

감지 패턴:
```ts
// shims/index.ts
if (typeof globalThis.Bun === "undefined") {
  globalThis.Bun = buildShim(); // 위 모듈 합성
}
```

### 3-2. esbuild 빌드 파이프라인 (Node 타깃)

```
패키지/coding-agent/src/sdk.ts         ← 엔진 진입점
     └─ esbuild bundle ─────────────────→ dist-node/sdk.js
              platform: "node"
              target: "node22"
              format: "esm"
              loader: { ".md": "text", ".html": "text" }  ← G항(158곳) 일괄 해소
              define: { "Bun.env": "process.env" }         ← N항(64곳) 일괄 해소
              inject: ["packages/jwc/src/shims/index.ts"] ← 셰임 사전 주입
              external: ["better-sqlite3", "xxhash-wasm", "json5", "strip-ansi", "node-tar"]
```

tsc는 타입체크 전용(`noEmit: true`). 실제 emit은 esbuild.

`Bun.env` → `process.env` 치환은 esbuild `define`으로 **빌드타임 0비용 처리**. `.md`/`.html` 텍스트 임베드(G항)도 esbuild `loader`가 빌드타임 인라인으로 처리하므로 런타임 셰임 불필요.

### 3-3. `Bun.semver` 처리 (M항)

`coding-agent/src/cli.ts`의 Bun 버전 체크는 Node 진입점에서 제외. Node 빌드 진입점은 `sdk.ts`를 직접 참조하므로 `cli.ts`는 번들 대상이 아니다. 문제 없음.

## 참고: API 카테고리 건수 요약 + 3대 리스크

**카테고리별 src 합계 (비TUI 기준):**

> `Bun.file`(173) > `Bun.env`(64) > `Bun.sleep`(49) > `Bun.write`(83) > `Bun.spawn`(38) > `Bun.hash`(37) > 텍스트 임베드(158) > `bun:sqlite`(17) > `Bun.serve`(7) > `Bun.Archive`(9) > `bun:test`(815줄 별도)

**3대 리스크 항목:**

1. **G항 — 텍스트 임베드 158곳** (`import … with { type: "text" }`): 빌드타임 전용 문법이므로 런타임 셰임 불가. esbuild `loader` 설정이 누락되면 모든 MD 프롬프트가 `undefined`가 된다. 단일 설정 실수가 전체 시스템 프롬프트를 무너뜨리므로 빌드 검증 단계가 필수.

2. **H항 — `Bun.JSONL.parseChunk` (utils/src/stream.ts)**: JSONL 스트리밍 파서는 `utils` 공통 레이어 최하단이며 모든 AI 스트림이 통과한다. 자체 구현이 Bun의 경계 처리(`start, stop` 파라미터) + 에러 복원력을 완전히 재현하지 못하면 스트림 분기가 발생한다. `stream.test.ts` 1,662줄이 유일한 보호막.

3. **D항 — `bun:sqlite` 핵심 DB 2곳** (`ai/src/auth-storage.ts`, `coding-agent/src/memories/storage.ts`): `better-sqlite3`는 API가 근접하지만 타입 시그니처(`Statement`, `SQLQueryBindings`)가 다르다. `auth-storage.ts`는 인증 전체를 담당하므로 교체 실패 시 어떤 프로바이더도 연결되지 않는다. 타입 어댑터 레이어 필요.
