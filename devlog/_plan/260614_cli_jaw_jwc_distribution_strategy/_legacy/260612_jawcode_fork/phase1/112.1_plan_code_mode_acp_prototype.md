## 1. ACP 표면 실사 — UI 필요 vs 제공 vs 갭

코드 경로: `packages/coding-agent/src/modes/acp/` (acp-agent.ts, acp-event-mapper.ts, acp-client-bridge.ts, acp-mode.ts).

### 1.1 ACP가 제공하는 것

| 카테고리 | ACP 표면 | 세부 |
|---|---|---|
| **세션 수명** | `initialize` → `newSession` / `loadSession` / `resumeSession` / `closeSession` / `unstable_forkSession` | 각 응답에 `sessionId`, `configOptions`, `models`, `modes` 포함. bootstrap race guard 50ms 후 `available_commands_update` + `session_info_update` 자동 발송. |
| **세션 목록** | `listSessions(cwd?)` | 페이지 단위 50건. `_gjc/sessions/listAll` ext 메서드로 글로벌 목록(최대 5000건). `_gjc/projects/list`로 cwd별 집계. `_gjc/chats/byCwd`로 특정 폴더 최근 세션. |
| **스트리밍 청크** | `agent_message_chunk` / `agent_thought_chunk` | 텍스트 델타·thinking 델타 실시간 전달. messageId로 동일 메시지 묶음. |
| **도구 셀** | `tool_call` (시작, `pending`) → `tool_call_update` (`in_progress`) → `tool_call_update` (`completed`/`failed`) | kind: read/edit/delete/move/execute/search/fetch/think/other. title, rawInput, locations(절대 경로), content 포함. |
| **diff 콘텐츠** | `tool_call_update.content[].type === "diff"` | edit 도구 완료 시 `{path, oldText, newText}` 자동 생성. |
| **plan 업데이트** | `plan` (sessionUpdate) | todo_write 완료·todo_reminder 이벤트 → `{entries: [{content, priority, status}]}`. |
| **권한 요청** | `requestPermission(toolCall, options, signal)` | 클라이언트가 `initialize.clientCapabilities`에 명시하면 모든 도구 실행 전 권한 게이트 발동. `options[].kind`: allow-once/allow-session/deny. |
| **모드** | `setSessionMode` / `setSessionConfigOption(mode)` | default ↔ plan 2가지. `current_mode_update` sessionUpdate 발송. |
| **모델** | `unstable_setSessionModel` / `setSessionConfigOption(model)` | 세션 내 모델 변경. `getAvailableModels()` → `configOptions` |
| **thinking** | `setSessionConfigOption(thinking)` | off / 세션 지원 레벨. `thinking_level_changed` 이벤트 → `config_option_update` 자동 발송. |
| **슬래시 커맨드** | `available_commands_update` | 빌트인(`/clear`, `/compact`, `/model` 등) + 커스텀 + 파일 기반 + `/skill:*`. |
| **elicitation** | `unstable_createElicitation` (form mode: select/confirm/input) | `clientCapabilities.elicitation.form` 선언 시 스킬/익스텐션이 UI 입력 요청. |
| **터미널** | `createTerminal` (clnt cap `terminal: true`) | bash 도구가 `ClientBridgeTerminalHandle`을 통해 클라이언트 터미널 세션 사용. |
| **파일 읽기/쓰기** | `readTextFile` / `writeTextFile` (clnt cap `fs.*`) | 클라이언트 측 파일 브리지. |
| **사용량** | `usage_update` (턴 종료 시) | contextWindow, tokens, cost(USD). `_gjc/usage` ext 메서드로 전체 리포트. |
| **익스텐션** | `_gjc/extensions` / `_gjc/extensions/toggle` | 익스텐션 목록 조회·활성화 토글. |
| **세션 히스토리 리플레이** | `loadSession` 응답 후 전체 메시지 replay 발송 | 완료된 tool_call + agent/user 청크 재전송. |
| **취소** | `cancel(sessionId)` | abort race + `stopReason: "cancelled"`. |
| **transport** | `ndJsonStream(stdout, stdin)` — stdio NDJSON | `runAcpMode`가 `process.stdout`/`process.stdin` 직결. |

## 2. 프로세스 관리 — child 수명 정책

### 2.1 기본 구조

```
cli-jaw 서버 (Node, 포트 3457)
  └── CodeSessionManager (신규 서비스 클래스)
        ├── child: ChildProcess  ← `jwc acp` 1개 (ACP 연결 1개)
        ├── acpConnection: AgentSideConnection (NDJSON framing)
        └── sessions: Map<sessionId, ManagedSession>  ← ACP 레벨 멀티세션
```

**핵심 결정**: P1에서는 **1 ACP 연결 = 1 child**. ACP 레벨에서 `newSession`을 여러 번 호출해 멀티세션을 관리. Child 재시작 시 모든 인메모리 세션 상태가 날아가므로 cli-jaw 서버가 sessionId를 기억해두고 `resumeSession`으로 복구.

### 2.2 spawn 정책

```typescript
// cli-jaw/src/services/code-session-manager.ts (P1 스케치)
const child = spawn("jwc", ["acp"], {
  cwd: selectedCwd,          // 작업장 피커가 선택한 폴더
  stdio: ["pipe", "pipe", "inherit"],  // stdin/stdout 파이프, stderr는 상속(서버 로그)
  env: {
    ...process.env,
    JWC_SESSION_DIR: undefined,  // 기본 ~/.jwc/sessions 사용
  },
});
```

## 프로세스 모델 개정 — 상주 ACP 단일 child × N세션 [확정 260613 새벽, 사용자 조건 합치]

사용자 3조건(N 동시 세션 비간섭 · 저메모리 · 네이티브 가능?) 검증 중 결정적 사실 확인:
`acp-agent.ts:377` `#sessions = new Map<string, ManagedSessionRecord>()` — **jwc ACP 서버는 이미
단일 프로세스에서 N개 AgentSession을 운영**한다 (G4 갭 해소 — spec 추정이 아니라 구현 사실).

따라서 세션당 child 1개(v1 안)를 폐기하고 **상주 `jwc acp` child 1개 + `newSession` N개**로 개정:

| 조건 | 충족 |
|------|------|
| N 동시 세션, 상호 인터셉트 없음 | ✅ 세션별 레코드 격리 + I/O 멀티플렉싱 (단, 전역 cwd 가정 감사 선행 — 아래) |
| 메모리 | ✅ 엔진 1카피, 세션당 MB 단위 (in-process급) |
| idle 세션 메모리 회수 (사용자 요구) | ✅ `closeSession` = 레코드 해제 — 프로세스 kill 없이 즉시 회수 |
| 크래시 격리 | ✅ cli-jaw 서버와 프로세스 분리 (in-process 대비 보너스) |
| 100 밴드(Node 포팅) 의존 | ✅ 불요 — bun child라 즉시 가능 |

파급: in-process 승격(transport 스왑)은 "child 1개 제거" 최적화로 격하 — 우선순위 하락.
100 밴드의 실수요는 110(boss chat, cli-jaw 서버 내부)으로 좁혀짐.

**선행 작업 — 멀티 세션 격리 감사**: 전역 cwd 가정(`get/setProjectDir` 91곳, `process.cwd()` 79곳
비테스트 src)이 세션 간섭 후보. ACP 경로가 이미 N세션을 돌리므로 치명 간섭은 낮을 것으로 추정되나
전수 감사 필요 — 110.2 후보.

**동시 세션 상한 (성능·메모리 가드 — 사용자 토론 260613)**: ACP 단계의 비용은 세션당 bun
프로세스(엔진 중복 로드, 수십~100MB)이므로 동시 child 상한 **기본 4** (설정 후보:
`code.maxConcurrentSessions`). 초과 생성 시 가장 오래 idle인 세션부터 reap 후 생성. in-process
승격 시 상한 제거 (세션당 비용이 MB 단위로 소멸).

**idle-reap 정책 (P1 단순화)**:
- `closeSession` 호출 후 모든 세션이 닫히면 30초 idle timer 기동 → 만료 시 `SIGTERM` → 2초 대기 → `SIGKILL`.
- 사용자가 새 세션을 열면 타이머 취소 + 이미 terminated면 재spawn.
- P2에서 세션 개수 기반 멀티 child 정책으로 확장 가능.

**재시작 정책**:
- child 비정상 종료(`exit code !== 0`) → 서버 측 연결 오류 SSE(`{type: "code:session_error", reason: "child_exit"}`) 발송 → 클라이언트가 재연결 버튼 표시.
- 자동 재시작 없음 (P1) — 상태 손실 없이 재spawn하려면 sessionId 보존이 필요하나, 세션 파일은 디스크에 있으므로 재spawn 후 `resumeSession`으로 복구 가능. P2에서 자동 복구 구현.

### 2.3 D112-2 일회용 세션 원칙 정합

D112-2 확정사항: jwc 세션은 일회용이며 자체 메모리 축적 없음. ACP 프로토타입에서의 정합:

- **memory.backend 확인**: `settings-schema.ts` 실사 결과 `memory.backend` 기본값은 `"off"`. 별도 `--no-memory` 플래그는 없음. 기본값 그대로 사용하면 consolidation/hindsight 모두 비활성. **추가 플래그 불필요**.
- gjc stage1/phase2는 `hindsight` 또는 `local` 백엔드 활성화 시에만 실행. 기본(`off`)이면 자동 실행 없음.
- cli-jaw가 `jwc acp`를 spawn할 때 `JWC_MEMORY_BACKEND=off` env를 명시적으로 주입하면 설정 파일과 무관하게 비활성 보장 (방어적 설계).

## 3. UI 구성요소 매핑

### 3.1 작업장 피커

```
[ Local ] [ /path/to/repo ] [ main ] [ worktree: feat/x ]
```

| 요소 | 데이터 출처 | 구현 |
|---|---|---|
| "Local" | 고정 레이블 (홈 디렉터리 또는 최근 폴더) | 상태 없음 |
| 폴더 경로 | `folderPicker` 다이얼로그 (Electron `dialog.showOpenDialog`) / 웹은 `<input type="file" webkitdirectory>` | — |
| 브랜치 | 서버 `GET /api/code/git-info?cwd=…` → `{branch, repoName, worktrees[]}` | G2 해결 |
| worktree 목록 | 동일 API `worktrees[]` | G1 해결 |

**서버 구현 스케치**:
```typescript
// cli-jaw/src/routes/code.ts
router.get("/api/code/git-info", async (req, res) => {
  const cwd = req.query.cwd as string;
  const branch = execSync(`git -C "${cwd}" rev-parse --abbrev-ref HEAD`).toString().trim();
  const repoName = path.basename(execSync(`git -C "${cwd}" rev-parse --show-toplevel`).toString().trim());
  const worktrees = parseWorktreeList(execSync(`git -C "${cwd}" worktree list --porcelain`).toString());
  res.json({ branch, repoName, worktrees });
});
```

### 3.2 세션 사이드바

**데이터 흐름**: `_gjc/chats/byCwd` ext 메서드 → `SessionInfo[]` (sessionId, title, cwd, updatedAt, messageCount, size)

```
Recents
  ├── [세션 1 제목] ·  14 msgs  · 2h ago
  ├── [세션 2 제목] ·   3 msgs  · 1d ago
```

### 3.3 스트리밍 채팅 — 도구 셀 접기

**상태 머신 (각 toolCallId에 대해)**:

```
ACP 이벤트                    UI 상태
─────────────────────────    ────────────────────────
tool_call (pending)       →  [확장 셀] 제목 + kind 아이콘 + rawInput 스피너
tool_call_update          →  [확장 셀] + 부분 결과 스트리밍
  (in_progress)
tool_call_update          →  [컴팩트 셀] 제목 + status 아이콘 (접힘)
  (completed/failed)           클릭 시 diff/content 확장 가능
```

**컴포넌트 구조** (네이티브 React, D112-1 확정):

```tsx
// 도구 셀
<ToolCell
  toolCallId={id}
  title={title}
  kind={kind}             // 아이콘 매핑: read/edit/execute/search/…
  status={status}         // pending → in_progress → completed/failed
  locations={locations}   // 파일 경로 → Electron shell.openPath
  diff={diffContent}      // oldText/newText → monaco-diff 또는 인라인
  isExpanded={expanded}   // 클릭 토글
/>

// thinking 셀 — 동일 접기 패턴
<ThoughtCell
  messageId={messageId}
  text={accumulatedThought}
  isExpanded={thinkingExpanded}
/>
```

**커밋 폴딩 의미론 (083 문법 재현)**:
- `completed` 전환 시 셀을 즉시 컴팩트 상태로 접음.
- 사용자가 수동 전개한 셀은 `isExpanded` 로컬 상태로 유지.
- 세션 히스토리 리플레이 시 모든 과거 도구 셀은 즉시 컴팩트(`status: "completed"`) 상태로 렌더.

### 3.4 권한 다이얼로그

ACP 흐름:
1. `jwc acp` child → `requestPermission(toolCall, options)` → cli-jaw 서버 수신
2. cli-jaw 서버 → SSE `{type: "code:permission_request", ...}` → 클라이언트
3. 클라이언트 → 다이얼로그 표시 → 사용자 선택 → REST `POST /api/code/sessions/:id/permission`
4. cli-jaw 서버 → ACP `requestPermission` 응답 → jwc 실행 재개

**다이얼로그 구성**:
```
━━━━━━━━━━━━━━━━━━━━━
🔐 도구 실행 권한 요청
tool: edit · /src/index.ts
─────────────────────
[이 파일만 허용]  [세션 전체 허용]  [거부]
━━━━━━━━━━━━━━━━━━━━━
```

`options[].kind` → 버튼 레이블 매핑: `allow-once` → "이 파일만 허용", `allow-session` → "세션 전체 허용", `deny` → "거부".

**P1 제약**: `clientCapabilities.elicitation.form`은 P3까지 선언하지 않음 — 스킬 UI elicitation은 P1 범위 외.

### 3.5 모델 팝업 — ACP 위임 vs D113 REST

| 방식 | 장점 | 단점 |
|---|---|---|
| **ACP 위임** (`unstable_setSessionModel`) | Code 모드 세션 범위 정확. ACP 연결만 있으면 됨. 추가 REST 없음. | 세션 외부(새 세션의 기본 모델) 설정 불가. 사용 가능 모델 목록은 `configOptions`에서 읽음. |
| **D113 REST** (`/api/jwc/sessions/:id/model`) | 110/111 in-process 이후 아키텍처와 통일. 스냅샷+무효화 동기화 패턴 재사용. | 110/111이 게이트 → P1에서 사용 불가. |

**P1 결정**: ACP 위임 방식 채택. `configOptions`에서 `{id: "model", options: [{value, name}]}` 읽어 팝업 렌더. `/model` 슬래시 커맨드도 ACP `available_commands_update`에서 광고됨 — 대안 경로.

**P3 이후**: D113 REST로 마이그레이션 시 `CodeSessionTransport` 추상화 뒤에서 스왑 — UI 무변경.

```typescript
export interface CodeSessionTransport {
  // 세션 수명
```

### 4.2 구현체 전환 계획

| 구현체 | 시점 | 내부 동작 |
|---|---|---|
| `AcpStdioTransport` | **P1 ~ P2** | `jwc acp` child spawn + NDJSON + 서버 측 ACP SDK |
| `InProcessTransport` | **P3 (100/110 완료 후)** | in-process `AgentSession` + 이벤트 구독. `runAcpMode` 없이 직결 |

**스왑 완료 기준**: UI(React 컴포넌트)는 `CodeSessionTransport` 타입만 참조. `AcpStdioTransport` → `InProcessTransport` 교체 후 UI 파일 변경 0.

## 5. 단계 경로

### P1 — 단일 세션 e2e (ACP stdio)

**범위**:
- cli-jaw 서버에 `CodeSessionManager` + `AcpStdioTransport` 구현
- 서버 라우트: `GET /api/code/git-info`, `POST /api/code/sessions` (newSession), `POST /api/code/sessions/:id/prompt`, `DELETE /api/code/sessions/:id` (closeSession)
- 서버 SSE: `GET /api/code/events` — 기존 `/api/events` 패턴 그대로, `code:` prefix 이벤트 추가
- Electron 렌더러: Code 모드 탭 신설 (jaw ↔ Code 토글)
- 컴포넌트: `WorkspacePicker`, `ChatPane`, `ToolCell`, `ThoughtCell`, `PermissionDialog`
- 권한 다이얼로그 e2e (서버 중계)
- 모델 팝업 (ACP configOptions 위임)

**완료 기준**: `jaw 모드 → Code 탭 클릭 → 폴더 선택 → 새 세션 → 텍스트 입력 → 스트리밍 텍스트 + 도구 셀 렌더 → 도구 완료 후 컴팩트 접힘 → 권한 다이얼로그 동작`

### P2 — 멀티 세션 + 사이드바

**범위**:
- 세션 사이드바 (cwd별 목록, Recents, `_gjc/projects/list`)
- 멀티 세션 탭 (`newSession` 복수 → Map 관리)
- `resumeSession` + 히스토리 리플레이 렌더
- child idle-reap 자동 재spawn
- 세션 삭제 (`G5`)

**완료 기준**: 탭 2개 동시 활성 → 각 세션 독립 스트리밍 → 사이드바 세션 클릭으로 이전 대화 복원

### P3 — 권한 정교화 + 모델/thinking + elicitation

**범위**:
- elicitation form 지원 (`clientCapabilities.elicitation.form: {}` 선언)
- thinking 토글 UI
- 터미널 인라인 렌더 (xterm.js, `G6`)
- 즐겨찾기 (`G3`)
- D113 REST 모델 팝업 마이그레이션 (110/111 완료 후)
- `InProcessTransport` 스왑

## 7. 열린 질문

1. **G4 멀티세션 구조**: 단일 ACP child에서 `newSession` 복수 호출 시 child 내부 `ManagedSessionRecord` Map이 세션 격리를 보장하는가 → acp-agent.ts 실사 결과 `#sessions: Map<string, ManagedSessionRecord>` 확인됨. **격리 가능** — 단, 하나의 child가 오래 실행되면 child 재시작 시 모든 세션이 날아가므로 cli-jaw 서버가 sessionId를 DB에 기억해야 함.
2. **권한 다이얼로그 동시 요청**: 두 세션이 동시에 권한 요청 시 UI가 큐로 처리해야 하는가, 아니면 세션별 별도 표시인가 → P2 착수 전 결정.
3. **웹(PWA) vs Electron 피커**: `dialog.showOpenDialog`는 Electron 전용. 웹은 `<input type="file" webkitdirectory>`가 대안이나 경로 문자열 취득이 제한됨 → P1은 Electron만 지원, 웹 경로 입력은 P2에서 수동 경로 텍스트 입력 폴백 제공.
4. **`memory.backend=off` 강제**: 기본값이 이미 `"off"`이므로 env 주입 없이도 consolidation 비활성. 사용자가 `~/.jwc/settings.json`에서 직접 `local` 또는 `hindsight`로 바꾼 경우에도 cli-jaw spawn 시 `JWC_MEMORY_BACKEND_OVERRIDE=off`를 주입해야 D112-2 보장 → **P1 구현 시 명시적 env 주입 채택**.
