> 소속: [99.00.00 MOC](./99.00.00_moc_stabilization.md) 패키지 99.03. 선행 진단: 050 밴드 구현 완료 후에도

> 입력: 사용자 "지침 패치가 많이 됐는데도 pabcd 진행해가 주입이 안 된다" + "/orchestrate P를 에이전트도
> 나도 실행할 수 있어야" + "주입 명칭도 ralplan이 아니라 pabcd 규격으로" + "워크플로 표면 전면 개정" [확정].

IPABCD 런타임(CLI·슬래시·상태머신·스테이지 프롬프트)은 050 밴드에서 전부 구현됐지만, **모델이 그 존재를
알 수 있는 층이 0개**다. 시스템 프롬프트는 "exactly four default workflow skills"로 다른 워크플로 라우팅을
금지하고, dev-pabcd 스킬은 jaw 브랜드에서 의도적으로 차단되며(skills.ts:164, R14), 스테이지 프롬프트는
사용자가 `/orchestrate <stage>`를 칠 때만 1회 주입된다. 실증: jwc에 "너가 할 수 있는 것들"을 물으면
ralplan/ultragoal/team 어휘로만 답한다 (260612 스크린샷). 본 플랜은 cli-jaw의 주입 구조를 동형 이식해
discovery(M1)·지속성(M2)·자가 전이(M3)를 복원하고, 표면 어휘를 IPABCD 규격으로 재편(re-facing)한다.

## 1. 조사 결론 (병렬 8기, 260612)

### cli-jaw 원본 — 4층 push 구조 (전부 jwc에 부재했음)

| 층 | cli-jaw 메커니즘 | 근거 |
|---|---|---|
| 상주 안내 | 시스템 프롬프트에 `orchestration.md` 섹션 + dev-pabcd 가이드 **경로 포인터**("⛔ 단계 전 필독") | builder.ts:540-553 |
| 자가 전이 | "**YOU advance phases by running the exact `cli-jaw orchestrate I\|P\|A\|B\|C\|D` shell command. No other method.**" — 모델이 Bash로 직접 전이 | orchestration.md:47 |
| 매 턴 헤더 | `getPrefix()` — I/P/A 유저 턴·A/B 워커 턴마다 `[PLANNING MODE — User Feedback]`류 prefix | pipeline.ts:417-419, state-machine.ts:240-261 |
| plan 재주입 | A/B/C 동안 매 턴 DB(`orc_state`)의 plan 전문 prepend → **컴팩션 면역** | pipeline.ts:421-424 |

스킬 본문·A/B/C/D 스테이지 프롬프트는 **pull**(모델이 읽기/CLI 실행으로 획득). 즉 원본 철학 =
"상주 안내는 작게, 상태 헤더는 매 턴, 본문은 pull".

### jwc 엔진 — 매 턴 주입 레일이 이미 4개 존재 (재사용 가능)

plan 모드(매 턴 PLAN.md custom message), goal 모드(`goal-mode-active.md` 렌더 + `<system-reminder>`
continuation), 메모리 `beforeAgentStartPrompt` 훅, TTSR. 신규 아키텍처 불요 — **custom message 레일에
`readPabcdState()` 리더만 얹으면 M2가 성립**한다.

### 가드 블래스트 — 산문 개정은 파괴 0건

- `"exactly four"`를 핀하는 테스트·스크립트 **없음**. 카운트 가드 3종(check-visible-definitions /
  verify-g002-gates / rebrand-inventory)은 **번들 스킬 디렉토리 수**만 검사 — orchestrate는 native 표면이라
  비대상 (R14 정합).
- 불변 문장 2개: `<decomposition>`의 "delegate bounded slices to \`executor\`",
  `<runtime-state>`의 "committed repo-visible \`.jwc\` defaults are not the source of truth"
  (default-gjc-definitions.test.ts:288-309) — **건드리지 말 것**.
- GJC 산문 금지 (agent-identity-leak.test.ts:45).
- system-prompt.md는 fork-delta HARD-EDIT 파일 (085.5-M2) → **fork-delta.md 동행 갱신 + 본 문서가
  "do not add" 규칙 개정의 페이퍼트레일**.

### 명칭 — hard rename ❌ / re-facing ⭕ [제안]

하드 리네임(ralplan→pabcd-p 등)은 14+ 파일 연쇄 + `.jwc/plans/ralplan/` 등 **디스크 경로 마이그레이션**까지
필요. re-facing(시스템 프롬프트 라벨에 IPABCD 우산만 씌우기)은 기능 식별자(`name=`/`user-entrypoint=`/
`cli-runtime=`/경로) 무변경으로 블래스트 0. D050-13/14·R14·D050-22 전부 호환.

## 2. 설계

### M1 — 시스템 프롬프트 `<jwc-runtime>` 전면 개정 (discovery)

`packages/coding-agent/src/prompts/system/system-prompt.md:19-84`:

1. **orchestrate 표면 등재**: `<public-workflow-surface>`에 5번째 블럭 — 단 스킬이 아니라 native 표면임을 명시:
   `<native-workflow name="orchestrate" user-entrypoint="/orchestrate <i|p|a|b|c|d>" cli-runtime="native: jwc orchestrate" alias="pabcd">`
   IPABCD 단계 문법·게이트(A: audit PASS, B: verification DONE)·상태 파일 요약.
2. **"exactly four" 개정**: "four default workflow skills **plus the native IPABCD orchestration surface**"
   + "Do not add ... without an explicit product decision" 유지 (본 문서 = 그 product decision).
3. **re-facing**: 4 스킬 산문에 IPABCD 우산 — "IPABCD I-stage engine."(jaw-interview),
   "IPABCD P-stage consensus engine."(ralplan), "goal ledger"(ultragoal), "B-stage coordinated
   execution"(team). 속성·invocation 문자열 무변경.
4. **`<routing>` 재편**: 각 행에 단계 라벨 prefix + 신규 행:
   - "User asks to run pabcd / 단계 진행 (e.g. 'pabcd 진행해', '/orchestrate p 해줘') → run
     \`jwc orchestrate <stage>\` via the shell tool yourself; the stdout IS the stage prompt — follow it."
   - cli-jaw 동형 자가 전이 문장: "YOU advance IPABCD phases by running the exact \`jwc orchestrate\`
     command. No other method."
5. 메모리 동사 행은 99.01 구현 커밋에서 추가 (본 M1은 자리만 설계).

같은 커밋: `structure/fork-delta.md` HARD-EDIT 행 갱신, 신규 테스트(아래 §검증).

### M2 — 매 턴 스테이지 헤더 재주입 (지속성) [채택 제안 — §결정 1]

- 위치: agent-session의 plan/goal 모드 custom message 경로와 동렬 (`#buildGoalModeMessage` 패턴).
- 동작: 턴 시작 시 `readPabcdState(cwd, sessionId)` (lenient, `orchestrate-state.ts:317`) → active면
  `[PABCD — B: BUILD · audit=pass]` 1줄 + 다음 액션 힌트(게이트 미충족 시)를 custom message로 prepend.
- 비용: 턴당 1~2줄 — cli-jaw `getPrefix()` 동형. 컴팩션 후에도 매 턴 재생성이라 생존.
- 경로는 `pabcdStatePath()` 헬퍼 경유 (β `.gjc→.jwc` 스윕과 자동 정합).
- plan 전문 재주입(cli-jaw Layer 3 동형)은 **M2b 후속** — `plan_ref` 파일 존재 시 a/b/c 단계에서 prepend.
  1차는 헤더만.

### M3 — 스테이지 프롬프트에 자가 전이 안내 보강

`prompts/jaw/orchestrate-*.md` 각 말미에 다음 단계 전이 명령 명시 (cli-jaw STATE_PROMPTS 동형):
"When the user approves, run \`jwc orchestrate a\` yourself." 현재는 사용자 슬래시 의존.

## 3. 결정 [확정 — 사용자 260612]

1. [확정] **M2 채택** — cli-jaw는 단계 prefix(I/P/A 유저 턴·A/B 워커 턴)와 plan 전문(A/B/C)을 **매 턴 push**하는
   것이 기본. M2는 옵션이 아니라 동형 이식의 필수 조각 (사용자 재확인: "cli-jaw는 매 턴 주입되는 거 아니야?").
2. [확정] **re-facing** — 명칭은 소개 문구만 IPABCD 우산으로 재편, 기능 식별자·디스크 경로 무변경.
   hard rename(14+ 파일 + `.jwc/plans/ralplan/` 마이그레이션)은 별도 밴드 후보로만 보존.

## 4. 완료 기준

- 신규 세션에서 "pabcd 진행해" → 모델이 \`jwc orchestrate i|p\`를 스스로 실행하고 스테이지 프롬프트를 따름 (e2e).
- "너가 할 수 있는 것들" 질문에 IPABCD 문법이 1급으로 등장 (260612 스크린샷 회귀 확인).
- 스테이지 active 상태에서 컴팩션 후에도 다음 턴에 단계 컨텍스트 유지 (M2).
- 기존 가드 green: agent-identity-leak / default-gjc-definitions(불변 문장 2개) / 카운트 가드 3종 무변경 통과.
- 신규 테스트: system-prompt orchestrate 표면 존재 + routing 행 + (M2) 헤더 주입 단위 테스트.
