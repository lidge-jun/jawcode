## 1. 엔진 사실 — 핵심 발견

**local memories는 "검색 가능한 메모리 DB"가 아니라 "자동 요약 파이프라인"이다.**

- 활성 조건: `memory.backend === "local"` 또는 legacy `memories.enabled` (`memories/index.ts:1089-1095`); `localBackend.buildDeveloperInstructions()`가 `buildMemoryToolDeveloperInstructions()`로 위임(`memory-backend/local-backend.ts:16-23`)
- **stage1**: 과거 세션 rollout jsonl → 모델 추출 `{raw_memory, rollout_summary, rollout_slug}` → `stage1_outputs` upsert (`index.ts:232-260`, `storage.ts:327-339`; claim 조건 `storage.ts:136-245` — `source_kind cli|app`만, idle 12h+, 30일 이내, 최대 64건)
- **phase2**: cwd별 워터마크 잡(`storage.ts:247-260`) → `listStage1OutputsForGlobal()` (`storage.ts:492-520`) → consolidation 모델 → **`MEMORY.md` + `memory_summary.md` + `skills/` 재생성**(`index.ts:720-815`)
- **read-path**: `memory_summary.md`를 5000토큰 한도로 잘라 developer instructions로 매 프롬프트 리빌드에 주입 (`index.ts:150-175`, `sdk.ts:1570`, `memory-backend/types.ts:38-46`)
- **SQLite 스키마 3테이블뿐**: `threads` / `stage1_outputs` / `jobs` (`storage.ts:47-88`) — **FTS 없음, 임베딩 없음, 검색 인덱스 없음**
- backend 공통 인터페이스(`memory-backend/types.ts:26-78`): `start/buildDeveloperInstructions/clear/enqueue(+optional beforeAgentStartPrompt/preCompactionContext)` — **search/read/save 동사가 인터페이스에 없음**
- hindsight(원격 백엔드)에는 있음: `recall/retain/retainBatch/reflect/listMemories/getDocument…` (`hindsight/client.ts:228-389`)

## 착수 전 실측 보강 (260612 조사)

260612 병렬 코드 조사(`structure/memory_pipeline.md`)로 확인된 사실을 구현자 트립 방지용으로 정리한다. 설계 방향은 변경 없음 — 접합점 정밀도만 보강.

### 1. 검색면은 전부 신규

코드베이스 어디에도 메모리 검색 함수가 없다. `stage1_outputs`에 FTS/LIKE 래퍼가 없고, `buildMemoryToolDeveloperInstructions`(`memories/index.ts:150`)는 `memory_summary.md` 1파일만 읽는다. `memories/local-query.ts`의 `searchLocalMemories` / `readLocalMemoryArtifact` / `saveLocalMemoryManual` / `buildLocalTaskSnapshot` / `contextLocalMemory` 전부 from scratch 신규 작성이다.

### 2. manual save 함정

`markStage1SucceededWithOutput`(`storage.ts`)은 실행 중 잡의 `ownershipToken`을 요구하므로 manual 행에 부적합하다. `saveLocalMemoryManual`은 `stage1_outputs`에 **직접 `INSERT...ON CONFLICT` SQL** + `upsertThreads`(`storage.ts:103`) + `enqueueGlobalWatermark`(`storage.ts:247`, `forceDirtyWhenNotAdvanced:true`) 조합으로 구현한다. 저장 후 `session.refreshBaseSystemPrompt()`(`agent-session.ts:3805`)를 호출해야 즉시 반영된다.

### 3. Task Snapshot(M6) 시그니처 변경 필요

`buildMemoryToolDeveloperInstructions`는 현재 프롬프트 텍스트에 접근할 수 없다. M6 구현 시 optional `promptText` 파라미터를 추가하거나, M1 라우터에서 별도 호출하는 방식으로 최소 변경한다. 신규 템플릿 파일은 `prompts/memories/task-snapshot.md`(M7)로 분리한다.

### 5. chat search

`collectThreads`는 jsonl 헤더 1줄만 읽는다. 세션 본문 검색은 **본문 grep 헬퍼를 신규** 작성해야 한다. 세션 디렉토리 접근은 `getSessionsDir` + `session.sessionManager.getSessionDir()`(`session-manager.ts:2696`) 조합으로 한다.

cli-jaw 원본은 활성 세션 한정 LIKE이지만, jwc는 세션 횡단 jsonl grep이라 커버리지가 오히려 넓다. 출력 포맷은 cli-jaw의 `[created_at] (role) content[:300]` 동형을 권장한다.

### 6. cli-jaw 패리티 주의 2건

**(a) 동의어 시드 하드코드 고려** — cli-jaw `memory_synonyms` 테이블에 `pabcd ↔ plan/audit/build/check/done` 1그룹이 있다. LIKE 1차에서도 이 그룹만큼은 하드코드 expand를 기본값으로 고려한다(FTS5 도입 전까지 한국어 업무 검색어 히트율 보강).

**(b) recency 계산 방식 교체** — cli-jaw는 파일명 날짜 regex에 의존한다. jwc는 `stage1_outputs.generated_at` 컬럼으로 대체하면 더 견고하다(파일명 규약 비의존).
