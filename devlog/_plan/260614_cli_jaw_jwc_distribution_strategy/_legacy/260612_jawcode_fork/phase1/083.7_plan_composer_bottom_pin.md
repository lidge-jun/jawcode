지금 jwc TUI에서 컴포저(상태레일+에디터+풋터 묶음)는 "프레임의 마지막 줄들"이다. 프레임이
터미널보다 짧으면 입력창이 내용 끝에 떠 있고, 위쪽 셀이 자라거나 접힐 때마다 입력창이 같이
오르내린다(083.6). 이 계획은 트랜스크립트와 컴포저 사이에 **뷰포트 잔여 높이만큼 늘어나는
스페이서(ViewportFill)** 를 끼워 입력창을 항상 터미널 바닥에 고정한다 — Claude Code/Codex와
같은 모델. 내용은 위에서 아래로 스페이서를 잠식하며 자라고, 접힘은 스페이서가 도로 늘어나는
것으로 흡수되므로 **컴포저 행은 상수**가 된다. 차등 렌더러의 핵심 불변식("한 번 방출된 줄은
움직이지 않는다")을 깨지 않는 것이 설계의 중심이다.

## 0. 설계 결정 — 왜 "사이 스페이서"(B1)이고 "내용 하단 흡착"(B2)이 아닌가

| 변형 | 배치 | 차등 렌더 비용 | 판정 |
|---|---|---|---|
| **B1 (채택)** | `[트랜스크립트][▼ViewportFill▼][컴포저]` — 내용은 위 고정, 컴포저는 바닥 고정 | append = 빈 줄 k개가 내용으로 바뀜(국소 diff). 기존 줄 불이동 | ✅ |
| B2 (기각) | `[▼여백▼][트랜스크립트][컴포저]` — 최신 출력이 항상 입력창 바로 위 (CC 체감과 더 유사) | append마다 **트랜스크립트 전체가 위로 시프트** → 매 렌더 전화면 diff, 스크롤백 일관성 모델(움직이지 않는 줄 = 스크롤백 후보) 붕괴 | ❌ 렌더러 재설계급 |

B1에서도 사용자 보장("입력창이 안 움직인다")은 동일하게 성립한다. B2의 체감(출력이 입력창에
붙음)은 프레임이 뷰포트를 넘는 순간부터는 B1도 자연히 동일해진다(스페이서=0, 하단 앵커).
차이는 짧은 세션 초반뿐이며, 그 구간을 위해 렌더러 불변식을 깨는 것은 비대칭 비용이다.

**프레임 조립** — `tui.ts:1117 #doRender()`:

```
:1120  height = terminal.rows
:1132  newLines = this.render(width)        ← TUI extends Container, 자식 줄 평탄화
:1136  overlayStack 합성 (#compositeOverlays)
```

핵심 제약: `Component.render(width)`는 **height를 모른다**. 잔여 높이 계산은 TUI 레벨에서만
가능 → 컴포넌트는 센티널 1줄만 방출하고, TUI가 조립 직후 실제 빈 줄 N개로 치환하는 구조가
필요하다(커서 마커와 같은 수법, 전례 있음).

**오늘의 출렁임 소비자들** (패치 후 영향 확인 대상):

- `clearOnShrink` (`tui.ts:1280` 부근): `newLines.length < previousLines.length`면 fullRender(true).
  → 핀 적용 시 프레임≤뷰포트 구간에서 총높이가 **항상 height로 상수**가 되어 이 경로가 그
  구간에서 아예 발화하지 않게 된다 (부수 효과: 전체 클리어 깜빡임 감소 — 086.2와 시너지).
- `#extractCursorPosition`/`#cursorControlSequence`: 최종 newLines 기준 절대행 계산 — 치환을
  커서 추출 **이전**에 끝내면 무수정으로 정합.
- 오버레이: `#compositeOverlays(newLines, width, height)` — 뷰포트 기준 배치이므로 치환 후
  합성하면 오히려 좌표가 안정된다 (치환을 :1136 합성 **이전**에 수행해야 하는 두 번째 이유).

## 2. 구현 설계

### 2.1 NEW `packages/tui/src/components/viewport-fill.ts`

```ts
/** Sentinel line a ViewportFill renders; TUI expands it to the viewport remainder. */
export const VIEWPORT_FILL_SENTINEL = "\x00__jwc_viewport_fill__\x00";

export class ViewportFill implements Component {
    #enabled = true;
    setEnabled(v: boolean): void { this.#enabled = v; }
    invalidate(): void {}
    render(_width: number): string[] {
        return this.#enabled ? [VIEWPORT_FILL_SENTINEL] : [];
    }
}
```

- 센티널은 제어문자 포함 고유 문자열 — 실내용과 충돌 불가, `visibleWidth` 계산 대상이 되기 전에
  치환되므로 폭 안전.
- `setEnabled(false)`면 0줄 — 설정 off/레거시 모드에서 컴포넌트 제거 없이 무력화.

### 2.2 `tui.ts` — 센티널 치환 (조립 직후, 오버레이 합성 전: :1132와 :1136 사이)

```ts
newLines = this.#expandViewportFill(newLines, height);
```

`#expandViewportFill(lines, height)` 규칙:

1. 첫 번째 센티널만 유효 — `fill = max(0, height - (lines.length - 1))` 개의 `""` 로 치환.
   (`-1` = 센티널 자신. fill 0이면 흔적 없이 제거.)
2. 두 번째 이후 센티널은 무조건 제거 (오용 가드 — 중첩 컨테이너에서 실수로 2개 꽂힌 경우).
3. 센티널이 없으면 원본 반환 (비활성/레거시 경로 zero-cost).
4. `renderMetrics`에 `viewportFill` 헬퍼 타이밍 기록 (기존 recordHelper 패턴).

치환 위치가 :1136 이전이어야 하는 이유: ① 오버레이가 최종(뷰포트 크기) 프레임 위에 합성,
② 커서 마커 추출(:1141)이 최종 절대행으로 계산, ③ 이후 모든 차등/풀렌더 분기가 치환 결과만
보므로 **렌더러 코어 분기는 단 한 줄도 수정하지 않는다**.

### 2.3 `interactive-mode.ts` — 마운트 (1줄 삽입 + 설정 와이어)

```ts
this.ui.addChild(this.chatContainer);        // :516 기존
this.ui.addChild(this.#viewportFill);        // NEW — 클러스터 전체(:517-524)가 바닥으로
this.ui.addChild(this.pendingMessagesContainer);

```

- pending/status/todo/btw를 클러스터에 포함시키는 이유: 이들의 높이 변동(예: todo 갱신)도
  스페이서가 흡수해 에디터가 안 움직인다. 단 이들이 자라면 채팅 가시 영역이 줄어드는 것은
  지금과 동일 (바닥 기준 위로 자람).
- `updateEditorChrome()`·`#syncEditorMaxHeight` 등 기존 에디터 지오메트리 경로 무수정.

### 2.4 설정 + 이스케이프 해치

- `settings-schema.ts`: `tui.composerPin: boolean`, **default true**, ui tab "interface",
  설명 "Pin the composer to the terminal bottom (fills the gap above with blank lines)".
- 적용: interactive-mode 초기화에서 `this.#viewportFill.setEnabled(settings.get("tui.composerPin") !== false)`,
  설정 변경 핸들러에서 토글 + `requestRender(true)`.
- env `PI_NO_COMPOSER_PIN=1` → 강제 off (디버깅/리그레션 비교용, `$flag` 패턴).
- **gjc 모드 diff-0**: ViewportFill은 jwc 마운트 경로에만 꽂히고 tui 코어 변경은 센티널이
  없으면 no-op이므로, 업스트림 리베이스 충돌면은 interactive-mode 1줄 + tui.ts 1줄 + 신규 파일.

## 11. B1 → B2-lite 전환: fill을 채팅 위로 (260612 23:3x — goal 지시 버그 2건의 공통 해소)

사용자 goal 보고 2건 — ① "접기 로직 때문에 맨 위 상단으로 올라가는 버그", ② "슬래시 커맨드 화면이
채팅창 아래(=입력창 근처)가 아니라 상단에 렌더, 짧은 TUI에서 버튼 안 보임" — 은 **둘 다 B1 배치의
상단 정렬이 원인**이다: 핀이 컴포저만 바닥에 내리고 콘텐츠는 상단에 남겨, 압축/접힘 후 콘텐츠가
패널 꼭대기로 가고(①), 슬래시 출력(chatContainer append)이 입력창에서 멀리 떨어진다(②).

**§0의 B2 기각 근거 재검토 → 무효 판정**: "append마다 전화면 diff + 스크롤백 모델 붕괴"는
fill이 양수인 구간(frame ≤ viewport)에만 줄 이동이 발생한다는 사실을 놓쳤다 — 그 구간에서는
**아무 줄도 스크롤백에 진입한 적이 없으므로** 일관성 모델이 깨지지 않고, diff 비용도 터미널 높이로
유계(수십 행)라 무시 가능하다. fill이 0이 되는 순간 줄 위치가 동결되고 이후는 표준 스크롤.

**구현 (1줄 이동)**: `interactive-mode.ts` 마운트 순서를 `[banner][**fill**][chat][pending…composer]`로

1. 초과 구간 floor 갭(§9)이 이제 **상단(스크롤백 쪽)에 적립**되어 뷰포트에서 안 보임 — §10 압축의
   역할이 "시각 복구"에서 "스크롤백 블랭크 청소 + floor 무한성장 방지"로 격하 (트리거는 유지).
2. 짧은 세션에서 환영 배너는 상단, 대화+입력은 하단 — 표준 데스크톱 채팅 배치.
