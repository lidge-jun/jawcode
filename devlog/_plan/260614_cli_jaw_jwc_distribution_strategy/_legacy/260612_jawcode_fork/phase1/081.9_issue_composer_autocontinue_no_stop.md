# 081.9 — issue: 턴 종료 후에도 안 멈춤 — 컴팩션 auto-continue × composer 과잉 진행

> 상태: 🔨 옵션 E 구현 완료 (2026-06-12, fork — §6). A/C/D는 보류: 사용자 판단 "auto-continue는 정상 로직".
> 소속: 081 밴드 (composer 행동 규율). 관련: [081.8](./081.8_issue_composer_anchor_fabrication.md) (앵커 규율), 081.2 (환각 타이틀).
> 입력: 사용자 "턴이 끝나고 안 멈춰서, 멈춰야 되는데 '다음 작업으로 이어가겠습니다' 이런 일이 발생 — 조사만."

composer로 작업하다 보면 할 일을 다 끝내고 결과 보고까지 한 턴이 **혼자 다시 살아나서**
"다음 작업으로 이어가겠습니다"라며 새 일을 벌인다. 화면에는 `⠦ Auto context-full
maintenance… (esc to cancel)` 스피너가 떠 있다. 분석 결과 이건 모델 단독의 폭주가
아니라 **3층 합작**이다: ① jwc가 threshold 컴팩션 성공 후 기본 설정으로 "Resume
work…" developer 메시지를 주입해 에이전트를 강제 재가동하고(`compaction.autoContinue`
기본 true), ② composer가 그 프롬프트의 "할 일 없으면 짧게 말하고 끝내라" 가드를
무시한 채 끝낸 일을 재검증·확장하며, ③ 우리가 넣은 모델 엔트리의 `maxTokens: 64000`이
컴팩션 임계값을 컨텍스트의 68%(136k)까지 끌어내려 이 사이클이 **자주** 돈다.

## 2. 메커니즘 — 누가 "계속해"라고 했나

**① 호스트 설계 (주범)**: threshold 유지보수는 *성공한 턴 직후*에만 발동하는데, 성공
직후 `compaction.autoContinue !== false`이면 무조건 auto-continue 프롬프트를 developer
메시지로 넣고 `agent.continue()`로 재가동한다 (gjc 내장 문서 compaction.md, threshold
maintenance 절). 즉 "턴이 끝났는데 다시 시작"은 **버그가 아니라 기본 동작**이고, 멈출지
말지는 전적으로 프롬프트 속 가드 문장("nothing left → say so")에 대한 **모델 판단**에
맡겨져 있다. 사용자 설정 파일이 없으므로(`~/.gjc`에 settings.json 부재) 전부 기본값.

**② 모델 습성 (공범)**: composer는 그 가드를 무시하고 "최근 의도 재개"를 "작업 재검증
+ 확장 명령"으로 해석한다. 38회 도구 호출이 그 증거. 081.2(환각 타이틀)·081.8(앵커
날조)과 같은 "지시보다 관성" 계열 습성이다. goal 모드 컨텍스트(rec 78 `<goal_context>`
"Goal mode is active… the task to pursue")가 요약을 거쳐 살아남아 "추구할 objective"를
계속 공급하는 것도 연료가 된다.

**③ 빈도 증폭기 (설계 불일치 — 우리 손 + 업스트림)**: 컴팩션 임계값은
`contextWindow − effectiveReserveTokens` 이고, reserve는
`max(15%·window, reserveTokens=16384, maxOutputTokens)` 다
(`packages/agent/src/compaction/compaction.ts:214,239`). 우리가 #518로 넣은 엔트리가
`contextWindow: 200000, maxTokens: 64000`이라 reserve가 **64k**로 뛰며 임계값이
**136k(68%)** 까지 내려온다. rec 421의 160.9k는 이를 한참 넘긴 값 — 긴 세션에선
컴팩션→auto-continue→재검증(토큰 추가 소모)→다시 임계 초과의 순환이 빨라진다.

**③-보강 (후속 분석)**: "모델별로 봐야 하는 것 아닌가"를 추적한 결과, reserve는 이미
모델별 값(`agent-session.ts:6521`의 `this.model?.maxTokens`)을 쓴다. 진짜 문제는
**예약과 실제 요청이 서로 다른 수를 본다는 것**:

| 측 | 수식 | composer 값 |
|---|---|---|
| 컴팩션 예약 | `model.maxTokens` (카탈로그 최대치) | **64,000** |
| 실제 요청 출력 캡 | `options.maxTokens \|\| min(model.maxTokens, 32000)` (`packages/ai/src/stream.ts:566`) | **32,000** |

즉 jwc는 한 번에 32k 넘는 출력을 요청한 적이 없는데 컴팩션은 64k를 비워둔다 —
컨텍스트 32k가 영구 낭비되고 임계가 136k로 불필요하게 내려온다(정합 시 168k).
이 불일치는 **업스트림도 동일**(_upstream_gjc `stream.ts:567`, `agent-session.ts:6515`
`maxOutputTokens = this.model?.maxTokens ?? 0`)하며, 카탈로그 maxTokens가 32k를 넘는
모든 모델(클로드 64k, gpt 계열 128k 출력 등)에서 임계가 과도하게 내려가는 일반 결함이다.
참고로 모델별 튜닝 노브는 설정엔 없고(`compaction.thresholdTokens`는 전역) 사실상
카탈로그 `maxTokens`가 유일한 모델별 입력이다.

채택 결과: **E 구현** (사용자 "32k 이 로직 개선해"). A/C/D 보류 — auto-continue는 정상 로직으로 인정,
문제는 "기다려야 할 때도 그냥 진행"하는 composer 판단력 쪽이므로 빈도 감소(E)로 1차 대응.

## 6. 구현 (2026-06-12, fork)

단일 소스 헬퍼 `effectiveMaxOutputTokens(model, requested?)` 신설
(`packages/ai/src/stream.ts`, `DEFAULT_MAX_OUTPUT_TOKENS_CAP = 32000`, `@gajae-code/ai`로 재수출):

| 호출처 | 변경 전 | 변경 후 |
|---|---|---|
| `stream.ts` `mapOptionsForApi` (실요청) | `options?.maxTokens \|\| min(model.maxTokens, 32000)` 인라인 | 헬퍼 호출 (시맨틱 동일) |
| `agent-session.ts` 컴팩션 체크 | `this.model?.maxTokens ?? 0` | `effectiveMaxOutputTokens(this.model)` |
| `context-usage.ts` 상태줄 임계·버퍼 ×2 | `model?.maxTokens ?? 0` | 동일 헬퍼 — UI와 실동작 일치 |

검증: 신규 `packages/ai/test/effective-max-output-tokens.test.ts` 5/5 pass (캡·통과·명시 요청 우선·
`\|\|` 시맨틱 보존·composer 임계 168k 수식 확인). 회귀: coding-agent 컴팩션 5파일 55 pass,
status-line 7파일 52 pass, ai 전체에서 신규 실패 0 (http-inspector 2건은 무변경 기준에서도 실패하는
환경 의존). biome clean. anthropic thinking 예산 경로(stream.ts:634, 32k 초과 가능)는 의도적으로
미정렬 — 컴팩션이 살짝 일찍 도는 안전한 방향이고 composer/xai엔 비해당.
