## 2. 검색 엔진 — FTS5 3-테이블 + 랭킹 공식

스키마 (`indexing.ts:128-150`): `chunks`(본문+relpath/kind/줄범위/해시) + `chunks_fts`(fts5, `tokenize='unicode61'`) + `chunks_trigram`(fts5, `tokenize='trigram'` — **CJK 전용 경로**) + `memory_synonyms`.

- 청킹: md를 `#`/`##`/`###` 헤딩 단위로 분할, 각 청크 앞에 `Source:/Kind:/Header:` 메타 prefix (`indexing.ts:45-120`)
- 검색 흐름 (`indexing.ts:429-561`): CJK ≥3자→trigram / Latin→BM25(+synonym 확장, 최대 16어)+trigram을 **RRF(k=60)** 융합 / 실패 시 LIKE fallback
- 최종 점수 = BM25 + kind 우선치(§1 표) + 반감기 recency boost + exact(-2.0)/header phrase(-1.0) — 낮을수록 좋음, 상위 8 반환 (`indexing.ts:310-354`, `runtime.ts:116`)
- 출력 포맷: `<relpath>:<start>-<end>\n<snippet ≤700자>` (`indexing.ts:356-365`)

**071 보강 [확정]**: gjc local 검색 신규 구현(071 §2 `searchLocalMemories`)은 1차에서 **SQL LIKE + artifact scan + kind 우선치 + recency 2요소**만 이식한다. FTS5/RRF는 후속. CJK trigram 경로는 1차 무대응 — LIKE가 한국어에도 자연 동작하므로 별도 tokenizer를 도입하지 않는다. 결정 근거: 인터뷰 260612 01:36.

## 5. 쓰기 파이프라인 — flush·reflect (gjc stage1·phase2의 대응물)

| cli-jaw | 메커니즘 | gjc 대응 |

| **flush** (`src/agent/memory-flush-controller.ts:85-154`) | 어시스턴트 응답 `flushEvery`(기본 10)건마다 별도 에이전트가 1-3문장 요약 → `episodes/live/YYYY-MM-DD.md`에 `## HH:MM` append, 무가치면 SKIP | **stage1** (idle 12h+ 세션에서 추출, 071 §1) — 트리거가 다름(턴 카운트 vs idle) |
