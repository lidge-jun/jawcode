# 99.20.01 — research: 선다형 ask 입력 UX 통일 (meta 경로 전역화)

> 상태: ✅ **구현 완료 (260612 밤)** — §7 구현 기록. 조사: Sonnet 서브에이전트 코드 분석.
> 소속: [99.20.00 MOC](./99.20.00_moc_frontend_uiux.md). 관련: [082.3 v2](./082.3_plan_interview_ask_docked_input.md) ✅.
> 결정: **[확정 260612 — 사용자] 선다형 ask는 어느 상황에서든 meta(v2) 입력 UX로 통일** —
> "선다로 내는 걸 meta로 하도록, 어느 상황에서든. 이게 UX 통일성에 맞다."
> 증거: Screenshot 2026-06-12 at 10.10.32 PM.png — 인터뷰 세션 중 meta 누락 ask가
> 번호 없는 옵션 + 레거시 `Other (type your own)` 2단계 경로로 렌더된 실측.

082.3 v2(번호 옵션 + 리스트 마지막 슬롯 인라인 composer 입력)는 완성됐지만, 활성 게이트가
`isStructuredInterviewQuestion(q.meta)` 단일 판정이라 모델이 meta를 빼먹으면 같은 선다형
질문이 전혀 다른 UX(레거시 Other 2단계)로 떨어진다. 같은 인터랙션이 호출 파라미터 하나로
두 얼굴을 갖는 것 — 본 조사는 이 게이트를 풀어 **모든 single-select ask가 v2 UX**가 되도록
하는 변경의 전모를 코드 전수 분석으로 확정했다. 082.3 §5의 "일반 ask Other 회귀 없음" 비목표는
본 결정으로 **폐기**된다 (일반 ask도 v2로 끌어올림).

핵심 한 줄: **`ask.ts:576`의 `shouldNumberOptions = jawInterviewPrompt !== null`을
`!(q.multi ?? false)`로 교체하는 것이 런타임 변경의 전부.** 인터뷰 헤더(라운드 띠)는 meta
없이는 만들 수 없으므로 기존 meta 게이트를 유지 — "입력 UX"와 "인터뷰 헤더"의 게이트 분리가
설계의 본질이다. 작업 부하의 80%는 `ask.test.ts`의 레거시 패턴 교체.

| 동작 | 위치 | meta 없이 가능? | 처분 |

| (b) 헤더 `formatInterviewSelectorPrompt` | `ask.ts:575` (`structured-renderer.ts:61-78`) | ❌ `meta.kind/round/component` 직접 접근 | meta 게이트 유지 |

| (f) selectedOptions 역매핑 | `ask.ts:603-608` | ✅ (a)에 종속 | 자동 동반 |

## 3. 확정 설계

```ts
// ask.ts:576 — 변경 전
const shouldNumberOptions = jawInterviewPrompt !== null;
// 변경 후: 모든 single-select에 번호+v2 슬롯
const shouldNumberOptions = !(q.multi ?? false);
```

- `displayQuestion = jawInterviewPrompt ?? q.question` — **유지** (헤더는 meta 게이트)
- `scrollTitleRows` — `jawInterviewPrompt !== null` 게이트 **유지**
- multi-select — Other 행 + 레거시 경로 **현행 유지** (D082-3-6 승계; v2 multi는 추후 별건)
- `extension-ui-controller.ts:35` dead `OTHER_OPTION` 제거
- `hook-selector.ts`·`jaw-interview-gate.ts` — **무변경**

## 8. 후속 수정 (260612 밤 — 사용자 실사용 피드백 2건)

전역화 직후 실사용에서 v2 슬롯 자체의 잔존 문제 2건 발견 (082.3 때부터 있던 것이 노출 확대로 표면화):

1. **슬롯 라벨** — `N. 출력창` → **`N. Type your own`** 통일 (`ask.ts` 명시 라벨 + `hook-selector.ts:539`
   기본값 + extension-ui-controller 주석). 한국어 하드코딩 제거 — 레거시 Other 문구 계승.
2. **이중 하이라이트 버그** — 슬롯에 포커스/입력 중일 때 마지막 옵션(4번)도 선택된 것처럼 렌더.
   원인: `#updateList()`가 슬롯 포커스 시 `FocusAwareList`에 마지막 옵션 인덱스를 **포커스로** 전달
   (스크롤 앵커 용도였으나 렌더러가 구분 못 함). 수리: `setState(..., slotFocused)` 플래그 신설 —
   슬롯 포커스 중 옵션 행은 전부 비선택 렌더(커서·accent 제거), 스크롤 앵커는 유지. 커서는 슬롯
   헤딩에만 1개.

**검증**: `hook-selector-list-slot.test.ts` 라벨 갱신 + 이중 커서 회귀 테스트 신규 1건
("clears the option cursor while the output panel is focused") — ask·hook-selector 4스위트
**62 pass / 0 fail** · tsc exit 0.

3. **박스 버킷 렌더 (∪자 — 상단 보더 소실)** — 슬롯 포커스 시 옵션 박스 윗줄이 사라져 버킷이 됨
   (스크린샷 260612 밤 2). 컴포넌트 출력은 무결함을 확인(단독·VirtualTerminal 전체 파이프라인·오버플로
   레짐 3중 재현 — 전부 정상) → **086 계열 diff 아티팩트**가 슬롯 포커스의 ±1행 높이 변화로 트리거되는
   것으로 판정 (스크린샷 상단의 깨진 텍스트 잔상도 동일 계열 증거). 수리: **셀렉터 높이 고정** —
   `#syncOutputPanelPresentation`에서 에디터 행을 포커스 여부와 무관하게 상시 렌더 (비포커스 에디터는
   커서 마커를 안 내보냄 `editor.ts:761`). 082.3 §8.1 목업 원안(슬롯에 composer 박스 상시 표시)과도
   일치. 트리거 제거이며 렌더러 근본 원인은 086.1/086.2 소유 유지.
4. **에디터 행 유령 말줄임 (`…` + 우측 보더 어긋남)** — `Bun.stringWidth`가 제로폭 커서 마커
   APC(`\x1b_pi:c\x07`)를 **5칸**으로 계산 → `wrapOutline`이 에디터 행을 실제보다 넓다고 보고
   5칸 잘라내고 `…` 부착 (실측 재현). 수리: `fitOutlineRow()` 신설 — 마커 제외 폭으로 측정·패딩,
   마커는 보존 (`OutlinedList`·`FocusAwareList` 공용).

**검증 2차**: 높이 고정+박스 닫힘+`…` 부재 회귀 테스트 신규 1건 ("keeps a constant height and a
closed box when slot focus toggles") — 관련 6스위트 **81 pass / 0 fail** · tsc exit 0 ·
VirtualTerminal 오버플로 레짐에서 open→슬롯 포커스 프레임 높이 동일 확인.

5. **박스 모서리 미연결 (진짜 정체 — 사용자 진단 260612 밤)** — "박스 자체가 안 이어지는 거였네":
   셀렉터 outline이 `┌┐└┘` 모서리 없이 **풀폭 가로선 2개 + 세로바**로 그려져, 터미널 폰트에서
   분리된 선 3개로 읽힘 (3·4의 버킷 체감의 실체 — 같은 화면의 DynamicBorder 구분선들과 구분 불가).
   컴포저는 `setClosedBorderBox`로 이미 모서리 박스를 쓰고 있어 표면 간 불일치이기도 함
   ([99.20.02](./99.20.02_audit_interaction_grammar.md) 계열). 수리: `outlineBoxChrome()` 신설 —
   상단 `┌─…─┐`·하단 `└─…─┘`로 연결 (`OutlinedList`·`FocusAwareList.#wrapOutline` 공용).
   `BASELINE_OUTLINED_RENDER_80_STRIPPED` 바이트 베이스라인도 모서리 기준으로 갱신
   (박스 상/하단 2행만 변경 — 공유 소비처 plan-mode·세션삭제 확인 등 동일 적용).

**검증 3차 — 실 PTY e2e (260612 밤, "아직도 안 고쳐짐" 재보고 대응)**: 실제 `jwc` 바이너리를
tmux PTY(170×40, TMUX unset + TERM_PROGRAM=vscode)에서 기동, 임시 `/asktest` 디버그 커맨드로
`ctx.showHookSelector` 실경로 호출 → **박스 4변 완전 닫힘 · 슬롯 입력(`> hello▏`) 중에도 유지 ·
`…` 없음 · 단일 커서** 전부 확인 (디버그 커맨드는 검증 후 제거). 재보고 스크린샷(23:20)은
픽스(23:15) **이전에 기동된 프로세스**의 화면 — bun은 프로세스 시작 시 소스를 로드하므로
워크스페이스 symlink여도 실행 중 세션에는 미반영. **jwc 재시작으로 해소.** 부가 확인:
TUI 파이프라인의 `#extractCursorPosition`이 truncate 전에 마커를 제거하므로 (`tui.ts:1209-1216`)
파이프라인 레벨 폭 계산은 추가 수정 불요 — fitOutlineRow(컴포넌트 레벨)로 충분.
