# 083.6 — issue: 자동접힘이 입력창을 위아래로 출렁이게 함 (조사 기록)

> 상태: ✅ **해소 (260612 밤)** — 옵션 B(하단 고정 컴포저) 구현 완료, [083.7 §8](./083.7_plan_composer_bottom_pin.md) 참조.

> 소속: 083 밴드 (출력 밀도). 원인 제공: [083.1](./083.1_plan_tool_autocollapse.md) 도구 자동접힘
> + [083.5](./083.5_plan_thinking_collapse.md) thinking 접힘. 관련: [086.2](./086.2_issue_scrollback_fragments_redraw.md) (clearOnShrink 재인쇄).
> 입력: 사용자 "텍스트 자동접힘 때문에 텍스트 입력창이 위아래로 왔다 갔다 해."

도구/thinking 자동접힘(083.1/083.5)은 셀 높이를 **턴이 진행되는 중간에** 줄인다. jwc TUI에서
입력창(컴포저)은 화면 하단에 고정된 위젯이 아니라 **트랜스크립트 바로 뒤에 이어 그려지는 줄**이라,
프레임 전체 높이가 터미널보다 짧은 동안에는 위 내용의 높이 변화가 1:1로 입력창의 화면 위치
변화가 된다. 스트리밍으로 셀이 자라면 입력창이 한 줄씩 내려가고, 접힘이 발화하면 그만큼 위로
튀어 오른다 — 도구가 연쇄되는 턴에서는 이게 도구 수만큼 반복된다.

---

## 1. 기전 (코드 사실)

**접힘 발화 지점 — 전부 턴 중간** (`modes/controllers/event-controller.ts`):

| 위치 | 트리거 | 효과 |
|---|---|---|
| :423, :543 | **새 도구 시작** | 직전 도구 `setMinimized(true)` — preview(헤더+미리보기 N줄+힌트)→1줄 |
| :267 | 새 assistant 메시지 시작 | 직전 도구 배치의 마지막 도구 접힘 |
| :351 | 도구 뒤 텍스트/thinking 도착(세그먼트 정착) | 마지막 도구 + 직전 thinking 접힘 (083.5) |

**높이 변화가 입력창 위치가 되는 이유** (`tui.ts` 렌더 모델):

- 프레임(트랜스크립트+컴포저)이 위에서부터 그려지고, 컴포저는 마지막 컴포넌트.
  프레임 높이 < 터미널 높이인 동안 컴포저 아래는 빈 공간 — **하단 고정이 아니다**
  (사용자 스크린샷에서 입력창 아래 여백 확인).
- 스트리밍 중 도구 preview·thinking tail이 자라면 컴포저가 줄 단위로 하강, 접힘 발화 시
  Δ(preview 높이−1)만큼 상승. 도구 k개 연쇄 = 출렁임 k회.
- 프레임 > 터미널이 되면 뷰포트가 하단 고정되어 시각적 출렁임은 멎지만, 대신 축소 시
  `clearOnShrink` 경로(`tui.ts:1280` 부근)가 **전체 클리어+재인쇄**를 발화 — 깜빡임으로 형태만 바뀜.

즉 083.1/083.5의 "이전 셀은 즉시 접는다" 리듬 자체가 높이 진동의 원천이고, TUI의
"컴포저 = 흐름의 마지막 줄" 배치가 그 진동을 입력창 위치로 전달한다.

## 3. 재현 / 관찰

- 새 세션(프레임 < 뷰포트) + 도구 3개 이상 연쇄하는 프롬프트 → 입력창이 도구마다 상승.
