## 1. 증상

긴 추론(thinking) + 도구 다발 턴에서, 최종 응답이 찍히기 직전에 화면이 위로 확 끌려 올라간다.
083.7 sticky gap 이후 스트리밍 중 점프는 사라졌지만 **message_end ~ agent_end 구간**에서 재발.

## 2. 원인 체인 (실코드 앵커)

## 3. 레퍼런스가 푸는 방식

| | Codex (rust tui) | Claude Code |
|---|---|---|
| 모델 | **inline viewport + `insert_history`** — 확정 라인만 SetScrollRegion+`\x1bM`(reverse index)로 실 스크롤백에 1회 write, **이후 픽셀 불변** (`insert_history.rs:193-244`) | **alt-screen 전체 소유** + frontFrame/backFrame diff — 스크롤백에 아예 안 씀 |
| 접기 | 완료 시점에 compact form을 **한 번만** 커밋 (`ExecCell::should_flush`) | entry가 **생성 시점부터 접힌 상태** (`collapsed_read_search`), retroactive 수축 없음 |
| 교훈 | **"커밋된 것은 다시 그리지 않는다"** | **"히스토리를 소유하거나, 건드리지 말라"** |

jwc는 정상 스크린에서 가상 버퍼 전체를 diff 렌더 — 위쪽 콘텐츠 사후 변경이 구조적으로 가능하고, 그게 일어나는 순간 3J 풀클리어로 도망간다.

## 4. 수리 제안

### 단기 (현 렌더러 유지)

- **S1. 뷰포트-위 불변 규칙**: trailing thinking 수축(②)·마지막 도구 접기(⑤)를 **컴포넌트 상단이 뷰포트 안에 있을 때만** 수행. 스크롤아웃된 블록은 확장 상태 그대로 둔다(Codex의 "커밋 후 불변"과 동형 — 접힌 모습은 다음 세션 replay부터 적용). 판정은 TUI가 컴포넌트별 렌더 구간을 아는 frame 조립 시점에 가능.
- **S2. `compactViewportFill()` 시점 이동**: turn end(⑤) → **다음 프롬프트 제출 직전**(input-controller submit 경로). 갭 수거를 사용자가 어차피 화면이 바뀌는 순간으로 옮겨 점프를 은닉. turn end 호출 제거.
- **S3. 위-뷰포트 변경 풀렌더에서 3J 제거**: `firstChanged < viewportTop`·`viewportFill compact` 경로의 `fullRender(true)`를 멀티플렉서 경로처럼 **2J/3J 없는 in-place 재페인트**로 강등. 스크롤백 보존 → "빨림" 체감 제거 (히스토리에 과거 프레임 잔상이 남는 트레이드오프는 Codex inline 모드와 동일하게 수용).

S1+S2만으로 ②⑤의 트리거 자체가 사라지고, S3는 잔존 케이스의 안전망.

### 장기 (정공법)

- **L1. insert_history 커밋 레인**: 확정 셀(99.20.04 커밋 폴딩의 접힌 도구 1줄, 083.3/PR#555 세그먼트 분할의 완료 세그먼트)을 SetScrollRegion+reverse index로 실 스크롤백에 1회 write 후 **가상 버퍼에서 제거**. diff 렌더러는 라이브 존(활성 세그먼트+라이브 도구+컴포저)만 소유 → 프레임 상시 소형, `firstChanged < viewportTop` 자체가 불가능, 3J 영구 퇴출. 커밋 시점 의미론(폴딩·세그먼트)은 이미 랜딩되어 있어 **렌더러 쪽 짝만 만들면 됨**.

## 6. 병존가치 평가 [260613 — 사용자 "시간 무관, 병존가치를 생각하라"]

**결론: S-트랙과 L1은 병존한다. 단 "임시 패치 → 정공법 후 폐기"가 아니라 역할이 재정의되는 병존이다.**

### 6a. S-트랙은 L1 이후에도 죽지 않는 이유

1. **가상 버퍼 레인은 영구 존속한다.** L1(스크롤 리전+reverse index)은 터미널 능력 의존적이다 —
   conpty(Windows)·구형 터미널·일부 멀티플렉서 조합에서 깨질 수 있고, Codex조차 inline insert_history와
   alt-screen을 병행 유지한다(`tui.rs:697-715`). jwc도 `커밋 레인 불가 → 가상 버퍼 레인 폴백`이 필요하고,
   그 폴백 레인의 품질이 곧 S1~S3이다. **S-트랙 = 폴백 레인의 정식 사양.**
2. **S1의 원칙은 L1 안에서 살아남는다.** "뷰포트 위 불변"(S1)과 "커밋 후 불변"(L1)은 같은 불변식의
   다른 집행 지점이다 — S1은 렌더 시점 검사로, L1은 커밋 시점 봉인으로 집행한다. S1을 먼저 구현하면
   그 판정 로직(컴포넌트별 렌더 구간 추적)이 L1의 커밋 경계 계산에 그대로 재사용된다.
3. **S2/S3는 레인 무관 위생이다.** compactViewportFill의 시점 이동(S2)과 3J 제거(S3)는 가상 버퍼
   레인이 존재하는 한 항상 옳다. L1 기본화 이후에도 폴백 레인에서 그대로 유효.

### 6b. L1의 독립 가치 — 이 버그를 넘어서

1. **렌더 비용 상한.** 현 디퍼렌셜 렌더러는 매 프레임 `#previousLines` 전체를 비교한다(tui.ts:1520대) —
   비용이 O(트랜스크립트 길이)로 장기 세션에서 단조 증가. L1은 프레임을 라이브 존으로 한정해
   **O(viewport) 상한**을 영구 보장. 긴 세션 성능 문제의 근본 해소.
2. **버그 클래스 전체 소멸.** `firstChanged < viewportTop`(:1448), clearOnShrink(:1357), 3J 폭풍,
   리사이즈 시 전체 리플로우 — 모두 "위쪽을 다시 그릴 수 있다"에서 파생. 커밋 레인에서는 발생 자체가 불가능.
3. **이미 절반이 랜딩돼 있다.** 커밋 시점 의미론(99.20.04 커밋 폴딩 = 완료 도구의 최종형 1줄,
   083.3/PR#555 세그먼트 분할 = 완료 세그먼트)이 커밋 단위를 정확히 정의해 둠. 렌더러 쪽 짝만 비어 있다.

### 6c. 진짜 트레이드오프 — 인-챗 인터랙티비티

스크롤백에 커밋된 픽셀은 토글이 불가능하다. 현 jwc UX 중 **커밋된(과거) 셀에 대한** ctrl+o/ctrl+t 전역 스윕,
도구 포커스 모드(ctrl+shift+up)는 커밋 레인에서 라이브 존으로 스코프가 줄어든다. 대응:

- 과거 셀 탐색·확장은 **alt+t 트랜스크립트 오버레이**(`app.tools.transcript`, 기구현)로 일원화 —
  Codex의 ctrl+t pager, CC의 alt-screen 소유와 동일한 제품 포지션.
- ctrl+o/t·포커스 모드는 **현재 턴(라이브 존) 전용**으로 재정의. 실사용 토글 대상은 거의 항상 직전 도구다.

### 6d. 확정 권고

```
ui.renderLane: "commit"(기본, 능력 감지 통과 시) | "virtual"(폴백)
```

- 순서: **S2+S3 즉시**(레인 무관 위생, 즉효) → **S1**(불변식 + 컴포넌트 렌더 구간 추적 = L1 선행 자산)
  → **L1 커밋 레인**(별도 플랜 083.9) → 능력 감지로 기본 레인 전환.
- S-트랙은 L1 완성 후에도 virtual 레인 사양으로 영구 유지. 삭제 대상 아님.
