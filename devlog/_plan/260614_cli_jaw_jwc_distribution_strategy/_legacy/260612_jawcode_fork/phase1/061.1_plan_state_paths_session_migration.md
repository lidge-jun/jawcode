### [확정 제안] 전략 A (상수 교체 + 원-타임 자동 마이그레이션)

**단, D4 결정을 먼저 번복해야 착수 가능하다.** D4는 "업스트림 리베이스 용이성"을 이유로 확정됐으므로, 아래 세 전제를 충족하면 번복이 정당화된다:

1. M1 완성 후 업스트림 리베이스 빈도가 낮아진 시점에 착수
2. 테스트 대량 수정을 단일 커밋(C-state)으로 원자화
3. 롤백 스토리 명확화 (§5 참조)

전략 B는 `gjc-runtime/` 내부 하드코딩 62줄이 오버라이드를 무시하므로 절반짜리 해결책이 된다. 전략 C는 macOS `realpathSync` 경로 비교(`dirs.ts:46`)와 충돌한다.

### M4 — 자동 마이그레이션 런처 (startup hook)

**대상**: `packages/coding-agent/src/` 엔트리포인트 (SDK 초기화 경로, `sdk.ts` 또는 `cli/args.ts`)

신규 함수 `migrateConfigDirOnce(home: string): void`:

```
1. old = path.join(home, ".gjc"), new = path.join(home, CONFIG_DIR_NAME)
2. new 존재하면 skip (멱등)
3. old 존재하면 fs.renameSync(old, new) — 실패 시 best-effort (EXDEV이면 copy+rm)
4. 프로젝트 레벨: startup 시 path.join(cwd, ".gjc") 존재하면 동일 rename
```

**테스트**: 신규 `test/migration-config-dir.test.ts` — old only / new only / both / neither 4개 케이스.

## 4. 마이그레이션 시퀀스

### 4-1. 디렉터리 마이그레이션 (M4 동작 순서)

```
[바이너리 시작] → migrateConfigDirOnce(homedir)  (~/.gjc → ~/.jwc rename)
              → migrateConfigDirOnce(cwd)       (.gjc → .jwc rename)
              → DirResolver 초기화 (.jwc 기반) → 정상 시작
```

**동시 접근 방어**: rename은 POSIX 원자적. rename 직후 구 바이너리(gjc)가 `.gjc`를 재생성하는 경쟁 대응 — `.gjc/.jwc-migrated` sentinel 파일 + 구 바이너리 접근 경고.

### 4-2. 기존 세션 마이그레이션 세부

**세션 저장 구조** (`session-manager.ts:738-745`): `~/.gjc/agent/sessions/<encoded-cwd>/<snowflake>.jsonl`. rename 시 하위 구조 그대로 이전, JSONL 내부 경로 참조 없음 → **내용 수정 불필요**.

| 상황 | 위험 | 대응 |
|------|------|------|
| `GJC_SESSION_FILE`이 절대 경로로 설정된 in-flight 프로세스 | rename 후 goal mode 쓰기 실패 | M4에서 rename 후 현재 프로세스 env 갱신; subprocess 재시작 필요 |
| 세션 재개 코드의 경로 인코딩 | 내용 무관 — 무영향 | 없음 |
| 진행 중 `.gjc/state/sessions/<id>/` | rename 원자적 — 단절 없음 | 없음 |

**세션 재개 검증 체크리스트**: ① `~/.jwc/agent/sessions/<encoded-cwd>/` 존재 ② `GJC_SESSION_FILE` 새 경로 ③ `jwc state jaw-interview` 정상 읽기 ④ `jwc goal status` 정상 읽기.
