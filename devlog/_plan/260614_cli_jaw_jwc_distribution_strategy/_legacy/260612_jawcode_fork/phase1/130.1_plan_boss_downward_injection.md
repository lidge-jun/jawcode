## 0. 목적과 범위

130 MOC 스코프 B 확정 결정(D130-1, 260612)은 두 가지를 못 박았다:

1. **스코프 분리**: cli-jaw `orc_state`(DB 1행, 전사 boss 파이프라인 정본)와 jwc `pabcd-state.json`(cwd·세션당 N개, 로컬 파이프라인)은 **같은 사실의 중복 저장이 아니라 다른 스코프의 상태**다. 동기화가 아닌 **단방향 주입만**.
2. **레일 재사용**: 주입은 99.03 M2에서 착지한 per-turn pabcd-stage-context 커스텀 메시지 레일을 그대로 확장한다. 신규 아키텍처 불필요.

이 문서는 레일의 실제 착지 코드를 앵커로 삼아, 어댑터 설계·튜닝 확정안·스코프 A 임베디드 옵션 표·상향 보고 경로·완료 기준을 서술한다. **코드 변경 없음**—설계 문서에 한정.

### 1-2. 상태 읽기 계층

```
readPabcdStateWithFallback(cwd, sessionId)
  └─ 1순위: readPabcdState(cwd, sessionId)   → .jwc/state/sessions/{id}/pabcd-state.json
  └─ 2순위: readPabcdState(cwd)              → .jwc/state/pabcd-state.json  (비스코프 경로)
```

비스코프 경로가 존재하는 이유: `jwc orchestrate` 셸 실행(99.03 자가 전이 패턴)은 세션 ID 없이 쓰기 때문에 TUI 소비자(스테이지 헤더, 상태선)가 항상 볼 수 있어야 한다. (`orchestrate-state.ts:329-333` 주석 참조)

### 1-3. 포맷 스트링 (현행)

```
buildPabcdStageContent(envelope, goal?) → string | null

출력 예:
"[GOAL: 결제 모듈 재설계 · PABCD — B: BUILD · verification=pending]\n
Implement the plan. On verification DONE, run `jwc orchestrate c`."
```

- 비활성(`envelope.active = false`) 또는 `COMPLETE` 단계: `null` 반환 → 레일 자동 침묵.
- 게이트 칩은 A(`audit_status`) · B(`verification_status`)에만 표시.
- goal prefix는 99.08-A에서 추가 — `PabcdHeaderGoal.objective`(최대 40자 잘림).

### 1-4. CustomMessage 형식

```ts
{

  customType: "pabcd-stage-context",
  content: "<위 포맷 문자열>",
  display: false,
  attribution: "agent",
  timestamp: Date.now(),
}
```

**이 객체가 확장 지점이다.** boss 상태를 주입하려면 `content` 생성 로직만 교체하거나 보강하면 된다.

### 1-5. 매 턴 주입 경로

`agent-session.ts L4822`:

```ts
const pabcdStageMessage = await this.#buildPabcdStageMessage();
if (pabcdStageMessage) {
    messages.push(pabcdStageMessage);
```

목적 메시지 직전에 push — goal 모드 메시지(`L4818-4820`) 바로 뒤, prepend 메시지(`L4826-4828`) 직전 슬롯.

## 2. Boss 상태 공급 — 어댑터 설계

### 2-1. 공급 방식 선택: (a) vs (b)

| | (a) 콜백 주입 `getBossState` | (b) 파일·엔드포인트 폴링 |
|---|---|---|
| 방식 | `createAgentSession()` 옵션으로 `getBossState?: () => BossStageSnapshot` 콜백 전달 | jwc 임베디드 세션이 매 턴 cli-jaw HTTP `/api/orc-state` 폴링 |
| 커플링 | in-process: 함수 참조 직전달, 직렬화 0 | 프로세스 경계 횡단, HTTP 지연, 인증 |
| 오프라인 | 서버 생존 = jwc 생존, 항상 사용 가능 | 서버 재기동 시 404 창 발생 |
| 확장 | 멀티 인스턴스(`.cli-jaw-34xx`)별 격리 trivial | 포트 발견 필요 |
| 구현 비용 | `CreateAgentSessionOptions` 옵션 1개 추가 + `#buildPabcdStageMessage` 분기 | 엔드포인트 설계 + 인증 + 재시도 |

**권고: (a) 콜백 주입.** in-process 시대 원칙(D1 확정)과 일치하며 구현 비용이 최소. (b)는 Phase 3 이후 원격 jwc 세션 지원 시 검토.

### 2-2. 어댑터 인터페이스 스케치

```ts
// packages/coding-agent/src/sdk.ts (CreateAgentSessionOptions 확장)
interface BossStageSnapshot {
  stage: OrcStateName;         // 'I' | 'P' | 'A' | 'B' | 'C' | 'D' | 'IDLE'
  gates: {
    auditStatus?: AuditVerdict;        // 'pending' | 'pass' | 'fail'
    verificationStatus?: VerificationVerdict;  // 'pending' | 'done' | 'needs_fix'
    userApproved?: boolean;
  };
  goal?: string;               // orc_state ctx.originalPrompt 요약 (선택)
}

// CreateAgentSessionOptions에 추가
getBossState?: () => BossStageSnapshot | null;
```

```ts
// cli-jaw src/agent/jwc-runtime.ts (어댑터 측)
import { getState, getCtx } from '../orchestrator/state-machine.js';

function makeBossStateProvider(): () => BossStageSnapshot | null {
  return () => {
    const stage = getState();         // DB에서 직접 읽기 (동기 SQLite)
    if (stage === 'IDLE') return null;
    const ctx = getCtx();
    return {
      stage,
      gates: {
        auditStatus: ctx?.auditStatus,
        verificationStatus: ctx?.verificationStatus,
        userApproved: ctx?.userApproved,
      },
      goal: ctx?.originalPrompt?.slice(0, 40) ?? undefined,
    };
  };
}

// createAgentSession 호출 시
const { session } = await createAgentSession({
  systemPrompt: buildSystemPrompt(),
  getBossState: makeBossStateProvider(),
  // …기타 옵션
});
```

cli-jaw의 `getState()` / `getCtx()`는 `state-machine.ts:95-108`의 동기 SQLite 읽기다. 매 턴 호출 부담 없음 — cli-jaw 자신이 이미 매 요청마다 동일 DB를 읽는다.

### 2-3. pabcd-stage-header 확장 지점

`#buildPabcdStageMessage`에서 boss 상태가 있으면 boss 정보를 우선 소비하고, 로컬 `pabcdStatePath` 읽기는 보조 경로로 내려간다:

```
입력 우선순위:
1. getBossState() → BossStageSnapshot  (콜백이 주입된 경우)
2. readPabcdStateWithFallback(cwd, sessionId)  (로컬 파일)
```

boss 상태가 `null`(IDLE)이면 로컬 pabcd 파일로 폴백 — 독립 실행 jwc 세션의 현행 동작 유지.

## 3. 튜닝 확정안 ①~④

### 튜닝 ① — 이중 라벨 포맷

boss와 로컬이 동시에 활성인 경우의 표기:

```
[BOSS — B: BUILD · audit=pass] · [LOCAL — P: PLANNING]
Implement the plan. On verification DONE, run `jwc orchestrate c`.
```

**확정안**: 이중 라벨 채택. 단, 세션이 boss 임베디드로만 생성된 경우(로컬 pabcd 비활성)는 boss 라벨 단독:

```
[BOSS — B: BUILD · verification=pending]
Implement the plan. On verification DONE, run `jwc orchestrate c`.
```

규칙:
- boss 상태가 존재하면 `[BOSS — {stage}: {label}{gates}]` 첫 번째.
- 로컬 pabcd가 별도로 활성이면 ` · [LOCAL — {stage}: {label}]` 접미.
- NEXT_HINT는 boss 단계 기준 (boss가 최종 진행 방향이므로).

`buildPabcdStageContent` 시그니처 확장:
```ts
export function buildPabcdStageContent(
  envelope: NativePabcdEnvelopeParsed,           // 로컬 (존재 시)
  goal?: PabcdHeaderGoal | null,
  bossSnapshot?: BossStageSnapshot | null,       // 추가
): string | null
```

### 튜닝 ② — 주입 시점

**확정안: 매 턴 (M2 동형).**

`#buildPabcdStageMessage`는 이미 매 턴 호출된다. boss 상태도 동일 경로에서 읽음 — 비용 추가 없음. 단계 변화 시 `sendPabcdStageContext()`의 즉시 전송도 그대로 유지 (전이 직후 명시적 push).

단계가 변하지 않아도 매 턴 헤더가 갱신되는 이유는 cli-jaw 원본과 동일: **컴팩션 면역** — 긴 대화에서 컨텍스트가 잘려도 헤더는 항상 최신 턴에 존재.

### 튜닝 ③ — HUD 세그먼트 우선순위

**확정안: boss 정보가 HUD 첫 번째 세그먼트, 로컬은 괄호.**

TUI 상태선(workflow-readers.ts)에서 boss 세그먼트가 보이면 로컬 세그먼트를 괄호 표기로 강등:

```
[B: BUILD]boss  ([P: PLANNING]local)
```

boss가 없으면(IDLE 또는 콜백 미주입): 현행 로컬 표시 그대로.

구현은 workflow-readers.ts의 `readWorkflowState()` 반환 구조에 `bossSnapshot?: BossStageSnapshot` 필드를 추가하고, TUI가 이를 우선 렌더링하는 분기를 두는 방식으로 처리한다.

### 튜닝 ④ — 자가 전이 단락 훅: BashTool 인터셉트 스케치

**문제**: 임베디드 jwc 세션에서 모델이 `jwc orchestrate B`를 Bash로 실행하면 in-process 세션 외부의 셸에서 실행되어 cli-jaw `orc_state` DB를 건드리지 않는다(서브프로세스가 별도 cli-jaw 서버를 바라봄).

**확정안**: BashTool 인터셉트 — 임베디드 세션의 Bash 실행을 가로채 cli-jaw 상태머신 API를 직접 호출.

스케치:

```
BashTool 실행 요청
  ↓
인터셉터: /^(jwc|gjc) orchestrate ([IPABCD])$/  매칭?
  ├─ YES → cli-jaw API 직접 호출
  │         import { setState, canTransition, getState } from '../../orchestrator/state-machine.js'
  │         canTransition(getState(), stage) → ok?
  │         YES: setState(stage, ctx)  → broadcast('orc_state', …)
  │         NO: 모델에 거부 이유 반환
  └─ NO  → 기존 Bash 실행 흐름
```

인터셉터 등록 방법 두 가지:

| 방법 | 장점 | 단점 |
|---|---|---|
| A. BashTool preExecHook (customTools 확장) | 기존 Bash 표면 유지, 모델 어휘 변경 불필요 | jawcode AgentSession BashTool 훅 지점 확인 필요 |
| B. 전용 도구 등록(`orchestrate` 커스텀 툴) | 명시적, 타입 안전 | 모델이 jwc orchestrate Bash 대신 새 툴 이름을 학습해야 함 |

**권고: A (BashTool 인터셉트)**. 모델이 NEXT_HINT에 적힌 `jwc orchestrate <stage>` 문자열을 그대로 실행하도록 설계되어 있으므로 어휘 변경 없이 in-process 라우팅이 가능.

실제 인터셉터는 `jwc-runtime.ts`의 `createAgentSession` 호출 시 `customTools` 또는 `extensions`를 통해 BashTool 래퍼를 주입한다. 공식 훅 표면이 아직 실사 미완료이므로 착수 전 `agent-session.ts`의 BashTool 등록 지점을 확인해야 한다(111 열린 질문 2 승계).

## 5. 상향 보고 경로

> 130 MOC: "상향은 sync가 아니라 **보고**"

worker verdict(Bash stdout에 PASS/FAIL/DONE/NEEDS_FIX 토큰)는 기존 cli-jaw `parseWorkerVerdict()` + `dispatch` 결과 경로를 통해 boss가 수신 → boss가 `setState()` 호출 → `orc_state` DB 갱신.

임베디드 jwc 세션이 별도로 boss DB를 직접 갱신하지 않는다. 세션이 생성한 worker 결과는 `orchestrate verdict` 경로와 동형으로 cli-jaw boss에 보고되고, boss가 단독으로 상태머신을 전진시킨다.

```
임베디드 jwc 세션
  → worker 결과 텍스트 반환 (PASS/FAIL/DONE/NEEDS_FIX)
  → cli-jaw JawRuntime (이벤트 버스 매핑, 110 MOC)
  → broadcast('orc_state', …) — cli-jaw 기존 경로
  → boss orc_state DB 갱신
```

역방향 N→1 머지 문제는 이 설계로 원천 차단된다: 세션은 보고만 하고 상태 기록은 boss 전담.

## 6. 완료 기준

### 6-1. M2 done ③ 부분 (스코프 A)

- Web UI에서 `~/.cli-jaw/skills`의 실스킬 1개가 발동된다 (e2e).
- 임베디드 세션의 시스템 프롬프트에 스킬 목록이 포함됨을 확인.

### 6-2. Boss 주입 e2e (스코프 B, D130-1 핵심)

- cli-jaw P 단계에서 임베디드 jwc 세션 헤더에 `[BOSS — P: PLANNING]` 표시.
- 단계 전이(`orchestrate B`) 후 헤더가 `[BOSS — B: BUILD]`로 갱신됨을 확인 (즉시 전송 경로).
- boss가 IDLE일 때 헤더가 로컬 pabcd 파일로 폴백됨을 확인 (회귀 방지).

### 6-3. 감사 체크리스트

- [ ] `getBossState` 콜백이 null을 반환할 때 현행 동작과 동일한지 (로컬 pabcd 폴백 회귀 없음).
- [ ] BashTool 인터셉터가 비-orchestrate 명령을 통과시키는지 (false positive 없음).
- [ ] 임베디드 세션의 자체 스킬 발견이 비활성인지 (`disableExtensionDiscovery`).
- [ ] 매 턴 boss 상태 읽기가 SQLite 잠금 충돌을 일으키지 않는지 (cli-jaw 동시 쓰기).
- [ ] 세션 종료 시 `dispose()` 호출로 자원 해제 (메모리 누수 확인).
