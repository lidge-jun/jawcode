# 99.20.07 — audit: 트랜지언트 크롬의 채팅 인라인 삽입 vs 하단 도킹 표면 전수 감사

> 상태: 조사 ✅ (260613) / ✅ **P1~P4 수리 완료** (260613 — 99.00.04 S7~S10: P3 showStatus 11곳·P4 Loader·P1 LoginDialog 도킹·P2 ScrollablePanel 7종. P5는 설계상 제외, quota-panel 통합은 병렬 랜딩 정리 이월).
> 소속: [99.20.00 MOC](./99.20.00_moc_frontend_uiux.md). 입력: 사용자 "`/quota`가 채팅창에 삽입돼
> UI/UX를 해친다" → `/quota`·`/usage` 도킹 변환(§6) 후 "provider 로그인 누르면 위쪽에 뜬다,
> 하단이 더 적합하지 않나" 후속 관찰.

jwc TUI에는 두 표면이 있다: **chatContainer**(스크롤되는 대화 기록, 상단)와
**editorContainer**(컴포저가 사는 하단 — 셀렉터들이 `showSelector()`로 잠시 점유). 원칙은
"읽고 버리는 일시적 크롬은 하단, 다시 찾아볼 대화 산출물은 채팅"인데, 실제로는 다수의
일시적 크롬이 chatContainer에 낱개 삽입되어 transcript를 영구 오염시키고 있다. 최악은
OAuth 로그인: **선택창은 하단 → URL/진행/성공은 상단 채팅 → 코드 입력은 다시 하단**으로
시선이 한 플로우에서 3번 점프한다. 게다가 이 용도로 만들어진 도킹형
`LoginDialogComponent`가 **export만 된 채 한 번도 인스턴스화되지 않은 dead code**로
방치되어 있음이 확인됐다 (§3).

## 1. 표면 모델

| 표면 | 진입 | 수명 | 적합 용도 |
|------|------|------|-----------|
| chatContainer | `addChild` 직접 삽입 | 영구 (세션 기록) | 사용자/어시스턴트 메시지, 도구·bash·python 실행 산출물 |
| editorContainer | `SelectorController.showSelector()` (`selector-controller.ts:110`) | 일시 (`done()`으로 에디터 복원) | 셀렉터·위저드·read-once 패널 |
| status line | `ctx.showStatus()` (`ui-helpers.ts:61-84`, in-place 갱신으로 중복 방지) | 덮어쓰기 | 한 줄 진행/성공 알림 |

## 3. LoginDialogComponent 포렌식

- `components/login-dialog.ts` — 주석 "replaces editor during OAuth login flow". `showAuth` /
  `showPrompt` / `showProgress` / `showManualInput` / `showWaiting` / ESC 취소가 `login()`
  콜백 5종과 **1:1 대응**하는 완성품.
- `grep "new LoginDialogComponent"` → **0건**. `components/index.ts:18` export만 존재.
- `git log -S` → 도입 커밋은 `19f8c1f4` (OMP 베이스라인 임포트) 단 1건, 분기 후 무수정.
- **upstream도 미사용**: `devlog/_upstream_omp/.../selector-controller.ts:938`의
  `#handleOAuthLogin`도 transcript 삽입 방식 (차이: upstream은 `TranscriptBlock` 컨테이너로
  묶어 `ctx.present()`, jwc는 낱개 `addChild`). 즉 **설계 의도만 있고 배선이 한 번도 없었던
  upstream 유래 dead code** — 부활시켜 쓰면 P1이 컴포넌트 신작 없이 끝난다.

## 4. 도킹 인프라 제약 (수리 시 주의)

1. `showSelector`의 `done()`은 콜백에서 **명시 호출 필수** — LoginDialog는 `onComplete`로
   외부에서 주입받아야 함.
2. 도킹 중 에디터 포커스 부재 → **채팅 입력 불가**. OAuth는 수십 초~수 분 흐름이라
   브라우저 대기 중 블로킹이 trade-off (단, 현재도 로그인 중 의미 있는 입력은 없음).
3. `#handleOAuthLogin`의 `onPrompt`는 이미 `editorContainer`를 셀렉터 프레임 밖에서 직접
   조작(`selector-controller.ts:1179-1186`) — LoginDialog로 일원화하면 이 수동 복원도 제거.
4. `oauthManualInput`(`modes/oauth-manual-input.ts`)은 표면 비의존 Promise 래퍼
   (`builtin-registry.ts:909`에서 submit) — LoginDialog 내부 Input과 호환.
