# 082.1 — issue: 한글 IME에서 Ctrl 단축키·종료가 안 먹음

> 상태: ✅ **부분 수정 (종료 안전망 C안 260612 10시 + 자모 chord 힌트 D안 260612 오후)** / 일반 Ctrl 단축키는 Kitty 터미널 권고. 조사: Opus 서브에이전트.
> 수정 1 (C안): **ESC 2연타 → 종료 안전망**을 `custom-editor.ts`에 추가 — ESC는 IME 무관하게 항상 0x1b라,
> 한글 IME에서 ctrl+d/ctrl+c가 안 먹혀도 **항상 빠져나갈 수 있음**. 첫 ESC는 기존 interrupt, 윈도우(500ms) 내
> 두 번째 ESC가 onExit. 테스트 4건 추가. ⚠️ 일반 Ctrl 단축키 자모 remap은 **미채택**(아래 §결정).
> 수정 2 (D안, §D안 참조): **자모 chord 힌트** — Ctrl 바인딩 자리의 단독 자모가 들어오면 액션은 절대 발동하지
> 않고(오발동 0), 입력창 아래 상태줄에 "switch to English (한/A) or press esc esc to exit" 안내를 4초 표시.
> 소속: 082 밴드 (TUI 입력/IME, [082_moc_tui_input.md](./082_moc_tui_input.md)). 발현 표면 = TUI interactive 모드. **업스트림 gjc 버그(0 diff 확인)**.
> 입력: 사용자 "tui 종료나 ctrl 커맨드가 영문으로만 먹는다, Ctrl+ㅊ(=Ctrl+C) 안 먹는 문제".

macOS 입력 소스가 **한글(두벌식)** 일 때 jwc/gjc TUI에서 Ctrl 단축키와 종료(Ctrl+C/Ctrl+D)가 안 먹는다.
물리적으로 Ctrl + 'c' 키(한글 레이아웃에선 'ㅊ')를 눌러도 동작하지 않고, 자모가 입력창에 텍스트로 들어간다.

## 원인 메커니즘 (확정 95%)

**터미널이 0x03이 아니라 한글 자모 바이트를 보내고, parser가 그걸 Ctrl 코드로 인식 못 해 텍스트로 흘린다.**

바이트 흐름 (각 hop):
1. raw 모드 stdin (`packages/tui/src/terminal.ts:166` `setRawMode(true)`) — 멀티바이트 UTF-8을 그대로 Buffer 전달.
   **raw 모드라 Ctrl+C가 SIGINT를 안 만든다** → 종료도 named binding에만 의존.
2. decode (`packages/tui/src/stdin-buffer.ts`) → string.
3. dispatch: `packages/coding-agent/src/modes/components/custom-editor.ts:201` `handleInput` →
   `#matchesAction()`(`:103`) → `matchesKey(data, "ctrl+c")`. 실패 시 `:352 super.handleInput` →
   base Editor(`packages/tui/src/components/editor.ts:1317`)가 `extractPrintableText`로 뽑아 입력창에 삽입.
4. **핵심 손실 지점**: `packages/tui/src/keys.ts:523` `matchesKey` → native 매처. legacy(non-Kitty) 경로는
   **C0 컨트롤 바이트(0x01–0x1A)로만** Ctrl+letter를 인식. 한글 자모(ㅊ = U+314A, UTF-8 `0xe3 0x85 0x8a`)는
   컨트롤 바이트가 아니라 매칭 실패 → printable로 분류.

코드베이스에 **한글 자모 → QWERTY 역매핑이 전혀 없음**(`grep jamo|두벌식|hangulToQwerty` → 0건). 영문 키 가정이 구조적.

## 증거 (native 매처 직접 실행)

`/tmp/probe_keys.ts`로 native 매처 직접 호출:
```
===== LEGACY (Kitty OFF) =====
[raw 0x03]   utf8=0x03            matchesKey(ctrl+c)=true   printable=undefined
[ㅊ U+314A]  utf8=0xe3 0x85 0x8a  matchesKey(ctrl+c)=false  printable="ㅊ"   ← 손실
===== KITTY ON =====
[Ctrl+c 99;5u]                    matchesKey(ctrl+c)=true   ← ASCII 코드포인트 경로
[12618::99;5u (baseLayout=c)]     matchesKey(ctrl+c)=true   ← Kitty baseLayoutKey가 구제
[12618;5u (baseLayout 없음)]      matchesKey(ctrl+c)=false  ← baseLayout 없으면 또 실패
```

## 왜 영문에서만 되나

- **Legacy 경로(Kitty OFF)**: 영문 IME면 Ctrl+letter가 C0 바이트(Ctrl+C→0x03) 생성 → native 매처 매칭
  (`keys.test.ts:7-8`이 `String.fromCharCode(3)`→`ctrl+c` 보장). 한글 IME면 같은 물리 키가 조합 자모로
  전달되어 0x03이 안 생김 → 실패.
- **Kitty 경로(ON)**: `terminal.ts:357`에서 flag 7(alternate keys) push. `codepoint::baseLayoutKey;5u`로
  물리 키를 알려줘 비라틴 레이아웃도 매칭(`keys.test.ts:21-37`). **즉 Kitty + baseLayoutKey 전송 터미널
  (Kitty/Ghostty/WezTerm)에선 한글 IME여도 Ctrl+ㅊ가 동작할 가능성 높음.** 문제는 주로 **Kitty 미지원/미전송
  터미널(기본 Terminal.app, 일부 iTerm2 설정) + 한글 IME** 조합.

종료(`app.exit`=ctrl+d, `app.clear`=ctrl+c)도 raw 모드라 SIGINT 없이 named binding
(`config/keybindings.ts:70,74`)에만 의존 → 동일하게 취약. interactive 모드에 0x03/SIGINT 독립 종료 경로 없음.

## 결정 (260612 10시) — C 채택, B 미채택

- **C (ESC 2연타 종료 안전망) 채택·구현 완료.** `custom-editor.ts` handleInput에 추가:
  `matchesKey(data, "escape")`로 lone ESC 감지 → 윈도우(`DOUBLE_ESCAPE_EXIT_WINDOW_MS=500`) 내 두 번째
  ESC면 `onExit()`. 첫 ESC는 기존 interrupt 유지, 사이에 다른 키 오면 트래커 리셋. 테스트 4건.
  ESC(0x1b)는 IME 무관이라 **종료 불능 탈출을 100% 보장** (사용자 "tui 종료" 직접 해결).
- **B (자모→Ctrl 무조건 remap) 미채택.** `#matchesAction`은 exit/clear/모델선택 등 다수 ctrl 액션을
  텍스트 입력 전에 가로챈다(`custom-editor.ts:228-335`). 자모를 Ctrl로 무조건 되돌리면 **흔한 한글 자모가
  단축키를 오발동** — ㅇ(→d→exit), ㅣ(→l), ㅅ(→t), ㅐ(→o), ㅔ(→p) 등이 그대로 종료/초기화를 유발해
  한글 텍스트 입력이 깨진다. legacy는 Ctrl 신호가 없어 "Ctrl+ㅊ"와 "텍스트 ㅊ"를 구분 불가 → 안전한
  기본값 불가. (원하면 default-off 세팅으로 opt-in 추가 가능 — 후속.)
- **일반 Ctrl 단축키 = Kitty 프로토콜 터미널 권고.** Ghostty/Kitty/WezTerm은 baseLayoutKey로 한글 IME여도
  Ctrl+ㅊ가 이미 동작(코드 수정 0). Terminal.app 등 legacy는 종료 안전망(C)으로 커버.

## D안 — 자모 chord 힌트 (260612 오후 구현, 사용자 지시)

사용자 지시: "오작동은 막되, 유저가 ctrl+ㅊ 자리에 있는 작동을 하면 IME 안내를 밑에 띄워라."
B안(강제 remap)의 오발동 위험은 그대로 회피하면서, "왜 아무 일도 안 일어나지?"라는 무반응 혼란을 안내로 해소한다.

동작: 한글 IME에서 Ctrl+ㅊ를 누르면 → legacy 터미널이 자모 "ㅊ"만 전달 → 모든 단축키 매칭 실패 후
(즉 액션 오발동 가능성 0인 지점에서) **두벌식 역매핑으로 그 자모가 Ctrl 바인딩 키 자리(c)임을 감지** →
입력창 아래 상태줄(hook-status 라인)에 안내를 4초 표시하고 자모는 평소처럼 텍스트로 삽입:

```
ctrl+c needs the English layout — switch to English (한/A) or press esc esc to exit
```

구현 (전부 업스트림 무수정 추가, brand 무관 일반 수정):

- `custom-editor.ts` — `DUBEOLSIK_JAMO_TO_QWERTY` 33자 역매핑 테이블(쌍자음/복모음 포함) +
  `#hangulCtrlChordFor(data)`: 단독 1자 자모 && 그 QWERTY 자리에 ctrl chord 바인딩(액션 키 + 확장
  커스텀 키 전체 스캔, remap 추종) → chord 반환. `handleInput`의 **모든 인터셉터 실패 후, super 위임
  직전**에 검사 → `onHangulCtrlChordHint(jamo, chord)` 발화 후 fall-through (자모 삽입 유지).
- `input-controller.ts` — `showHangulImeHint(chord)`: `statusLine.setHookStatus("ime-hangul-chord", …)`
  4초 후 자동 해제(재발화 시 타이머 연장). hook-status 라인은 `sanitizeStatusText`가 ANSI를 벗기므로 평문.
- 테스트 8건: `custom-editor-keybindings.test.ts` 4건(ㅊ→ctrl+c 힌트+무발동+텍스트 삽입 유지 / ㅇ→ctrl+d /
  remap 추종(ctrl+y→ㅛ) / 음절·라틴·멀티차 chunk 침묵) + `input-controller-keybindings.test.ts` 2건(상태줄
  표시+4초 자동 해제 / 재발화 시 윈도우 연장) — 기존 스위트 그린.

한계: 자모 단독 입력(ㅋㅋㅋ 등)도 바인딩 자리 자모면 힌트가 잠깐 뜰 수 있음 — 액션은 절대 발동하지
않으므로 표시 노이즈뿐이고 4초 후 사라짐. 진짜 remap을 원하면 default-off 세팅 opt-in은 여전히 후속 옵션.

## 확정 실험 (터미널별 바이트 측정)

parser 측은 확정. 남은 변수는 "각 macOS 터미널이 한글 IME에서 어떤 바이트를 보내나" — 터미널+IME 의존.
```ts
// bun run; 한글 IME로 전환 후 Ctrl+c / Ctrl+d / Ctrl+ㅊ 눌러보기
process.stdin.setRawMode(true); process.stdin.resume();
process.stdin.on("data", b => {
  console.log("bytes:", [...b].map(x => "0x"+x.toString(16)).join(" "), "| str:", JSON.stringify(b.toString("utf8")));
  if (b[0] === 3) process.exit(0);
});
```
Terminal.app / iTerm2 / Ghostty 각각에서 영문·한글로 실행 → 0x03 vs 자모(`0xe3 0x85 0x8a`) 확인하면
터미널별 필요한 수정안이 최종 확정됨.
