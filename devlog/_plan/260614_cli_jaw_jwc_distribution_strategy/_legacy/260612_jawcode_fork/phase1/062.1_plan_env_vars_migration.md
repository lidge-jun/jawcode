## 배경

현재 모든 런타임 구성은 `GJC_*` 접두사를 사용하며, 그 위에 업스트림에서 이미 `PI_*` 접두사가 사용되고 있다. 본 문서는 `JWC_*`를 canonical로 도입하고 `GJC_*`를 read-fallback으로 유지하는 전략을 수립한다. (rg 기준 `packages/**/*.ts` 398건 히트, 고유 변수 ~181종)

## 2. 해석 지점 설계 — [확정 제안: A 단일 resolver]

환경 읽기 진입점: `packages/utils/src/env.ts`의 `$env`/`$flag()`/`$pickenv()`. 단, ~39곳이 `process.env.GJC_*` 직접 참조.

| 방식 | 장점 | 단점 |
|---|---|---|
| **A. 단일 resolver 헬퍼 (`$resolveEnv`)** | 변경 집중, 테스트 1개소, GJC_ 제거 경로 명확 | 직접 참조 ~39곳 별도 처리 |
| B. 전수 치환 (fallback 없음) | 깔끔 | 기존 사용자 `.env`/스크립트 파괴, ~400 사이트 |
| C. 모듈 로드 시 alias 주입 | 코드 수정 0 | 자식 프로세스 이중 노출, 직접 참조 잔존 |

## 4. GJC_BRAND_NAME 특별 케이스

| 읽기 사이트 | 동작 |
|---|---|
| `packages/utils/src/dirs.ts:23` | `APP_NAME = GJC_BRAND_NAME \|\| ENGINE_NAME` — 모듈 로드 시 freeze |
| `cli.ts:68` / `discovery/helpers.ts:33` | 브랜드 게이트 (helpers는 런타임 매 읽기 — 테스트 토글) |

**[확정 제안: B+C 병행]** — `APP_NAME = JWC_BRAND_NAME ?? GJC_BRAND_NAME ?? "jwc"` (포크 기본값 jwc — gjc bin 제거 확정과 정합) + `bin/jwc.js` 진입점 명시 SET 유지. `CONFIG_DIR_NAME`은 061.1 소관, `ENGINE_NAME`은 065.1 패밀리 H(보존 확정) 소관.
