## 1. 세션 수명주기 — 상주 vs lazy, resume/steer 재발명

### 결정: lazy 초기화 + 영속 상주

- **서버 기동 시 즉시 createAgentSession을 호출하지 않는다.** 첫 boss 메시지가 게이트웨이를 통과하는 시점에
  JawRuntime이 세션을 생성하고(lazy init) 이후 서버가 살아있는 한 그 세션을 재사용한다.
- 근거: 서버가 뜰 때마다 API 연결·MCP manager 초기화가 발생하는 비용을 첫 사용 전까지 유예.
  실질 사용 패턴(서버 기동 즉시 대화 시작)과 차이가 없고, 단위 테스트에서 세션 없는 서버 상태를 격리하기 쉽다.
- **Lazy의 단일 예외**: 서버 기동 플래그 `JAW_PREHEAT=true` 시 미리 세션을 생성해 첫 응답 지연을 줄인다(옵트인).

### 기존 spawn/resume 구분 → 상주 세션 매핑

| 구 경로 | 상주 세션 대체 |
|---------|-------------|
| 신규 spawn | `JawRuntime.ensureSession()` — 세션 없으면 createAgentSession, 있으면 재사용 |
| resume (session-ID 역추적) | 서버 재시작 시 jaw.db에서 메시지 로드 → `AgentSession` 재구성 (120 MOC 계약). resume-classifier 전 경로 우회 |
| steer (kill-respawn) | `session.prompt(text, { streamingBehavior: "steer" })` — 살아있는 루프에 직접 push |
| followUp | `session.prompt(text, { streamingBehavior: "followUp" })` |
| forceNew | `JawRuntime.recreate()` — 세션 dispose 후 재생성, 메시지 히스토리 초기화 |

### 컴팩션 소유권

- **기본값: gjc 자체 컴팩션에 위임.**
  cli-jaw `compact` 트리거가 들어오면 JawRuntime은 `session.compact()`(또는 동등 API)를 호출하는
  위임 호출만 수행 — 두 레이어가 동시에 컴팩션을 시도하지 않음.
- cli-jaw `autoCompactRefresh` 로직은 jwc 경로에서 **비활성**. gjc 컴팩션이 발화하면 이벤트를
  bus로 전파해 Web UI에 "컴팩션 발생" 알림만 올린다.

## 2. 프롬프트 합성 핸드오프

### getSystemPrompt() → createAgentSession({systemPrompt})

cli-jaw `getSystemPrompt()`는 A1·A2·메모리·orchestration·스킬·위임 규칙을 포함하는 완성된 문자열을 반환한다
(structure/prompt_flow.md §조립 순서 참조). jwc `createAgentSession`의 `systemPrompt` 파라미터는
**host override** 경로(sdk.ts 내부에서 기본 시스템 프롬프트를 대체)를 통해 이 값을 받는다.

```
cli-jaw getSystemPrompt()
    → systemPromptBlock (string)
    → createAgentSession({ systemPrompt: systemPromptBlock, agentDir })
```

핵심 규칙:

1. **단일 합성**: cli-jaw가 합성 책임자다. gjc 기본 시스템 프롬프트는 완전히 대체된다(override).
   중복 조각(아이덴티티, 스킬 서두)이 두 번 들어가지 않도록 합성 함수에 중복 제거 단계를 추가한다.
2. **스킬 주입 1차**: 130 MOC 스코프 A §1 기본값 — cli-jaw builder.ts 산출(스킬 목록 포함)을 systemPrompt에
   포함. 네이티브 030 디스커버리(2차)로 전환하기 전까지 이 경로를 유지한다.
3. **jwc 자체 consolidation 비활성** (D112-2 확정): 세션은 주입만 받는 소비자.
   `createAgentSession` 옵션에서 consolidation/자체 메모리 축적을 비활성 플래그로 제어한다.

### 매 턴 prefix → custom-message 레일

cli-jaw `getPrefix()`(오케스트레이션 헤더, 단계 라벨)와 plan 재주입(A/B/C 동안 매 턴 DB plan 전문)은
**신규 spawn마다 프롬프트에 붙이던 방식**이었다. in-process 전환 후:

- `getPrefix()` 출력 → `session.prompt()`의 **사용자 메시지 앞에 prepend** (custom-message 레일).
  99.03 M2 헤더 레일(custom message `<system-reminder>` 패턴)을 재사용한다.
- D130-1 boss 단계 하향 주입(단계 변화 시 또는 매 턴)도 동일 레일: 메시지 body 앞에
  `[BOSS — {stage}]` 헤더 블록을 prepend.

## 3. 이벤트 → 서버 매핑

### 수신: AgentSessionEvent (in-process, 직렬화 0)

| AgentSessionEvent 종류 | 의미 |
|----------------------|------|
| `message_start` | 어시스턴트 응답 시작 |
| `message_update` (delta) | 텍스트/thinking 델타 스트리밍 |
| `message_end` | 완료 — 이 시점에 jaw.db 1회 기록 |
| `tool_execution_start` | 도구 호출 시작 |
| `tool_execution_update` | 도구 실행 중간 (진행 로그) |
| `tool_execution_end` | 도구 완료 + 결과 |
| `thinking_delta` / `thinking_end` | Extended thinking (원본 구조체, 암호화 포함) |
| `compaction` | 컴팩션 발생 |
| `agent_end` / `error` | 전체 완료 / 에러 |

### 송신: bus.broadcast → SSE (Web UI)

원칙: **cli-jaw 기존 이벤트 타입 우선 — Web UI 무수정 목표.**

| AgentSessionEvent | bus 이벤트 타입 | 비고 |
|------------------|--------------|------|
| `message_update` (text delta) | `agent_stream` (기존) | 텍스트 청크 그대로 |
| `message_end` | `agent_done` (기존) | 완성 텍스트 + 토큰 통계 |
| `tool_execution_start` | `tool_log` (기존, 신규 포맷) | `{ name, input, status: "start" }` |
| `tool_execution_end` | `tool_log` (기존, 신규 포맷) | `{ name, output, status: "done" }` |
| `thinking_delta` | `agent_thinking` (**신규**) | Web UI에 thinking 접기 블록 필요 시만 |
| `compaction` | `agent_compact` (**신규**) | Web UI 알림용 |
| `agent_end` | `agent_done` (기존) | code=0 |
| `error` | `agent_error` (기존) | message 필드 |

### messages 테이블 쓰기 (jaw.db)

현재 spawn 경로: CLI stdout 파싱 후 messages 삽입. in-process 전환 후:

- **쓰기 주체**: JawRuntime 내 subscribe 콜백.
- **쓰기 시점**: `message_end` 이벤트 수신 시 1회 원자적 커밋.
  스트리밍 중간 저장 금지(AGY 진행문 저장 버그 교훈 — 120 MOC 계약 승계).
- **도구 블록**: `tool_execution_end` 이벤트 수신 시 별도 tool_log 행 삽입.
- 기존 NDJSON 파싱 스택(`dispatchNdjsonLine` / `discriminate` / `extractFromEvent`) — cli='jwc' 경로에서 **전부 우회**.

## 4. busy/동시성 — in-process busy 플래그

111 열린 질문 3(gateway.ts `isAgentBusy()` 편입)에 대한 설계 결정:

### 현행 패턴 추정 (gateway.ts 실사 미완 — 착수 전 확인 필수)

spawn 경로는 `activeProcess !== null` 또는 `mainSpawnStarting` 플래그로 busy를 판정한다.
in-process 경로는 `ChildProcess`가 없으므로 **별도 busy 플래그**가 필요하다.

### 설계

```typescript
// jwc-runtime.ts 내
let _streaming = false;   // session.prompt() 실행 중 true

export function isJwcStreaming(): boolean { return _streaming; }
```

- `spawnAgent` 어댑터가 `child: null, promise`를 반환할 때, promise가 resolve되는 동안 `_streaming = true`.
- gateway.ts `isAgentBusy()`는 `cli === 'jwc'` 분기에서 `isJwcStreaming()`을 호출하도록 확장.
- **큐 의미론 보존**: 스트리밍 중 새 메시지 도착 시 cli-jaw 현행 queueCtrl 동작(거부 또는 큐잉) 그대로.
  어댑터가 `promise` 수명으로 큐 게이트 계약을 표현하므로 queueCtrl 내부 변경 불필요.

## 5. 직원 dispatch 경계

D112-2 "cli-jaw=인스턴스, jwc=세션" 원칙의 dispatch 함의:

| 대상 | 경로 | 변경 |
|------|------|------|
| **boss 대화** | JawRuntime in-process (`cli='jwc'`) | ✅ 본 플랜 대상 |
| **직원(worker) spawn** | 기존 외부 CLI spawn 유지 | ❌ 변경 없음 |

경계 명확화:

- `mainManaged = true` → jwc 경로.
- `isEmployee = true` (empSid 있음) → 기존 spawn 경로(`discriminate()` → 외부 CLI 바이너리).
- 어댑터 삽입 지점: `spawn.ts:767` `resolveMainCli()` 반환값이 `'jwc'`일 때 분기.
  직원 dispatch는 `resolveMainCli()`가 달라 이 분기에 진입하지 않으므로 경계가 자연 격리된다.

## 6. 롤백 스위치

111 §6 "단계적 전환: settings cli='jwc' 옵트인" 을 구체화:

```jsonc
// ~/.cli-jaw/settings.json
{
  "cli": "jwc",            // 활성화 — jwc in-process 경로
  // "cli": "claude",      // 폴백 — 구 spawn 경로 복원
}
```

- `resolveMainCli()` 반환값 `'jwc'` = 새 경로, 그 외 = 구 경로.
- **bridge 모드 폴백** (111 §착수 전 실측 D6): 완전 폴백 전 중간 단계로,
  cli='jwc-bridge' 값으로 bridge 모드(`jwc --mode bridge` HTTP+SSE)를 선택 가능하게 예비 슬롯 보유.
  in-process 완전 검증 전 1릴리스 완충 옵션.
- 롤백 스위치는 M2 done ① 검증 이후 **150 밴드(기본 승격)** 전까지 1릴리스 유지.

## 수명주기 결정 요약

**Lazy 초기화 + 서버 수명 동안 상주.** 첫 boss 메시지 도착 시 세션을 생성하고, 이후 서버 재시작 시에는
jaw.db에서 메시지를 로드해 세션을 재구성한다. resume/steer 구분은 소멸하고, steer는 살아있는 루프에
`session.prompt(…, { streamingBehavior: "steer" })`로 직접 push한다. 컴팩션 소유권은 gjc 위임.

## 가장 위험한 통합점

**gateway.ts `isAgentBusy()` — in-process busy 플래그 통합.**

111 실측 보강에서 "gateway.ts 실사 미완"으로 명시된 유일한 미확인 앵커다. 기존 경로는
`activeProcess !== null`(ChildProcess 존재 여부)로 busy를 판정하는데, in-process 경로는
ChildProcess가 전혀 없다. 만약 gateway.ts가 cli 값이 아닌 `activeProcess` 참조로 busy를 단락하면
**모든 boss 메시지가 거부 없이 통과** — 동시 두 번 호출이 in-process 세션에 동시에 진입해
메시지 순서 오염과 jaw.db 커밋 충돌을 일으킨다. 착수 전 gateway.ts 실사를 선행 조건으로 지정한다.
