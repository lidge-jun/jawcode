## 0. 왜 audit인가

상주 ACP/in-process 호스트는 **단일 프로세스에서 서로 다른 cwd의 N세션**을 돈다. `process.cwd()`/`getProjectDir()`로 작업 디렉터리를 잡는 코드는 세션 A의 작업을 세션 B의 cwd에서 실행하는 교차오염을 일으킨다. 단, **모든 cwd 참조가 결함은 아니다** — CLI 진입점은 사용자 셸 cwd가 정답.

## 1. 분류 기준 (핵심)

| 부류 | cwd 소스 | 판정 |
|---|---|---|
| **A. CLI 진입점** (`jwc orchestrate`/`goal`/`chat`/`ralplan`/`team` top-level `run()`) | `process.cwd()` = 사용자 셸 | ✅ **올바름 — 수정 금지** |
| **B. in-session 슬래시/도구** (세션 핸들 존재) | `runtime.session.sessionManager.getCwd()` | ⚠️ **결함 — B1 패턴으로 수리** |
| **C. 프로세스 부트스트랩** (1회 기동, 세션 무관) | `process.cwd()`/`getProjectDir()` | ✅ 무해 (단일 세션 가정 전 1회) |

### 2-B. 부류 B (결함 후보 — 수색 대상)

- ✅ **수리됨**: `slash-commands/builtin-registry.ts:333` `/orchestrate` (B1, 260613) — `getCwd()` 전환.
- ⬜ **수색 잔여**: in-session 컨텍스트에서 `process.cwd()`/`getProjectDir()`를 쓰는 다른 슬래시 커맨드·도구. 후보 grep:
  - `grep -rn "process.cwd()\|getProjectDir()" packages/coding-agent/src/slash-commands packages/coding-agent/src/tools packages/coding-agent/src/session` → 각 호출처를 부류 A/B/C로 판정.
  - 특히 `tools/*` 중 세션 cwd가 아닌 전역 cwd를 쓰는 파일(파일 도구가 세션 워크스페이스 밖을 보면 교차오염).
