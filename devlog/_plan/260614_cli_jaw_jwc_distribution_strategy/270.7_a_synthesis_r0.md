## Original design intent

Keep 270 narrowly scoped to Manager selector + `/api/cli-registry` JWC exposure. Do not switch cli-jaw default runtime, do not implement Code mode UI, and do not attach existing Jaw mode JWC runtime.

## Local evidence gathered after audit

- `../cli-jaw/src/http/response.ts:12-14` defines `ok(res, data, extra)` as `{ ok: true, data, ...extra }`; `/api/cli-registry` assertion path must be `body.data.jwc`.
- `../cli-jaw/src/cli/readiness.ts:154` `DEFAULT_ORDER` omits `jwc`.
- `../cli-jaw/tests/unit/cli-registry.test.ts:214-217` currently expects every `CLI_KEYS` entry to appear in `DEFAULT_ORDER`.
- `../cli-jaw/public/manager/src/settings/pages/Agent.tsx:169` derives `cliOptions` from `Object.keys(perCli)`.
- `../cli-jaw/public/manager/src/settings/pages/components/agent/RuntimeHeader.tsx:50-56` filters `cliOptions` by `PRIMARY_CLIS` but preserves `cliOptions` order, so adding `jwc` to `PRIMARY_CLIS` alone does not guarantee placement after `claude-e` / before `agy`.

## Shared root cause

The plan still treated metadata membership as equivalent to runtime/UI behavior. The actual code has two additional ordering owners: `readiness.ts DEFAULT_ORDER` for default fallback selection and `RuntimeHeader` option construction for visible primary ordering.
