## WONDER — what is still missing?

- The GitHub skills repo URL is configurable but still assumes `https://github.com/lidge-jun/cli-jaw-skills.git` as the default upstream; a future spec should decide whether that repo is official product infrastructure or a temporary bridge.
- Registry-faithful mode intentionally skips `jawcode/sdk` import because the currently published registry native package can fail in that path; same-batch local natives cover SDK import. A future release should remove that exception after the dependent registry native version is known-good.
- Native probes are strongest on the maintainer's macOS arm64 environment and on `macos-14` CI, but Linux-specific native behavior is still covered only by install/build/type checks rather than equivalent runtime probes.
- The release workflow still performs raw `npm publish` for `packages/jwc`; workspace-wide publisher dry-run is contract-tested rather than executed in the workflow to avoid mutating package manifests before publish.
- No real GitHub network clone was exercised in the final local validation; local fixture and CI workflow coverage verify the bootstrap mechanics, while actual network availability remains an environment assumption.

## REFLECT — how to improve the spec next time

- Add an explicit acceptance criterion for failure-mode behavior: postinstall network failure must not create an empty active skill root or break fallback discovery.
- Add an explicit acceptance criterion that validation must pass with `CI=true`, because GitHub Actions injects that env by default.
- Make runner architecture a first-class requirement for native probes instead of treating native probes as a generic CI step.
- Specify exactly which publish path is canonical: raw `packages/jwc` publish workflow vs workspace-wide `ci-release-publish.ts`.
- Separate "registry-faithful install" from "same-batch local workspace release" in the ontology so SDK/native expectations are unambiguous.
- Add a release infra decision for the GitHub skills repository ownership, availability, and versioning policy.
