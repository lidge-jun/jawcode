## Why This Is A Gap

GJC progressed from "Telegram can answer current asks" to Telegram-driven lifecycle control: create, close, resume/list sessions through a daemon-side control plane. JWC has no Telegram session lifecycle surface.

This is more powerful than remote answers and should not be bundled into the first Telegram parity pass.

Lifecycle operations:

| operation | risk |
|---|---|
| create session | can spawn a local agent process from Telegram. |
| resume session | can reattach to existing work. |
| close session | can terminate or disable a live agent. |
| list/recent activity | may expose repo/session metadata. |

## Suggested Split

1. Read-only `/sessions` or recent activity command.
2. Close current mapped session only.
3. Resume known session by safe id.
4. Create new session from allowlisted cwd/profile.

## Done Gate

- [ ] All lifecycle commands are paired-chat-only.
- [ ] Unknown chat/topic/session fails closed.
- [ ] Idempotency ledger prevents duplicate create/close/resume.
- [ ] Audit JSONL records request/result without secrets.
- [ ] Create/resume path cannot escape allowed project/session roots.
- [ ] Close cannot kill unrelated user processes.
