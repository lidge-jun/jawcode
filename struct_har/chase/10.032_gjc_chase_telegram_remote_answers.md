## Why This Is A Gap

GJC lets a Telegram user answer pending asks through inline buttons or free-text replies, handles custom/Other input, strips duplicate numbering, shows typing/activity acknowledgements, and preserves ask readability under redaction. JWC has only local TUI/CLI ask handling.

This is user-visible and high value: it is the minimal "phone can unblock the agent" behavior.

## Suggested Split

1. Core SDK ask/reply race without Telegram rendering.
2. Telegram inline keyboard rendering and callback routing.
3. Free-text answer/custom input support.
4. Redaction + verbosity commands.
5. Activity/ack UX.

## Done Gate

- [ ] Remote button answer resolves the active local ask and later local answer is ignored.
- [ ] Local answer resolves ask and Telegram later reply is rejected/already answered.
- [ ] Free-text/custom answer can satisfy an active ask where allowed.
- [ ] Redaction does not redact ask question/options.
- [ ] Telegram option labels are not double-numbered.
- [ ] Assistant lead-in is emitted before ask prompt.
- [ ] Inbound acknowledgements are idempotent and scoped to the mapped session.
