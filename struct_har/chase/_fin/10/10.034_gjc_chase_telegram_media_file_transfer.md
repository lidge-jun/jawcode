# 10.034 — gjc chase: Telegram media, attachments, and `telegram_send`

## Why This Is A Gap

GJC can send agent-produced images/files to Telegram, accept inbound Telegram media attachments, and expose a `telegram_send` tool that sends a workspace-contained file as a Telegram document. JWC has no equivalent file/media transfer path.

This is useful but expands the egress surface. It should come after basic notification and ask-answer parity.

| tests | `telegram-send-tool.test.ts`, `notifications-threaded-render.test.ts`, `notifications-threaded-inbound.test.ts` |

GJC `telegram_send` safety boundary:

| rule | detail |
|---|---|
| workspace-confined | requested path resolves through realpath under session cwd. |
| regular files only | directories and missing files rejected. |
| symlink escape blocked | `realpath` + `path.relative` confinement. |
| connected session required | no active Telegram sink => error. |
| caption optional | no raw token exposure. |

## JWC Reconcile Notes

| issue | decision pressure |
|---|---|
| egress risk | A model can ask to send a file; workspace-only confinement is mandatory. |
| binary size | Base64 file payloads may be large; define size limits if JWC differs. |
| inbound media | Decide whether inbound images become user message attachments or are ignored first. |
| discoverability | `telegram_send` should be discoverable only when Telegram notifications are configured and connected. |

## Suggested Split

1. Outbound image/file frame render only.
2. `telegram_send` tool with workspace confinement.
3. Inbound media attachments to user-message injection.
4. Size limits and MIME allowlist hardening.
