## Why This Is A Gap

GJC progressed from "Telegram can answer current asks" to Telegram-driven lifecycle control: create, close, resume/list sessions through a daemon-side control plane. JWC has no Telegram session lifecycle surface.

This is more powerful than remote answers and should not be bundled into the first Telegram parity pass.

## Suggested Split

1. Read-only `/sessions` or recent activity command.
2. Close current mapped session only.
3. Resume known session by safe id.
4. Create new session from allowlisted cwd/profile.

## JWC Final Close — 2026-06-28 (slices A phase 11 + B/C phase 52)

Closed: the **safe JWC lifecycle control surface**. Telegram-driven process execution
(create/resume/close) remains explicitly **deferred** to a later C4 architecture/security decision, as
this card always stipulated. What is implemented and tested:

- **10.033-A** (phase 11): `lifecycle-command-parser.ts` — inert intents `{list|new|close_current|
  resume(safeId)}`; no cwd/prompt/profile/shell/env accepted from Telegram.
- **Authorization gate** (phase 52): `executeLifecycleIntent` is fail-closed — paired-chat-only, with
  per-session ownership required for resume/close; unknown chat/session ⇒ reject.
- **10.033-B** (phase 52): `LifecycleSessionSummary` read-only listing (id/title/updatedAt only — no
  secrets, paths, or prompt bodies).
- **10.033-C** (phase 52): `LifecycleIdempotencyLedger` (bounded, replay returns first outcome) +
  `buildLifecycleAuditRecord` JSONL record that never carries the bot token or message text.
- Mutating intents (`new`/`resume`/`close_current`) are authorized, idempotency-tracked, and audited,
  but resolve to `execution_deferred` — they never spawn, attach to, or kill a process.
