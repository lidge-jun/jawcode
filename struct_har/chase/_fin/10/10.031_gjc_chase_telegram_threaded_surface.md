## Why This Is A Gap

GJC maps each running session to a Telegram forum topic in the paired private chat, sends an identity header, renders context/turn frames, and routes replies by `message_thread_id`. JWC has no per-session Telegram topic registry or threaded rendering surface.

This card depends on [10.030](./10.030_gjc_chase_telegram_managed_daemon.md).

## Suggested Split

1. Pure topic registry + serializer.
2. Pure HTML/threaded renderer for identity/context/turn/config.
3. Inbound fail-closed router.
4. Runtime integration in Telegram daemon.
5. Topic lifecycle: create, rename, delete/fallback.

## Done Gate

- [ ] Session topic is created or flat fallback is explicitly used without dropping notifications.
- [ ] Duplicate concurrent topic creation is deduped.
- [ ] Inbound update from wrong chat/no topic/unknown topic/duplicate update is ignored.
- [ ] In-topic free text injects only to the mapped session.
- [ ] Identity header is sent once per active topic.
- [ ] Topic deletion on session shutdown is best-effort and never breaks delivery.
