## 왜 gap인가

Upstream GJC dev added a `#contentBlocksCache` WeakMap to `AssistantMessage` that caches rendered text and thinking block components. When the source text hasn't changed, the cached Markdown component is reused instead of re-creating it. This is a pure performance optimization — JWC currently re-renders on every update.

## Source facts

| fact | upstream evidence |
|---|---|
| `#contentBlocksCache` WeakMap keyed by content block | `assistant-message.ts` — avoids re-creating Markdown components |
| `#renderTextBlock` method with cache check | returns cached component when text unchanged |
| `#renderThinkingBlock` method with cache check | same pattern for thinking blocks |
| `renderDeepInterviewAssistantText` rendering path | interview-specific text rendering in `#renderTextBlock` |

## JWC reconcile notes

- JWC has `thinking collapse/expand` feature (agent-side) that upstream lacks — cache must coexist with collapse state.
- The cache is independent of collapse — cache key is the content block identity, not the expanded/collapsed state.
- Performance impact is proportional to message count; most visible in long conversations.
