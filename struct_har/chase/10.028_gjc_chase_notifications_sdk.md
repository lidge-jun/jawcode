## Why This Is A Gap

GJC now has a transport-agnostic notifications SDK: each running session exposes a local authenticated WebSocket endpoint, writes a discovery file, emits action-needed/session/stream frames, and accepts remote replies or injected user messages. JWC currently has no equivalent `packages/coding-agent/src/notifications/` module and no Rust/N-API `NotificationServer` surface.

This is the base layer for Telegram parity. Do not start from Telegram Bot API code until this SDK contract exists or is consciously rejected.

## Source Facts

| capability | upstream anchor |
|---|---|
| Rust protocol crate | `crates/gjc-notifications/src/protocol.rs` |
| loopback WebSocket server | `crates/gjc-notifications/src/server.rs` |
| action registry / reply lifecycle | `crates/gjc-notifications/src/actions.rs` |
| endpoint discovery file | `crates/gjc-notifications/src/discovery.rs` |
| N-API wrapper | `crates/pi-natives/src/notifications.rs` |
| TS notification core | `packages/coding-agent/src/notifications/index.ts` |
| SDK docs | `docs/notifications-sdk.md` |

Important wire frames:

| direction | minimal frames | extended threaded frames |
|---|---|---|
| server -> client | `action_needed`, `action_resolved`, `reply_rejected` | `identity_header`, `context_update`, `turn_stream`, `image_attachment`, `file_attachment`, `activity`, `inbound_ack`, `session_ready`, `session_closed`, `config_update`, `hello`, `pong` |
| client -> server | `reply` | `user_message`, `config_command`, `hello`, `ping` |

Endpoint discovery in GJC:

```text
<repo>/.gjc/state/notifications/<sessionId>.json
```

JWC equivalent must be:

```text
<repo>/.jwc/state/notifications/<sessionId>.json
```

## JWC Reconcile Notes

| issue | decision pressure |
|---|---|
| Rust crate name | Use `jwc` public docs, but internal crate naming may either fork to `jwc-notifications` or retain vendored names only if never public. |
| state path | Must be `.jwc/state/notifications`, not `.gjc`. |
| token in discovery file | Required for local clients; must be `0600`/atomic and never logged. |
| reply semantics | Remote ask answer must resolve the same gate path as local ask; reject if resolver unavailable. |
| forward compatibility | Unknown frames must be tolerated. |
| replay | Buffered ask/session-ready frames are replayable; idle/context streams are not durable history. |

## Suggested Split

1. Port protocol types + TS-facing adapter with JWC names.
2. Add loopback server and discovery file writer.
3. Wire session startup/shutdown to create/remove endpoint files.
4. Wire ask gate forwarding and `action_resolved` only after real resolution.
5. Add idle/session-ready/context optional frames only after minimal ask path is proven.
