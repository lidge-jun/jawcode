## Why This Is A Gap

GJC ships a managed Telegram daemon that scans notification endpoint files, owns the Telegram `getUpdates` long poll, sends per-session messages, and routes inbound updates back to sessions. JWC has no daemon runtime or singleton poller for Telegram.

This card depends on [10.028](./_fin/10/10.028_gjc_chase_notifications_sdk.md) and [10.029](./10.029_gjc_chase_notify_config_cli.md).

Key invariants:

| invariant | detail |
|---|---|
| one token/chat owner | Telegram allows one active `getUpdates`; daemon prevents 409 conflict by lock/state ownership. |
| roots registry | sessions register roots in daemon state, daemon scans `.gjc/state/notifications/*.json`. JWC must use `.jwc`. |
| heartbeat | owner has `heartbeatAt`; stale owners can be replaced only when pid/deadness rules pass. |
| no overlap reload | reload writes owner-scoped control request, signals old daemon, waits for pid death before spawning fresh poller. |
| bounded retries | transient network errors retry; persistent poll failure backs off. |
| session scan independent of long poll | scan interval is separate so asks do not wait for a long `getUpdates` poll to return. |

## Suggested Split

1. Port root registration + singleton state files with JWC names.
2. Add hidden `jwc notify daemon-internal` smoke.
3. Add daemon scanner that connects to notification endpoints.
4. Add Telegram poll loop with retry/backoff and owner lock.
5. Add `daemon status/reload/stop` only if product wants public daemon control.
