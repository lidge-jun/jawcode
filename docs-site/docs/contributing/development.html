Development Setup

Get a working local development environment for Jawcode. This guide covers cloning, dependencies, running from source, and the test/verification workflow.

Prerequisites

ToolVersionNotes
BunLatestRequired for builds, tests, and running from source
Node.js>= 18Required for npm compatibility layer
GitAnyRequired for cloning and repo-aware features
Rust toolchainLatest stableRequired only if modifying crates/pi-natives
tmuxAnyOptional; needed for --tmux mode and jwc session

Clone and install

git clone https://github.com/lidge-jun/jawcode.git
cd jawcode

# Install all workspace dependencies
bun install

# Install bundled default skills and workflow definitions
bun run install:defaults

bun install resolves the monorepo workspace. bun run install:defaults copies bundled skill definitions into the local .jwc/ runtime directory for development testing.

Run from source

# Run the CLI directly
bun packages/jwc/bin/jwc.js --help

# Or with a specific command
bun packages/jwc/bin/jwc.js --version
bun packages/jwc/bin/jwc.js --smoke-test
bun packages/jwc/bin/jwc.js --list-models

You can also link the binary globally for convenience during development:

bun link
jwc --version

Repository structure

Jawcode is a monorepo with multiple packages. The primary product surface is packages/coding-agent/:

PackageDescription
packages/coding-agentMain JWC CLI application
packages/aiMulti-provider LLM client with streaming
packages/agentAgent runtime with tool calling and state management
packages/tuiTerminal UI library with differential rendering
packages/nativesNative text/image/grep bindings
packages/statsLocal observability dashboard (jwc stats)
packages/utilsShared utilities
crates/pi-nativesRust native helpers (NAPI-RS)

Bundled workflow skills and role agents are source files, not committed .jwc copies:

packages/coding-agent/src/defaults/jwc/skills/<name>/SKILL.md   # workflow skills
packages/coding-agent/src/prompts/agents/<role>.md              # role agents

Testing

Tests use Bun's built-in test runner with Vitest-compatible assertions.

Running tests

# Run all tests
bun test

# Run tests for a specific package
bun test packages/coding-agent/

# Run a single test file
bun test packages/coding-agent/test/default-jwc-definitions.test.ts

Testing philosophy

Test externally observable contracts: behavior, output shape, state transition, error mapping, or regression-prone parsing boundaries. Specifically:

  • Avoid placeholder tests and tautologies.
  • Avoid broad not.toThrow() assertions — they prove nothing useful.
  • Avoid duplicated coverage across test files.
  • Avoid long-lived global mutations in test setup.
  • Avoid mock.module() — prefer vi.spyOn(...) with explicit cleanup.
  • Do not write runtime tests for compile-time guarantees — those belong in type checks.

Type checking

# Type check all packages (never use tsc/npx tsc directly)
bun check
# or
bun run check:ts

Verification gates

After modifying workflow definitions, skill registrations, or public-facing names, run the full gate suite:

bun scripts/check-visible-definitions.ts
bun scripts/verify-g002-gates.ts
bun scripts/rebrand-inventory.ts --strict
bun test packages/coding-agent/test/default-jwc-definitions.test.ts

All four must pass before committing. See the Contributing Guide for details on what each gate checks.

Commit and release

  • Stage only intended files. Preserve unrelated work.
  • Add changelog entries under ## [Unreleased] in packages/*/CHANGELOG.md. Never edit released sections.
  • Release flow: bun run release after all changelogs and verification are complete.

Environment variables

At minimum, set one LLM provider key:

export ANTHROPIC_API_KEY="sk-ant-..."   # or any of the 46 supported providers
