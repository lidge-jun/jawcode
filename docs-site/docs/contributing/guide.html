Contributing Guide

Jawcode enforces strict code-quality rules across all packages. Every contribution — whether from a human or an automated agent — must follow these conventions. The rules exist to keep the codebase predictable and grep-friendly.

Code quality rules

No any

Never use any unless absolutely necessary. If an external API forces an untyped boundary, add an explicit cast with a comment explaining why. Prefer unknown and narrow with type guards.

No ReturnType<>

Write the actual type name instead of deriving it with ReturnType<typeof fn>. Explicit types are easier to search, rename, and reason about.

No inline imports

All imports must be top-level static declarations. The following patterns are banned:

// BANNED — dynamic import
const mod = await import("./heavy.ts");

// BANNED — inline type import
type Foo = import("./types").Foo;

// CORRECT — top-level
import type { Foo } from "./types";
import { heavy } from "./heavy.ts";

ES #private fields

Use ES native #private fields. Do not use the TypeScript private, protected, or public keywords on fields or methods, except for constructor parameter properties where TypeScript requires the modifier:

class Agent {
  #state: AgentState;          // correct — ES private
  #loop(): void { ... }        // correct

  constructor(private readonly id: string) {} // allowed — ctor param property
}

Promise.withResolvers()

Use Promise.withResolvers() instead of the new Promise((resolve, reject) => ...) constructor pattern. It produces cleaner code and avoids the nested-callback shape.

Barrel exports

Prefer export * from "./module" in barrel files. If star exports create ambiguity (duplicate names from different modules), remove the redundant path rather than switching to named re-exports.

Prompts in .md files

All prompts live in static .md files imported with the with { type: "text" } assertion. Never build prompts inline in TypeScript code:

// CORRECT
import systemPrompt from "./prompts/system-prompt.md" with { type: "text" };

// BANNED — inline string template
const prompt = `You are an agent that...`;

Check node_modules for types

When consuming an external library, check its actual type declarations in node_modules instead of guessing the API surface. This prevents phantom type mismatches and keeps imports accurate.

models.json is generated

Never edit packages/ai/src/models.json directly. Modify the generator, descriptors, or resolvers, then regenerate:

bun --cwd=packages/ai run generate-models

Bun and filesystem conventions

Prefer Bun APIs where they are cleaner:

OperationUseAvoid
File read/writeBun.file(), Bun.write()readFileSync, writeFileSync
Spawn commandsBun Shell ($`cmd`)child_process
SleepBun.sleep(ms)timeout promises
JSON5/JSONLBun.JSON5, Bun.JSONLad-hoc parsers
String widthBun.stringWidth, Bun.wrapAnsicustom ANSI wrapping

Use namespace imports for Node built-in modules:

import * as fs from "node:fs/promises";
import * as path from "node:path";
import * as os from "node:os";

Use node:fs/promises for directory operations. Avoid redundant parent-directory creation before Bun.write() — Bun creates parent directories automatically.

Worker scripts

Spawn workers with the compile-safe hybrid pattern that works in both source and compiled binary modes:

import { isCompiledBinary } from "@gajae-code/pi-utils";

const worker = isCompiledBinary()
  ? new Worker("./packages/<pkg>/src/<worker>.ts", { type: "module" })
  : new Worker(new URL("./<worker>.ts", import.meta.url).href, { type: "module" });

Every worker entry must also be listed as an extra compile entrypoint in packages/coding-agent/scripts/build-binary.ts.

Logging and TUI safety

Do not use console.log, console.warn, or console.error in packages/coding-agent/. Console output corrupts TUI rendering. Use the centralized logger from @gajae-code/pi-utils.

All text displayed in tool renderers must be sanitized:

  • Tabs to spaces via replaceTabs()
  • Truncation via truncateToWidth() / ui.truncate()
  • Home paths shortened via shortenPath()
  • Previews bounded by shared preview constants

Verification gates

After any workflow-definition or rebrand-surface change, run the full gate suite:

# All four gates — run in order
bun scripts/check-visible-definitions.ts
bun scripts/verify-g002-gates.ts
bun scripts/rebrand-inventory.ts --strict
bun test packages/coding-agent/test/default-jwc-definitions.test.ts
GateWhat it checks
check-visible-definitionsOnly the four public workflow skills + two tool-help skills are registered as defaults
verify-g002-gatesG002 rebrand contract — all public-facing names use jwc/.jwc
rebrand-inventory --strictNo stale upstream names leak into user-visible surfaces
default-jwc-definitions.testBundled skill/agent definitions load correctly and match expected shapes

General rules

  • Never run tsc or npx tsc; use bun check / bun run check:ts.
  • For focused changes, prefer targeted tests first, then type/lint/build checks.
  • Commit completed changes proactively; stage only intended files and preserve unrelated work.
  • Package changelogs live at packages/*/CHANGELOG.md. Add entries under ## [Unreleased]; never edit released sections.
  • Release flow: bun run release after changelogs and verification.
