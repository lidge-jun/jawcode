Subagent System Overview

Jawcode delegates bounded work to subagents — independent agent instances that run asynchronously while the main session continues answering the user, doing independent work, or simply waiting.

Fork / Spawn / Resume

The subagent lifecycle has three core operations: 

Main Session
  |
  |-- spawn(executor, assignment) --> Subagent A  (running)
  |-- spawn(executor, assignment) --> Subagent B  (running)
  |                                       |
  |   ... main keeps working ...          |-- yield(result)
  |                                       |
  |-- await(A, timeout) <-- result        |
  |-- resume(B, "correction") ----------> B re-enters with CORRECTION_DEANCHORING
  |                                       |-- yield(revised_result)
  |-- inspect(B) <-- revised_result
  • spawn — launches a detached subagent with a self-contained assignment. The main session is not blocked; it returns immediately.
  • await — waits for completion with a timeout. Timeout is an observation window, not a failure — the subagent keeps running.
  • resume / steer — sends a message into a running, paused, or completed subagent. When the message corrects a prior answer, the correction de-anchoring protocol activates.

Async by Default

Every task launch returns immediately as a detached background subagent. While subagents work, the main session can:

  • Continue answering the user
  • Do independent work (edits, searches, other spawns)
  • Simply wait — there is no obligation to stay busy

The main session must never duplicate work that a subagent is already doing. If research or implementation was delegated to a subagent, do not perform the same searches or edits yourself — wait for their result.

Five Role Agents

Jawcode bundles five callable role agents. Each is a specialization of the base subagent runtime with different constraints and output contracts:

Agent Mode Purpose
executor Read-write Default subagent. Implementation, fixes, research. Uses the parent model via self-fork.
executor_ext Read-write External/fresh model lane. Different model perspective or independent verification.
architect Read-only Architecture review. CLEAR / WATCH / BLOCK status with severity-rated findings.
planner Read-only Sequencing, acceptance criteria, risk mapping, and execution handoff.
critic Read-only Plan critique. OKAY / ITERATE / REJECT with evidence-backed justification.

Routing Rules

The main session routes work to subagents based on these rules:

  • Generic parallel work — executor (default)
  • User requests external/fresh/model-diverse work — executor_ext
  • User names a specific model + generic worker — executor_ext with explicit provider/modelId[:effort]
  • User explicitly requests specialist lens — architect, planner, or critic
  • IPABCD B-stage slices — executor unless the user requests external lane

Specialist role agents (architect, planner, critic) are never auto-routed. They activate only when the user explicitly asks for that lens.

Shared Guardrails

All subagents inherit a common set of guardrails from the subagent-system-prompt.md template:

  • Completion contract — no fabricated outputs, no suppressed tests, verification claims must match what was actually run
  • Repo safety — never revert, stash, commit, push, or delete user work unless the assignment explicitly requires it
  • Authority-bias prevention — verify the parent's factual claims (file paths, symbol names, behavior descriptions) with own tool calls
  • Sunk-cost awareness — if significant effort yields no progress, evaluate whether the approach itself is wrong instead of continuing
  • Confirmation-bias prevention — seek disconfirming evidence; if first checks all support one hypothesis, run at least one that would falsify it
  • Correction de-anchoring — on resume with a correction, discard prior analysis and generate 3+ fresh hypotheses from different angles

Coordination

Subagents coordinate through two mechanisms:

  • IRC — lightweight peer-to-peer messaging between live agents. Use for quick questions; not for long-form content transfer.
  • Filesystem — subagents read/write to the shared repository. When working in isolated worktrees, they never modify files outside their tree.

The main session can manage subagents through the subagent tool: list, inspect, await, pause, resume, steer, and cancel.
