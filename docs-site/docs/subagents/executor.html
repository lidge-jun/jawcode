Executor & Executor Ext

The two read-write subagent roles. executor is the default self-fork lane; executor_ext is the external/fresh model lane for independent verification or model-diverse work.

Executor (Default Fork)

The executor is the workhorse subagent. It converts a scoped assignment into a working, verified outcome. Key characteristics:

  • Model: inherits the parent session model (self-fork). This is prompt-cache-friendly — the same model means the static system prompt prefix is already cached.
  • Thinking level: medium
  • Fork context: can receive a sanitized snapshot of the parent conversation as read-only background
  • Scope: read-write. Can edit files, run commands, commit.

Execution Loop

1. Inspect relevant files, tests, and conventions
2. Make a compact file-level plan for non-trivial changes
3. Implement the minimal correct change
4. Run focused checks if the parent assigns verification
5. Remove debug leftovers and report changed files + evidence

Constraints

  • Keep diffs small, reversible, and aligned to existing patterns
  • Do not broaden scope or invent abstractions
  • Explore first, ask last — ask only when progress is impossible or the next decision is destructive, credentialed, or materially scope-changing
  • Respect repository instructions (AGENTS.md); deepest/nearest wins
  • No new dependencies unless explicitly requested

Failure Recovery

When an approach fails, the executor diagnoses why before switching tactics. It reads the full error, checks assumptions, and tries a focused fix. It does not retry the identical action blindly. After materially different failed approaches, it stops adding risk and reports the blocker. If significant effort yields no progress, it evaluates whether the approach itself is wrong — sunk effort is not a reason to continue.

For corrections received via resume or steer, the executor follows the CORRECTION_DEANCHORING protocol instead.

Goal Red-Team Mode

When explicitly labeled as a QA/red-team executor, the agent shifts from implementation to adversarial verification:

  • Starts from the approved plan and acceptance criteria
  • Exercises the real user-facing invocation (browser automation for web, terminal for CLI, black-box tests for API)
  • Tries to break the work with adversarial cases
  • Reports a structured QA matrix: contractCoverage, surfaceEvidence, adversarialCases, artifactRefs

Executor Ext (External Model Lane)

executor_ext reuses the same executor behavior but selects a different model. It exists for three use cases:

Use Case How to Invoke
User wants an external/fresh perspective executor_ext with .model omitted — uses the configured EXECUTOR_EXT target
User names a specific model executor_ext with .model: "provider/modelId[:effort]"
Independent verification of executor work executor_ext — a different model is less likely to share the same blind spots

Model Routing

# Configured external executor (uses EXECUTOR_EXT env/config)
{ agent: "executor_ext", tasks: [{ id: "Review", assignment: "..." }] }

# One-off named model
{ agent: "executor_ext", tasks: [{
  id: "Review",
  assignment: "...",
  model: "openai/gpt-4.1:high"
}] }

The executor_ext lane is a callable task role and model target, not a workflow skill. It does not appear in the .jwc defaults.

Executor vs Executor Ext

Property executor executor_ext
Model Parent model (self-fork) Configured EXECUTOR_EXT or explicit provider/modelId
Cache affinity Shares parent prefix cache Separate cache (different model)
Default routing All generic parallel work Only on explicit user request
Read-write Yes Yes
Fork context Allowed Allowed

Delegation Rules

Executors default to direct execution inside their assigned scope. They do not recursively delegate unless the assignment explicitly permits it and the subtask is independent. The main session controls fan-out — executors are leaves, not orchestrators.
