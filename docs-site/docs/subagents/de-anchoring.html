Correction De-Anchoring

When a subagent is resumed or steered with feedback that its previous answer was wrong, the CORRECTION_DEANCHORING protocol forces it to abandon its prior framing and investigate from scratch.

The Problem: Anchoring Bias

When an LLM produces an answer and is then told "that's wrong, try again," it tends to patch its prior conclusion — making small adjustments within the same frame rather than genuinely reconsidering. This is anchoring bias: the original answer acts as a cognitive anchor that constrains the revised response.

In the subagent context, this is especially dangerous because:

  • The subagent has its entire prior analysis in context
  • The prior analysis may have been confidently presented with evidence
  • Minor edits to a wrong conclusion rarely reach the right one
  • The parent agent already judged the prior answer insufficient

The Protocol

When a subagent is resumed or steered with feedback that its previous answer was wrong, incomplete, or missed the root cause: 

[CORRECTION_DEANCHORING]

1. Set aside your prior analysis entirely.

2. Generate at least 3 materially different hypotheses
   or debugging angles you have not yet explored.

3. Investigate each with fresh tool calls and evidence.

4. Only then form a new conclusion from the evidence
   — not from editing your old one.

Step-by-Step

Step What Happens Why
1. Discard Prior analysis is treated as a failed hypothesis, not a starting point Breaks the anchor. The old frame is poisoned.
2. Diverge Generate 3+ materially different hypotheses or angles Forces exploration of the solution space, not local search around the old answer
3. Investigate Each hypothesis gets fresh tool calls — read, search, ast_grep Evidence-based, not reasoning-based. The tools are the source of truth.
4. Conclude New conclusion is formed from the new evidence only The conclusion follows the evidence, not the other way around

When It Activates

The protocol activates only when the resume/steer message corrects the prior answer. It does not activate for:

  • Simple additive requests: "also check X", "add Y to the list"
  • Scope expansions: "now do the same for file B"
  • Clarifications: "I meant the other module"

The subagent determines activation by reading the intent of the correction message. If the message says the previous analysis was wrong, incomplete, or missed the root cause, the protocol activates.

Example

# Main session spawns executor to find a bug
spawn(executor, "Find why user sessions expire after 5 minutes")

# Executor concludes: "The session TTL is hardcoded to 300s in config.ts"
yield({ data: { cause: "hardcoded TTL", file: "config.ts", line: 42 } })

# Main session: "That's wrong — I checked config.ts, TTL is 3600s"
steer(executor, "Your config.ts analysis is wrong. TTL is 3600s there.")

# CORRECTION_DEANCHORING activates:
# 1. Discard: config.ts TTL is no longer the hypothesis
# 2. Generate 3+ new angles:
#    - Redis session store expiry settings
#    - Load balancer sticky session timeout
#    - Cookie maxAge mismatch
#    - Token refresh race condition
# 3. Investigate each with fresh tool calls
# 4. Conclude from new evidence

Interaction with Failure Recovery

The executor's normal <failure_recovery> handles approach failures — when a tool call fails or an implementation doesn't work. CORRECTION_DEANCHORING handles external corrections — when the parent tells you your conclusion was wrong.

Scenario Protocol
Tool call returns an error failure_recovery — diagnose why, try focused fix
Implementation doesn't compile failure_recovery — read error, check assumptions
Parent says "your answer is wrong" CORRECTION_DEANCHORING — 3+ fresh hypotheses
Parent says "also check X" Neither — simple additive request

Design Rationale

This protocol was developed after observing that resumed subagents consistently produced minor variations of their original (wrong) answer. The 3-hypothesis minimum forces genuine divergent exploration. The "fresh tool calls" requirement prevents the subagent from reusing cached mental models built on the wrong premise.

The conditional activation ("only on corrections, not additive requests") prevents wasted effort — a simple "also check X" message doesn't need the full de-anchoring ceremony.
