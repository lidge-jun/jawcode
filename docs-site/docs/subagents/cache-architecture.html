Cache Architecture

Jawcode's subagent spawning is prompt-cache-aware by design. The system prompt is split at a static/dynamic boundary so the cacheable prefix is shared across spawns, and resumed agents reuse cached session context instead of rebuilding from scratch.

The Prefix Cache Problem

LLM providers (Anthropic, DeepSeek, and others) cache the prefix of a conversation — the leading bytes of the system prompt and message history. If the prefix stays identical across API calls, the provider serves it from cache at reduced cost and latency. If even one byte changes in the prefix, the entire cache misses.

For coding agents that spawn subagents, this creates a challenge: each subagent has a different assignment, different context, and different tools. Naively, every spawn is a cold start with zero cache hits.

Static / Dynamic Boundary

Jawcode solves this by splitting the subagent system prompt into two zones: 

+------------------------------------------+
|  STATIC PREFIX (cacheable)               |
|  - Role definition                       |
|  - Constraints                           |
|  - Execution loop                        |
|  - Success criteria                      |
|  - Failure recovery                      |
|  - Completion contract                   |
|  - Correction de-anchoring               |
|  - Tool specifications                   |
+------------------------------------------+  <-- cache boundary
|  DYNAMIC SUFFIX                          |
|  - Assignment text                       |
|  - Shared context                        |
|  - Fork context snapshot                 |
|  - IRC peer list                         |
|  - Worktree path                         |
|  - Output schema                         |
+------------------------------------------+

The static prefix is identical across all subagents of the same role. Every executor spawn shares the same prefix; every architect spawn shares a different but internally consistent prefix. Only the dynamic suffix varies per task.

Append-Only Context

The main session uses an append-only context manager (AppendOnlyContextManager) that stabilizes the byte prefix sent to the LLM across turns:

  • StablePrefix — snapshots the system prompt and tool specifications on first build. Subsequent calls reuse the cached copy unless the live state's fingerprint changes.
  • Append-only message log — messages are only appended, never reordered or mutated. This means prior-turn bytes are stable and only the new user message is a cache miss each turn.

The tool spec is sanitized to exclude per-turn _i intent fields, keeping the tool definition bytes identical across turns.

Spawn Cache Flow

# Turn 1: First executor spawn
Main --spawn(executor, task_A)--> Provider
  Cache: MISS (prefix written to cache)
  Tokens: full system prompt + tools + assignment

# Turn 2: Second executor spawn (different task)
Main --spawn(executor, task_B)--> Provider
  Cache: HIT on static prefix
  Tokens: only assignment delta is new

# Turn 3: Resume executor A with correction
Main --resume(A, correction)--> Provider
  Cache: HIT on prefix + prior conversation
  Tokens: only the correction message is new

# Turn 4: Architect spawn (different role)
Main --spawn(architect, review)--> Provider
  Cache: MISS (different prefix for architect role)
  Tokens: full architect system prompt + tools + assignment

Token Accounting

Jawcode tracks cache performance at the per-call and aggregate level:

Metric Description
inputTokens Total cost-bearing input: base + cacheRead + cacheWrite
cachedInputTokens Tokens served from provider cache (reduced cost)
cacheWriteTokens Tokens written to provider cache on this call
outputTokens Generated tokens
reasoningOutputTokens Tokens consumed by chain-of-thought reasoning

These metrics are visible in the TUI footer, session receipts, and the jwc stats observability dashboard. The telemetry system emits OpenTelemetry-compatible attributes for cache hit ratios: 

gen_ai.usage.cache_read.input_tokens
gen_ai.usage.cache_creation.input_tokens

Handoff Cache Preservation

When the compaction system hands off to a fresh context (e.g., after hitting the context window limit), it passes the live agent's system prompt and tool list verbatim so the provider hits the cached prefix on the first turn of the new context. This is explicitly documented in the handoff interface: 

interface HandoffOptions {
  /** Live agent system prompt — passed verbatim
      so providers hit the cached prefix. */
  systemPrompt: string[];
  /** Live agent tool list — same purpose.
      Forced to toolChoice: "none". */
  tools?: AgentTool[];
}

Self-Fork Cache Affinity

The executor role uses model: self, which means it inherits the parent session's model. This is a deliberate cache optimization: the same model means the same tokenizer, which means the same byte representation of the static prefix. If the parent uses claude-sonnet-4-20250514, all executor spawns use the same model and share the provider's prefix cache.

executor_ext breaks this affinity by design — it uses a different model (the configured EXECUTOR_EXT target or an explicit provider/modelId). This means a separate prefix cache, which is the cost of getting an independent model's perspective.

Provider Session Affinity

Some providers support session-level cache affinity through a providerSessionId. Jawcode forwards this identifier so that multiple API calls within the same agent session share a provider-side cache slot. This is separate from the prefix cache — it allows providers to maintain session-scoped transport and computation caches. 

interface AgentOptions {
  /** Provider-facing cache/session affinity identifier. */
  providerSessionId?: string;
  /** Shared provider state map for session-scoped
      transport/session caches. */
  providerSessionState?: Map<string, ProviderSessionState>;
}
