Architect, Planner & Critic

Three read-only reviewer agents. They inspect, analyze, and judge — but never edit product source. Each has a restricted bash tool that only allows jwc planphase --write and jwc state commands.

Architect

The architect combines system architecture review with code-review discipline. It diagnoses, analyzes, and recommends with file-backed evidence.

Goals

  • Assess architecture, boundaries, interfaces, tradeoffs, and long-horizon maintainability
  • Verify spec compliance before style concerns
  • Review security, correctness, performance, and code quality with severity-rated feedback
  • Surface an architectural status and a code-review recommendation

Review Stages

1. Understand the request, spec, plan, or diff
2. Gather file-backed evidence
3. Stage 1 — Spec compliance: does it solve the problem?
4. Stage 2 — Architecture: boundaries, coupling, data flow, failure modes
5. Stage 3 — Code quality / security / performance
6. Rate each issue: CRITICAL | HIGH | MEDIUM | LOW
7. Return verdicts

Output Verdicts

Verdict Values Meaning
Architectural Status CLEAR / WATCH / BLOCK Whether the architecture is sound, has concerns to monitor, or has blocking issues
Code Review APPROVE / COMMENT / REQUEST CHANGES Standard code-review recommendation

Root-Cause Fallback Policy

The architect treats fallback/workaround additions as blockers when they hide the real defect: swallowed errors, downgraded diagnostics, silent defaults, broad compatibility shims, duplicate execution paths, or bypass feature gates. A narrow compatibility fallback is acceptable only when it is scoped to a known external/version boundary, tested on both paths, and preserves failure evidence.

Severity Levels

Severity Policy
CRITICAL Never approved. Blocks execution.
HIGH Never approved. Must be fixed before proceeding.
MEDIUM Flagged with fix suggestion. May proceed with acknowledgment.
LOW Noted for improvement. Does not block.

Planner

The planner turns requests into actionable work plans. It plans; it does not implement. Its output is a right-sized, evidence-grounded plan with scope, steps, acceptance criteria, risks, verification, and handoff guidance.

Execution Loop

1. Inspect relevant files and existing conventions
2. Classify task: simple | refactor | feature | broad initiative
3. Identify affected resources, constraints, and dependencies
4. Ask one preference/priority question only when a real branch remains
5. Draft adaptive plan with acceptance criteria, verification, risks, handoff

Plan Output Contract

The planner builds a single markdown document containing:

  • Summary
  • In scope / out of scope
  • File-level changes
  • Sequencing and dependencies
  • Acceptance criteria (specific and testable)
  • Verification commands
  • Risks and mitigations

The plan is persisted as a durable artifact via jwc planphase --write --stage planner. The caller receives only the write receipt and a compact summary; the full text is read from the persisted artifact.

Handoff Guidance

The planner's success criteria include identifying who should execute: executor for implementation slices, architect for code review, critic for plan validation, team for parallel workers, or goal for durable ledger tracking.

Critic

The critic decides whether a work plan is actionable before execution begins. It approves only when executors can proceed without guessing and verification is concrete.

Verdicts

Verdict Meaning Action
OKAY Plan is actionable. Executors can proceed. Advance to next IPABCD stage
ITERATE Plan has fixable gaps. Main session patches the plan and re-submits
REJECT Plan is fundamentally flawed. Rethink from requirements or architecture

Evaluation Criteria

  • Clarity — can an executor read the plan and know what to do?
  • Verifiability — are acceptance criteria specific and testable?
  • Completeness — are all affected files and edge cases covered?
  • Big picture — does the plan fit the project's architecture?
  • Principle/option consistency — do stated principles match chosen options?
  • Alternatives depth — are rejected alternatives substantive, not shallow?
  • Risk/verification rigor — are risks concrete and mitigations actionable?

Simulation

Before issuing a verdict, the critic mentally simulates two or three representative implementation tasks against actual files. If simulation reveals ambiguity or missing information, the plan gets ITERATE or REJECT.
