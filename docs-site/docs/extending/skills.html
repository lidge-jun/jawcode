Skills

Skills are self-contained workflow definitions that the agent loads on demand. Each skill is a Markdown file (SKILL.md) that defines phases, gates, artifacts, and behavioral rules for a specific workflow pattern.

Bundled workflow skills

Jawcode ships with exactly four default workflow skills. This is an intentional design constraint — JWC improves by making this small surface better, not by adding more defaults.

SkillInvocationPurposeArtifacts
jaw-interview /skill:jaw-interview IPABCD I-stage. Socratic requirements gathering with ambiguity scoring and topology confirmation. .jwc/specs/jaw-interview-{slug}.md
plan /skill:plan or jwc orchestrate p IPABCD P-stage. Consensus planning with Critic review loop. The /skill:plan entrypoint is a compatibility alias; the native jwc orchestrate surface is the primary interface. .jwc/plans/planphase/
goal /skill:goal or jwc goal Durable goal ledger with evidence checkpoints. Tracks multi-goal execution across sessions. .jwc/goal/ (goals.json, ledger.jsonl, brief.md)
team /skill:team IPABCD B-stage coordinated execution. Tmux-backed parallel workers with shared state. .jwc/state/team/

Skill structure

Each skill is defined by a SKILL.md file in a named directory. The skill shape includes:

FieldDescription
nameUnique identifier (e.g. jaw-interview)
descriptionOne-line summary displayed in skill listings
filePathAbsolute path to the SKILL.md file
baseDirDirectory containing the skill (for relative imports)
sourceProvider that discovered this skill
hideWhether to hide from user-facing listings
contentEmbedded Markdown content (for bundled skills)

Skill discovery

Jawcode discovers skills from multiple sources, evaluated in priority order:

  1. Bundled defaults — The four workflow skills are compiled into the binary via TypeScript imports. They are always present.
  2. cli-jaw global root — When running under the jaw brand with ~/.cli-jaw/skills present, this directory replaces the native user root. Skills here take precedence over user-level native skills.
  3. Project-level — {cwd}/.jwc/skills/ and ancestor directories. Project skills override user-level skills of the same name.
  4. User-level — ~/.jwc/agent/skills/. The native user skills directory.
  5. Custom directories — Paths configured in settings under customDirectories, scanned as provider custom:user.

Collision handling

When two skills share the same name, the first one discovered wins (based on the priority order above). The duplicate is skipped and a warning is recorded. This means project-level skills can shadow user-level or global skills of the same name.

Deterministic ordering

The final skill list is sorted by compareSkillOrder to ensure consistent ordering regardless of filesystem enumeration order.

Creating custom skills

To create a custom skill, add a directory with a SKILL.md file to any discovered skill path:

# Project-level custom skill
mkdir -p .jwc/skills/my-workflow
cat > .jwc/skills/my-workflow/SKILL.md << 'EOF'
# My Workflow

## Phase 1: Gather context
Read the relevant files and understand the problem.

## Phase 2: Implement
Make the minimal correct change.

## Phase 3: Verify
Run the affected tests.
EOF

The skill is now available via /skill:my-workflow in any Jawcode session within that project.

User-level custom skills

# Available in all projects
mkdir -p ~/.jwc/agent/skills/code-review
# Add SKILL.md with your review workflow

Skill invocation

Skills are invoked via the /skill:name slash command during a session:

/skill:jaw-interview
/skill:goal
/skill:my-workflow

When a skill is active, the agent reads its full content and follows its instructions exactly. Skills define phases, gates, and artifacts that constrain the agent's behavior until the skill workflow completes.

Skill discipline rules

  • Never ignore a skill invocation or any skill text.
  • Read-only skills (jaw-interview, planner, architect, critic) must not edit product source, commit, or run mutating commands.
  • When a task fits a bundled skill, recommend invoking it; on user approval, invoke it.
  • When no skill is active and the action is non-destructive and clearly correct, perform it directly.

Skill vs. slash command vs. hook

ExtensionTriggerScopeUse case
Skill /skill:name Multi-phase workflow with gates and artifacts Interview, planning, goal tracking, team coordination
Slash command /name Single-shot content injection Settings, model selection, quick actions
Hook Lifecycle event Background processing on events Custom logging, notifications, auto-formatting
