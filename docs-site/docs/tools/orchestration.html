Orchestration Tools

Five tools manage parallel work and background processes: task launches subagents, subagent controls them, background manages background rows, monitor streams events from long-running scripts, and job provides legacy async job control.

task

Launches subagents to parallelize workflows. Subagents run asynchronously — the main agent continues working while they execute. Results are delivered automatically when complete.

Parameters

ParameterRequiredDescription
agentYesRole agent: executor, executor_ext, planner, architect, or critic
tasksYesArray of task objects
contextNoShared background prepended to every assignment
spawnPlanNoRequired before batches of more than 4 tasks

Task object

FieldRequiredDescription
idYesCamelCase identifier, max 32 characters
descriptionYesUI label only — subagent never sees this
assignmentYesComplete, self-contained instructions. One-liners are prohibited.
modelNoModel hint: "self", "provider/modelId[:effort]", or "cheap:provider"
inheritContextNoFork parent conversation: none, receipt, last-turn, bounded, full

External/fresh executor with no named modelexecutor_extOmit (uses configured target)

Prompt-cache-aware design

The spawn/resume cycle is designed around prompt caching. Static system prompt content forms a cacheable prefix shared across all subagents in a session. When a subagent is resumed, the cached session context is reused, reducing latency and redundant token processing with each cycle.

Example

// Spawn two parallel executors
task({
  agent: "executor",
  context: "Working on the auth module refactor",
  tasks: [
    {
      id: "RefactorLogin",
      description: "Refactor login handler",
      assignment: `# Target
src/auth/login.ts

# Change
Extract validation logic into a separate function.
Add input sanitization for email field.

# Acceptance
- Login handler calls validateInput() before processing
- Email is normalized to lowercase`
    },
    {
      id: "RefactorSession",
      description: "Refactor session manager",
      assignment: `# Target
src/auth/session.ts

# Change
Replace manual token generation with crypto.randomUUID().

# Acceptance
- Session tokens use crypto.randomUUID()
- Existing tests pass`
    }
  ]
})

subagent

Lists, inspects, awaits, pauses, resumes, steers, or cancels detached task subagents. Use after task launches to manage running work.

Actions

ActionDescription
listSnapshot all visible subagents (running, paused, queued, terminal)
inspectInspect specific subagents by ids
awaitWait for subagents to complete (set timeout_ms)
pauseGraceful safe-boundary pause
resumeResume a paused subagent (with optional message)
steerSend a message to a running subagent
cancelStop a subagent (only when actually failed or off-track)

Verbosity levels

LevelOutput
receiptStatus + 280-char preview (default)
previewUp to 2000 characters
fullUp to 12000 characters (requires explicit ids)

Correction de-anchoring

When a subagent is resumed with a correction ("your answer was wrong"), it activates a de-anchoring protocol: it discards its prior analysis entirely, generates 3+ materially different hypotheses from fresh angles, investigates each with new tool calls, and only then forms a new conclusion. This prevents anchoring bias where the agent merely patches its original flawed conclusion.

Example

// Check on running subagents
subagent({ action: "list" })

// Wait up to 30 seconds for a specific subagent
subagent({ action: "await", ids: ["50-RefactorLogin"], timeout_ms: 30000 })

// Steer a running subagent with new information
subagent({ action: "steer", id: "50-RefactorLogin",
           message: "Also handle the OAuth2 callback endpoint" })

// Resume a terminal subagent with a correction
subagent({ action: "resume", id: "50-RefactorLogin",
           message: "The root cause was wrong. Re-investigate from scratch." })

background

Manages background work rows created by other launch tools (task, bash async, monitor). This tool does not launch new work; it inspects and manages existing rows.

Operations

OperationDescription
listEnumerate current background rows
detailInspect kind, status, timing, previews, output refs
followRead retained output with pagination (nextOffset)
cancelCancel work (only when user asks or work is obsolete)
settingsRead background-related settings/keybindings (read-only)

The TUI footer panel (bg ... ctrl+j) and this tool are two views over the same background work surface. Always call list or detail when accuracy matters — do not claim a background state from memory.

monitor

Starts a background monitor that streams stdout lines from a long-running script as notifications. Each line becomes a separate event delivered into the conversation between turns.

When to use

  • One notification ("tell me when the build finishes") — use bash with async: true
  • Many ongoing events (logs, polling, file watching) — use monitor

Parameters

ParameterRequiredDescription
commandYesShell command to monitor
kindYes"log", "poll", "watch", or "other"
descriptionYesHuman-readable description for listings
timeoutNoMax seconds before auto-shutdown
persistentNoKeep running past the current turn (default false)

Examples

// Tail a log file for errors
monitor({
  command: "tail -F server.log | grep -i error",
  kind: "log",
  description: "Watch server.log for errors"
})

// Watch directory for changes
monitor({
  command: "fswatch -r dist/",
  kind: "watch",
  description: "Watch dist/ for file changes"
})

job (legacy)
