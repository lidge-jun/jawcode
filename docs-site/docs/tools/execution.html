Execution Tools

Five tools run code and interact with the outside world: bash for shell commands, eval for in-process Python/JavaScript, browser for Chromium automation, debug for DAP debugging, and computer_use for desktop app control.

bash

Executes shell commands for terminal operations like git, bun, cargo, and python. This is the only way to run external processes, but dedicated tools should be preferred over shell equivalents.

Parameters

ParameterRequiredDefaultDescription
commandYes—Shell command to execute
cwdNoProject rootWorking directory (use instead of cd dir &&)
envNo—Extra environment variables as key-value pairs
ptyNofalseRun in PTY mode (for sudo, ssh, interactive input)
timeoutNo300Wall-clock timeout in seconds (1 to 3600)

Rules

  • Use cwd to set directory, not cd dir && ...
  • Prefer env: { NAME: "..." } for multiline or quote-heavy values
  • Quote variable expansions: "$NAME"
  • Never pipe through | head or | tail — output is already truncated with full result in artifact://
  • Never redirect with 2>&1 or 2>/dev/null — stdout and stderr are merged
  • Never use shell for operations that have dedicated tools (cat, grep, find, sed)

Async mode

Set async: true to run commands in the background. The call returns immediately with a job ID; the result is delivered as a notification when the command completes. Use for long-running builds, dev servers, or any command where the agent should continue working.

Async mode does not extend or disable the timeout. For daemons that must survive beyond the timeout, use nohup ... & or setsid ... &.

Example

// Run tests in a specific package
bash({ command: "bun test", cwd: "packages/agent" })

// Git operations with env
bash({ command: "git commit -m \"$MSG\"", env: { MSG: "feat: add parser" } })

// Background dev server
bash({ command: "bun run dev", async: true, timeout: 3600 })

eval

Run code in persistent kernels. State persists within each language across cells and across tool calls. Ideal for data exploration, quick calculations, or testing logic without writing to disk.

Cell fields

FieldRequiredDescription
languageYes"py" for IPython, "js" for JavaScript VM
codeYesCell body (JSON-encoded, no fences)
titleNoShort label shown in transcript
timeoutNoPer-cell timeout in seconds (1-600, default 30)
resetNoWipe this cell's language kernel before running

Built-in helpers

Available in both Python and JavaScript kernels with the same API:

display(value)                    # Render a value in output
print(value, ...)                 # Print to cell text output
read(path, offset?, limit?)       # Read file contents
write(path, content)              # Write content to file
append(path, content)             # Append to file
tree(path?, max_depth?, hidden?)  # Render directory tree
diff(a, b)                        # Unified diff between files
env(key?, value?)                 # Get/set environment variables

Best practices

  • One logical step per cell (imports, define, test, use)
  • Pass multiple small cells in one call
  • On failure, resubmit only the fixed cell
  • Python cells run in IPython with a live event loop — use top-level await, not asyncio.run()

browser

Controls a real Chromium tab for JavaScript execution, interactive actions, and authentication flows. For static web content, prefer read with a URL.

Actions

ActionPurpose
openAcquire a named tab, optionally navigate to a URL
closeRelease tab(s)
runExecute JavaScript with page/tab helpers
actStructured interaction steps (preferred for routine actions)

Act verbs

The act action accepts an array of structured steps:

browser({
  action: "act",
  actions: [
    { verb: "navigate", url: "https://example.com" },
    { verb: "observe" },
    { verb: "click", id: 42 },
    { verb: "fill", id: 15, value: "hello" },
    { verb: "screenshot" }
  ]
})

Rules

  • Must call open before run or act
  • After navigation, prior element IDs from observe are invalidated
  • Default to observe over screenshot for understanding page state

debug

Debugger access through the Debug Adapter Protocol (DAP). Supports launching, attaching, breakpoints, stepping, thread/stack inspection, and expression evaluation.

Key actions

ActionDescription
launchStart a debug session (program required)
attachConnect to existing process (pid or port)
set_breakpointSet breakpoint by file+line or function name
continueResume execution
step_over / step_in / step_outSingle-step execution
pauseInterrupt a running program
threads / stack_trace / scopes / variablesInspect runtime state
evaluateEvaluate expression (with optional context: "repl")
terminateEnd the debug session

Example

// Launch and debug a Python script
debug({ action: "launch", adapter: "debugpy", program: "scripts/job.py" })
debug({ action: "set_breakpoint", file: "scripts/job.py", line: 42 })
debug({ action: "continue" })

// Inspect state when stopped
debug({ action: "stack_trace" })
debug({ action: "variables" })

computer_use

Control desktop applications through the lazy cua-driver backend. The driver process starts only when this tool is first called. This is a discoverable tool, activated via search_tool_bm25.

Key actions

ActionDescription
list_appsDiscover running applications
observe / window_stateInspect window state with PID and window ID
clickClick by element_index (preferred) or coordinates
type_textType text into the focused element
press_keySend a key press
scrollScroll with dx/dy

Always call list_apps first to discover running apps, then observe or window_state to get element indices before interacting.
