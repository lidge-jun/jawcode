C: Check Stage

The Check stage runs mechanical verification gates, then performs adversarial review with 3-way reject routing. It ensures that implemented code actually works, passes tests, and meets the plan's acceptance criteria at the integration level.

Two-phase check

Phase 1: Mechanical gates

Automated checks run against the codebase:

# Type checking
bun run check:ts

# Affected test suites
bun test <affected-files>

# Lint and format verification
bun run check

All mechanical gates must pass before Phase 2 begins. If any fail, the agent fixes the issues and re-runs.

Phase 2: Adversarial review

After mechanical gates pass, the agent conducts an adversarial review:

  • Re-read every changed file against the plan's acceptance criteria
  • Look for edge cases, error handling gaps, and missing test coverage
  • Check for security issues (OWASP top 10: injection, XSS, SSRF, path traversal)
  • Verify no debug leftovers, TODO stubs, or placeholder implementations remain
  • Confirm observable behavior matches what was specified

3-way reject routing

If the Check stage finds issues, it routes the rejection back to the appropriate stage based on the issue type:

Issue typeRoute toRationale
Code bug, missing edge case, test failure B (Build) Implementation error; fix the code
Acceptance criteria unclear, plan gap, sequencing error P (Planning) Plan was flawed; revise and re-audit
Requirements ambiguity, missing domain context, scope mismatch I (Interview) Spec was incomplete; gather more requirements

This routing ensures problems are fixed at their source. A spec ambiguity is not patched at the code level; it goes back to requirements gathering.

Verification scope matching

The Check stage enforces a strict rule: verification scope must match claim scope.

  • If the plan claims 10 acceptance criteria, all 10 must have direct evidence
  • If a deliverable lacks current-state evidence, the work is not complete
  • Verification claims must match what was actually run (no fabricated outputs)
  • If verification was not run, it must be declared explicitly

Completion contract

The Check stage enforces the completion contract:

  • No partial work presented as complete
  • No suppressed tests or warnings
  • No fabricated outputs or tool results
  • No stubs, placeholders, or TODO-only code shipped as delivered features
  • Faithful bidirectional reporting: failures reported honestly, successes stated without unnecessary hedging

Gate

The C stage gate requires:

  • All mechanical checks pass (types, tests, lint)
  • Adversarial review finds no blockers
  • Every acceptance criterion has direct evidence
  • No 3-way reject routing triggered (or all routed issues resolved)
