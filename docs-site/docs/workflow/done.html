D-stage: Done

The final IPABCD stage. After C-stage mechanical gates and adversarial review pass, D-stage summarizes the full cycle and closes the orchestration.

What happens in D-stage

The agent reports four things in one line each:

  1. P (Plan) — what was planned, how many files, which acceptance criteria.
  2. A (Audit) — audit verdicts from Planner and Architect subagents.
  3. B (Build) — what was implemented, how many commits, verifier outcome.
  4. C (Check) — mechanical gate results (typecheck, tests, lint) and adversarial review outcome.

Then two structured reflections follow.

WONDER

WONDER asks: "What's still missing?"

  • What edge cases were NOT covered by acceptance criteria?
  • What assumptions turned out wrong during build?
  • Are there integration risks the plan didn't anticipate?

WONDER items are not failures — they are honest admissions of scope gaps. A clean WONDER section means the spec was tight. A long one means the next cycle should start with orchestrate i to sharpen the spec.

REFLECT

REFLECT asks: "How would we improve the spec?"

  • Which acceptance criteria need sharpening?
  • Which constraints were missing?
  • What would change if we revised the ontology?

REFLECT feeds forward into the next IPABCD cycle. If the same project continues, REFLECT items become interview questions for the next I-stage.

Closing the orchestration

After the summary and reflections are written, the agent runs:

jwc orchestrate d

This writes a closing receipt to .jwc/state/sessions/<session-id>/pabcd-state.json and returns the orchestration to idle. The summary + reflections text is the user-facing artifact; the state file records the machine-readable receipt.

When issues surface

If WONDER or REFLECT reveal significant gaps:

  • Spec-level issue — suggest orchestrate i to re-interview.
  • Plan-level issue — suggest orchestrate p to re-plan.
  • No significant issues — state completion plainly. No ceremony.

Goal-mode continuation

When D-stage closes inside an active jwc goal, the orchestration returns to idle but the goal may not be complete. The agent checks whether the goal objective is fully achieved:

  • Work remains — re-enter planning with orchestrate p for the next piece. PABCD cycles repeat until the goal is done.
  • Goal complete — run jwc goal done to close the ledger.
  • Direction change needed — pause and ask the user.

This is how HOTL (Human-on-the-Loop) works in practice: PABCD cycles loop inside a goal until every deliverable has evidence.
