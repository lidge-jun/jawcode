IPABCD Workflow Overview

Jawcode structures every non-trivial task through six sequential stages. Each stage has an explicit gate that must pass before the next stage begins. The pipeline prevents the agent from guessing requirements, shipping untested code, or silently reducing scope.

The six stages

StageNamePurposeGate
I Interview Socratic requirements gathering via jaw-interview. Asks targeted questions, scores ambiguity mathematically, and produces a spec under .jwc/specs/. Ambiguity score below resolved threshold
P Planning Main session authors a plan. A Critic subagent reviews it in a consensus loop: OKAY / ITERATE / REJECT. Critic verdict = OKAY; plan written to .jwc/plans/planphase/
A Plan Audit Independent Planner and Architect subagents audit the approved plan for sequencing gaps and architectural risks. audit_status = pass from both auditors
B Build Main session implements the plan. A read-only verifier subagent confirms each deliverable. verification_status = done
C Check Mechanical gates (bun run check + affected tests) plus adversarial review with 3-way reject routing. All mechanical checks pass; adversarial review finds no blockers
D Done Cycle summary with WONDER (observations) and REFLECT (process improvements) sections. Summary written; state closed

Entry points

Not every task needs every stage. The agent classifies work by risk and enters at the appropriate stage:

ClassificationEntryExample
C0 trivial Direct edit Fix a typo in a comment
C1 single-file Direct edit + verification Fix a bug in one function
C2 one subsystem Compact local plan, then build Add a new CLI flag
C3 cross-domain P stage Add a new provider integration
C4 high-risk P + A stages Auth system changes, schema migration
C5 vague I stage "I have an idea for a new feature"

Reject routing

The C stage (Check) can reject work back to three different stages depending on the issue type:

C stage finds a code issue     → route back to B (Build)
C stage finds a plan issue     → route back to P (Planning)
C stage finds a spec issue     → route back to I (Interview)

This 3-way reject routing ensures problems are fixed at their source rather than patched over at the wrong level.

State management

IPABCD state lives in a session-scoped JSON file:

.jwc/state/sessions/<session-id>/pabcd-state.json

The jwc orchestrate command scopes to the live session automatically via JWC_SESSION_ID. You never pass a session ID manually. Advance stages by running:

jwc orchestrate i   # Start interview
jwc orchestrate p   # Start planning
jwc orchestrate a   # Start audit
jwc orchestrate b   # Start build
jwc orchestrate c   # Start check
jwc orchestrate d   # Close cycle

To abandon the pipeline from any stage:

jwc orchestrate reset

HITL and HOTL modes

The pipeline supports two interaction modes:

  • HITL (Human-in-the-Loop) — the agent pauses at each gate and waits for explicit user approval before advancing. Default for high-risk work.
  • HOTL (Hands-off-the-Loop) — the agent advances through gates automatically, stopping only on failures or ambiguity. Used when the user explicitly trusts the pipeline.
