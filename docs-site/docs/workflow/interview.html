I: Interview Stage

The Interview stage transforms vague ideas into crystal-clear specifications through Socratic questioning with mathematical ambiguity scoring. It is implemented by the jaw-interview skill.

When to use

  • User has a vague idea and wants thorough requirements gathering
  • Task is complex enough that jumping to code would waste cycles on scope discovery
  • User wants mathematically-validated clarity before committing to execution
  • User explicitly requests interview mode (/skill:jaw-interview)

Skip the Interview stage when the user provides detailed, specific requirements with file paths, function names, and acceptance criteria. Enter at P (Planning) instead.

How it works

Phase 0: Resolve ambiguity threshold

Before any questions, the interview resolves the ambiguity threshold from settings. This determines when the interview is "done enough" to proceed. Three depth modes:

ModeThresholdUse case
--quick0.3Rough outline sufficient
--standard0.2 (default)Most features
--deep0.1Mission-critical specifications

Round 0: Topology enumeration

Before scoring begins, a one-time topology enumeration gate confirms the top-level component list and locks it into state. For brownfield projects, the interview explores the existing codebase first to ask informed questions rather than making the user rediscover their own architecture.

Rounds 1-N: Socratic questioning

Each round follows this loop:

  1. Identify the weakest clarity dimension across all locked topology components
  2. Ask 1-3 targeted questions aimed at that dimension (only independent questions are batched)
  3. Score the user's answer against four dimensions
  4. Apply negativity bias: vague, hedging, or undetailed answers downgrade the dimension score
  5. Display the ambiguity HUD with current scores

Four clarity dimensions

DimensionWhat it measures
GoalWhat the user wants to achieve; success criteria
ConstraintsBoundaries, limitations, non-functional requirements
CriteriaAcceptance criteria, edge cases, verification approach
OntologyDomain terms, entity relationships, data model stability

External audit checkpoints

Mathematical scoring runs at specific checkpoints rather than every round:

  • When internal tracker estimates every dimension at 0.8+
  • Before honoring an explicit user skip (mandatory)
  • Before Phase 4 crystallization (mandatory)

Challenge agents

At specific round thresholds, challenge agents activate to shift perspective and pressure-test the emerging specification from different angles.

Output

When ambiguity drops below the threshold, the interview crystallizes a specification file:

.jwc/specs/jaw-interview-{slug}.md

The spec contains all gathered requirements, topology, decisions, and the final ambiguity score. It is marked pending approval until the user explicitly approves it.

Handoff

The normal handoff chain is:

jaw-interview spec
  → jwc orchestrate p (planning with spec reference)
  → pending approval
  → explicitly approved execution

The interview skill is read-only: it must not mutate product code, commit, or delegate execution. It writes only to .jwc/specs/ and interview state.

Invocation

# Inside a jwc session
/skill:jaw-interview

# With depth flag
/skill:jaw-interview --deep "I want to build a notification system"

# Via CLI
jwc jaw-interview
