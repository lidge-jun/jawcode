Human Control Model

Jawcode offers two control modes for the same IPABCD pipeline. The difference is who decides when to advance to the next stage.

HITL: Human-in-the-Loop

Plain jwc orchestrate is HITL by default. The agent plans, audits, and builds in PABCD stages, but the important handoffs expect a human to look and approve.

How it works

  1. Agent completes a stage (e.g. P-stage plan with Critic OKAY).
  2. Agent stops and presents the result to the user.
  3. User reviews, asks questions, requests changes, or approves.
  4. On approval, user tells the agent to advance: "go to A-stage" or "orchestrate a".
  5. Repeat for each stage transition.

When to use HITL

  • Learning the workflow for the first time.
  • High-risk changes (security, auth, data migration, public API).
  • When you want to review the plan before any code is written.
  • When you want to inspect audit verdicts before build begins.
# HITL session — user advances each stage
jwc

> "Plan a refactor of the auth module"
# Agent writes plan, stops at pending-approval
> "Looks good, run audit"
# Agent runs A-stage, stops with verdicts
> "Build it"
# Agent implements, stops with verifier result
> "Check"
# Agent runs C-stage gates

HOTL: Human-on-the-Loop

jwc goal wraps the same PABCD loop as HOTL. The goal gives the agent permission to keep moving with evidence checkpoints, while the human watches progress and steps in only to steer, pause, or cancel.

How it works

  1. User sets a goal: "Implement feature X with tests and docs".
  2. Agent enters PABCD autonomously — I, P, A, B, C, D — recording evidence at each stage.
  3. If the goal requires multiple PABCD cycles, the agent loops back to P-stage after D-stage closes.
  4. Agent records milestones in the goal ledger: .jwc/goal/ledger.jsonl.
  5. Human can intervene at any time: steer, pause, cancel, or inspect the ledger.
  6. Agent calls jwc goal done only when every deliverable has evidence.

When to use HOTL

  • Large tasks that span multiple PABCD cycles.
  • When you trust the pipeline's gates (audit, check, adversarial review) to catch issues.
  • When you want to do other work while the agent executes.
  • Batch operations where manual approval at each stage would be tedious.
# HOTL session — agent advances autonomously
jwc

> "Goal: build docs site with 55 pages, deploy to GitHub Pages"
# Agent enters I-stage, interviews if needed
# Agent enters P-stage, writes plan, runs Critic
# Agent enters A-stage, dual audit (Planner + Architect)
# Agent enters B-stage, delegates to subagents
# Agent enters C-stage, runs mechanical gates
# Agent enters D-stage, writes WONDER + REFLECT
# If deliverables remain, loops back to P-stage
# When all done, runs: jwc goal done

Intervention commands

In HOTL mode, you always retain control:

ActionWhat happens
Steer Send a message to redirect the agent's approach without stopping it.
Pause Agent stops at the next safe boundary. Goal state preserved. Resume later.
Cancel Goal is dropped. Partial work remains on disk for manual review.
Inspect Read .jwc/goal/ledger.jsonl for the full evidence trail.

The key insight

HITL and HOTL run the same IPABCD pipeline with the same gates. The difference is only who presses "next." In HITL, you press it. In HOTL, the goal presses it — and you watch, ready to grab the wheel.
