A: Plan Audit Stage

The Audit stage provides independent review of the approved plan by two specialist subagents: a Planner and an Architect. Both must pass before implementation begins.

Dual audit model

Two subagents audit the plan in parallel, each with a different lens:

AuditorFocusOutput
Planner Sequencing, acceptance criteria completeness, risk mapping, execution handoff quality PASS or FAIL with findings
Architect Architectural risks, contract stability, dependency analysis, performance implications PASS / FAIL with CLEAR / WATCH / BLOCK status per finding

Finding severity levels

The Architect rates each finding with a status:

StatusMeaningBlocks?
CLEAR No concerns; approach is sound No
WATCH Minor concern; proceed with awareness No
BLOCK Architectural risk that must be resolved Yes

Audit process

  1. Fetch audit prompts: 
    jwc orchestrate audit-prompt planner
jwc orchestrate audit-prompt architect
  2. Spawn both auditors as subagents (parallel execution)
  3. Each auditor reads the plan, inspects source code, and writes findings
  4. Both auditors are read-only: they inspect but do not modify source files
  5. Record verdicts in PABCD state

Solo vs dual mode

For documentation or frontend-only changes where architectural risk is low, the audit may run in solo mode with only the Architect. The default is dual mode (both Planner and Architect).

Gate

The A stage gate requires:

audit_status = pass

This means:

  • All auditors return PASS
  • No BLOCK-level findings remain unresolved
  • WATCH-level findings are noted but do not block

Failure path

If an auditor returns FAIL:

  • The agent reviews the blocking findings
  • Routes back to P (Planning) to address the issues
  • A new plan revision goes through the Critic loop again
  • Re-audit with fresh subagent instances

Auditor constraints

Both the Planner and Architect are PABCD-lifecycle-centered read-only agents. They can:

  • Read source files and plan artifacts
  • Run sanctioned workflow CLI commands (jwc orchestrate, jwc state)
  • Write audit verdicts to workflow state

They cannot:

  • Edit product source files
  • Override environment variables
  • Perform direct handoffs between stages
  • Run arbitrary shell commands
