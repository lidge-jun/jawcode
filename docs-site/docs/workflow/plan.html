P: Planning Stage

The Planning stage produces an implementation plan through a consensus loop with a Critic subagent. The main session authors the plan; the Critic reviews it until it reaches OKAY or is rejected. Plans remain pending approval until the user explicitly approves execution.

Entry conditions

  • Direct entry: requirements are clear enough (C3+ classification) that planning is needed but not a full interview
  • From I stage: an approved spec exists under .jwc/specs/ and the interview handed off
# Advance to planning
jwc orchestrate p

# With spec reference from interview
jwc orchestrate p --spec-ref .jwc/specs/jaw-interview-my-feature.md

The Critic consensus loop

The planning stage runs iterative rounds:

  1. Main session writes or revises the plan
  2. Critic subagent reviews the plan and returns a verdict
  3. If ITERATE: main session addresses findings and resubmits
  4. If REJECT: fundamental issues require rethinking the approach
  5. If OKAY: plan is approved with optional minor nits

Critic verdicts

VerdictMeaningAction
OKAY Plan is actionable; execution can proceed without guessing Write pending-approval.md under .jwc/plans/planphase/
ITERATE Specific gaps need addressing; the approach is sound Main session patches the plan and resubmits to Critic
REJECT Fundamental issues; the approach needs rethinking Main session rewrites the plan or escalates to user

What the Critic checks

The Critic approves only when execution can proceed without guessing and verification is concrete:

  • Every acceptance criterion is testable and unambiguous
  • File paths and symbols are verified (not assumed from memory)
  • Verification plan covers each acceptance criterion
  • No orphan files or dangling references
  • Dependencies between plan steps are explicit
  • Edge cases and error paths are addressed

Plan format

Plans are written as structured Markdown under .jwc/plans/planphase/:

.jwc/plans/planphase/<run-id>/
  020_plan.md          # The implementation plan
  030_synthesis.md     # Critic findings + patches
  040_final.md         # Final approved version
  pending-approval.md  # Written after OKAY verdict

A plan typically includes:

  • Scope: what changes and what does not
  • Acceptance criteria: numbered, testable criteria (AC1, AC2, ...)
  • File-level changes: exact files, symbols, and operations (add/modify/delete)
  • Verification plan: commands and scenarios that prove each AC
  • Stream/batch structure: parallelizable work groups for subagent dispatch
  • Risk and rollback: what could go wrong and how to recover

Boundary rules

The planning stage is read-only for product source. Before explicit execution approval:

  • No editing product source files
  • No running mutation-oriented shell commands
  • No committing, pushing, or opening PRs
  • No delegating implementation tasks to subagents

The only files written are plan artifacts under .jwc/plans/.

Dev skill integration

During planning, the agent reads relevant dev skills for quality:

  • /skill:dev for base development contract
  • /skill:dev-architecture for architectural patterns
  • Domain-specific dev skills matching the files being planned

Gate

The P stage gate is satisfied when:

  1. Critic returns OKAY verdict
  2. pending-approval.md is written
  3. User explicitly approves execution (HITL) or pipeline auto-advances (HOTL)
