Team Mode

jwc team is the tmux-based multi-worker execution mode. It splits the current tmux window into a leader pane and one or more worker panes, each running an independent JWC CLI session. Workers coordinate through shared state files and CLI APIs — not through the leader's conversation.

Team vs native subagents

JWC has two parallelism surfaces. Choose the right one:

AspectNative subagents (executor)Team mode (jwc team)
Lifetime Bounded to one reasoning burst Durable — survives beyond a single burst
Visibility Background (results returned to parent) Visible tmux panes you can watch and inspect
State In-memory, ephemeral Persisted in .jwc/state/team/
Coordination IRC messages between peers Shared state files, mailbox, CLI APIs
Worktrees Shares parent worktree Can use dedicated git worktrees
Best for Quick parallel tasks (research, bounded edits) Long-running coordinated execution, multi-lane delivery

Quick start

# Default: 3 executor workers
jwc team "analyze feature X and report flaws"

# Explicit worker count and type
jwc team 3:executor "debug flaky integration tests"

# Single worker
jwc team 1:executor "ship end-to-end fix with verification"

Prerequisites

  • tmux installed (tmux -V)
  • Current session is inside tmux ($TMUX is set)
  • JWC CLI is available (jwc --version)

Architecture

When jwc team launches:

  1. Detects the current tmux leader context.
  2. Initializes team state under .jwc/state/team/<team-name>/.
  3. Splits the tmux window: worker 1 horizontally to the right, workers 2..N vertically stacked in the right column.
  4. Applies main-vertical layout — leader on the left (~50%), workers stacked on the right (~50%).
  5. Launches each worker with environment variables: GJC_TEAM_NAME, GJC_TEAM_WORKER_ID, GJC_TEAM_STATE_ROOT.
  6. Returns control to the leader for monitoring.

Lifecycle commands

# Read-only status snapshot
jwc team status <team-name>

# Mutating monitor: recover stale claims, integrate worker commits, sync HUD
jwc team monitor <team-name>

# Reconnect/inspection with liveness recovery
jwc team resume <team-name>

# Graceful shutdown (only when all tasks are terminal)
jwc team shutdown <team-name>

# List known teams
jwc team list

Shutdown rules

Do not run shutdown while workers are actively writing. Wait for terminal task state:

  • pending = 0
  • in_progress = 0
  • All tasks are completed, failed, or blocked
  • No pending integration request or conflict

Shutdown outcome depends on state:

ConditionPhase
All tasks verified, all workers gracefully stoppedcomplete
Tasks complete but integration still pendingawaiting_integration
Tasks failed or lack evidencefailed
Work still pending/in-progresscancelled

Worker protocol

Each worker follows this sequence:

  1. Send startup ACK: jwc team api worker-startup-ack
  2. Claim a task: jwc team api claim-task
  3. Work on the task, updating status periodically.
  4. Transition task to terminal: jwc team api transition-task-status with completion evidence.
  5. Commit or leave changes in the worker worktree — the leader integrates them.

Worktree integration

When workers use dedicated git worktrees, the leader's monitor/resume commands automatically integrate worker commits:

  • Dirty worktrees are auto-checkpointed before integration.
  • Clean-ahead history is merged into the leader with a runtime merge commit.
  • Diverged history is cherry-picked into the leader.
  • Idle/done/failed workers are cross-rebased onto the updated leader after integration.
  • Conflicts are aborted, recorded, and reported to the leader mailbox.

Goal bridge

Team mode integrates with goal mode. When jwc team launches with an active goal:

  • Workers receive goal context (objective, active goal ID) but do not own goal state.
  • Workers provide task status and verification evidence only.
  • The leader uses terminal team evidence plus a fresh goal snapshot to run the quality gate.
  • Workers must not run jwc goal checkpoint — checkpoint authority stays with the leader.

Typed lanes

For complex multi-worker tasks, use explicit lane sections instead of inline splits:

### Lane A — Delivery
Implement delivery-only changes and evidence.

### Lane B — Verification
Run verification suite and report results.

Inline "Split lanes: A..., B..." sentences are rejected because they cause every worker to receive the same broad task.

Integration conflict Worker and leader diverged Run jwc team monitor, inspect integration-report.md
Shutdown says awaiting_integration Pending worker commits not yet merged Run jwc team resume first to trigger integration
