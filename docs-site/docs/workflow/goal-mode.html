Goal Mode

Goal mode (jwc goal) is the durable execution ledger that wraps IPABCD into autonomous, multi-cycle operation. It tracks objectives, records evidence, and only reports completion when every deliverable is verified.

Lifecycle

A goal moves through these states:

create -> active -> [pause] -> active -> complete
                         \-> drop (abandoned)

Operations

OperationWhat it doesWhen to use
create Starts a new goal with an objective string. Requires no existing active or paused goal. Beginning of a large task.
get Returns the current goal state, usage (tokens, time), and latest evidence. Anytime — inspection is always safe.
resume Re-activates a paused goal so work can continue. After a pause, when you're ready to continue.
pause Suspends the active goal and stops continuation steers. In-memory state is updated immediately. When deliverables are done but programmatic completion is blocked, or when the user should decide the next step.
complete Marks the goal complete after verifying every deliverable against current evidence. Only when all acceptance criteria have direct proof.
drop Discards the current goal without completing it. When the objective is no longer relevant.

Evidence trail

Goal mode maintains a durable evidence trail in .jwc/goal/:

FileFormatPurpose
goals.json JSON Current goal state: objective, status, milestones, timestamps.
ledger.jsonl JSONL (append-only) Every update, evidence checkpoint, pause, and resume event.
brief.md Markdown Human-readable summary of the current goal.

Recording milestones

The agent records evidence at meaningful boundaries using the CLI:

jwc goal update "B-stage complete: 11 pages generated" \
  --evidence "11 HTML files in docs-site/docs/, sidebar updated, typecheck clean"

Evidence is mandatory for every update. Updates without evidence are rejected.

PABCD integration

Goal mode does not bypass IPABCD — it wraps it. When a goal is active:

  1. Agent enters orchestrate p to plan the next piece of work.
  2. PABCD stages (P, A, B, C, D) execute normally with all gates.
  3. D-stage closes the cycle. Agent checks: is the goal fully achieved?
  4. If work remains, agent loops back to orchestrate p.
  5. If complete, agent runs goal complete with final evidence.
Goal: "Build docs site with 55 pages"
  |
  +-- PABCD cycle 1: Infrastructure + 11 pages (Getting Started, Architecture)
  |     P -> Critic OKAY -> A -> dual PASS -> B -> verifier DONE -> C -> green -> D
  |
  +-- PABCD cycle 2: 10 workflow pages
  |     P -> Critic OKAY -> A -> PASS -> B -> DONE -> C -> green -> D
  |
  +-- PABCD cycle 3: 15 tool + provider + subagent pages
  |     ...
  |
  +-- goal complete (all 55 pages verified)

Continuation steers

While a goal is active, the runtime generates continuation steers — automatic prompts that keep the agent working toward the objective. The agent does not wait for user input between PABCD cycles.

Pausing the goal (goal pause) stops these continuation steers immediately. The in-memory runtime state is updated, so no further prompts are generated. Resuming (goal resume) restarts them.

Quality gate

Before goal complete succeeds, the agent must audit the current repo state against every concrete deliverable. The completion contract requires:

  • Reading the files that were supposed to be created or modified.
  • Running the relevant checks (tests, typecheck, lint) that cover the changes.
  • Matching the verification scope to the claim scope — no partial-work-as-complete.

If any deliverable lacks direct current-state evidence, the goal stays active.

CLI usage

# Start a goal
jwc goal create "Implement feature X with tests and docs"

# Check status anytime
jwc goal get

# Record a milestone
jwc goal update "Unit tests passing" --evidence "bun test src/feature.test.ts: 12/12 pass"

# Pause (agent or user)
jwc goal pause

# Resume
jwc goal resume

# Complete (only when verified)
jwc goal complete

# Abandon
jwc goal drop

Agent-initiated pause

The agent can pause a goal through the 2-tap audit gate:

jwc goal pause --agent --audit "All deliverables implemented, awaiting user review"

This records the pause reason in the ledger and signals that the agent chose to stop — not that something failed. The user can inspect the ledger, review the work, and resume or complete.
