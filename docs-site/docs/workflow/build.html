B: Build Stage

The Build stage is where implementation happens. The main session executes the approved plan, optionally dispatching work to parallel executor subagents. A read-only verifier subagent confirms each deliverable before the stage gate is satisfied.

Execution model

The main session implements the plan directly. For multi-file changes, work is organized into streams (parallelizable groups) defined in the plan:

# Example stream structure from a plan
Stream 1: CSS foundation + landing page    (main session)
Stream 2: Subpages A, B, C                (executor subagents)
Stream 3: README improvements              (executor subagent)
Stream 4: Deploy infrastructure            (executor subagent)

Subagent dispatch

The main session dispatches independent work slices to executor subagents. Each subagent receives:

  • Specific file scope (3-5 files maximum per subagent)
  • Exact instructions from the plan
  • Shared context about conventions and constraints
  • Instructions to skip formatters and verification (parent runs those once at the end)

Fork-based executor

The default executor subagent uses the fork lane: it inherits the parent's model and cached prompt context. This is prompt-cache-aware by design:

  • Static prompt prefix (system prompt, skills, repo context) is shared and cacheable
  • Only the dynamic assignment payload is unique per subagent
  • Each spawn reuses the cached prefix, reducing latency and redundant tokens

External executor

The executor_ext subagent uses a different model or provider for independent verification or model-diverse implementation:

# Different model's perspective
executor_ext with model: "openai/gpt-4.1"

# Configured external target (from settings)
executor_ext  # uses EXECUTOR_EXT target

Subagent coordination

Subagents run asynchronously. While they work, the main session can:

  • Continue implementing its own stream
  • Answer user questions
  • Inspect subagent progress
  • Steer or correct subagents via IRC messaging

The main session must not duplicate work that subagents are already doing.

Correction de-anchoring

When a subagent is resumed with a correction ("your previous answer was wrong"), it activates the de-anchoring protocol:

  1. Set aside the prior analysis entirely
  2. Generate at least 3 materially different hypotheses
  3. Investigate each with fresh tool calls
  4. Form a new conclusion from evidence, not from editing the old one

This prevents anchoring bias where subagents merely patch their previous wrong answer instead of genuinely reconsidering.

Verifier

After implementation, a read-only verifier subagent checks each acceptance criterion from the plan:

VerdictMeaningAction
DONE All acceptance criteria verified Proceed to C stage
NEEDS_FIX Specific criteria not met Main session fixes issues, then re-verifies

Commit discipline

During the Build stage:

  • Commit completed work proactively when authorized
  • Inspect status and stage only intended files
  • Preserve unrelated user work
  • Resolve or stop on conflicts instead of overwriting
  • Never revert, stash, or delete user work unless explicitly asked

Gate

The B stage gate requires:

verification_status = done

All acceptance criteria from the plan must be confirmed by the verifier subagent.
