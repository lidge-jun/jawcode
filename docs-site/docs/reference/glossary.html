Glossary

Definitions for terms used throughout Jawcode documentation. Sorted alphabetically.

A

abyss-bite

The default JWC dark TUI theme. Dark navy background with cyan accent colors (#00d1da) and an orange bite highlight (#ff5a2e). Named after the deep-sea + shark-jaw visual identity. Light terminals use abyss-bite-light instead.

architect

A read-only role agent that performs architecture and code review. It produces structural assessments with severity ratings (CLEAR, WATCH, BLOCK) and is PABCD-lifecycle-centered but directly callable when requested. The architect cannot edit product files.

authority-bias prevention

A subagent guard that requires agents to verify factual claims from their parent assignment using their own tool calls, rather than trusting them blindly. Prevents errors from propagating through delegation chains.

B

blast radius

The scope of impact of a change or action. Jawcode requires agents to assess reversibility and blast radius before acting. Local, reversible actions (file edits, local tests) proceed freely. Hard-to-reverse or externally visible actions (force push, rm -rf, drop table, send external messages, delete branches) require explicit user confirmation each time.

C

compaction

The process of summarizing older conversation context to fit within the model's context window. When the conversation grows too long, compaction preserves key facts (file paths, decisions, errors) while discarding routine tool-call narration. Agents must extract and record important tool results before compaction clears them.

confirmation bias

A cognitive bias where an agent seeks only evidence that confirms its first hypothesis. Jawcode counters this by requiring agents to run at least one check that would falsify their hypothesis before declaring a root cause.

correction de-anchoring

See de-anchoring.

critic

A read-only role agent that performs plan critique. It evaluates plans for actionability and produces verdicts: OKAY (plan is executable), ITERATE (fixable gaps remain), or REJECT (fundamental issues require replanning). Approves only when execution can proceed without guessing and verification is concrete.

D

de-anchoring

A protocol activated when a subagent is resumed or steered with a correction indicating its previous answer was wrong. Instead of patching the prior conclusion (which would be biased by anchoring), the agent must: (1) set aside the prior analysis entirely, (2) generate at least 3 materially different hypotheses from unexplored angles, (3) investigate each with fresh tool calls, (4) form a new conclusion from the evidence alone. Does not apply to simple additive requests.

dev-work classification

A risk-routing system that classifies implementation work before coding begins. Six levels from C0 (trivial config touch) through C5 (vague/ambiguous requirements). Higher classes require more ceremony: C0-C1 are direct edits, C2 uses a compact local plan, C3-C4 require orchestration and audit, C5 requires jaw-interview first. When signals match multiple classes, the higher class wins.

E

executor

The default general-purpose subagent role. Used for parallel implementation, research, investigation, and bounded task slices. Executors run asynchronously — the main agent can continue working or responding to the user while executors process in the background. Uses the normal execution-model fork lane with prompt-cache-aware context sharing.

executor_ext

The external/fresh executor lane. Identical behavior to executor but selects a different model — either the configured EXECUTOR_EXT target or an explicit provider/modelId[:effort] override. Used when the user explicitly requests external, fresh, or model-diverse executor work. Provides independent verification through a different model's perspective.

F

fork context

A sanitized snapshot of the parent conversation that can be copied into a subagent at spawn time. Modes include none (default), receipt, last-turn, bounded, and full. Fork context is prompt-cache-aware: the static prefix is shared across spawned agents to minimize redundant token processing.

G

goal mode

A durable execution ledger under .jwc/goal/. Goals persist across turns with evidence checkpoints and quality-gate completion contracts. The agent records milestones with jwc goal update (evidence mandatory) and can pause via agent-initiated 2-tap audit gates. Goal mode does not bypass PABCD — if the objective requires structured workflow, the agent runs jwc orchestrate stages within the goal.

H

HITL (Human-in-the-Loop)

Execution mode where the agent pauses at each PABCD stage boundary for explicit user approval before proceeding. The user reviews and approves each stage transition. Used for high-risk or unfamiliar work where human judgment is needed at every step.

HOTL (Human-on-the-Loop)

Execution mode where the agent advances through PABCD stages autonomously, notifying the user of progress but not waiting for approval at every boundary. The user monitors and can intervene but does not need to approve each step. Used for well-understood work where the plan is already approved.

I

IPABCD

The six-stage structured workflow pipeline: Interview, Plan, Audit, Build, Check, Done. Each stage has specific gates and verdicts. The Check stage features 3-way reject routing: code issues route back to Build, plan issues to Plan, spec issues to Interview. The pipeline is operated via jwc orchestrate <stage>.

J

jaw-interview

The IPABCD I-stage engine. A Socratic requirements-gathering workflow with mathematical ambiguity scoring and topology confirmation. Produces approved specs under .jwc/specs/. It is a requirements workflow and must not mutate product code. Normal handoff: spec to jwc orchestrate p for planning.

K

k-writing

Korean UX writing rules for Jawcode documentation and UI strings. Direct translation with natural phrasing — technical terms keep their English form, explanations use natural Korean. Avoids bureaucratic Sino-Korean vocabulary. Example: use "저장했어요" not "요청 처리가 완료되었습니다".

P

planner

A read-only role agent focused on sequencing, acceptance criteria, risk mapping, and execution handoff. PABCD-lifecycle-centered but directly callable when the user requests a planner lens. Cannot edit product files.

prompt cache

LLM providers cache the static prefix of prompts across requests. Jawcode's subagent architecture exploits this: the system prompt, role definition, and shared context form a cacheable prefix. Fork/spawn/resume cycles reuse this cached prefix, reducing latency and redundant token costs with each interaction.

R

role agent

A callable task agent with a specific role definition. Jawcode bundles five: executor, executor_ext, architect, planner, and critic. Role agents are not workflow skills and are not committed as repo-visible .jwc defaults. They are loaded from packages/coding-agent/src/prompts/agents/.

S

sunk-cost guard

A prompt rule that prevents agents from continuing a failing approach just because they have already invested effort. If significant effort yields no progress, agents must evaluate whether the approach itself is wrong rather than doubling down. Sunk effort is not a reason to continue.

T

team mode

Tmux-backed coordinated execution with multiple workers. Uses shared state under .jwc/state/team/, mailbox/dispatch APIs, optional git worktrees, lifecycle control, and explicit verification lanes. Activated via the team workflow skill when approved work benefits from parallel persistent workers.

TUI

Terminal User Interface. Jawcode's TUI is built on packages/tui with differential rendering. Features include the abyss-bite gradient welcome banner with sweep animation, shine effects, tool-call folding, composer styling, and B2-lite viewport scroll with sticky gap.

W

workflow skill

A bundled workflow definition that provides a complete interaction pattern. Jawcode has four public workflow skills: jaw-interview, plan, goal, and team. Skills are loaded from packages/coding-agent/src/defaults/jwc/skills/ and activated via /skill:<name>. Two additional tool-help skills (browse, search) are hidden utility references.
