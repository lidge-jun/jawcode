Permissions & Tool Restrictions

Jawcode assigns different tool surfaces to different agent roles. The main session agent has full tool access, while read-only subagents (architect, planner, critic) operate under strict restrictions.

Tool assignment by role

RoleToolsBash access
Main session All tools (read, edit, write, bash, browser, task, etc.) Unrestricted
executor All tools (same as main) Unrestricted
executor_ext All tools (same as main) Unrestricted
architect read, search, find, lsp, ast_grep, web_search, bash, report_finding Restricted (allowlist only)
planner read, search, find, lsp, ast_grep, web_search, bash, report_finding Restricted (allowlist only)
critic read, search, find, lsp, ast_grep, web_search, bash Restricted (allowlist only)

Bash allowlists for read-only agents

Read-only agents (architect, planner, critic) can only run bash commands that match specific allowed prefixes. All other bash commands are blocked at the tool level.

Architect allowed prefixes

jwc planphase --write
jwc state

The architect may use jwc planphase --write to persist audit verdicts (passing the verdict markdown inline via --artifact, not as a file path) and jwc state for workflow state read/write/contract commands.

Blocked operations for read-only agents

  • Product-source writes (edit, write tools, or bash writes)
  • Direct handoffs between workflow stages
  • State clears or resets
  • Artifact file-path ingestion
  • Environment variable overrides
  • General shell work (git, npm, file manipulation)

Read-only contract

The read-only contract for architect, planner, and critic is enforced at the prompt level:

Read-only: never write, edit, format, commit, push,
or mutate files.

Exception: you may use the restricted bash tool only for
sanctioned jwc workflow CLI persistence and jwc workflow
state read/write/contract commands.

Read-only agents may receive a forked parent-conversation snapshot as background context. The read-only contract is unchanged regardless of what the snapshot contains — the agent never performs edits inferred from the snapshot.

Tool discovery and load modes

Not all tools are loaded into every session by default. Jawcode uses a discovery system with load modes:

Load modeBehavior
always Tool is always available in the tool list
discoverable Tool is hidden until the agent searches for it via search_tool_bm25

Discoverable tools reduce prompt size and prevent tool overexposure. The agent uses BM25-style relevance search to activate hidden tools when needed. Once activated, they remain available for the rest of the session.

MCP tool restrictions

MCP (Model Context Protocol) servers provide external tools that are loaded at runtime. Jawcode applies several safety measures:

  • Startup timeout — MCP servers must respond within 250ms. Servers that exceed this timeout are aborted (unless cached).
  • Schema normalization — Tool schemas are normalized via normalizeSchemaForMCP. Broken $ref references cause provider 400 errors rather than silent failures.
  • Tool cache — Server tool definitions are cached in agent.db to avoid repeated startup costs.
  • Conflict detection — MCP tools that conflict with built-in tool names are skipped with an error log.

Computer use safety tiers

When computer use is enabled (via cu-mcp), a deterministic tier system restricts what the agent can do based on the frontmost application:

App categoryTierAllowed actions
Browser, tradingreadScreenshot, observe only
Terminal, IDEclickScreenshot, observe, click
MediablockedNone
OtherfullAll actions

System key combos (Cmd+Q, etc.) are blocked regardless of tier. The tier system is enforced server-side and is model-agnostic.
