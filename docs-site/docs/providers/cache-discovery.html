Model Cache & Discovery

Jawcode maintains a local model cache and supports runtime discovery from local inference servers. This page covers the cache database, the static fingerprint system, discovery protocols, and the ModelRegistry merge pipeline.

Model Cache Database

Discovered and fetched model lists are persisted in a SQLite database: 

~/.jwc/agent/models.db

The cache uses schema v3 with per-provider rows. Each row stores the full model list JSON, an authoritative flag, and a static_fingerprint column.

Static Fingerprint

The fingerprint hashes the static catalog slice that was merged into the cached row. On subsequent loads:

  1. If the network fetch is skipped (cache hit), check the fingerprint
  2. If the in-memory static catalog fingerprint matches the cached one, return cached rows verbatim — no static+dynamic merge needed
  3. If fingerprints differ (catalog updated), re-merge and update the cache

The fingerprint is memoized per process via a WeakMap keyed by the static-models array reference, so repeated cold-start calls do not re-hash.

Cache Staleness

The cache can serve stale compatibility flags after a catalog update. If a model returns persistent 400 errors after you change its compat in the catalog, check the cache: 

# Inspect cached compat for a specific model
sqlite3 ~/.jwc/agent/models.db \
  "SELECT models FROM model_cache WHERE provider='xai'" \
  | python3 -m json.tool | grep -A3 composer

# Nuclear option: delete the provider cache row
sqlite3 ~/.jwc/agent/models.db \
  "DELETE FROM model_cache WHERE provider='xai'"

ModelRegistry Merge Pipeline

On every refresh, ModelRegistry builds the final model list through this pipeline:

  1. Load built-in models — static catalog from @gajae-code/ai
  2. Load models.yml — custom provider config
  3. Apply provider overrides — baseUrl, headers, requestTransform, disableStrictTools, cacheRetention to built-in models
  4. Apply modelOverrides — per provider+model ID metadata changes
  5. Merge custom models — same provider+ID replaces existing; otherwise append
  6. Load cached/discovered models — Ollama, llama.cpp, LM Studio, plus built-in provider managers; then re-apply model overrides

Provider defaults vs per-model overrides:

  • Provider headers are baseline; model headers override by key
  • modelOverrides can change: name, reasoning, input, cost, contextWindow, maxTokens, headers, compat, contextPromotionTarget
  • compat is deep-merged for nested routing blocks

Runtime Discovery

Jawcode discovers local inference servers automatically. If a provider is not explicitly configured in models.yml, an implicit discoverable entry is created with default settings.

Explicit Discovery Configuration

Override defaults or add discovery to any custom provider: 

providers:
  ollama:
    baseUrl: http://192.168.1.100:11434
    api: openai-responses
    auth: none
    discovery:
      type: ollama

  llama.cpp:
    baseUrl: http://127.0.0.1:9090
    api: openai-responses
    auth: none
    discovery:
      type: llama.cpp

Supported discovery.type values: ollama, llama.cpp, lm-studio. Discovery requires a provider-level api.

Extension Provider Registration

Extensions and SDK embedders can register providers at runtime via pi.registerProvider(...):

  • Model replacement/append for a provider
  • Custom stream handler registration for new API IDs
  • Custom OAuth provider registration

This is used by the jawcode/sdk embedding surface for host applications that need to inject custom providers.

Model Availability

There is a distinction between registered and available models:

  • getAll() — returns the full loaded registry (built-in + custom + discovered)
  • getAvailable() — filters to models that are keyless or have resolvable auth

A model can exist in the registry but not be selectable until credentials are configured. The /model TUI command and --list-models CLI both show only available models by default.

Diagnostics

Quick commands for debugging model cache and discovery issues: 

# Check cached compat for a model
sqlite3 ~/.jwc/agent/models.db \
  "SELECT models FROM model_cache WHERE provider='xai'" \
  | python3 -m json.tool | grep -A3 composer

# Check session errors
grep -c "anchors do not match" \
  ~/.jwc/agent/sessions/<slug>/<session>.jsonl

# Live model verification
jwc -p --no-session --no-tools \
  --model "xai/grok-composer-2.5-fast" \
  --thinking high \
  "Reply with exactly: OK"

Note: Jawcode uses packages/jwc/dist/jwc.bundle.js when present. After catalog changes, re-bundle with cd packages/jwc && bun run bundle to avoid stale bundle surprises.
