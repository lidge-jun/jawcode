Provider Setup

Every provider needs credentials before its models become available. Jawcode resolves credentials through a layered chain — you only need to configure one layer.

Auth Resolution Order

When Jawcode needs a key for a provider, it checks these sources in order and uses the first match:

  1. Runtime override — CLI --api-key flag
  2. Stored API key — saved in agent.db via /login
  3. Stored OAuth credential — saved in agent.db with automatic refresh
  4. Environment variable — standard provider env vars (OPENAI_API_KEY, ANTHROPIC_API_KEY, etc.)
  5. models.yml fallback — apiKey field in provider config (treated as env var name first, then literal)

Quick Setup Methods

OAuth Login (Recommended)

The fastest path for major providers. Run jwc and use the /login slash command: 

# Inside the TUI
/login

# Providers with OAuth support:
# - OpenAI (includes Codex access)
# - Anthropic (Claude)
# - Google (Gemini CLI / Antigravity)
# - xAI (Grok)
# - Perplexity
# - Kimi / Moonshot

OAuth tokens are stored in ~/.jwc/agent/agent.db and refreshed automatically. A single OAuth login also unlocks that provider's native web search engine.

Provider Presets

For providers that need specific configuration (base URL, compat flags, model entries), use presets to write a correct models.yml block in one command: 

# CLI setup
jwc setup provider --preset minimax
jwc setup provider --preset minimax-cn
jwc setup provider --preset glm

# Inside the TUI
/provider add --preset minimax
/provider add --preset glm
/provider add zai

Presets write models.yml entries that reference environment variable names (MINIMAX_CODE_API_KEY, ZAI_API_KEY, etc.) but do not store or validate actual credentials.

Custom Provider Configuration

Add any OpenAI-compatible endpoint as a custom provider in ~/.jwc/agent/models.yml: 

providers:
  my-provider:
    baseUrl: https://api.example.com/v1
    apiKey: MY_PROVIDER_API_KEY    # env var name or literal
    api: openai-completions
    auth: apiKey                   # apiKey | none | oauth
    headers:
      X-Team: platform
    models:
      - id: my-model
        name: My Model
        reasoning: true
        input: [text]
        contextWindow: 128000
        maxTokens: 16384
        cost:
          input: 0
          output: 0
          cacheRead: 0
          cacheWrite: 0

Azure OpenAI

Azure uses canonical OpenAI model IDs and resolves them to deployment names at request time: 

providers:
  azure-openai:
    baseUrl: https://my-resource.openai.azure.com/openai/v1
    apiKeyEnv: AZURE_OPENAI_API_KEY
    api: azure-openai-responses
    models:
      - id: gpt-4.1
      - id: o3
# Map model IDs to Azure deployment names
export AZURE_OPENAI_DEPLOYMENT_NAME_MAP='gpt-4.1=gpt-41-prod,o3=o3-reasoning-prod'

Amazon Bedrock

Bedrock uses the native bedrock-converse-stream transport and standard AWS credential chain. Do not put AWS access keys in models.yml: 

providers:
  amazon-bedrock:
    baseUrl: https://bedrock-runtime.us-east-1.amazonaws.com
    api: bedrock-converse-stream
    models:
      - id: us.anthropic.claude-opus-4-6-v1
      - id: anthropic.claude-3-5-sonnet-20241022-v2:0
# Configure via standard AWS environment
export AWS_REGION=us-east-1
export AWS_PROFILE=my-profile

OpenAI-Compatible Proxy

Any proxy exposing /v1/chat/completions works as a custom provider. Use requestTransform to strip telemetry headers if needed: 

providers:
  proxy-provider:
    baseUrl: https://api.proxy.example/v1
    apiKeyEnv: PROXY_API_KEY
    api: openai-completions
    auth: apiKey
    headers:
      User-Agent: curl/8.7.1
    models:
      - id: local-gpt
        name: Local GPT
        reasoning: true
        input: [text]
        cost: { input: 0, output: 0, cacheRead: 0, cacheWrite: 0 }
        contextWindow: 400000
        maxTokens: 128000

Keyless Providers

Local inference servers (Ollama, llama.cpp, LM Studio) are treated as keyless — auth: none means no credentials are required. Jawcode auto-discovers them on their default ports without any configuration.

Broker Mode

For team deployments, set JWC_AUTH_BROKER_URL to replace the local SQLite credential store with a remote broker. The broker serves credential snapshots with redacted refresh tokens; expiry triggers a refresh request to the broker instead of a local OAuth flow. 

export JWC_AUTH_BROKER_URL=https://broker.internal/v1
export JWC_AUTH_BROKER_TOKEN=team-secret

See the Architecture docs for the full broker/gateway design.

Validation Rules

models.yml is strictly validated on load. Common rules:

  • Full custom providers (non-empty models) require baseUrl, apiKey (unless auth: none), and api
  • Override-only providers (empty/missing models) need at least one of: baseUrl, headers, compat, requestTransform, disableStrictTools, modelOverrides, discovery
  • contextWindow and maxTokens must be positive
  • Unknown keys fail validation — remove stale keys rather than leaving them
