Authentication

Jawcode authenticates with LLM providers via API keys, OAuth tokens, or cloud IAM credentials. All credentials are stored in the local SQLite database at ~/.jwc/agent/agent.db.

Auth Priority

AuthStorage.getApiKey() resolves credentials in this order:

  1. Runtime override — setRuntimeApiKey() (programmatic, for SDK embedders)
  2. Stored credentials — agent.db SQLite store (set via /login or jwc login)
  3. Environment variables — provider-specific env vars
  4. Custom provider resolver — fallback for custom provider configurations

Interactive Login

Use the /login command within a session or jwc login from the terminal to store API keys persistently in the local database:

# From terminal
jwc login

# Within a session
/login

Stored keys persist across sessions and take precedence over environment variables.

OAuth Authentication

Jawcode supports OAuth for providers that offer it. The OAuth flow opens a browser for authorization and stores refresh tokens locally.

# Start OAuth flow for a provider
jwc login --provider anthropic --oauth

AWS Bedrock

For AWS Bedrock, Jawcode uses the standard AWS credential chain:

VariableDescription
AWS_ACCESS_KEY_IDAWS access key
AWS_SECRET_ACCESS_KEYAWS secret key
AWS_SESSION_TOKENSession token (for temporary credentials)
AWS_REGION or AWS_DEFAULT_REGIONAWS region for Bedrock endpoint
AWS_PROFILENamed profile from ~/.aws/credentials

Bedrock models appear automatically when valid AWS credentials are present. No /login needed.

# Example: use a specific profile
export AWS_PROFILE=my-bedrock-profile
export AWS_REGION=us-east-1
jwc

jwc

Auth Broker

For team deployments, Jawcode supports a remote auth broker that proxies credentials:

# config.yml
auth.broker.url: https://auth.myteam.com
auth.broker.token: my-team-token

Or via environment variables:

export GJC_AUTH_BROKER_URL=https://auth.myteam.com
export GJC_AUTH_BROKER_TOKEN=my-team-token

When an auth broker is configured, RemoteAuthCredentialStore is used instead of the local SQLite store. The broker handles credential rotation and team-level access control.

SDK Auth Wiring

SDK embedders can provide their own auth storage:

import {
  createAgentSession,
  discoverAuthStorage,
  ModelRegistry,
} from "@gajae-code/coding-agent";

// Auto-discover (local SQLite or broker)
const authStorage = await discoverAuthStorage();
const modelRegistry = new ModelRegistry(authStorage);
await modelRegistry.refresh();

const { session } = await createAgentSession({
  authStorage,
  modelRegistry,
  model: modelRegistry.getAvailable()[0],
});

The discoverAuthStorage() function checks for broker configuration first, then falls back to local SQLite.

Troubleshooting

IssueSolution
No models available Run /login or set provider environment variables. Check disabledProviders in settings.
Bedrock models not showing Verify AWS_REGION is set to a Bedrock-supported region. Check IAM permissions for bedrock:InvokeModel.
Vertex models not showing Run gcloud auth application-default login. Ensure GOOGLE_CLOUD_PROJECT is set.
Auth broker connection failed Verify broker URL is reachable. Check token validity. Environment variables override config.yml.
