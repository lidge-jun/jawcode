Prompt Assembly

Jawcode assembles its system prompt from static Handlebars templates, dynamic context, and per-turn injection rails. Understanding this pipeline is essential for customization, embedding, and debugging unexpected agent behavior.

Assembly Pipeline

The full prompt is built once at session start and cached. It is only rebuilt when tools change or refreshBaseSystemPrompt() is called explicitly. 

createAgentSession()
  -> rebuildSystemPrompt(toolNames, tools)
     -> resolveMemoryBackend(...).buildDeveloperInstructions(...)
     -> buildSystemPromptInternal(...)
        -> loadSystemPromptFiles(SYSTEM.md)
        -> loadProjectContextFiles(AGENTS.md, etc.)
        -> buildWorkspaceTree(...)
        -> loadSkills(...)
        -> render system-prompt.md or custom-system-prompt.md
        -> optional project-prompt.md block
  -> options.systemPrompt(defaultPrompt) override/append

Template Files

Three Handlebars templates control the prompt shape. The engine picks one based on whether a custom prompt override is provided.

FileRoleKey Slots
system-prompt.md Default system prompt. Contains identity, runtime config, workflow surface, tools, skills, soul, and communication sections. {{systemPromptCustomization}}, {{#if toolInfo.length}}, {{#if skills.length}}
custom-system-prompt.md Rendered when customPrompt is provided. Contains only customization, custom prompt, append, context files, and skills. {{customPrompt}}, {{appendPrompt}}, contextFiles, skills
project-prompt.md Appended as a separate block after the main prompt render. Rendered via projectPromptTemplate, pushed to systemPrompt[]

system-prompt.md Sections

The default template is organized into named XML-like sections. Each section has a specific responsibility and can reference Handlebars conditionals.

SectionPurpose
<identity>Agent name, role, and operating contract
<authority>RFC 2119 keyword semantics, tag interpretation rules
<jwc-runtime>Public workflow surface (4 skills + IPABCD orchestrate), role agents, routing rules, dev-work classification
<communication>Output formatting, progress reporting, confirmation checkpoints
<completion-contract>Fabrication prevention, security, verification claims
<repo-safety>Blast radius assessment, reversibility gates
<tools>Dynamic: tool inventory, priority rules, AST tools, subagent guidance
<skills>Dynamic: loaded skill descriptions with paths
<search-mandate>Search-before-answering rules, scaling tiers, citation requirements
<workflow>Scope, before-editing, decomposition, and verification disciplines
<soul>Personality, freedom scope, behavioral guardrails

SYSTEM.md Customization

Users can customize the system prompt without editing templates. The runtime scans for SYSTEM.md files at two levels:

LevelLocationPrecedence
User-level ~/.jwc/agent/SYSTEM.md Fallback when no project-level file exists
Project-level .jwc/SYSTEM.md (nearest config dir) Takes priority over user-level

The resolved content fills the {{systemPromptCustomization}} slot in the template, appearing near the top of the final prompt.

CLI Overrides

Two CLI surfaces provide runtime prompt injection:

SurfaceBehavior
--system-prompt Replaces the first default prompt block entirely
APPEND_SYSTEM.md Discovery: project-level first, then global. Appended after default prompt blocks.

Dynamic Sections

Tools

Tool descriptions are rendered from the active tool set. Priority: explicit toolNames parameter, then tools map keys, then the hardcoded default set (read, bash, eval, edit, write).

Skills

Skills are included only when the read tool is available. Hidden skills (hide: true) are excluded. Even if options.skills = [] is passed, the four default workflow skills (jaw-interview, plan, goal, team) are always injected by withEmbeddedDefaultGjcSkills().

Context Files

AGENTS.md and similar context files are loaded from the working directory tree. The workspace tree itself is generated with a 5-second timeout and included as a truncated directory listing.

Per-Turn Injection Rails

While the system prompt is cached, ten rails inject state into the model context on every turn:

#RailTimingContent
1Memory backend beforeAgentStartPrompt Task snapshot + hindsight memories/mental models
2Plan mode Before user message PLAN.md body + acceptance criteria
3Goal mode Custom message + periodic continuation Active goal objective, token usage, PABCD hints
4Skill injection /skill: invocation or yield Full SKILL.md body via sendCustomMessage
5TTSR (prose) Stream interrupt retry Hidden custom message from ttsr-interrupt.md
6TTSR (tool) afterToolCall hook Tool reminder prepended to result content
7Plan decision Plan-to-execute transition Decision reminder steer
8Todo eager prelude Before prompt() Todo template with forced tool choice
9Todo stop reminder Final stop (no tool calls) Incomplete todo reminder + auto-continue
10PABCD stage header Every prompt() turn Current stage, gate chips, goal objective prefix

Subagent Prompt

Subagents receive a separate prompt from subagent-system-prompt.md. This template includes:

  • Role-specific identity block (executor, architect, planner, critic)
  • Shared context from the parent agent
  • Completion contract with fabrication/suppression guards
  • Repo safety rules (no revert/stash/commit/push/delete)
  • Correction de-anchoring protocol for resume/steer corrections
  • Authority-bias prevention (verify parent claims with own tool calls)
  • Confirmation-bias prevention (seek disconfirming evidence)
