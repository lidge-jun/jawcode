Agent Loop

The agent loop is the core execution engine in packages/agent/src/agent-loop.ts. It drives the turn cycle: accept a user message, call the LLM, dispatch tool calls, append results, and decide whether to continue or yield back to the user.

Turn Cycle

Each turn follows this sequence: 

  User message (or steering/follow-up)
          |
          v
  [1] syncContextBeforeModelCall()
      Refresh prompt/tool context from live state
          |
          v
  [2] streamAssistantResponse()
      Transform AgentMessage[] -> Message[] at the LLM boundary
      Stream tokens via @gajae-code/ai streamSimple()
      Emit message_start / streaming / message_end events
          |
          v
  [3] Parse assistant response
      Extract tool calls from message content
          |
          +--- No tool calls ---> Yield (turn complete)
          |
          v
  [4] executeToolCalls()
      Parallel dispatch of tool calls
      Each tool: validate args -> execute -> coerce result
      Emit tool_start / tool_result events
          |
          v
  [5] Append tool results to context
          |
          v
  [6] Check for steering messages
      (User may have typed while tools were running)
          |
          +--- Steering messages ---> Back to [1]
          |
          v
  [7] Check shouldPause()
          |
          +--- Pause requested ---> Yield with "paused" status
          |
          v
  [8] Back to [1] (auto-continue with tool results)

Double Loop Structure

The implementation uses a double loop in runLoopBody():

  • Inner loop — Runs while there are tool calls to process or pending steering messages. This is the tool-call-and-continue cycle that keeps the agent working without user intervention.
  • Outer loop — After the inner loop completes (agent would yield), checks for follow-up messages via config.getFollowUpMessages(). If follow-ups exist (e.g., goal continuation prompts, HOTL steering), the inner loop restarts. Otherwise the agent yields. 
while (true) {                          // Outer loop
    while (hasToolCalls || pending) {    // Inner loop
        // Process pending messages
        // Stream assistant response
        // Execute tool calls
        // Check for steering messages
    }
    // Agent would stop here
    onBeforeYield()
    followUps = getFollowUpMessages()
    if (followUps) continue              // Restart inner loop
    break                                // Exit — yield to user
}

Tool Choice Queue

When the assistant response contains multiple tool calls, they are dispatched through executeToolCalls(). The function:

  1. Filters tool calls from the assistant message content
  2. For each tool call, finds the matching AgentTool by name
  3. Validates arguments against the tool's schema using validateToolArguments()
  4. Executes tools — potentially in parallel depending on tool configuration
  5. Coerces results via coerceToolResult() to normalize the output shape
  6. Returns tool result messages that maintain the tool_use/tool_result pairing required by the API

If the agent is aborted during tool execution, placeholder tool results are created via createAbortedToolResult() to maintain the API's pairing contract.

Auto-Continue Logic

The agent continues automatically when:

ConditionMechanism
Tool calls present in response hasMoreToolCalls = true keeps the inner loop running
Steering messages from tool execution steeringMessagesFromExecution set by tools that inject follow-up messages (e.g., subagent steer/resume)
User typed while agent was working config.getSteeringMessages() polled after each turn
Follow-up messages after yield config.getFollowUpMessages() checked in outer loop — goal mode continuation, HOTL prompts

The agent stops when:

  • No tool calls and no pending/follow-up messages
  • config.shouldPause() returns true (cooperative pause)
  • The response has stopReason of "error" or "aborted"
  • The abort signal fires

Event Stream

The loop emits structured events through an EventStream<AgentEvent>:

EventWhen
turn_startBeginning of each turn (except the first)
message_startUser/steering message injected
message_endMessage injection complete
tool_startTool execution begins
tool_resultTool execution completes
turn_endTurn complete (with assistant message + tool results)
agent_endLoop exits — includes telemetry, step count, stop reason

Subagent Fork / Spawn

Subagents are spawned through the task tool in packages/coding-agent/src/task/. The spawn mechanism:

Fork architecture

Main session (agent-loop running)
    |
    |--- task tool call (spawn subagent)
    |       |
    |       +---> New agent-loop instance
    |             - Own AgentContext (append-only)
    |             - Shared system prompt prefix (cacheable)
    |             - Assignment + forked context snapshot (dynamic)
    |             - Own tool set
    |             - Runs async — main loop continues
    |
    |--- (main agent keeps working / chatting)
    |
    |--- subagent yield event
    |       |
    |       v
    |    Result delivered as tool result to main session

Prompt cache design

Subagent prompts are structured with a static/dynamic boundary for LLM prompt caching:

  • Static prefix — Role definition, system prompt, tool descriptions. Identical across all subagents of the same role. This is the cacheable portion.
  • Dynamic suffix — Assignment text, shared context, forked conversation snapshot. Appended after the cacheable prefix.

On resume/steer, the existing session context (including the cached prefix) is reused. Only the new steering message is appended. This makes resume significantly cheaper than a fresh spawn.

Role agents

Five bundled role agents are defined in packages/coding-agent/src/task/agents.ts:

RoleModeCache behavior
executor Read/write — full tool access Self-fork lane: inherits parent model, forked context snapshot is cache-affine
executor_ext Read/write — full tool access External model lane: uses configured EXECUTOR_EXT model target or explicit per-task selector
architect Read-only for product files Shared static prefix with other read-only roles
planner Read-only for product files Shared static prefix with other read-only roles
critic Read-only for product files Shared static prefix with other read-only roles

Correction de-anchoring

When a subagent is resumed with a correction (the parent says "your previous answer was wrong"), a de-anchoring protocol activates to prevent anchoring bias. The subagent:

  1. Sets aside the prior analysis entirely
  2. Generates at least 3 materially different hypotheses from different angles
  3. Investigates each with fresh tool calls
  4. Forms a new conclusion from evidence — not from editing the old one

This protocol does not activate for simple additive requests ("also check X") where the prior work is not being corrected.

Telemetry

The loop tracks per-run telemetry through AgentTelemetry:

  • Step count per run
  • Tool invocation counts (including skipped/aborted tools)
  • Token usage per model call
  • Coverage data: which tools were invoked, which were skipped
  • Run-level rollup via agentLoopDetailed() / agentLoopContinueDetailed()

Entry Points

FunctionUse
agentLoop(prompts, context, config) Start a new loop with prompt messages
agentLoopContinue(context, config) Continue from current context without adding a message
agentLoopDetailed(...) Like agentLoop but captures run-level telemetry
agentLoopContinueDetailed(...) Like agentLoopContinue with telemetry capture

All variants accept an optional AbortSignal for cooperative cancellation and an optional StreamFn for custom streaming behavior.
